## Task Summary

分析 假全螢幕(影片/簡報/遊戲)遮蔽偵測方案與生效驗證方法

---

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 症狀
全螢幕播放影片（YouTube）、簡報（Keynote）、遊戲時，透明遮罩時鐘仍覆蓋其上，未讓位。用戶先前認為「修改沒成功」。

### 根因鏈（含原生碼證據）
1. 設定來源 `lib/main.dart:62-65`：
   `windowManager.setVisibleOnAllWorkspaces(true, visibleOnFullScreen: false)`
2. 追入 `window_manager-0.4.3/macos/Classes/WindowManager.swift:460-465`：
   - `setCollectionBehavior(visible, .canJoinAllSpaces)`
   - `setCollectionBehavior(visibleOnFullScreen, .fullScreenAuxiliary)`
   即 `visibleOnFullScreen: false` → 移除 `.fullScreenAuxiliary` collection behavior。
3. always-on-top `lib/main.dart:58` → `WindowManager.swift:360`：`mainWindow.level = .floating`。

### 關鍵技術事實
`.fullScreenAuxiliary` 只決定「浮動視窗要不要加入『native fullscreen app 的獨立 Space』」。
macOS「全螢幕」有兩類：
- native fullscreen：點視窗左上綠燈鈕，系統建立**獨立 Space**。`fullScreenAuxiliary` 僅對此有意義。
- 假全螢幕（影片/簡報/遊戲）：一個**鋪滿螢幕的普通視窗，不建獨立 Space**。`fullScreenAuxiliary` 對其**完全不適用**。

### 結論
v1.2.0 的設定對 native fullscreen 正確生效，但用戶遭遇的是假全螢幕——該設定在技術上**不可能**讓時鐘讓位。這不是「沒生效」的 bug，而是**用錯工具 + false positive 完成**（CHANGELOG 1.2.0 與 README.md:112 均宣稱「全螢幕不遮蔽」為設計行為，且 README 誤稱假全螢幕「佔據獨立 Space」——技術描述錯誤）。

## 重現實驗結果
### 驗證方法設計（三層，回答「如何驗證此動作生效」）

過去無法驗證生效的根本原因：現有 widget/unit test 只能斷言 Dart 常數 `AppWindow.visibleOnFullScreen == false`，**無法證明** NSWindow.collectionBehavior 真的被套用，也無法區分兩類全螢幕。這是 quality-baseline 規則 1「測試綠燈 ≠ runtime 正確」的典型陷阱。

#### 層級 1：原生狀態觀測（最低成本，先做）
在 MainFlutterWindow.swift 或設定後印出：
`NSLog("[overlay] collectionBehavior=\(self.collectionBehavior.rawValue) level=\(self.level.rawValue)")`
驗證 `.canJoinAllSpaces`(bit 1) 有設、`.fullScreenAuxiliary`(bit 256) 未設。
→ 直接排除/確認「設定靜默失效」這條可能。

#### 層級 2：實機行為雙場景（必須分兩類全螢幕——過去漏掉的關鍵）
| 案例 | 操作 | 預期（設定生效時） | 對應根因 |
|------|------|------------------|---------|
| A. native fullscreen | Safari/TextEdit 點綠燈鈕進獨立 Space | 時鐘消失 | fullScreenAuxiliary=false 生效 |
| B. 假全螢幕 | YouTube 影片全螢幕 / Keynote 播放 | 時鐘仍在 | 此設定管不到（需新機制） |

判讀：若 A 通過、B 失敗 → 不是 bug，是需求邊界。要遮蔽假全螢幕需「層級 3 的新偵測機制」。

#### 層級 3：新偵測機制的驗收（IMP 階段，避免再次 false positive）
未來 IMP 實作「假全螢幕覆蓋偵測 → 隱藏時鐘」後，驗收**必含 runtime 層**：
- 實機開 YouTube 全螢幕，肉眼確認時鐘讓位；退出全螢幕後時鐘復現。
- 禁止只以「常數值正確 + widget test 綠燈」宣告完成（PC-165 false positive 防護）。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 假全螢幕覆蓋偵測方案評估

目標：偵測「目標螢幕被一個非本 app 的視窗鋪滿（假全螢幕）」→ 透過既有 method channel 通知 Dart 端切換 `_clockVisible=false`，退出時復現。沿用 1.2.0 已建立的「Flutter 層條件渲染」隱藏機制（不動原生視窗，避免崩潰）。

### 方案 A：CGWindowListCopyWindowInfo 覆蓋判定
作法：列舉 `kCGWindowListOptionOnScreenOnly` 視窗，過濾掉本 app 與系統層（Dock/menubar），判斷是否存在某視窗 bounds 覆蓋目標螢幕整個 frame 且 window layer == 0（normal）。
- 優：通用，能抓所有類型假全螢幕（影片/簡報/遊戲），判定最精確。
- 缺：需事件或輪詢觸發；判定邏輯較複雜（需排除自身、處理多螢幕座標）。
- 風險（待原生驗證）：只取 bounds+layer 應不需 Screen Recording 權限（取 window owner name 才需要）；需實機確認 macOS 14/15 行為。

### 方案 B：menu bar / visibleFrame 偵測（NSScreen）
作法：監聽螢幕參數變化，比較目標 `NSScreen.visibleFrame` 與 `frame`——假全螢幕通常隱藏 menu bar 與 Dock，使 visibleFrame == frame。
- 優：輕量、事件驅動、無需任何權限。
- 缺：多螢幕時不精確（他螢幕全螢幕也可能影響）；少數不隱藏 menubar 的假全螢幕會漏判；無法定位是哪個 app。
- 風險：誤判率較高，單獨使用不足。

### 方案 C：NSWorkspace 通知（active app / space 變化）
作法：監聽 `NSWorkspace.activeSpaceDidChangeNotification` 等。
- 優：低成本事件源。
- 缺：資訊不足以判斷「螢幕是否被鋪滿」，無法單獨解題。
- 定位：作為方案 A 的**觸發源**（減少輪詢），非獨立方案。

### 推薦：A 為主 + C 為觸發源
以 NSWorkspace 通知（C）+ 螢幕參數變化作為觸發時機，觸發時跑一次 CGWindowList 覆蓋判定（A）得精確結論，避免持續輪詢耗 CPU。方案 B 的 visibleFrame 比較可作為 A 結果的快速前置過濾。最終經 method channel 回報 Dart 端切換 `_clockVisible`。
