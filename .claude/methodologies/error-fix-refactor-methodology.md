# 錯誤修復和重構方法論

## 核心概念

錯誤修復遵循 TDD 原則：**程式服務測試，測試服務需求**。修復錯誤時保持測試不變，調整程式實作。

---

## 錯誤分類

### 程式實作錯誤

程式不符合測試需求，但測試需求正確。

**判斷標準**：
- 測試需求明確且未變更
- 相同輸入產生錯誤輸出

**處理**：保持測試不變 → 修正程式實作

### 架構變更需求

程式架構必須調整，現有測試需對應修改。

**判斷標準**：
- 需求文件已更新且與測試不符
- 業務流程變更影響多個模組

**處理**：確認文件更新 → PM 評估 → 調整測試 → 修正程式

---

## 執行步驟

### 程式錯誤

1. 檢查測試需求
2. 保持測試不變
3. 調整程式實作
4. 驗證所有測試通過

### 架構變更

1. 確認需求文件已更新
2. PM 代理人評估變更範圍
3. 列出需修改的測試
4. 執行修正

---

## 觀測原則

只觀測公開行為（Public API 輸入輸出），不觀測內部實作。

---

## 檢查清單

### 修復前

- [ ] 錯誤類型已判斷（程式錯誤 vs 架構變更）
- [ ] 需求文件狀態已確認
- [ ] 受影響測試已識別

### 修復後

- [ ] 所有測試通過（100%）
- [ ] 無內部狀態曝露
- [ ] 無行為旁路實作
- [ ] 修改有測試保護
