# 兩階段深度反思方法論

## 核心理念

**第一直覺的根因解答不一定準確。真實根因需要兩階段檢驗：Phase 1 多假設 Reality Test 挖掘深層，Phase 2 WRAP 檢驗結論避免自我確認偏誤。**

## 為何需要兩階段

### 單階段反思的失敗模式

直接問「為什麼我會這樣做？」產出的答案通常是表層：
- 「是習慣」
- 「訓練資料如此」
- 「比較方便」

這些答案**無法解釋「為何不是 B/C/D」**，只解釋「為何是 A」。防護方向因此錯誤。

### W15-005 實證

案例：PM 選 `cat /tmp/x.md | append-log` 而非 heredoc。

| 階段 | 產出 | 評估 |
|------|------|------|
| 單階段直覺 | 「shell 習慣」| 表層，解釋不了為何不用 heredoc |
| Phase 1 深度 | 3 層根因：檔案感物化 → 認知負擔規避 → 主動認知策略 | 可解釋完整動機 |
| Phase 2 WRAP | 方案從「3 獨立 Ticket」修正為「D+E 組合」 | 避免 bulk failure mode |

## Phase 1：深度反思（多假設 Reality Test）

### 步驟 1：列出 5+ 候選假設

強制列舉 5 個以上可能動機，**不能只列 1 個**。常見類型：

| 類型 | 範例 |
|------|------|
| 被動歸因 | 訓練資料/習慣/文化 |
| 主動策略 | 認知負擔規避/步驟數最小化 |
| 環境限制 | 工具能力/權限邊界 |
| 心智模型 | 物化/模組化錯覺 |
| 摩擦規避 | 避免精確匹配/避免失敗重試 |
| 事後合理化 | 降低風險/便於除錯（容易混淆為當時動機）|

**禁止只列 1-2 個假設就下結論**——這是單階段反思的陷阱。

### 步驟 2：Reality Test 逐個驗證

對每個假設問：「**我實際思考流程是否匹配這個假設**？」

| 匹配度 | 判定 |
|-------|------|
| 高度匹配 | 當時確實這樣想 |
| 部分匹配 | 有這傾向但非主因 |
| 弱匹配 | 事後合理化，非當時動機 |
| 不匹配 | 排除 |

**誠實標準**：匹配度是自我觀察結果，不是「聽起來合理」。

### 步驟 3：多層深因挖掘

對高度匹配的假設，繼續問「為什麼會這樣想？」至少挖 2 層。

| 層級 | W15-005 範例 |
|------|-------------|
| 表層 | shell 習慣 |
| 第一層深因 | 檔案感物化（長文字視為檔案）|
| 第二層深因 | 認知負擔規避（增加步驟換每步簡潔）|
| 真實根因 | 主動認知策略（非被動習慣）|

**深度標準**：停下的訊號是「若這就是真因，為何不是 B/C/D？」能完整回答。

### 步驟 4：區分被動 vs 主動根因

這個區分**決定防護方向**：

| 根因類型 | 防護方向 |
|---------|---------|
| 被動歸因（習慣/訓練）| 加規則提醒 |
| 主動策略（認知優化）| 加 heuristic check 在決策點介入 |
| 環境限制 | 擴充工具/白名單 |
| 心智模型錯誤 | 教育 + 重新框架 |

## Phase 2：結論後 WRAP 檢驗

### 為何需要

Phase 1 產出的「真實根因」仍可能是**第一直覺的延伸**——深度反思本身也有偏誤：
- 確認偏誤（傾向找符合自己敘事的根因）
- 錨定偏誤（被最先想到的假設綁架）
- 過度推論（2 層深度可能還不夠）

### 方案擴增

將「Phase 1 產出的根因」當作**一個候選方案**，用 WRAP 擴增其他解釋：

| Widen 層 | 搜尋 |
|---------|-----|
| L0 | 既有知識庫（error-patterns、memory）是否有類似根因？ |
| L1 | 同領域其他案例的根因分佈 |
| L2 | 反面框架（Consider the Opposite）|
| L3 | 跨領域類比 |

### Reality Test

| 檢驗 | 問題 |
|------|------|
| 基本率 | 此類根因在文獻/經驗中的頻率 |
| 來源核對 | 根因論述引用的權威 source 是否真實 |
| Steelman | 用最有力方式陳述競爭根因 |

### Consider the Opposite

問：「如果我相信的真實根因正好相反，會怎樣？」
- W15-005 案例：「如果根本不是認知負擔規避，而是主動最佳化呢？」→ 結論改變，但用戶未質疑，故保留
- 若反面論述有力 → 回 Phase 1 補假設重挖

### 防護方向驗證

根因決定的防護方案，再過一次 WRAP：
- 方案集是否真多元？（非只是整合層變體）
- 機會成本？
- Premortem：3 大失敗原因？

**W15-005 實證**：原「3 獨立 Ticket」提議經 WRAP 後發現：
- 框架概括測試失敗（全在「建文字規則」）
- 高 bulk-ignored 風險
- 修正為 D+E 組合（WRAP skill 整合 + bash 規則五）

## Phase 3：權威方法論對照檢查

> 新增背景：W15-008 產出 PC-088 v1 框架後，用戶指出與 `.claude/methodologies/friction-management-methodology.md` 衝突。Phase 1+2 未捕捉此衝突，顯示 Phase 3 必要性。

### 核心問句

「我的結論用的概念/詞彙/框架，與本專案既有權威方法論是否一致？」

### 對照檢查步驟

1. **識別結論用詞**：列出結論中的關鍵詞（如「bias」「對抗」「架構偏誤」）
2. **查找權威 source**：搜尋 `.claude/methodologies/` 和 `.claude/rules/` 是否有相關主題
3. **概念對照**：權威 source 如何定義這些概念？
4. **衝突偵測**：結論框架是否與權威 source 衝突？
5. **若有衝突**：回 Phase 1 補假設，用權威 source 詞彙系統重構結論

### 常見衝突類型

| 類型 | 症狀 | 檢查 source |
|------|-----|-----------|
| 摩擦力 | 用「bias 要對抗」框架 | friction-management-methodology |
| 認知負擔 | 用「懶惰/偏好」框架 | cognitive-load-design-methodology |
| 代理人職責 | 用「能力不足」框架 | agent-authoring-guide / pm-role |
| 溝通品質 | 用「表達問題」框架 | ai-communication-rules |

### 實證案例

**PC-088 v1 vs v2**：
- v1 用「LLM bias 要對抗」→ 與 friction-methodology 衝突
- v2 改用「步驟數估算偏誤 + 決策點摩擦」→ 相容
- 差別：v2 承認短路徑偏好是對的，問題在估算不準

## 終止條件

反思流程可結束的判準（滿足 A + B 任一即可停止）：

### A. 深度門檻（Phase 1/2/3 全通過）

- Phase 1：能完整回答「若真因為 X，為何不是 B/C/D？」
- Phase 2：WRAP 四階段走完，tripwire 無觸發
- Phase 3：結論與 `.claude/methodologies/` + `.claude/rules/` 無衝突

### B. 用戶閘門

- 用戶明確表達「這個深度夠了」「可以停了」
- AskUserQuestion 確認繼續反思，用戶選擇「停止」

**禁止條件**：

- 不可以 PM 單方判斷「差不多了」而未過 A 或 B
- 不可連續 > 3 層 ANA spawn ANA 不停（Layer 3 警示）

**Layer 4 元條件保護（防遞迴）**：

- 禁止再 spawn 任何「終止條件的終止條件」ANA
- 若未來發現本設計有缺口，直接修改方法論不另開 ANA

## 觸發條件

本方法論適用於：

| 情境 | 識別 |
|------|-----|
| error-pattern 根因分析 | 寫 PC-XXX 的「根因」章節 |
| 設計決策回顧 | Phase 4 重構評估發現設計問題 |
| 代理人失敗歸因 | incident-responder 分析 |
| 用戶質疑分析深度 | 用戶說「這解釋太表層」「再想想」|
| ANA Ticket 執行 | analysis task 的核心產出 |
| 方法論/規則建立前 | 確認防護方向正確 |

## 反模式

| 反模式 | 症狀 | 正解 |
|-------|------|------|
| 只列 1-2 假設 | 第一直覺即結論 | 強制 5+ 假設 |
| 挖 1 層就停 | 「是習慣」型答案 | 至少 2 層深因 |
| 跳過 Reality Test | 「聽起來合理」即接受 | 強制自我觀察驗證 |
| 跳過 Phase 2 | Phase 1 結論直接落地 | 強制 WRAP 檢驗 |
| 防護方向錯置 | 主動策略卻只加規則 | 依根因類型配防護 |
