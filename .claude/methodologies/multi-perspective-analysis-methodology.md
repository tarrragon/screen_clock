# 多視角並行分析方法論

本文件定義多視角並行分析模式，將單一視角分析升級為多代理人並行分析，提升分析品質和決策正確性。

---

## 核心理念

> **單一視角分析容易遺漏問題，多視角並行可交叉驗證，提高決策品質。**

傳統的分析模式依賴單一代理人（如 incident-responder）進行問題分析，這種模式存在以下侷限：

| 侷限 | 說明 |
|------|------|
| 視角盲點 | 單一代理人受限於其專業領域，可能遺漏其他層面的問題 |
| 效率瓶頸 | 多個分析角度需要序列執行，時間成本高 |
| 決策品質 | 最終決策依賴單一來源判斷，缺乏交叉驗證 |

多視角並行分析模式通過並行派發多個代理人，從不同視角同時分析問題，然後彙整報告做出更全面的決策。

---

## 適用場景

### 觸發條件

**符合任一條件即可考慮啟用多視角分析**：

| 條件 | 閾值 | 說明 |
|------|------|------|
| 認知負擔指數 | > 10 | 問題複雜度高，需要多視角 |
| 風險等級 | >= 高 | 高風險問題需要交叉驗證 |
| 安全敏感 | 是 | 涉及認證、授權、敏感資料 |
| 影響範圍 | > 3 模組 | 跨模組問題需要多視角 |
| 原因不明確 | 是 | 根因不清楚時需要多角度分析 |

### 場景清單

| 場景 | 觸發條件 | 建議視角 | 報告消費者 |
|------|---------|---------|-----------|
| 測試失敗分析 | 原因不明顯的測試失敗 | 程式碼、測試、架構、需求 | SA → PM |
| 架構決策 | 多個技術方案需要評估 | 技術、效能、維護性、擴展性 | SA → PM |
| 需求分析 | 新功能需求不明確 | 用戶、技術、業務、UseCase | SA → PM |
| 效能問題 | 效能瓶頸定位 | 程式碼、資料庫、網路、資源 | ginger → PM |
| 安全審查 | 安全敏感功能 | OWASP、認證、授權、資料 | security → PM |
| 重構決策 | 大規模重構前評估 | 程式碼、測試、架構、依賴 | cinnamon → PM |
| 技術債務評估 | TD 優先級判斷 | 風險、成本、收益、緊迫性 | SA → PM |
| 系統級問題 | 跨模組的複雜問題 | 各相關模組視角 | SA → PM |

### 不適用場景

| 場景 | 原因 | 建議做法 |
|------|------|---------|
| 明顯的單點問題 | 原因明確，無需多視角 | 直接派發對應代理人 |
| 簡單的格式/筆誤 | 複雜度低 | 直接修復 |
| 緊急 P0 問題 | 時間緊迫 | 先修復後分析 |
| 認知負擔 < 5 | 簡單問題 | 單一代理人分析 |

## 執行流程

### 流程圖

```
問題/任務進入
    |
    v
[第負一層] 評估是否需要多視角分析?
    |
    +-- 否（簡單問題）--> 單一代理人分析 --> 派發修復
    |
    +-- 是（符合觸發條件）
        |
        v
    [決定分析視角]
        |
        v
    [並行派發] 多個分析代理人
        |
        +-- 程式碼視角代理人 --> 報告 A
        +-- 測試視角代理人 --> 報告 B
        +-- 架構視角代理人 --> 報告 C
        +-- ... --> 報告 N
        |
        v
    [報告彙整] SA 或彙整代理人
        |
        v
    產出彙整報告
        |
        v
    [PM 決策] 根據彙整報告決定派發
        |
        v
    派發修復/實作代理人
```

### 步驟說明

| 步驟 | 負責者 | 動作 | 產出 |
|------|-------|------|------|
| 1. 觸發判斷 | PM | 評估是否符合多視角觸發條件 | 決定啟用/不啟用 |
| 2. 視角選擇 | PM | 根據場景選擇必選和可選視角 | 視角清單 |
| 3. 並行派發 | PM | 同時派發多個分析代理人 | 派發記錄 |
| 4. 並行分析 | 各代理人 | 從各自視角分析問題 | 視角報告 |
| 5. 報告彙整 | SA/彙整者 | 整合多份報告、交叉驗證 | 彙整報告 |
| 6. 決策派發 | PM | 根據彙整報告決定修復方案 | 派發決策 |

## 報告彙整機制

### 彙整責任

| 場景類型 | 報告消費者（彙整者） | 理由 |
|---------|-------------------|------|
| 測試失敗/程式碼問題 | saffron-system-analyst | 需要架構視角整合 |
| 效能問題 | ginger-performance-tuner | 需要效能專業判斷 |
| 安全問題 | clove-security-reviewer | 需要安全專業判斷 |
| 一般性問題 | saffron-system-analyst | SA 負責系統級整合 |

### 交叉驗證規則

| 驗證項 | 驗證方法 | 不一致時處理 |
|-------|---------|-------------|
| 根因判斷 | 比對各視角的根因推斷 | 列出分歧，由 PM 決定 |
| 修復建議 | 檢查建議是否相容 | 衝突建議需要選擇或合併 |
| 優先級 | 取最高優先級 | 明確標註來源視角 |
| 風險評估 | 合併所有風險 | 取最高風險等級 |

### 衝突處理策略（分類處理）

> **核心原則**：不同類型的衝突需要不同的解決策略。預設聯集會導致加法偏見——每增加一個審查視角，產出物只膨脹不收縮。

| 衝突類型 | 處理策略 | 舉證責任 |
|---------|---------|---------|
| 根因不一致 | 列出所有可能根因，建議逐一排查 | 各視角舉證 |
| 加法 vs 加法（都想新增） | 聯集（合併） | 無需額外舉證 |
| **加法 vs 減法（一方想保留、一方想刪除）** | **預設採減法，保留方需舉證** | **加法方（想保留的人）** |
| 減法 vs 減法（都想刪除） | 交集（刪最少的，保守刪除） | 無需額外舉證 |
| 優先級分歧 | 採用最高優先級 | 明確標註來源視角 |
| 範圍分歧 | **依上方衝突類型分類處理**，不再一律聯集 | 依衝突類型決定 |

**「加法 vs 減法」規則說明**：當審查者 A 說「刪除這個章節」而審查者 B 說「保留」時，B 必須提出保留的具體理由（不是「可能有用」）。PM 禁止透過折衷合併（如「保留一半」）來迴避二元決策。

> **來源**：多視角發想結構性加法偏見分析的歷史教訓

## 並行安全檢查

### 分析任務並行安全檢查清單

```markdown
## 多視角並行分析安全檢查

- [ ] 各視角分析範圍無重疊（不會重複分析相同內容）
- [ ] 各代理人可獨立完成分析（無需互相等待）
- [ ] 報告格式統一（便於彙整）
- [ ] 彙整者已確定（知道誰負責整合）
```

### 與實作任務並行的差異

| 維度 | 實作任務並行 | 分析任務並行 |
|------|-------------|-------------|
| 檔案重疊 | 必須無重疊 | 可讀取相同檔案（只讀） |
| 依賴關係 | 需要明確標記 | 無依賴（各視角獨立） |
| 輸出衝突 | 會衝突 | 不衝突（各自產出報告） |
| 彙整需求 | 無 | 需要彙整者整合報告 |

## 與決策樹的整合

### 第負一層：並行化評估（擴展）

在現有「任務拆分」判斷後，增加「分析並行化」判斷：

```
接收任務/問題
    |
    v
[任務拆分判斷] 可拆分給多個代理人?
    |
    +-- 是 --> 並行派發實作代理人
    |
    +-- 否 --> [分析並行化判斷] 需要多視角分析?
               |
               +-- 是 --> 並行派發分析代理人 --> 彙整 --> 派發修復
               |
               +-- 否 --> 單一代理人處理
```

### 第六層：事件回應流程（擴展）

在 incident-responder 分析後，增加「多視角分析選項」：

```
錯誤發生
    |
    v
[強制] 執行 /pre-fix-eval
    |
    v
[判斷] 需要多視角分析?
    |
    +-- 是（原因不明/複雜度高/風險高）
    |   |
    |   v
    |   [多視角並行分析流程]
    |
    +-- 否（原因明確/簡單問題）
        |
        v
        [單一 incident-responder 分析]
```

## 實施檢查清單

### PM 啟用多視角分析前

- [ ] 確認符合觸發條件（認知負擔 > 10 / 風險 >= 高 / 安全敏感）
- [ ] 確定分析場景類型
- [ ] 選擇必選和可選視角
- [ ] 確定報告消費者（彙整者）
- [ ] 完成並行安全檢查

### 分析代理人產出報告時

- [ ] 遵循標準報告格式
- [ ] 包含發現項、根因推斷、建議
- [ ] 包含風險評估
- [ ] 標註與其他視角的關聯

### 彙整者產出彙整報告時

- [ ] 整合所有視角報告
- [ ] 完成交叉驗證
- [ ] 處理衝突和分歧
- [ ] 產出綜合建議和派發建議

### PM 根據彙整報告決策時

- [ ] 閱讀彙整報告
- [ ] 確認根因結論
- [ ] 決定修復方案
- [ ] 派發修復代理人
