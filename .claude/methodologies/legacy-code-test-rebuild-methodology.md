# Legacy Code 測試重建方法論

## 核心概念

Legacy Code 測試重建是一套系統化流程，用於接手現有專案後建立或修復測試覆蓋。核心理念：先分類根因，再聚焦執行，避免過度工程。

**核心價值**：區分「知識問題」和「基礎設施問題」，用最小成本解決真正的根因。

> **來源**：Legacy Code 驗證 + 四視角審查實證經驗。

---

## 五步驟流程

### Step 1：UC/功能盤點

| 動作 | 產出 |
|------|------|
| 列出所有 Use Case 或核心功能 | UC 清單，標記各 UC 的測試狀態 |
| 確認功能範圍和優先級 | 優先級排序（核心功能優先） |
| 盤點現有測試覆蓋 | 測試覆蓋矩陣（UC x 測試層級） |

**產出物**：UC 測試覆蓋矩陣

```
| UC    | 單元測試 | Widget/UI 測試 | 整合測試 | 狀態   |
|-------|---------|---------------|---------|--------|
| UC-01 | 有      | 有（部分失敗） | 無      | 需修復 |
| UC-02 | 有      | 無            | 無      | 需補充 |
```

### Step 2：逐 UC 測試漏洞分析

| 動作 | 產出 |
|------|------|
| 逐 UC 執行測試 | 每個 UC 的失敗清單 |
| 記錄失敗模式和錯誤訊息 | 失敗模式分類表 |
| 識別重複出現的失敗模式 | 系統性問題清單 |

**關鍵**：記錄「失敗的原因」，不是「失敗的測試」。同一原因跨多個 UC 出現 = 系統性問題。

### Step 3：根因分類

這是整個流程最關鍵的步驟。將所有失敗歸類為三種根因：

| 根因類型 | 定義 | 解決成本 | 範例 |
|---------|------|---------|------|
| **知識問題** | 開發者不知道工具/慣例存在 | 低（更新文件/指南） | 不知道 WidgetTestHelper、不知道自訂元件 |
| **基礎設施問題** | 工具/Helper/Fixture 不存在 | 中（建立工具） | 缺少測試 Helper、缺少 Fixture |
| **設計問題** | 架構缺陷導致無法測試 | 高（重構架構） | 緊耦合、無介面抽象、全域狀態 |

**判斷規則**：

```
失敗的測試
    |
    v
工具/Helper 存在嗎？
    |
    +-- 不存在 → 基礎設施問題
    |
    +-- 存在 → 開發者知道要用嗎？
        |
        +-- 不知道 → 知識問題
        |
        +-- 知道但用不了 → 設計問題
```

**歷史實證**：5 個系統性問題中 3 個是知識問題、1 個是基礎設施問題、1 個是設計問題。更新代理人知識（1 個 commit）比建立 6 張基礎設施票有效 10 倍。

### Step 4：多視角精簡方案

對初步方案進行多視角審查，確保不過度工程：

| 視角 | 檢查重點 |
|------|---------|
| Consistency | 方案是否與現有程式碼模式一致 |
| Impact | 每個方案的風險/收益比 |
| Good Taste | 是否過度工程、是否解決真正問題 |
| 語言代理人 | 框架/語言層面的可行性和慣例 |

**精簡原則**：
- 測試全部通過 → 分散建立是美觀問題，不是缺陷
- 工具已存在但不被使用 → 更新知識，不重建工具
- Mock 按需補齊是正確策略 → 不需要全量審計

### Step 5：聚焦執行

只做真正解決根因的工作：

| 根因類型 | 執行動作 |
|---------|---------|
| 知識問題 | 更新 CLAUDE.md 測試規範 + 代理人知識庫 |
| 基礎設施問題 | 建立/改善 Helper、Fixture、Mock |
| 設計問題 | 建立重構 Ticket，安排到後續版本 |

## 專案類型變體

### 前端專案（Flutter/React/Vue）

**特有挑戰**：
- 自訂 UI 元件（AppButton 等）與標準元件的斷言差異
- 國際化（l10n）上下文在測試中的初始化
- 響應式佈局的測試尺寸處理
- UI 框架特定的測試 Helper（WidgetTestHelper 等）

**Step 2 重點**：
- 檢查 Widget/Component 測試是否使用標準 Helper
- 檢查 l10n/i18n 是否正確配置
- 檢查自訂元件是否被正確斷言

**Step 3 常見分佈**：知識問題 60%、基礎設施 30%、設計 10%

### 資料庫專案（有 DB 的後端/全端）

**特有挑戰**：
- 過時的 Mock Data 與實際 Schema 不一致
- Migration 腳本未被測試覆蓋
- ORM Model 與實際欄位不同步
- 測試資料庫的隔離和清理

**Step 2 重點**：
- 比對 Mock Data 和實際 DB Schema
- 檢查 Migration 是否有對應測試
- 驗證 Repository 測試使用的是真實 DB 還是 Mock

**Step 3 常見分佈**：基礎設施 50%、設計 30%、知識 20%

### 無測試專案

**特有挑戰**：
- 沒有現有測試可作為基準
- 程式碼可能沒有介面抽象，難以 Mock
- 沒有測試文化和慣例

**流程調整**：
- Step 1 變為「功能盤點 + 可測試性評估」
- Step 2 變為「嘗試為核心 UC 寫第一個測試，記錄障礙」
- Step 3 常見分佈：設計問題 60%、基礎設施 30%、知識 10%
- 優先解決設計問題（引入介面、依賴注入），否則無法測試

### 混合專案（前後端 + DB）

**流程調整**：
- 前後端分別執行 Step 1-3
- Step 4 合併審查（避免前後端方案衝突）
- 共用 Fixture 策略統一設計

## 反模式

| 反模式 | 說明 | 正確做法 |
|--------|------|---------|
| 大爆炸重構 | 一次替換所有測試資料（1000+ 處） | 按 UC 拆分，逐步替換 |
| 全量審計 | 審計所有 Mock 的完整性 | 按需補齊，遇到失敗再修 |
| 過度抽象 | 為 19 處使用建立新工具類 | 直接使用現有 API |
| 基礎設施先行 | 先建工具再看是否需要 | 先分類根因，知識問題不需要工具 |
| 忽略根因分類 | 所有失敗都歸為「測試不完整」 | 區分知識/基礎設施/設計三類 |

## 檢查清單

### Step 1 完成標準
- [ ] UC/功能清單完整
- [ ] 測試覆蓋矩陣建立
- [ ] 優先級排序完成

### Step 3 完成標準
- [ ] 每個失敗都被分類為知識/基礎設施/設計之一
- [ ] 知識問題佔比已量化
- [ ] 基礎設施問題有具體工具需求
- [ ] 設計問題有重構 Ticket

### Step 4 完成標準
- [ ] 多視角審查完成（含語言代理人）
- [ ] 過度工程方案已識別並刪除
- [ ] 最終方案只包含解決根因的工作
