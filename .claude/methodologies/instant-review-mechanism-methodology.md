# 即時 Review 機制方法論

## 核心概念

每完成一個 Ticket 立即觸發 Review，30 分鐘內完成，使用 16 項標準化檢查清單。

**三大原則**：
1. **即時觸發**：Ticket 完成即 Review，不累積
2. **快速完成**：目標 30 分鐘，最長 1 小時
3. **標準檢查**：16 項檢查清單，確保一致性

## 執行步驟

### 觸發 Review

1. 完成 Ticket 實作
2. 自我檢查通過（驗收條件、測試 100%、`dart analyze` 0 錯誤）
3. 標記 Ticket 為「Review 中」

### 執行 Review（30 分鐘）

| 類別 | 時間 | 檢查項 |
|-----|------|-------|
| **功能正確性** | 10 分鐘 | 功能實現、驗收條件、邊界處理、錯誤處理 |
| **架構合規性** | 8 分鐘 | Clean Architecture、依賴方向、Interface-Driven、架構債務 |
| **測試通過率** | 5 分鐘 | 單元測試、整合測試、覆蓋率 >80%、無 skip |
| **文檔同步性** | 2 分鐘 | 工作日誌、設計決策、API 文檔、README |
| **記錄結果** | 5 分鐘 | Review 記錄 |

### 偏差糾正

發現問題時：

1. 暫停當前 Ticket
2. 記錄偏差（描述、影響、根因）
3. 建立修正 Ticket
4. 執行修正
5. 再次 Review

## 檢查清單

### 功能正確性（4 項）

- [ ] Ticket 描述的功能是否實現？
- [ ] 驗收條件是否全部滿足？
- [ ] 是否有未處理的邊界情況？
- [ ] 錯誤處理是否完整？

### 架構合規性（4 項）

- [ ] 是否符合 Clean Architecture 分層原則？
- [ ] 依賴方向是否正確（內層不依賴外層）？
- [ ] 是否使用 Interface-Driven 開發？
- [ ] 是否有架構債務產生？

### 測試通過率（4 項）

- [ ] 相關單元測試是否 100% 通過？
- [ ] 相關整合測試是否 100% 通過？
- [ ] 測試覆蓋率是否達標（> 80%）？
- [ ] 是否有測試被 skip？

### 文檔同步性（4 項）

- [ ] Ticket 工作日誌是否更新？
- [ ] 設計決策是否記錄？
- [ ] API 文檔是否同步？
- [ ] README 是否需要更新？

## 時效標準

| 階段 | 標準 | 超時處理 |
|-----|------|---------|
| 開始 Review | 2 小時內 | 發送提醒 |
| 完成 Review | 30 分鐘內 | 分析原因 |
| 最長 Review | 1 小時 | 升級處理（Ticket 太大？） |

## 問題優先級

| 優先級 | 問題類型 | 處理方式 |
|-------|---------|---------|
| **P0（阻塞）** | 功能錯誤、架構偏差、測試失敗 | 必須修正才能通過 |
| **P1（重要）** | 邊界缺失、文檔不完整 | 建立修正 Ticket |
| **P2（建議）** | 命名改善、程式風格 | 記錄建議，不阻塞 |
