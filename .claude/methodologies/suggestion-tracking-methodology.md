# 建議追蹤方法論

本文件定義調查/分析報告中建議的追蹤機制，確保每個建議都被明確處理。

---

## 問題背景

### 問題重現案例

**歷史調查報告範例**：

```
建議行動：
1. 建立後續 Ticket A 補齊缺失功能（優先）
2. 建立後續 Ticket B 改進驗收流程（次要）
```

**實際執行**：

- 只建立 Ticket A
- Ticket B 被遺漏
- 沒有記錄說明為何只做 1/2

### 根本原因

缺乏系統化的建議追蹤機制：

- 建議產生後沒有強制追蹤
- 沒有明確的採納/拒絕決策流程
- 建議與驗收條件沒有連結

---

## 設計目標

確保每個建議都被：

| 要求     | 說明                                       |
| -------- | ------------------------------------------ |
| 明確記錄 | 列出所有建議，有編號、來源、內容           |
| 明確處理 | 標記採納/拒絕/延後，有決定時間和決定者     |
| 明確追蹤 | 採納的建議轉為驗收條件，有對應關係         |
| 明確理由 | 拒絕的建議說明原因，延後的建議說明目標版本 |

---

## 建議狀態定義

| 狀態   | 代碼       | 說明             | 要求                 |
| ------ | ---------- | ---------------- | -------------------- |
| 待決定 | `pending`  | 建議尚未被處理   | 必須在任務執行前處理 |
| 採納   | `adopted`  | 決定實作此建議   | 必須轉為驗收條件     |
| 拒絕   | `rejected` | 決定不實作此建議 | 必須記錄拒絕理由     |
| 延後   | `deferred` | 排入後續版本處理 | 必須記錄目標版本     |

---

## 建議追蹤格式

### 標準格式（Markdown 表格）

```markdown
## Suggestion Tracking

### 來源：{來源文件/報告}

| #   | 建議內容   | 狀態     | 決定理由     | 對應驗收條件 | 決定者 | 決定時間   |
| --- | ---------- | -------- | ------------ | ------------ | ------ | ---------- |
| 1   | {建議內容} | adopted  | {理由}       | AC-001       | PM     | 2026-01-30 |
| 2   | {建議內容} | rejected | {拒絕理由}   | -            | PM     | 2026-01-30 |
| 3   | {建議內容} | deferred | 排入 v0.32.0 | -            | PM     | 2026-01-30 |
```

### 欄位說明

| 欄位         | 必填 | 說明                              |
| ------------ | ---- | --------------------------------- |
| #            | 是   | 建議編號（從 1 開始）             |
| 建議內容     | 是   | 建議的具體內容                    |
| 狀態         | 是   | pending/adopted/rejected/deferred |
| 決定理由     | 是   | 為何採納/拒絕/延後                |
| 對應驗收條件 | 條件 | 採納時必填，對應的 AC 編號        |
| 決定者       | 是   | 做出決定的角色                    |
| 決定時間     | 是   | 做出決定的日期                    |

## 建議狀態流轉

```
建議產生
    |
    v
[pending] 待決定
    |
    v
PM/決定者 評估
    |
    +-- [adopted] 採納
    |       |
    |       v
    |   轉為驗收條件（AC）
    |       |
    |       v
    |   納入 Acceptance Criteria 表格
    |
    +-- [rejected] 拒絕
    |       |
    |       v
    |   記錄拒絕理由
    |
    +-- [deferred] 延後
            |
            v
        記錄目標版本
            |
            v
        建立追蹤 Ticket（可選）
```

---

## 採納時的轉換規則

### 建議 → 驗收條件轉換

當建議被採納時，必須：

1. **建立對應的驗收條件**：在 Ticket 的 Acceptance Criteria 中新增

2. **標註來源**：驗收條件的「來源」欄位引用建議來源

3. **建立雙向連結**：
   - 建議追蹤表 → 對應驗收條件編號
   - 驗收條件表 → 引用建議來源

### 轉換範例

**原始建議**：

```markdown
| #   | 建議內容           | 狀態    | 對應驗收條件 |
| --- | ------------------ | ------- | ------------ |
| 1   | 建立專責驗收代理人 | adopted | AC-001       |
```

**轉換後的驗收條件**：

```markdown
| #      | 項目                       | 來源                  | 確認方法           | 狀態 |
| ------ | -------------------------- | --------------------- | ------------------ | ---- |
| AC-001 | 建立 acceptance-auditor.md | 建議 #1 (報告來源) | 檔案存在且內容完整 | [ ]  |
```

## 拒絕時的記錄要求

### 拒絕必須說明理由

| 拒絕類型 | 理由範例                                        |
| -------- | ----------------------------------------------- |
| 範圍不符 | 「此建議超出當前 Ticket 範圍，建議另開 Ticket」 |
| 成本過高 | 「實作成本過高，ROI 不足」                      |
| 技術限制 | 「目前架構不支援，需要先完成 X」                |
| 優先級   | 「優先級較低，排入技術債務」                    |
| 已有替代 | 「已有 Y 功能可達成相同目的」                   |

### 拒絕記錄格式

```markdown
| #   | 建議內容            | 狀態     | 決定理由                        |
| --- | ------------------- | -------- | ------------------------------- |
| 2   | 建立自動化驗收 Hook | rejected | 目前架構不支援，需先完成前置 Ticket |
```

---

## 延後時的追蹤要求

### 延後必須標註目標版本

| 延後類型   | 目標版本 | 追蹤方式             |
| ---------- | -------- | -------------------- |
| 下個 Patch | v0.31.1  | 記錄到 todolist.yaml |
| 下個 Minor | v0.32.0  | 建立追蹤 Ticket      |
| 未定       | Backlog  | 記錄到 todolist.yaml |

### 延後記錄格式

```markdown
| #   | 建議內容       | 狀態     | 決定理由     | 追蹤                  |
| --- | -------------- | -------- | ------------ | --------------------- |
| 3   | 建立驗收自動化 | deferred | 排入下個版本 | Ticket: {未定} |
```

## 驗收流程整合

### 驗收時的建議追蹤檢查

acceptance-auditor 在驗收時必須檢查：

1. **建議追蹤區塊存在**：Ticket 中有 Suggestion Tracking 區段
2. **所有建議已處理**：沒有 pending 狀態的建議
3. **採納建議已轉換**：所有 adopted 建議都有對應 AC
4. **拒絕建議有理由**：所有 rejected 建議都有決定理由
5. **延後建議已追蹤**：所有 deferred 建議都有目標版本

### 驗收報告中的建議追蹤結果

```markdown
### 建議追蹤檢查

| #   | 建議來源    | 建議內容        | 決定     | 驗證結果         |
| --- | ----------- | --------------- | -------- | ---------------- |
| 1   | 歷史報告 | 建立驗收代理人  | adopted  | AC-001 已完成    |
| 2   | 歷史報告 | 建立自動化 Hook | deferred | 已記錄到後續版本 |
```

## Ticket 模板更新

### 新增建議追蹤區塊

在 Ticket 模板中，Acceptance Criteria 之後新增：

```markdown
---

## Suggestion Tracking

### 來源：{來源}

| #   | 建議內容 | 狀態    | 決定理由 | 對應 AC | 決定者 | 決定時間 |
| --- | -------- | ------- | -------- | ------- | ------ | -------- |
| 1   | -        | pending | -        | -       | -      | -        |

### 狀態說明

- `pending`: 待決定
- `adopted`: 採納，已轉為驗收條件
- `rejected`: 拒絕，必須說明理由
- `deferred`: 延後，建立追蹤 Ticket
```

## 強制規則

### 必須遵守

| 規則           | 說明                                     |
| -------------- | ---------------------------------------- |
| 不可忽略建議   | 所有建議都必須明確處理（不可留 pending） |
| 不可無理由拒絕 | 拒絕必須說明理由                         |
| 採納必須轉換   | 採納的建議必須轉為驗收條件               |
| 延後必須追蹤   | 延後的建議必須有目標版本和追蹤方式       |

### 違規處理

| 違規行為                     | 處理方式               |
| ---------------------------- | ---------------------- |
| 有 pending 建議就完成 Ticket | 驗收失敗，要求補充     |
| 拒絕建議無理由               | 驗收失敗，要求補充理由 |
| 採納建議無對應 AC            | 驗收失敗，要求建立 AC  |
