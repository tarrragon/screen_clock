## 前言：設計哲學導讀

### 為什麼需要設計哲學？

在程式開發的世界中，我們常常被告知要遵循各種原則：

- DRY（Don't Repeat Yourself）
- SOLID 原則
- Clean Code
- 設計模式

但很少有人告訴我們**為什麼**這些原則重要。

> **所有程式碼設計原則的終極目標是什麼？**

答案是：**降低閱讀者的認知負擔**。

## 1. 核心概念：認知負擔

### 什麼是認知負擔？

認知負擔（Cognitive Load）是指閱讀者在理解程式碼時需要**同時記住**的資訊量。

人類的工作記憶（Working Memory）有其限制：

> **Miller's Law**：人類工作記憶一次只能處理約 7±2 個項目。

當程式碼要求閱讀者同時記住超過這個數量的概念時，理解和維護就會變得困難。

### 認知負擔的影響

| 認知負擔 | 閱讀體驗 | 維護成本 | Bug 風險 |
|---------|---------|---------|---------|
| 低 | 流暢，如同閱讀文章 | 低 | 低 |
| 中 | 需要反覆閱讀 | 中等 | 中等 |
| 高 | 痛苦，需要大量筆記 | 高 | 高 |
| 極高 | 幾乎無法理解 | 極高 | 極高 |

### 核心洞見

> **好的程式碼不是讓電腦容易執行，而是讓人類容易理解。**

電腦不在乎程式碼寫成什麼樣子，但人類在乎。

## 2. 認知負擔的三種類型

認知心理學將認知負擔分為三種類型：

### 2.1 內在認知負擔（Intrinsic Load）

**定義**：任務本身的固有複雜度。

**程式碼範例**：
- 演算法的本質複雜度
- 業務邏輯的固有複雜度
- 技術限制帶來的複雜度

**處理方式**：無法消除，但可以通過良好的設計分解。

### 2.2 外在認知負擔（Extraneous Load）

**定義**：由於不良設計導致的額外複雜度。

**程式碼範例**：
- 糟糕的命名
- 過長的函式
- 深層巢狀
- 隱藏的副作用

**處理方式**：這是我們主要要消除的！

### 2.3 相關認知負擔（Germane Load）

**定義**：用於建立心智模型的有益負擔。

**程式碼範例**：
- 學習新的設計模式
- 理解架構決策的原因
- 建立領域知識

**處理方式**：這是有益的，應該保留並支持。

### 設計目標

```
目標：最小化外在認知負擔，讓閱讀者的認知資源用於
     理解真正重要的事情（內在負擔）和建立心智模型（相關負擔）
```

## 3. 認知負擔的來源

### 3.1 變數狀態追蹤

每個變數都是閱讀者需要「記住」的項目。

**問題程式碼**：
```dart
void processOrder(Order order) {
  var items = order.items;
  var total = 0.0;
  var discount = 0.0;
  var tax = 0.0;
  var shipping = 0.0;
  var finalPrice = 0.0;
  var isValid = true;
  var errorMessage = '';
  // 閱讀者需要同時追蹤 8 個變數狀態
}
```

**改善方式**：
```dart
void processOrder(Order order) {
  final pricing = _calculatePricing(order);
  final validation = _validateOrder(order);
  // 閱讀者只需追蹤 2 個概念
}
```

### 3.2 呼叫層級追蹤

追蹤呼叫鏈需要「堆疊」記憶。

**問題**：
```dart
// 閱讀者需要追蹤 4 層呼叫才能理解邏輯
methodA() -> methodB() -> methodC() -> methodD()
```

**改善方式**：
```dart
// 扁平化，讓閱讀者在同一層級理解
methodA() {
  resultB = methodB();
  resultC = methodC(resultB);
  return finalProcess(resultC);
}
```

### 3.3 命名品質

不佳的命名增加「翻譯」負擔。

| 問題命名 | 認知負擔 | 改善命名 |
|---------|---------|---------|
| `d` | 需要猜測含義 | `discountAmount` |
| `temp` | 臨時？什麼的臨時？ | `previousBalance` |
| `data` | 什麼資料？ | `userProfile` |
| `process()` | 處理什麼？怎麼處理？ | `calculateTotalPrice()` |
| `handle()` | 處理什麼事件？ | `onUserLoginSuccess()` |

### 3.4 條件分支複雜度

每個分支都是需要考慮的「路徑」。

**問題**：
```dart
if (condition1) {
  if (condition2) {
    if (condition3) {
      // 閱讀者需要同時記住 3 個條件
    }
  }
}
```

**改善方式**：
```dart
// 使用 Guard Clause 扁平化
if (!condition1) return;
if (!condition2) return;
if (!condition3) return;

// 主邏輯在這裡，只有一層縮排
```

### 3.5 隱藏的副作用

函式名稱未反映其真實行為。

**問題**：
```dart
// 名稱說「取得」，但實際上會修改資料庫
User getUser(int id) {
  _database.updateLastAccessTime(id); // 隱藏副作用！
  return _database.findById(id);
}
```

**改善方式**：
```dart
// 分離查詢和命令
User findUserById(int id) {
  return _database.findById(id);
}

void recordUserAccess(int id) {
  _database.updateLastAccessTime(id);
}
```

## 4. 降低認知負擔的原則

### 原則 1：單一責任

> 一個函式只做一件事，讓讀者只需理解一個概念。

**自我檢查**：這個函式做了幾件事？

| 函式描述 | 責任數 | 評估 |
|---------|--------|------|
| 「計算總價」 | 1 | 符合 |
| 「計算總價並發送通知」 | 2 | 違反 |
| 「驗證、計算、儲存」 | 3 | 嚴重違反 |

### 原則 2：自說明命名

> 名稱本身就是文件，讓讀者不需要額外資訊。

**自我檢查**：不看實作，能從名稱理解功能嗎？

| 命名 | 自說明程度 | 改善 |
|------|-----------|------|
| `calc()` | 低 | `calculateMonthlyRevenue()` |
| `isValid` | 中 | `isEmailFormatValid` |
| `processUserRegistration()` | 高 | 維持 |

### 原則 3：避免副作用

> 函式的行為應該可預測，讓讀者不需要追蹤隱藏狀態。

**自我檢查**：這個函式有沒有修改輸入以外的東西？

| 行為 | 副作用 | 處理 |
|------|--------|------|
| 純計算 | 無 | 良好 |
| 修改參數 | 有 | 改為返回新物件 |
| 修改全域狀態 | 有 | 提取為獨立命令 |

### 原則 4：扁平化結構

> 減少巢狀，讓讀者不需要追蹤深層上下文。

**自我檢查**：最深的巢狀層級是多少？

| 層級 | 評估 | 處理 |
|------|------|------|
| 1-2 | 優良 | 維持 |
| 3 | 可接受 | 考慮重構 |
| 4+ | 需重構 | 使用 Guard Clause 或提取函式 |

### 原則 5：資訊就近原則

> 相關的資訊應該放在一起，讓讀者不需要跳轉尋找。

**自我檢查**：理解這段程式碼需要跳到幾個地方？

| 跳轉次數 | 評估 | 處理 |
|---------|------|------|
| 0-1 | 優良 | 維持 |
| 2-3 | 可接受 | 考慮整合 |
| 4+ | 需重構 | 重新組織 |

## 5. SOLID 原則的認知負擔視角

### 5.1 單一責任原則（SRP）

**傳統解釋**：一個類別只有一個改變的原因。

**認知負擔視角**：一個類別只代表一個概念，閱讀者只需建立一個心智模型。

| 類別 | 責任數 | 認知負擔 |
|------|--------|---------|
| `UserRepository` | 1 | 低 |
| `UserServiceAndValidator` | 2 | 中 |
| `UserManagerAndLoggerAndNotifier` | 3+ | 高 |

### 5.2 開放封閉原則（OCP）

**傳統解釋**：對擴展開放，對修改封閉。

**認知負擔視角**：新增功能不需要理解現有程式碼的所有細節。

**未遵循 OCP 時**：
```
新增折扣類型
    |
    +-- 需要理解 calculatePrice() 所有邏輯
    +-- 需要修改 switch/case
    +-- 需要理解所有現有折扣的互動
    |
    認知負擔：高（需要理解整個系統）
```

**遵循 OCP 時**：
```
新增折扣類型
    |
    +-- 只需理解 Discount 介面
    +-- 實作新的 DiscountStrategy
    +-- 註冊到系統
    |
    認知負擔：低（只需理解介面契約）
```

### 5.3 里氏替換原則（LSP）

**傳統解釋**：子類別可以替換父類別。

**認知負擔視角**：使用者不需要知道具體實作，減少需要考慮的情況。

### 5.4 介面隔離原則（ISP）

**傳統解釋**：不應該強迫依賴不需要的方法。

**認知負擔視角**：介面只包含相關概念，閱讀者不需要過濾不相關資訊。

### 5.5 依賴反轉原則（DIP）

**傳統解釋**：高層模組不應依賴低層模組。

**認知負擔視角**：透過抽象隔離複雜度，閱讀者可以分層理解系統。

## 6. 自我檢查清單

### 函式層級

- [ ] 函式長度 <= 15 行（理想 5-10 行）
- [ ] 參數數量 <= 3
- [ ] 巢狀深度 <= 3
- [ ] 區域變數 < 5
- [ ] 函式名稱是動詞片語
- [ ] 函式只做一件事
- [ ] 無隱藏副作用

### 類別層級

- [ ] 類別方法數 <= 10
- [ ] 公開方法數 <= 5
- [ ] 類別只代表一個概念
- [ ] 類別名稱是名詞

### 命名層級

- [ ] 無縮寫或僅使用通用縮寫
- [ ] 變數名稱是名詞
- [ ] 函式名稱是動詞
- [ ] 布林變數是 is/has/can 開頭
- [ ] 不使用 data/info/value 等模糊詞

### 認知負擔計算

```
認知負擔指數 = 變數數 + 分支數 + 巢狀深度 + 外部依賴數

| 指數 | 評估 |
|------|------|
| 1-5  | 優良 |
| 6-10 | 可接受 |
| 11-15| 需重構 |
| >15  | 必須重構 |
```

## 7. 與其他方法論的關係

| 方法論 | 認知負擔視角 |
|--------|-------------|
| DRY | 減少重複 = 減少需要記憶的版本 |
| KISS | 保持簡單 = 降低理解門檻 |
| YAGNI | 不做多餘的事 = 減少無關資訊 |
| Clean Code | 可讀性 = 低認知負擔 |
| TDD | 測試作為文件 = 提供學習支持 |
| 自然語言程式設計 | 程式碼如閱讀文章 = 利用語言直覺 |
