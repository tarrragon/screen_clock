## 前言：設計哲學導讀

DRY、SOLID、Clean Code、設計模式——這些原則為何重要？本方法論回答一個根本問題：**所有程式碼設計原則的終極目標是什麼？**

答案是：**降低閱讀者的認知負擔**。

> **核心洞見**：好的程式碼不是讓電腦容易執行，而是讓人類容易理解。電腦不在乎程式碼寫成什麼樣子，但人類在乎。

## 1. 核心概念：認知負擔

認知負擔（Cognitive Load）是指閱讀者在理解程式碼時需要**同時記住**的資訊量。人類的工作記憶（Working Memory）有其限制：

> **Miller's Law**：人類工作記憶一次只能處理約 7±2 個項目。

當程式碼要求閱讀者同時記住超過這個數量的概念時，理解和維護就會變得困難。

### 認知負擔的影響

| 認知負擔 | 閱讀體驗 | 維護成本 | Bug 風險 |
|---------|---------|---------|---------|
| 低 | 流暢，如同閱讀文章 | 低 | 低 |
| 中 | 需要反覆閱讀 | 中等 | 中等 |
| 高 | 痛苦，需要大量筆記 | 高 | 高 |
| 極高 | 幾乎無法理解 | 極高 | 極高 |

## 2. 認知負擔的三種類型

認知心理學將認知負擔分為三種類型。這是本方法論的核心深度概念——理解三者差異，才能精準判斷「哪些複雜度該消除、哪些該保留」。

| 類型 | 定義 | 程式碼對應 | 處理方式 |
|------|------|-----------|---------|
| 內在認知負擔（Intrinsic Load） | 任務本身的固有複雜度 | 演算法本質複雜度、業務邏輯固有複雜度、技術限制複雜度 | 無法消除，但可透過良好設計分解 |
| 外在認知負擔（Extraneous Load） | 不良設計導致的額外複雜度 | 糟糕命名、過長函式、深層巢狀、隱藏副作用 | **這是主要要消除的** |
| 相關認知負擔（Germane Load） | 用於建立心智模型的有益負擔 | 學習設計模式、理解架構決策原因、建立領域知識 | 有益，應保留並支持 |

### 設計目標

> 最小化外在認知負擔，讓閱讀者的認知資源用於理解真正重要的事情（內在負擔）和建立心智模型（相關負擔）。

## 3. 認知負擔的來源

五大來源各自如何拉高閱讀者需要同時記憶的項目數。每一類的完整正反程式碼範例見衛星檔。

| 來源 | 機制 | 改善方向 |
|------|------|---------|
| 變數狀態追蹤 | 每個區域變數都是要「記住」的項目 | 聚合為少數高階概念（如 pricing / validation） |
| 呼叫層級追蹤 | 追蹤呼叫鏈需要「堆疊」記憶 | 扁平化，讓閱讀者在同一層級理解 |
| 命名品質 | 不佳命名增加「翻譯」負擔 | 自說明命名，名稱即文件 |
| 條件分支複雜度 | 每個分支都是要考慮的「路徑」 | Guard Clause 扁平化巢狀 |
| 隱藏的副作用 | 函式名稱未反映真實行為 | 分離查詢（Query）與命令（Command） |

> **何時讀衛星檔**：需要對照具體的問題程式碼與改善方式（變數狀態追蹤 / 呼叫層級 / 命名品質 / 條件分支 / 隱藏副作用），讀 `.claude/references/cognitive-load-design-examples.md`「認知負擔來源的完整範例」節。

## 4. 降低認知負擔的原則

五大原則的核心要求、違反症狀與改善方向已在 rule 速查表（`.claude/rules/core/cognitive-load.md`「降低認知負擔的原則」節）。本節僅補充每原則對應的「自我提問」，作為設計時的快速反思。

| 原則 | 自我提問 |
|------|---------|
| 1. 單一責任 | 這個函式做了幾件事？ |
| 2. 自說明命名 | 不看實作，能從名稱理解功能嗎？ |
| 3. 避免副作用 | 這個函式有沒有修改輸入以外的東西？ |
| 4. 扁平化結構 | 最深的巢狀層級是多少？ |
| 5. 資訊就近 | 理解這段程式碼需要跳到幾個地方？ |

> **閾值與檢查清單**：各原則的具體門檻（巢狀 1-2 優良 / 3 可接受 / 4+ 需重構等）與完整函式/類別/命名檢查清單見 rule。

## 5. SOLID 原則的認知負擔視角

SOLID 五原則的傳統解釋人盡皆知，但「為何」遵循它們？認知負擔視角給出統一解答：每個原則都在減少閱讀者需要同時持有的概念數。

| 原則 | 傳統解釋 | 認知負擔視角 |
|------|---------|-------------|
| SRP（單一責任） | 一個類別只有一個改變的原因 | 一個類別只代表一個概念，閱讀者只需建立一個心智模型 |
| OCP（開放封閉） | 對擴展開放，對修改封閉 | 新增功能不需要理解現有程式碼的所有細節 |
| LSP（里氏替換） | 子類別可以替換父類別 | 使用者不需要知道具體實作，減少需要考慮的情況 |
| ISP（介面隔離） | 不應強迫依賴不需要的方法 | 介面只包含相關概念，閱讀者不需要過濾不相關資訊 |
| DIP（依賴反轉） | 高層模組不應依賴低層模組 | 透過抽象隔離複雜度，閱讀者可以分層理解系統 |

> **何時讀衛星檔**：需要各原則的責任數對照表、OCP 遵循/未遵循的流程示意，讀 `.claude/references/cognitive-load-design-examples.md`「SOLID 原則的認知負擔視角詳解」節。

## 6. 與其他方法論的關係

認知負擔是貫穿所有設計原則的共同目標，其他方法論皆可從此視角統一理解。

| 方法論 | 認知負擔視角 |
|--------|-------------|
| DRY | 減少重複 = 減少需要記憶的版本 |
| KISS | 保持簡單 = 降低理解門檻 |
| YAGNI | 不做多餘的事 = 減少無關資訊 |
| Clean Code | 可讀性 = 低認知負擔 |
| TDD | 測試作為文件 = 提供學習支持 |
| 自然語言程式設計 | 程式碼如閱讀文章 = 利用語言直覺 |
