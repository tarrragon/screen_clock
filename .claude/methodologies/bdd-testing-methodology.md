# BDD 測試方法論

## 核心概念

BDD 是測試行為而非實作的策略。測試從使用者視角描述系統行為，使用 Given-When-Then 結構。

**核心價值**：重構時測試保持穩定，因為測試只關心行為結果。

---

## Given-When-Then 結構

| 元素 | 定義 | 規範 |
|-----|------|------|
| **Given** | 系統初始狀態 | 明確、完整、可重現 |
| **When** | 使用者操作 | 單一動作、業務語言 |
| **Then** | 預期結果 | 可驗證、可觀察的行為 |

---

## 行為 vs 實作判斷

| 問題 | 是行為 | 是實作 |
|-----|-------|-------|
| 使用者能直接觀察到？ | Yes | No |
| 改變實作會影響測試？ | No | Yes |
| 產品經理需要關心？ | Yes | No |

---

## 分層測試策略

| 層級 | 測試策略 |
|-----|---------|
| UseCase (Layer 3) | BDD 測試（必須） |
| Domain (Layer 5) | 複雜規則 → 單元測試 |
| Behavior (Layer 2) | 複雜轉換 → 單元測試 |
| UI (Layer 1) | 關鍵流程 → 整合測試 |
| Interface (Layer 4) | 不測試 |

---

## Mock 策略

| 依賴類型 | Mock？ |
|---------|-------|
| Repository / Service / Event Publisher | Yes（外層依賴） |
| Domain Entity / Value Object | No（使用真實物件） |

---

## 執行步驟

1. 從需求提取 Given-When-Then 場景
2. 場景轉換為測試程式碼
3. 測試先行（Red → Green → Refactor）
4. 重構時測試保持穩定

---

## 檢查清單

### 場景完整性
- [ ] 有正常流程測試
- [ ] 有異常流程測試（至少 2 個）
- [ ] 有邊界條件測試（null、0、負數、上限）

### 測試品質
- [ ] 測試名稱使用業務語言
- [ ] Given-When-Then 結構清晰
- [ ] When 只有單一動作
- [ ] Then 驗證可觀察結果（非實作細節）

### Mock 策略
- [ ] 只 Mock 外層依賴
- [ ] Domain Entity 使用真實物件
- [ ] 沒有驗證內部方法呼叫次數

### 重構驗證
- [ ] 重構內部邏輯 → 測試無需修改？
- [ ] 改變演算法 → 測試無需修改？

## 行為鏈式推演框架（v1.2.0 新增）

### 核心概念

行為鏈式推演（Behavior Chain Derivation）是指測試設計時必須沿著使用者的操作序列逐步推演，每個步驟依賴前一步驟的結果，形成 A → B → C 的依賴鏈。

**問題根源**：粗糙測試設計跳過中間步驟，直接驗證最終結果，導致測試無法精確定位失敗原因。

**範例問題**：測試「點擊按鈕後頁面跳轉」，若只驗證跳轉成功，當按鈕不存在時測試會報錯但錯誤訊息令人困惑。

### 行為鏈定義

| 鏈節 | 意義 | 驗證對象 |
|------|------|---------|
| A（前置條件） | 行為發生前系統必須處於的狀態 | 元素存在、資料載入、權限具備 |
| B（行為觸發） | 使用者執行的單一操作 | 點擊、輸入、提交、導航 |
| C（結果確認） | 行為後系統狀態的可觀察變化 | UI 更新、資料變更、導航完成 |

### 行為鏈式推演規則（三項強制規則）

**規則 1：前置條件必須先驗證**

測試行為 B 之前，**必須**先以獨立斷言驗證 A（前置條件）存在或成立。

```
// 錯誤做法（跳過前置驗證）
When: 點擊「送出」按鈕
Then: 驗證表單已提交

// 正確做法（先驗後行）
Then（前置）: 「送出」按鈕存在且可互動
When: 點擊「送出」按鈕
Then: 驗證表單已提交
```

**規則 2：行為觸發必須明確單一**

每個 When 只描述一個原子操作，不合併多個動作。

```
// 錯誤做法（合併多動作）
When: 填寫表單並點擊送出

// 正確做法（單一動作）
When: 點擊「送出」按鈕
（填寫表單是另一個 Given 步驟）
```

**規則 3：結果確認必須可觀察**

Then 只驗證使用者可直接觀察到的狀態，不驗證內部實作細節。

```
// 錯誤做法（驗證內部實作）
Then: repository.save() 被呼叫一次

// 正確做法（驗證可觀察結果）
Then: 畫面顯示「儲存成功」訊息
```

### 推演停止條件

沙盤推演達到以下任一條件即可停止：

1. **行為分支已窮舉**：所有可能的操作路徑（正常、異常、邊界）都已列出對應場景
2. **前置條件已完整**：每個場景的前置狀態都可明確描述且可重現
3. **結果可觀察已確認**：每個 Then 都對應使用者可直接感知的系統狀態

**不應停止的情況**：「這個邊界條件很少發生」或「這個錯誤不太重要」不是停止理由。

### 行為鏈完整推演示例

**功能**：點擊「刪除」按鈕刪除一筆記錄

**反面案例（粗糙設計）**：

```
場景：刪除記錄
Given: 已登入系統
When: 點擊刪除按鈕
Then: 記錄被刪除
```

問題：未驗證按鈕是否存在、未驗證確認對話框、未驗證刪除後的 UI 反饋、未測試刪除失敗情況。

**正面案例（完整行為鏈）**：

```
場景 1：刪除記錄 - 正常流程
Given: 已登入系統，且清單頁面已載入，且清單中存在至少一筆記錄
Then（前置驗證）: 第一筆記錄旁的「刪除」按鈕存在且可點擊
When: 點擊第一筆記錄的「刪除」按鈕
Then（中間狀態）: 確認對話框出現，顯示「確定要刪除嗎？」文字
When: 點擊對話框的「確定」按鈕
Then: 對話框關閉，清單重新整理，第一筆記錄不再出現

場景 2：刪除記錄 - 取消操作
Given: 確認對話框已出現
When: 點擊對話框的「取消」按鈕
Then: 對話框關閉，清單不變，記錄仍然存在

場景 3：刪除記錄 - 刪除失敗（網路錯誤）
Given: 確認對話框已出現，且網路連線中斷
When: 點擊對話框的「確定」按鈕
Then: 顯示錯誤提示「刪除失敗，請重試」，記錄仍然存在

場景 4：刪除最後一筆記錄（邊界條件）
Given: 清單中只剩一筆記錄
Then（前置驗證）: 「刪除」按鈕存在
When: 完成刪除確認
Then: 清單顯示空狀態畫面「目前沒有記錄」
```

## 前置條件驗證強制規則（v1.2.0 新增）

### 強制規則

**每個測試步驟在執行行為（When）之前，必須以明確斷言驗證其前置條件成立。**

| 前置條件類型 | 驗證方式 | 範例 |
|------------|---------|------|
| UI 元素存在 | `expect(widget, findsOneWidget)` | 按鈕、輸入框、對話框 |
| 資料已載入 | 驗證清單長度 > 0 或特定項目存在 | 清單項目、表單初始值 |
| 系統狀態 | 驗證狀態變數或畫面文字 | 登入狀態、載入完成 |
| 權限具備 | 驗證操作按鈕未被禁用 | 編輯按鈕可點擊 |

### 前置驗證失敗的處理

前置條件驗證失敗時，測試應**立即停止**並給出清晰的失敗訊息，而非繼續執行行為步驟導致難以理解的錯誤。

```
// 前置驗證失敗的預期行為
前置: expect(deleteButton, findsOneWidget)  → 失敗訊息：「刪除按鈕不存在」
                                              ← 測試在此停止，不繼續執行 tap()
```

若跳過前置驗證直接執行行為，失敗訊息可能是「NullPointerException」或「element not found」，難以判斷真正原因是前置條件未滿足還是行為本身有問題。

### 前置條件驗證檢查清單

設計每個測試場景時，確認：

- [ ] Given 中的每個狀態都有對應的驗證斷言
- [ ] UI 元素在互動前先 `expect(element, findsOneWidget)`
- [ ] 資料操作前先確認資料存在或符合預期格式
- [ ] 多步驟場景中每個中間狀態都有驗證斷言
- [ ] 前置驗證的失敗訊息明確（能直接定位問題）

## 測試設計完成標準（v1.2.0 新增）

### 行為分支窮舉方法

按以下四個維度系統化列出所有行為分支：

| 維度 | 問題 | 典型場景 |
|------|------|---------|
| 正常流程 | 功能按預期工作時的路徑 | 成功新增、成功刪除、成功跳轉 |
| 異常流程 | 輸入無效或操作失敗時的路徑 | 驗證失敗、網路錯誤、伺服器錯誤 |
| 邊界條件 | 極端輸入或狀態下的行為 | 空清單、最大長度、單一記錄 |
| 中斷操作 | 使用者中途放棄的路徑 | 取消確認、離開表單、返回上頁 |

### 完成判斷標準

沙盤推演完成的可操作化定義：

1. 上述四個維度的場景都已至少有一個對應測試案例
2. 對每個場景問「這個 Then 還能細分為更小的可觀察狀態嗎？」— 若答案為否，推演停止
3. 對每個場景問「這個 Given 還有什麼前置條件沒有驗證？」— 若答案為否，推演停止
4. 整理所有場景後，無法再找到新的「如果...那麼...」句型組合

## 邊界條件系統化方法（v1.2.0 新增）

### 分類體系

| 類別 | 子類 | 典型邊界值 |
|------|------|-----------|
| 數值邊界 | 最小值 | 0、1、-1、Integer.MIN_VALUE |
| 數值邊界 | 最大值 | 上限值、上限+1、Integer.MAX_VALUE |
| 字串邊界 | 空值 | null、空字串 `""`、空白字串 `" "` |
| 字串邊界 | 長度 | 最大長度、最大長度+1 |
| 字串邊界 | 特殊字元 | 換行符、Unicode、SQL 注入字元 |
| 集合邊界 | 空集合 | `[]`、`null` |
| 集合邊界 | 單一元素 | `[item]` |
| 集合邊界 | 滿載 | 最大容量、最大容量+1 |
| 狀態邊界 | 初始狀態 | 首次使用、重置後 |
| 狀態邊界 | 終止狀態 | 已刪除、已完成、已過期 |
| 並發邊界 | 競爭條件 | 同時操作、重複提交 |

### 推導方法

對每個輸入參數或系統狀態，依序問：

1. 「這個值的合法範圍是什麼？」→ 識別最小值和最大值
2. 「合法範圍的邊界外一步是什麼？」→ 加入 boundary+1 場景
3. 「完全沒有值時會怎樣？」→ 加入 null/empty 場景
4. 「恰好只有一個時會怎樣？」→ 加入 single item 場景
