# 程式碼自然語言化撰寫方法論

## 核心概念

程式碼是給人閱讀的業務邏輯文件，不是給機器的指令縮寫。

---

## 理論基礎：認知負擔

> 參考：[認知負擔設計方法論]($CLAUDE_PROJECT_DIR/.claude/methodologies/cognitive-load-design-methodology.md)

### 為什麼自然語言化？

人類的工作記憶有限（約 7 個項目）。當程式碼需要額外的「翻譯」工作時，就會消耗這些寶貴的認知資源。

**自然語言化的目標**：讓程式碼像閱讀文章一樣自然，減少讀者的翻譯負擔。

### 認知負擔的來源

| 來源 | 問題 | 解決方式 |
|------|------|---------|
| 縮寫 | 讀者需要「展開」縮寫 | 完整命名 |
| 模糊詞 | 讀者需要「猜測」含義 | 具體命名 |
| 長函式 | 讀者需要「分段」理解 | 拆分函式 |
| 深巢狀 | 讀者需要「追蹤」上下文 | 扁平化 |

### 程式碼是故事

把程式碼想像成一個故事：

- **類別**是故事中的角色（名詞）
- **函式**是角色的行為（動詞）
- **變數**是角色持有的物品（名詞）
- **邏輯流程**是故事的情節

好的命名讓讀者能自然地「閱讀」這個故事，而不是「解碼」。

## 五大原則

### 原則一：自然語言命名

> **認知負擔視角**：好的命名消除「翻譯」負擔，讓讀者直接理解意圖。

| 元素 | 規則 | 範例 |
|-----|------|------|
| 函式 | 動詞 + 完整目標 | `calculateBookReadingProgress()` |
| 變數 | 內容物描述，無縮寫 | `authenticatedUser`（非 `usr`） |
| 類別 | 業務職責描述 | `BookMetadataEnrichmentService`（非 `Manager`） |

#### 命名的藝術：讓程式碼說故事

**1. 變數命名：名詞原則**

| 問題命名 | 問題 | 改善命名 |
|---------|------|---------|
| `d` | 需要猜測 | `discountAmount` |
| `temp` | 臨時什麼？ | `previousBalance` |
| `data` | 什麼資料？ | `userProfile` |
| `flag` | 什麼旗標？ | `isUserLoggedIn` |

**2. 函式命名：動詞原則**

| 問題命名 | 問題 | 改善命名 |
|---------|------|---------|
| `process()` | 處理什麼？ | `validateUserInput()` |
| `handle()` | 處理什麼事件？ | `onLoginButtonClick()` |
| `do()` | 做什麼？ | `sendEmailNotification()` |
| `check()` | 檢查什麼？ | `isEmailFormatValid()` |

**3. 布林命名：問句原則**

布林變數應該能讀成問句：

| 命名 | 讀法 | 評估 |
|------|------|------|
| `isValid` | Is valid? | 良好 |
| `hasPermission` | Has permission? | 良好 |
| `canEdit` | Can edit? | 良好 |
| `valid` | Valid? | 可接受 |
| `permission` | Permission? | 不佳 |

**4. 避免的命名模式**

| 模式 | 問題 | 替代方案 |
|------|------|---------|
| 匈牙利命名法 | `strName`, `intCount` | 直接用 `name`, `count` |
| 無意義前綴 | `theUser`, `aBook` | 直接用 `user`, `book` |
| 過度縮寫 | `usrMgr` | `userManager` |
| 數字後綴 | `user1`, `user2` | `primaryUser`, `secondaryUser` |

### 原則二：五行函式

- 每個函式 **5-10 行**
- 超過 10 行 → 必須拆分
- 名稱使用「和/或」→ 必須拆分

### 原則三：變數職責專一

- 每個變數只承載 **單一類型資料**
- 禁止同一變數在不同位置承載不同內容
- 每階段使用專用變數

### 原則四：事件驅動語意化

多個 if/else → 拆分為獨立事件處理函式：

```text
submitForm() 包含：驗證 + API 調用 + 結果處理
↓ 拆分為
validateData() → handleSuccess() / handleFailure()
```

### 原則五：可讀性優於簡潔性

價值排序：正確性 > 可讀性 > 可維護性 > 簡潔性

## 執行步驟

1. **命名檢查**：函式是否動詞開頭？變數是否無縮寫？
2. **長度檢查**：函式超過 10 行？→ 拆分
3. **職責檢查**：變數是否兼用？→ 分離
4. **事件檢查**：多個 if/else？→ 事件化
5. **可讀性測試**：5 分鐘內陌生人能理解？

## 檢查清單

### 命名
- [ ] 函式以動詞開頭
- [ ] 變數完整描述內容，無縮寫
- [ ] 類別說明業務職責

### 結構
- [ ] 每個函式 ≤ 10 行
- [ ] 無複合邏輯函式
- [ ] 變數職責單一

### 事件化
- [ ] if/else 已拆分為事件處理函式
- [ ] 函式名稱反映業務事件

### 驗證
- [ ] 陌生人 5 分鐘內能理解
- [ ] 翻譯成自然語言流暢
