# 工作日誌撰寫方法論

**版本**: v2.0.0
**建立日期**: 2026-01-10
**最後更新**: 2026-03-31
**核心原則**: 工具相容性優先，純文字狀態標記，進度即時記錄

---

## 方法論目的

定義工作日誌的撰寫規範，確保：
1. **格式一致性** - 所有工作日誌遵循統一結構
2. **工具相容性** - 避免 CLI 處理問題導致的 crash
3. **可維護性** - 便於搜尋、解析和自動化處理

### 背景

Claude Code CLI 在處理 markdown 表格中的特定 emoji 時會發生 Rust panic：
```
byte index 2 is not a char boundary; it is inside '⏳' (bytes 0..3) of `⏳ |`
```

此問題源於 Rust 字串切割時遇到多位元組 Unicode 字元的 byte boundary 問題。

---

## 第一原則：工具相容性

### 禁止項目

在 **markdown 表格單元格**中：
- 禁止使用 emoji 字元（如 `⏳`, `🔄`, `❌`, `🚫`）
- 禁止使用其他多位元組特殊符號作為狀態標記

**錯誤範例**：
```markdown
| Ticket ID | 狀態 |
|-----------|------|
| Wn-001    | ⏳   |  <!-- 禁止：會導致 CLI crash -->
| Wn-002    | 🔄   |  <!-- 禁止 -->
```

### 允許項目

- 表格外的段落中使用 emoji（標題、說明文字、列表項目）
- 純文字狀態標記

**正確範例**：
```markdown
| Ticket ID | 狀態 |
|-----------|------|
| Wn-001    | 待處理 |  <!-- 正確：純文字 -->
| Wn-002    | 進行中 |  <!-- 正確 -->

#### Wn-001 分析結果：已完成

上述 Ticket 已於 2026-01-10 完成。  <!-- 表格外可用 emoji -->
```

---

## 第二原則：狀態符號標準化

### 表格內狀態（純文字）

| 狀態語意 | 標準寫法 | 禁用符號 |
|----------|---------|----------|
| 等待處理 | `待處理` | `⏳` |
| 正在執行 | `進行中` | `🔄` |
| 執行完畢 | `已完成` | - |
| 主動取消 | `取消` | `❌` |
| 跳過不做 | `跳過` | `⏭️` |
| 被阻塞中 | `阻塞` | `🚫` |
| 失敗錯誤 | `失敗` | `💥` |

### 表格外狀態（可用 emoji）

在段落、標題、列表中可使用：
- `已完成` 項目標記
- `失敗` 或取消項目標記
- `警告` 說明

**範例**：
```markdown
#### Wn-001 分析結果（下一 Wave）：已完成

**核心發現**：
- 測試卡住問題不存在，所有 14 個測試 100% 通過
- 技術債務等級：無債務
```

## 第三原則：進度即時記錄

> **來源**：Legacy Code 步驟 4 執行中發現 worklog 完全未更新的歷史教訓，PC-033 僅被動防護。
> **審查結論**：這是規範問題，用規範解決。

### Worklog 雙職責

| 區段 | 職責 | 更新頻率 |
|------|------|---------|
| 版本目標 + Ticket 索引 | 企劃（靜態） | 版本開始、範圍變更時 |
| **進度追蹤** | 進度記錄（動態） | 每個重要事件發生時 |

進度追蹤記錄的是**決策和里程碑**，不是執行細節（細節在 ticket）。

### 需記錄到 worklog 的 5 類事件

| 事件類型 | 觸發條件 | 記錄格式 |
|---------|---------|---------|
| Ticket 完成 | ticket track complete | `{日期}: {id} 完成 -- {摘要}` |
| 任務拆分 | 建立子 Ticket 或拆分 | `{日期}: {parent} 拆分 -- {child1}, {child2}` |
| 額外發現 | 執行中建立非計畫 Ticket | `{日期}: 新增 {id} -- {原因摘要}` |
| UC/功能推進 | 一組相關 Ticket 完成 | `{日期}: UC-XX 步驟 Y 完成 -- {結果摘要}` |
| 阻塞/風險 | 發現阻塞或設計問題 | `{日期}: {id} 阻塞 -- {原因}` |

### 更新時機

| 時機 | 動作 |
|------|------|
| ticket track complete 後 | 追加一行進度到 worklog |
| 建立新 Ticket 後 | 追加一行進度到 worklog |
| 任務拆分決策後 | 追加一行進度到 worklog |
| session 結束前（handoff） | 確認本次 session 的進度已記錄 |

### 進度追蹤區段格式（bullet list）

```markdown
## 進度追蹤

- {日期}: UC-08 修復完成 -- {commit-hash}，失敗 259->141
- {日期}: UC-04 {ticket-id} 完成 -- 3/5 Widget 測試修復（14 個）
- {日期}: {ticket-id} 拆分 -- {子 ticket-id}（複雜 Provider）、{子 ticket-id}（UI 規範）
- {日期}: {ticket-id} 完成 -- Worklog 進度同步機制分析
```

> 使用 bullet list 而非 table：更易追加、diff 友好、條目超過 20 行後仍可讀。

### 密集衝刺後的敘事式摘要（合法替代格式）

> **來源**：完成大量 Ticket 但 worklog 缺少逐筆 bullet list 進度記錄的歷史教訓。

當 session 中密集完成多個 Ticket（如 Wave 收尾、批量修復），逐筆 bullet list 可能不切實際。此時允許使用**敘事式摘要**替代：

```markdown
### {日期}：{主題}衝刺

- {ticket-range} 可觀測性修復：全域錯誤處理、靜默失敗路徑消除、日誌覆蓋
- {ticket-range} 審查結論全部修復：AppLogger 遷移、ErrorHandler 防護
- {ticket-range} 完成：jsonDecode 回退、Timer/Stream 審計、AppLogger 實例化
```

**敘事式摘要要求**：
- 必須涵蓋所有完成的 Ticket ID（可用範圍表示如 W{n}-{start}~{end}）
- 必須包含關鍵成果摘要
- 應在 session 結束前或 handoff 時補寫

**優先級**：逐筆 bullet list > 敘事式摘要 > 無記錄。日常開發用 bullet list，密集衝刺後用敘事式摘要。

## 第四原則：標準結構

### 工作日誌必要章節

```markdown
# vX.Y.Z [版本名稱] 工作日誌

**版本**: vX.Y.Z
**開始日期**: YYYY-MM-DD
**狀態**: 進行中 / 已完成
**分支**: `feature/xxx` 或 `fix/xxx`

---

## 版本目標

[簡要描述本版本要達成的目標]

---

## 進度追蹤

- YYYY-MM-DD: [事件] -- [摘要]

---

## Phase 1: [階段名稱]

### [子任務或 Ticket 清單]

| Ticket ID | Action | Target | Agent | 狀態 |
|-----------|--------|--------|-------|------|
| X.Y.Z-W1-001 | [動詞] | [目標] | [代理人] | 待處理 |

---

## 技術筆記

[開發過程中的技術發現和決策記錄]

---

## 參考文件

[相關文件連結]
```

### Ticket 表格標準格式

| 欄位 | 說明 | 範例 |
|------|------|------|
| Ticket ID | 版本號-階段-序號 | `{version}-W1-001` |
| Action | 動詞（Analyze/Design/Fix/Implement） | `Analyze` |
| Target | 修改目標 | `Widget 測試基礎設施` |
| Agent | 執行代理人 | `sage-test-architect` |
| Dependencies | 依賴的其他 Ticket | `Wn-001`（格式說明） |
| 狀態 | 純文字狀態 | `待處理` |

---

## 驗證檢查清單

撰寫工作日誌時，確認以下項目：

- [ ] 所有表格單元格使用純文字狀態
- [ ] 無 `| ⏳ |`、`| 🔄 |`、`| ❌ |` 等模式
- [ ] 版本資訊完整（版本號、日期、狀態、分支）
- [ ] Ticket ID 格式正確（版本號-階段-序號）
- [ ] 結構符合標準模板

---

## Hook 自動驗證

本方法論由 `worklog-format-check.py` Hook 自動驗證：

- **觸發時機**：PostToolUse (Edit/Write)
- **檢測範圍**：`docs/work-logs/**/*.md`
- **檢測內容**：表格中的問題 emoji 模式
- **行為**：警告（非阻擋），輸出問題位置

## 結論

工作日誌是開發過程的重要記錄，必須確保：
1. **純文字狀態** - 表格內禁用 emoji，使用標準化純文字
2. **結構一致** - 遵循標準模板，便於搜尋和解析
3. **自動驗證** - 依賴 Hook 系統進行格式檢查

遵循本方法論可避免工具相容性問題，並提升工作日誌的可維護性。
