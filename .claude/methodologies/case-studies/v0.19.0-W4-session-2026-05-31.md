## SOP 組 1：Stale Ticket Cleanup

### 標準觸發條件

依 `feedback_stale_ticket_cleanup_sop`（memory）+ `PC-055`（Ticket AC 與實況漂移）：

- Ticket 建立至 claim 超過 7 天（pending stale）
- claim 前必先執行三步驗證：
  1. `grep` 目標檔案確認問題是否仍存在
  2. `git log` 確認是否有外溢修復（其他 commit 已順帶修復）
  3. 若外溢修復存在 → 走「PM bookkeeping 路徑」（驗證 + 紀錄收尾，無程式碼變更）

### 標準執行步驟（已被外溢修復時）

1. 執行三步驗證，確認外溢修復 commit SHA
2. claim ticket（start_at 記錄）
3. 更新 Problem Analysis（根因鏈 + 外溢修復追溯）
4. 更新 Solution（描述本 ticket 僅做「驗證 + 紀錄收尾」）
5. 更新 Test Results（驗證指令 + 結果）
6. complete（Executing Agent 標 rosemary-project-manager）

## SOP 組 2：PM 親修 tests/perf/

### 標準觸發條件

依 `feedback_pm_test_perf_direct_edit_sop`（memory）：

- 問題位於 `tests/perf/` 或 `tests/performance/`（`npm run test:perf` 範圍，非 npm test 主套件）
- 修改屬「規格層」（移除/調整斷言門檻、移除環境敏感斷言），無 `src/` 變更
- 修改量單一：單檔案 + 有限 Edit 操作（≤ 5 個 Edit）
- PM 自評：實作成本 ≤ 派 thyme 溝通成本

### 標準執行步驟

1. PM 前台執行三步診斷（grep / git log / 連跑 N 次確認 flaky）
2. 設計修改策略（路徑 A 放寬門檻 / 路徑 B 移除斷言 / 路徑 C 混合）
3. PM 前台 Edit 目標檔案（≤ 5 Edit）
4. 連跑 3 次 `npm run test:perf` 驗證修復後穩定
5. commit + complete

## SOP 組 3：分析代理人 Worktree Partial Commit

### 標準觸發條件

依 `feedback_analysis_agent_worktree_partial_commit`（memory）+ `PC-167`（分析代理人 worktree 無寫入，轉錄負擔）：

- 派發分析代理人（saffron-system-analyst / bay-quality-auditor 等）使用 worktree isolation
- 分析代理人無 `ticket create` 或 `ticket track` 寫入權限
- 代理人完成分析後，PM 需接手：(a) 轉錄結論至 ticket body；(b) 依結論 spawn 落地 IMP；(c) check-acceptance + complete

### 標準執行步驟

1. 派發時 prompt 含「讀取 ticket，分析後 append-log 至 Problem Analysis + Solution 章節（分階段）」指引
2. 代理人在 worktree 完成分析（output_file 回報）
3. PM 接手：
   - 讀取代理人 output_file
   - Transcribe 結論至 ticket Problem Analysis / Solution
   - 依 Solution spawn 規劃建落地 IMP（acceptance #N 若含）
   - check-acceptance --all + complete
4. 落地 IMP ticket 完成後，ANA 鏈正式收尾

## SOP 組 4：實作代理人 Self-Managed（新發現）

**背景說明**：本組為本 session 新發現的模式，目前無對應 memory feedback，本 case study 為首次記錄。包含三個子模式：

| 子模式 | 特徵 | Case |
|--------|------|------|
| Background dispatch（thyme 背景） | PM 前台完成診斷 + 規格，派 thyme 背景執行 auto-fix + manual fix | W4-009 |
| Self-managed（thyme 全自主） | PM 完成 Problem Analysis + 路徑選擇規格，thyme claim + 實作 + 驗證 + complete | W4-019 |
| Self merge --theirs（thyme 自主合併） | thyme 完成 commit 後自主執行 merge，不等待 PM | W4-008 |

### 標準觸發條件（子模式通用）

- Ticket 問題明確、修改範圍邊界清晰
- PM 已在 Problem Analysis 完成診斷（包含 stale 三步驗證、策略選擇、禁止範圍）
- thyme 收到 ticket 後不需進一步確認即可執行

- thyme 完成後在 Completion Info 標記「pending PM merge of worktree branch」→ PM 需確認 worktree merge，此環節是 self-managed 完成的最後一步（feedback_agent_monitor_wait_premature_exit + feedback_worktree_merge_after_agent）

### 案例 4.3：W4-008 — event-system-integration 計時斷言（Self Merge --theirs）

**觸發條件**：
- W4-003 thyme 執行期間發現同測試案例 5 處同性質絕對門檻斷言

**thyme 執行**：

| 動作 | 結果 |

**Self merge --theirs 特徵**：

W4-008 在 merge commit 記錄中使用 `--theirs` 策略（取 worktree 版本）。Completion Info 標記「self-verified」，表示 thyme 在 worktree 完成後自主判斷 merge 方向並執行，無需等待 PM 指示。

根據 `feedback_auto_stage_worktree_conflict` memory：auto-stage + worktree 並行 append-log 同檔 → merge conflict；thyme 以 `git checkout --theirs` 採 worktree SSOT 版本，體現對 conflict 解決策略的自主判斷。

**命中率**：thyme 自主執行完整 IMP 流程（WRAP + 執行 + 驗證 + commit + merge）。

**例外**：
- merge commit 合併點與 git log 記錄的 `6cb51839 merge(0.19.0-W4-008): event-system-integration 4 處絕對計時門檻斷言修復` 一致，auto-stage conflict 解決步驟完整

**教訓**：
- `--theirs` 決策正確性前提：thyme 必須確認 worktree 版本是 SSOT（worktree = 最新的實作結果）；若 main 也有相關變更，需先分析 conflict 內容再選策略
- 「self merge --theirs」是 self-managed 的延伸——thyme 不僅完成實作，也完成 worktree 生命週期收尾
- WRAP 三問記錄在 Problem Analysis（W / A / P 各問有記錄）→ 體現 claim 強制 WRAP 規則在 self-managed 場景下的正確應用

### 組 4 共同模式

| 維度 | W4-009 (background) | W4-019 (self-managed) | W4-008 (self merge) |
|------|--------------------|-----------------------|--------------------|
| PM 前台完成程度 | 盤點 + 策略規格 | ANA 觀察鏈 + Stage 規格 | 5 處分類 + 路徑推薦 |

- Self-managed 的前提不是「thyme 聰明」而是「PM 已完成診斷規格」

- Background / self-managed / self merge 三子模式的差異在 PM 參與程度，不在 thyme 實作能力

## 跨組共通教訓

### 教訓 1：PM 主動退出干預迴圈

W4-012 Completion Info 明記「整體執行成本：PM 4 個 Edit + 3 runs npm run test:perf ≈ subagent 派發成本」，作為 PM 親修決策的理由。核心邏輯：**PM 在判斷成本後，主動退出「必須派 thyme」的預設模式，選擇前台執行**。

適用條件：單檔、規格層修改、有限 Edit 數（≤ 5），且 PM 已完成診斷（不需 thyme 探索）。

**反模式**：PM 機械性地把所有修改都派給代理人，忽略派發本身的溝通成本（prompt 撰寫 + 回合消耗）。

### 教訓 2：Stale Ticket 三步驗證的分叉決策

三步驗證的第二步（git log 確認外溢修復）是整個 SOP 最重要的分叉點：

| 結果 | 路徑 | 範例 |
|------|------|------|
| 有外溢修復 | bookkeeping（驗證 + 紀錄 + complete） | W4-013 / W4-014 / W4-011 |
| 無外溢修復，問題仍真實存在 | 實際修復（IMP 或 PM 親修） | W4-012 |
| 無外溢修復，問題消失（環境變化） | 驗證 + 紀錄 + close with reason | （本 session 未出現此情況） |

git log 的 `--all` 範圍很重要：W4-011 的外溢修復來自 W1-095（跨 wave），只查 W4 範圍的 git log 會漏。

### 教訓 3：Auto-stage × Worktree 衝突的系統性防護

W4-008 使用 `git checkout --theirs` 解決 auto-stage × worktree 並行修改同檔的 merge conflict，與 `feedback_auto_stage_worktree_conflict` memory 記錄一致。

**系統性含義**：多個 worktree 並行 + auto-stage hook 是 merge conflict 的結構性來源，不是偶發事故。PM 在並行派發前需確認檔案所有權互斥（`feedback_parallel_agent_conflict` memory）。

### 教訓 4：分析代理人 spawn 的「最後一哩」責任

W4-015 的 PM 接手轉錄 + spawn W4-019 是 ANA 閉環的「最後一哩」：saffron 完成分析但不能建 ticket，PM 接手時需確認 spawned_tickets 實際建立（而非只有 Solution 表格）。這是 quality-baseline 規則 5 的實例：「ANA Solution spawn 規劃表格 ≠ ticket 已建立」。

### 教訓 5：PC-162 stale description 漂移的普遍性

W4-012 發現原 ticket 描述 line 291 失敗，實際失敗在 line 89。W4-011 的外溢修復範圍比原 ticket 描述更廣（全專案 ANA）。Stale ticket 的 description 隨時間漂移是普遍現象，**接手前必須以實際三步驗證結果為準，不可直接依原 ticket 描述執行**。
