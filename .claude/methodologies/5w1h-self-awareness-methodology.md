# 5W1H 自覺決策方法論：系統化決策框架

> **判斷範例與 Hook 程式碼（六維度完整正反範例 + Hook 系統整合）**：`.claude/references/5w1h-self-awareness-examples.md`（需對照具體正反範例落實某一維度判斷，或需實作 5W1H 檢查與敏捷重構合規 Hook 邏輯時讀取）

## 方法論目的

本方法論建立基於 5W1H 原則的系統化決策框架，確保每個開發決策都經過完整思考，消除重複實作和逃避行為。

### 核心立場

| 立場 | 內容 |
|------|------|
| 要求 | 每個 todo 建立前必須回答完整 5W1H；每個決策都有明確的責任歸屬和實作策略；零容忍任何逃避性思維 |
| 拒絕 | 未經 5W1H 思考的決策；重複功能的開發；逃避問題的權宜方案 |
| 標準 | 系統化思考取代直覺判斷；防止重複實作優於事後重構；完整解決問題優於症狀緩解 |

## 5W1H 判斷標準與執行規則

### Who（誰）- 責任歸屬判斷

**定義**：確定功能責任歸屬，防止重複實作，並明確區分執行者和分派者。

**格式要求**（敏捷重構合規）：
```markdown
Who: {執行代理人} (執行者) | {分派者} (分派者)
```
適用場景：代理人執行實作標記執行者與分派者；主線程自行執行職責標記「自行執行」；文件代理人執行同代理人格式。

**邊界**：包含 Domain 類別、Service 物件、既有功能模組、執行代理人、分派者；不包含外部依賴、第三方函式庫。

**判斷標準**：

| 情境 | 判定 |
|------|------|
| Domain 中已存在相同功能 | 禁止新建 |
| 責任明確歸屬於特定類別 | 允許執行 |
| 多個類別可能負責 | 必須先釐清歸屬 |
| 責任完全不明 | 禁止執行 |
| 執行者是主線程但任務是 Implementation | 違反敏捷重構原則 |

**執行規則**：遇到新功能需求 -> 搜尋現有 Domain 是否有相同功能 -> 檢查既有 Service 是否已實作 -> 確認測試覆蓋是否存在相關功能 -> 若存在則重用，若不存在則明確責任歸屬。

**驗證機制**：使用 Grep 搜尋相關功能實作；檢查測試檔案確認覆蓋範圍；查看 Domain 設計文件確認職責劃分。

> 正反判斷範例（代理人實作 / 主線程分派 / 文件代理人 / 未檢查既有功能等）見衛星檔「Who 判斷範例」節。

### What（什麼）- 功能定義判斷

**定義**：明確定義功能行為和邊界。

**邊界**：包含具體輸入輸出、業務行為、異常處理；不包含技術實作細節、效能最佳化。

**判斷標準**：

| 情境 | 判定 |
|------|------|
| 單一職責且明確定義 | 正確 |
| 包含多個不相關職責 | 必須拆分 |
| 職責模糊不清 | 禁止執行 |
| 與既有功能重疊 | 必須整合 |

**執行規則**：遇到功能定義 -> 確認輸入輸出明確定義 -> 檢查是否符合單一職責原則 -> 驗證與既有功能的邊界清晰 -> 若職責不明則重新定義。

**驗證機制**：能寫出明確的測試案例 Given-When-Then；功能描述不超過一個句子；無法拆分為更小的獨立功能。

> 正反判斷範例（ISBN 格式驗證 / 書籍處理職責過多）見衛星檔「What 判斷範例」節。

### When（何時）- 觸發時機判斷

**定義**：明確功能觸發的事件和時機。

**邊界**：包含使用者動作、系統事件、定時任務；不包含內部實作的方法呼叫。

**判斷標準**：

| 情境 | 判定 |
|------|------|
| 觸發事件明確且唯一 | 正確 |
| 多個觸發點但邏輯相同 | 正確 |
| 觸發時機不明確 | 必須釐清 |
| 副作用未識別 | 禁止執行 |

**執行規則**：遇到功能觸發設計 -> 識別所有可能的觸發事件 -> 確認事件處理的副作用 -> 驗證與 Event-Driven 架構的整合 -> 若時機不明則重新設計事件流。

**驗證機制**：能繪製完整的事件流程圖；所有副作用都有明確處理；與既有事件系統整合無衝突。

> 正反判斷範例（ISBN 掃描後驗證觸發 / 需要驗證的時候）見衛星檔「When 判斷範例」節。

### Where（何地）- 執行位置判斷

**定義**：確定功能執行的架構位置和 UseCase。

**邊界**：包含 UI 層、Domain 層、Infrastructure 層；不包含第三方服務內部實作。

**判斷標準**：

| 情境 | 判定 |
|------|------|
| UseCase 明確且位置正確 | 正確 |
| 位置錯誤但功能明確 | 重新定位 |
| 執行位置不明 | 必須找出 |
| 跨層級混亂 | 必須重新架構 |

**執行規則**：遇到功能執行位置問題 -> 根據 Clean Architecture 確定正確層級 -> 驗證與 DDD 聚合根的關係 -> 確認 UseCase 的呼叫鏈路 -> 若位置錯誤則重新設計架構。

**驗證機制**：符合 Clean Architecture 分層原則；DDD 聚合根邊界清晰；UseCase 呼叫路徑可追蹤。

> 正反判斷範例（Book Domain 驗證層 / 在需要的地方執行）見衛星檔「Where 判斷範例」節。

### Why（為何）- 動機驗證判斷

**定義**：驗證功能開發的真實需求和動機。

**邊界**：包含使用者需求、業務價值、技術必要性；不包含開發者偏好、技術炫技。

**判斷標準**：

| 情境 | 判定 |
|------|------|
| 有明確需求編號和文件 | 正確 |
| 有業務價值但無正式需求 | 必須補充需求 |
| 純技術優化無業務影響 | 評估必要性 |
| 逃避性動機 | 立即阻止 |

**執行規則**：遇到功能需求 -> 檢查是否有對應的需求文件 -> 確認業務價值和使用者場景 -> 識別是否為逃避問題的替代方案 -> 若動機不純則禁止開發。

**驗證機制**：需求文件中有對應條目；能說明具體的使用者價值；非用於迴避其他問題。

**逃避動機識別清單**：
- 「先做簡單的」-> 逃避複雜問題
- 「這個功能比較好實作」-> 迴避困難任務
- 「順便加個功能」-> 缺乏明確需求
- 「優化一下效能」-> 可能迴避功能問題

> 正反判斷範例（滿足書籍管理需求 / 增加驗證功能）見衛星檔「Why 判斷範例」節。

### How（如何）- 實作策略判斷

**定義**：確定實作方法和技術策略，並明確標記任務類型。

**格式要求**（敏捷重構合規）：
```markdown
How: [Task Type: {任務類型}] {具體實作策略}
```

**任務類型分類**：

| 任務類型 | 執行者 |
|---------|--------|
| Implementation - 程式碼實作 | 必須由執行代理人執行 |
| Dispatch - 任務分派 | 主線程執行 |
| Review - 驗收檢查 | 主線程執行 |
| Documentation - 文件更新 | 文件代理人或主線程執行 |
| Analysis - 問題分析 | 設計代理人或主線程執行 |
| Planning - 策略規劃 | 主線程或設計代理人執行 |

**邊界**：包含技術選擇、實作步驟、測試策略、任務類型標記；不包含具體程式碼細節。

**判斷標準**：

| 情境 | 判定 |
|------|------|
| TDD 測試先行策略 | 正確 |
| 完整的實作計劃 | 正確 |
| 直接寫程式無測試 | 違反流程 |
| Task Type: Implementation 但執行者是主線程 | 違反敏捷重構原則 |
| 包含架構債務 | 立即修正 |
| 臨時解法 | 禁止 |

**執行規則**：遇到實作規劃 -> 確認採用 TDD 測試驅動開發 -> 檢查是否產生架構債務 -> 驗證符合現有程式碼標準 -> 若策略不當則重新規劃。

**驗證機制**：測試案例先於實作程式碼；不產生任何技術債務；符合專案程式碼品質標準。

> 正反判斷範例（程式碼實作 / 任務分派 / 驗收檢查 / 未遵循 TDD / 任務類型不匹配）見衛星檔「How 判斷範例」節。

## 5W1H 強制檢查機制

### 完整性檢查清單

每個 todo 建立前必須回答：

- [ ] **Who**：責任歸屬明確，無重複功能
- [ ] **What**：功能定義清晰，符合單一職責
- [ ] **When**：觸發時機明確，副作用識別完整
- [ ] **Where**：執行位置正確，符合架構原則
- [ ] **Why**：需求依據充分，非逃避性動機
- [ ] **How**：實作策略完整，遵循 TDD 原則

缺失任何項目 -> 禁止建立 todo。

### 品質驗證標準

- 每個 W/H 都有明確答案，無模糊表述
- 所有判斷都是二元的（正確/錯誤）
- 實作策略不包含逃避性語言
- 需求追溯完整可驗證

### 敏捷重構合規性檢查

**目的**：防止主線程違反敏捷重構原則，確保執行者和任務類型正確匹配。

**Who 欄位檢查**：是否明確標記「執行者」和「分派者」；執行者是否符合任務類型要求；主線程是否執行了應由代理人執行的任務。

**How 欄位檢查**：是否包含 `[Task Type: XXX]` 標記；任務類型是否與執行者匹配。

**違反組合**：

| 組合 | 判定 |
|------|------|
| Task Type: Implementation + 執行者是主線程 | 違反 |
| Task Type: Dispatch + 執行者是代理人 | 違反 |
| 缺少執行者/分派者標記 | 格式錯誤 |
| Task Type: Implementation + 執行者是執行代理人 | 正確 |
| Task Type: Dispatch + 執行者是主線程 | 正確 |
| Task Type: Review + 執行者是主線程 | 正確 |
| Task Type: Documentation + 執行者是文件代理人或主線程 | 正確 |

> 敏捷重構合規性自動檢查的 Hook 程式碼（`check_agile_refactor_compliance`）見衛星檔「Hook 系統整合程式碼」節。

### 逃避行為識別

逃避語言檢測基於「[Claude 自檢與逃避預防方法論](./claude-self-check-methodology.md)」的完整違規詞彙表。檢測到任何逃避語言 -> 立即阻止決策並要求修正。

| 類別 | 代表詞彙 |
|------|---------|
| 品質妥協和逃避責任 | 太複雜、先將就、暫時性修正、症狀緩解、workaround、hack、temporary fix、skip for now |
| 簡化妥協 | 更簡單的方法、簡化處理、simpler approach、easier approach、simplify |
| 發現問題但不解決 | 架構問題先不管、只加個 TODO、技術債務之後處理、ignore the issue、code smell ignore |
| 測試品質妥協 | 簡化測試、降低測試標準、放寬測試條件、基本測試即可、simplify test、minimal test |
| 程式碼修改逃避 | 註解掉、停用功能、暫時關閉、comment out、disable、temporarily disable |
| 模糊不精確詞彙 | 智能、自動（無具體描述）、優化（無具體指標）、smart、auto (without details) |

> Hook 觸發時機、5W1H 完整性檢查邏輯（`check_5w1h_compliance`）與修復機制程式碼見衛星檔「Hook 系統整合程式碼」節。

## 執行驗證機制

**成功標準**：每個決策都有完整的 5W1H 記錄；無重複功能開發；零逃避行為；所有實作都基於明確需求。

**失敗處理**：

| 失敗類型 | 處理方式 |
|---------|---------|
| 缺失 5W1H | 補充完整後重新提交 |
| 發現重複功能 | 使用既有實作 |
| 逃避行為 | 按照永不放棄鐵律重新規劃 |
| 需求不明 | 先澄清需求再開發 |

**持續改進**：定期檢視 5W1H 執行品質；更新逃避語言識別清單；最佳化 Hook 檢查邏輯；補充判斷標準邊界案例。

## 方法論執行檢查清單

| 階段 | 確認項目 |
|------|---------|
| 執行前 | Hook 系統已正確配置；5W1H 檢查腳本運作正常；逃避語言清單完整更新 |
| 執行中 | 每個 todo 都經過 5W1H 檢查；無法回答的問題立即處理；重複功能被及時發現 |
| 執行後 | 開發決策品質提升；重複實作情況減少；逃避行為有效控制 |
