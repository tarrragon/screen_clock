# 驗收條件方法論

本文件定義驗收條件（Acceptance Criteria）的標準化格式規範，確保驗收條件可量化、可追溯、可驗證。

---

## 核心原則

> **驗收條件是契約**：用戶和執行者之間的明確約定，雙方都能清楚判斷任務是否完成。

### 4V 原則

| 原則 | 說明 | 判斷標準 |
|------|------|---------|
| **可驗證 (Verifiable)** | 能明確判斷完成/未完成 | 有具體確認方法 |
| **可量化 (Quantifiable)** | 明確列出項目數量 | 有編號和總數 |
| **可追溯 (Traceable)** | 引用設計文件來源 | 有來源連結 |
| **可記錄 (Documented)** | 要求完成證據 | 有狀態標記 |

---

## 驗收情境分類

不同的驗收情境需要不同的確認方法和證據類型。

| 情境 | 說明 | 典型項目 | 主要確認方法 |
|------|------|---------|-------------|
| **功能驗收** | 程式功能是否正確實作 | 子命令、API、Widget | 執行命令、執行測試 |
| **流程驗收** | 開發流程是否正確執行 | 測試通過、品質檢查 | 執行測試、狀態檢查 |
| **文件驗收** | 文件是否正確更新 | SKILL.md、worklog | 檔案檢查 |
| **建議驗收** | 建議是否被正確處理 | 採納/拒絕決定 | 清單核對 |

### 情境判定規則

| 驗收項目關鍵字 | 對應情境 |
|---------------|---------|
| 「實作」「建立」「功能」「子命令」 | 功能驗收 |
| 「測試通過」「lint」「analyze」「品質」 | 流程驗收 |
| 「更新」「撰寫」「文件」「README」 | 文件驗收 |
| 「建議」「採納」「拒絕」「處理」 | 建議驗收 |

## 確認方法矩陣

### 情境 x 確認方法對應表

| 情境 | 執行命令 | 執行測試 | 檔案檢查 | 狀態檢查 | 清單核對 | 程式碼審查 |
|------|---------|---------|---------|---------|---------|-----------|
| **功能驗收** | **主要** | **主要** | 輔助 | - | - | 輔助 |
| **流程驗收** | 輔助 | **主要** | - | **主要** | - | - |
| **文件驗收** | - | - | **主要** | - | 輔助 | - |
| **建議驗收** | - | - | 輔助 | - | **主要** | - |

**說明**：
- **主要**：該情境的首選確認方法
- **輔助**：可選的補充確認方法
- `-`：通常不適用

### 確認方法詳細說明

| 確認方法 | 執行方式 | 適用工具 | 輸出證據 |
|---------|---------|---------|---------|
| 執行命令 | 執行 CLI 命令驗證輸出 | `bash` | 命令輸出截圖/文字 |
| 執行測試 | 執行測試套件 | `flutter test`、`mcp__dart__run_tests` | 測試結果摘要 |
| 檔案檢查 | 確認檔案存在或內容 | `read`、`grep` | 檔案路徑、內容片段 |
| 狀態檢查 | 檢查系統狀態 | `dart analyze`、`git status` | 狀態輸出 |
| 清單核對 | 逐項核對清單 | 手動檢查 | 核對結果表格 |
| 程式碼審查 | 審查程式碼品質 | `read`、`grep` | 審查報告 |

## 驗收證據要求

### 證據類型定義

| 證據類型 | 說明 | 格式範例 |
|---------|------|---------|
| **命令輸出** | CLI 命令的執行結果 | `$ ticket track tree {ticket-id}` + 輸出內容 |
| **測試結果** | 測試執行的通過狀態 | `77/77 tests passed` |
| **檔案存在** | 確認檔案已建立 | `{path} exists` |
| **檔案內容** | 特定檔案的內容片段 | `{path} L{line_start}-{line_end}` |
| **區段存在** | 確認文件區段存在 | `{file} 包含「{section}」區段` |
| **狀態輸出** | 工具狀態檢查結果 | `dart analyze: 0 issues` |

### 各情境證據要求

| 情境 | 必要證據 | 可選證據 |
|------|---------|---------|
| 功能驗收 | 命令輸出 或 測試結果 | 程式碼片段 |
| 流程驗收 | 測試結果 或 狀態輸出 | 檢查日誌 |
| 文件驗收 | 檔案存在 或 檔案內容 | 區段存在 |
| 建議驗收 | 清單核對結果 | 對應 AC 連結 |

### 證據記錄格式

```markdown
| # | 項目 | 確認方法 | 證據 | 狀態 |
|---|------|---------|------|------|
| 1 | 實作 tree 子命令 | 執行命令 | `$ ticket track tree {ticket-id}` 顯示樹狀結構 | [x] |
| 2 | 單元測試通過 | 執行測試 | `77/77 tests passed` | [x] |
| 3 | 更新 SKILL.md | 檔案檢查 | `SKILL.md L97 狀態為 [x]` | [x] |
```

## 驗收條件格式規範

### 必要欄位定義

| 欄位 | 說明 | 必填 | 範例 |
|------|------|------|------|
| # | 項目編號 | 是 | 1, 2, 3... |
| 項目 | 驗收項目描述 | 是 | 「實作 tree 子命令」 |
| 來源 | 設計文件引用 | 是 | 「SKILL.md L97」 |
| 確認方法 | 如何驗證完成 | 是 | 「執行命令、檢查輸出」 |
| 狀態 | 完成狀態 | 是 | `[ ]` 或 `[x]` |

### 來源引用格式

| 引用類型 | 格式 | 範例 |
|---------|------|------|
| 檔案行號 | `{檔案名} L{行號}` | `SKILL.md L97` |
| 檔案區段 | `{檔案名} #{區段}` | `README.md #Usage` |
| Ticket | `{Ticket-ID}` | `W{n}-{seq}` |
| 方法論 | `{方法論名}` | `atomic-ticket-methodology` |
| 用戶需求 | `用戶需求: {描述}` | `用戶需求: 支援樹狀顯示` |

### 確認方法類型

| 類型 | 說明 | 範例 |
|------|------|------|
| 執行命令 | 執行特定命令並驗證輸出 | 「執行 `ticket track tree <id>`，顯示樹狀結構」 |
| 檢查檔案 | 檢查特定檔案內容 | 「檢查 SKILL.md 已更新實作狀態」 |
| 執行測試 | 執行測試套件 | 「執行 `flutter test`，100% 通過」 |
| 程式碼審查 | 審查程式碼品質 | 「程式碼符合命名規範」 |
| 功能驗證 | 手動操作驗證功能 | 「手動建立 Ticket 並驗證格式正確」 |

## 標準格式模板

### 基本格式（表格式）

```markdown
## Acceptance Criteria

### 功能驗收
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 1 | {項目描述} | {來源引用} | {確認方法} | [ ] |
| 2 | {項目描述} | {來源引用} | {確認方法} | [ ] |

### 流程驗收
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 1 | {項目描述} | {來源引用} | {確認方法} | [ ] |
```

### 簡化格式（清單式）

適用於簡單任務（< 5 個驗收項目）：

```markdown
## Acceptance Criteria

- [ ] **{項目1}** (來源: {引用}) - {確認方法}
- [ ] **{項目2}** (來源: {引用}) - {確認方法}
```

### 複雜格式（分類式）

適用於複雜任務（跨多個類別）：

```markdown
## Acceptance Criteria

### 1. 功能驗收 (N 項)
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 1.1 | {項目描述} | {來源引用} | {確認方法} | [ ] |
| 1.2 | {項目描述} | {來源引用} | {確認方法} | [ ] |

### 2. 品質驗收 (N 項)
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 2.1 | {項目描述} | {來源引用} | {確認方法} | [ ] |

### 3. 文件驗收 (N 項)
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 3.1 | {項目描述} | {來源引用} | {確認方法} | [ ] |

### 驗收摘要
- 功能驗收: 0/N 完成
- 品質驗收: 0/N 完成
- 文件驗收: 0/N 完成
- **總計: 0/M 完成**
```

## 好/壞範例對比

### 壞範例 1：模糊詞彙

```markdown
## Acceptance Criteria

- [ ] 功能實作完成
- [ ] 測試通過
- [ ] 文件更新
```

**問題**：
- 「完成」無法判斷標準
- 「通過」沒有說明哪些測試
- 「更新」沒有說明更新什麼

### 好範例 1：明確具體

```markdown
## Acceptance Criteria

### 功能驗收
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 1 | 實作 tree 子命令 | SKILL.md L97 | 執行 `ticket track tree <id>` 顯示樹狀結構 | [ ] |
| 2 | 實作 list 子命令 | SKILL.md L112 | 執行 `ticket track list` 顯示 Ticket 清單 | [ ] |

### 品質驗收
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 1 | 單元測試通過 | quality-baseline | 執行 `flutter test`，0 failures | [ ] |
| 2 | 無 lint 錯誤 | quality-baseline | 執行 `dart analyze`，0 issues | [ ] |

### 文件驗收
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 1 | 更新 SKILL.md 實作狀態 | W{n}-{seq} | 檢查 L97, L112 狀態為 `[x]` | [ ] |
```

### 壞範例 2：無量化

```markdown
## Acceptance Criteria

- [ ] 實作子命令

**問題**：

| 3 | 實作 status 子命令 | SKILL.md L125 | 執行命令驗證 | [ ] |

### 測試驗收 (2 項)
| # | 項目 | 來源 | 確認方法 | 狀態 |

| 2 | list 子命令單元測試 | test-pyramid | 測試檔案存在且通過 | [ ] |
```

## 驗收條件禁用規則

### 禁止規則

| 規則 | 說明 | 範例 |
|------|------|------|
| 禁止「或」連接 | 不可用「或」連接不同方案 | ❌「移除或實作」 |
| 禁止模糊目標 | 不可有相反的選項 | ❌「達成 A 或 B（A、B 相反）」 |
| 禁止缺乏業務意圖 | 必須說明為什麼要這樣做 | ❌ 只說「一致性」不說明方向 |
| 禁止最低成本陷阱 | 不可設計讓執行者選擇最省力方案 | ❌「更新文件（可移除或修正）」 |

### 違規範例

**❌ 錯誤：** 「SKILL.md 只列已實作指令，或實作缺失指令」

**問題分析：**
- 兩個相反的方案（移除 vs 實作）用「或」連接
- 執行者會選擇最低成本方案（移除）
- 業務意圖不明確（這些是規劃功能還是錯誤記錄？）

**✓ 正確做法：**
1. 先確認業務意圖（這些是規劃功能還是錯誤記錄？）
2. 選擇一個方案
3. 設計對應的驗收條件

**正確範例：**
- 若為規劃功能：「保留未實作指令描述，新增『規劃中』標記，建立實作 Ticket」
- 若為錯誤記錄：「移除未實作指令描述，記錄移除原因」

## 業務意圖要求

### 必填欄位

每個 Ticket 的驗收條件必須包含業務意圖說明：

| 欄位 | 說明 | 範例 |
|------|------|------|
| 業務目標 | 這個任務要達成什麼業務價值 | 「保留規劃功能，確保後續可實作」 |
| 期望結果 | 完成後應該是什麼狀態 | 「所有規劃功能都有實作 Ticket」 |
| 禁止結果 | 完成後不應該是什麼狀態 | 「不可損失規劃中的功能」 |

### 業務意圖確認問題

撰寫驗收條件前，必須回答以下問題：

| 問題 | 目的 | 範例回答 |
|------|------|---------|
| 這個任務要解決什麼問題？ | 確認問題定義 | 「SKILL.md 與實作不一致」 |
| 問題的根本原因是什麼？ | 防止只處理表面症狀 | 「文件記錄了規劃中的功能」 |
| 理想的解決結果是什麼？ | 確認期望狀態 | 「文件與實作一致，且保留規劃功能」 |
| 什麼結果是不可接受的？ | 設定禁止邊界 | 「損失規劃中的功能」 |

### 不一致問題的處理規則

當發現「文件 vs 實作」不一致時，必須先判斷不一致的性質：

| 情況 | 判斷依據 | 正確處理 | 錯誤處理 |
|------|---------|---------|---------|
| 規劃中的功能 | 文件有設計、有預期用途 | 保留標記 + 建立實作 Ticket | 移除描述 |
| 錯誤記錄 | 無設計依據、無預期用途 | 移除描述 | 建立實作 Ticket |
| 過時功能 | 曾經實作但已移除 | 標記為 deprecated | 直接移除 |
| 重命名/移動 | 功能存在但位置不同 | 更新文件引用 | 重複實作 |

## 方案評估矩陣

### 評估維度

選擇方案前，必須評估以下維度：

| 評估維度 | 問題 | 必填 |
|---------|------|------|
| 技術可行性 | 這個方案技術上可行嗎？ | 是 |
| **業務價值** | 這個方案符合業務意圖嗎？ | **是** |
| **副作用評估** | 這個方案會損失什麼？ | **是** |
| **替代方案** | 有沒有更好的方案？ | **是** |
| 長期影響 | 這個方案對未來有什麼影響？ | 建議 |

### 評估範例

**歷史事件回顧：**

| 方案 | 技術可行 | 業務價值 | 副作用 | 結論 |
|------|---------|---------|--------|------|
| 移除未實作指令 | ✓ 簡單 | ✗ 損失規劃功能 | **高** | **拒絕** |
| 實作缺失指令 | ✓ 可行 | ✓ 保留功能 | 無 | 考慮 |
| 保留標記 + 建立 Ticket | ✓ 可行 | ✓ 規劃清晰 | 無 | **採用** |

**關鍵洞見：** 「移除」方案技術上最簡單，但業務價值最低，副作用最高。

### 方案選擇規則

| 規則 | 說明 |
|------|------|
| 禁止只考慮技術可行性 | 必須同時考慮業務價值和副作用 |
| 最低成本方案需警惕 | 最簡單的方案往往有隱藏的業務成本 |
| 不可逆操作需謹慎 | 移除、刪除等操作需要確認是否正確 |
| 替代方案必須考慮 | 不可只評估單一方案 |

## 驗收條件設計檢查清單

> **重要**：在撰寫驗收條件前完成此檢查清單，避免設計缺陷導致驗收問題。

### 4V 原則檢查

- [ ] 每個項目都**可驗證**（有明確的確認方法，非「完成」「正常」）
- [ ] 每個項目都**可量化**（有具體數字或逐項清單，非「所有」「相關」）
- [ ] 每個項目都**可追溯**（有來源引用，如 SKILL.md L97）
- [ ] 每個項目都**可記錄**（結果可以寫入文件，有證據類型）

### 禁止規則檢查

- [ ] 沒有使用「或」連接多個選項（如「移除或實作」）
- [ ] 沒有使用模糊詞彙（「適當」「正常」「完成」「相關」）
- [ ] 沒有合併不同層級的需求在同一項目
- [ ] 沒有讓執行者可選擇最低成本方案的設計

### 業務意圖檢查

- [ ] 驗收條件與業務目標一致（解決正確的問題）
- [ ] 沒有「技術正確但業務錯誤」的風險
- [ ] 考慮了方案的負面副作用
- [ ] 不可逆操作（刪除、移除）已確認業務意圖

### 完整性檢查

- [ ] 每個需求都有對應的驗收條件
- [ ] 需要逐項列出的項目已**逐項列出**（非籠統描述如「所有標記移除」）
- [ ] 複合需求已拆分為獨立的驗收項目
- [ ] 涉及多個檔案/功能的任務，每個都有獨立驗收項目

## 驗收條件撰寫檢查清單

撰寫驗收條件時，檢查以下項目：

### 完整性檢查
- [ ] 每個項目都有編號
- [ ] 每個項目都有來源引用
- [ ] 每個項目都有確認方法
- [ ] 每個項目都有狀態標記

### 品質檢查
- [ ] 無模糊詞彙（「完成」「正常」「適當」）
- [ ] 有明確數量（N 個項目）
- [ ] 確認方法可執行
- [ ] 來源可追溯

### 格式檢查
- [ ] 使用標準表格格式
- [ ] 分類清晰（功能/品質/文件）
- [ ] 有驗收摘要（複雜任務）

---

## 常見模糊詞彙替換表

| 模糊詞彙 | 替換方式 | 範例 |
|---------|---------|------|
| 完成 | 具體描述完成標準 | 「tree 子命令可執行並顯示樹狀結構」 |
| 正常 | 定義預期行為 | 「輸入無效 ID 時顯示錯誤訊息」 |
| 適當 | 引用具體標準 | 「符合 quality-baseline.md 的測試覆蓋率要求」 |
| 足夠 | 給出具體數字 | 「至少 3 個測試案例覆蓋邊界情況」 |
| 良好 | 定義量化指標 | 「認知負擔指數 < 10」 |
| 符合規範 | 引用具體規範 | 「符合 language-constraints.md 的命名規則」 |

## 與 Ticket 系統整合

### Ticket 驗收條件位置

驗收條件應放在 Ticket 的 `acceptance` 欄位或 `## Acceptance Criteria` 區段：

```yaml
# YAML frontmatter 格式
acceptance:
  - 項目: 實作 tree 子命令
    來源: SKILL.md L97
    確認方法: 執行命令驗證
```

```markdown
# Markdown 區段格式
## Acceptance Criteria

| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 1 | 實作 tree 子命令 | SKILL.md L97 | 執行命令驗證 | [ ] |
```

### 驗收流程

```
執行任務
    |
    v
逐項完成驗收條件
    |
    v
標記狀態 [x]
    |
    v
全部完成? ─否→ 繼續執行
         |
         是
         |
         v
執行 /ticket track complete
```

## 與驗收代理人整合

### acceptance-auditor 驗收流程

驗收代理人（acceptance-auditor）在執行驗收時：

1. **格式檢查**：確認驗收條件符合 4V 原則
2. **情境分類**：識別每個驗收條件的情境類型
3. **方法執行**：根據確認方法矩陣選擇適當方法
4. **證據收集**：記錄每個驗收條件的通過證據
5. **報告產出**：產出標準格式的驗收報告

### 驗收報告與驗收條件的對應

```markdown
# 驗收報告

## 詳細驗收結果

### 驗收條件

| # | 項目 | 來源 | 情境 | 確認方法 | 證據 | 結果 |
|---|------|------|------|---------|------|------|
| 1 | {項目} | {來源} | 功能 | 執行命令 | {輸出} | 通過 |
| 2 | {項目} | {來源} | 流程 | 執行測試 | {結果} | 通過 |
```

> 詳見：@.claude/agents/acceptance-auditor.md

## 與建議追蹤整合

### 建議 → 驗收條件轉換

當建議被採納時，必須轉換為驗收條件：

| 建議追蹤欄位 | 對應驗收條件欄位 |
|------------|-----------------|
| 建議內容 | 項目 |
| 建議來源 | 來源（格式：「建議 #N ({來源})」） |
| - | 確認方法（根據情境選擇） |
| - | 狀態 |

### 轉換範例

**原始建議**：
```markdown
| # | 建議內容 | 狀態 | 對應 AC |
|---|---------|------|--------|
| 1 | 建立驗收代理人 | adopted | AC-001 |
```

**轉換後的驗收條件**：
```markdown
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| AC-001 | 建立 acceptance-auditor.md | 建議 #1 (報告來源) | 檔案檢查 | [ ] |
```

### 建議驗收的驗收條件

當 Ticket 包含建議追蹤時，驗收條件應包含：

```markdown
### 建議驗收
| # | 項目 | 來源 | 確認方法 | 狀態 |
|---|------|------|---------|------|
| 1 | 所有建議已處理（無 pending） | suggestion-tracking | 清單核對 | [ ] |
| 2 | 採納的建議已轉為 AC | suggestion-tracking | 清單核對 | [ ] |
| 3 | 拒絕的建議有記錄理由 | suggestion-tracking | 清單核對 | [ ] |
```

> 詳見：[suggestion-tracking-methodology]($CLAUDE_PROJECT_DIR/.claude/methodologies/suggestion-tracking-methodology.md)
