# 寫作方法論

## 一、方法論撰寫

### 讀者定位

讀者是專家，已經知道內容，只是忘記細節。

**你的任務**：幫讀者找回記憶。

**不是**：教導、說服、糾正觀念。

### 30 秒電梯理論

方法論必須 30 秒內讀完。

超出部分 → reference 到其他方法論。

讀者閱讀方法論是為了檢查和複習，不是學習完整理論。

### SKILL vs 方法論

| 類型 | 內容 |
|-----|------|
| **方法論** | 30 秒複習清單 |
| **SKILL** | 完整實作指南（範例、錯誤處理、決策依據） |

### 撰寫/改寫檢查

撰寫新方法論或改寫舊方法論時，評估：

- 是否有完整操作流程需要保存？ → 建立對應 SKILL
- 是否有程式碼範例、錯誤處理細節？ → 移到 SKILL
- 精簡後是否會流失關鍵資訊？ → 建立 SKILL 保存

## 二、經驗分享文章撰寫

> 來源：2026-03 測試經驗記錄的多次修改過程。每一條都是實際犯過的寫作錯誤。

### 核心定位

經驗分享是**記錄自己的經歷**，不是寫教材。讀者想看「發生了什麼、怎麼解決的」，不是聽人說教。

### 六項準則

#### 1. 語氣：經驗分享，不是教人

不要用「你應該」「要記得」這種由上往下的口氣。

| 修改前 | 修改後 |
|--------|--------|
| 教訓：測試應該驅動被測程式碼的真實路徑 | 後來改成從入口方法開始呼叫，Bug 隨即被發現 |
| 關鍵原則：如果你的功能涉及多個元件協作，只寫單元測試是不夠的 | （直接移除，用案例本身說明） |

#### 2. 語氣：陳述事實，不帶情緒

避免強調語氣、反問、自責/誇張的措辭。平鋪直敘地描述發生了什麼。

| 修改前 | 修改後 |
|--------|--------|
| 我到底哪裡搞砸了 | 經驗記錄 |
| 炸了四次 | 陸續發現四個 Bug |
| 但我偷懶了 | 但模擬的回傳資料只有一行純文字 |
| 測試全過是假象 | 測試通過不代表修改生效 |

#### 3. 用功能描述，不要用函式名稱

文章的讀者不一定看過這份程式碼。函式名稱可以出現在程式碼範例裡，但前後的解說要用功能語言。

| 修改前 | 修改後 |
|--------|--------|
| `_buildItemPrinterMapping` 沒有被執行 | 品項分派的程式碼沒有被執行 |
| `generator` 未初始化 | 印表機內部元件未初始化 |
| `_getLexemes('')` 對 `text[0]` 越界 | 解析文字時沒有處理空字串，直接取第一個字元導致越界 |

#### 4. 敘事角度：不是每件事都是「犯錯」

有些問題不是自己造成的，不需要用自責的角度來寫。

| 修改前 | 修改後 |
|--------|--------|
| 修完一個 Bug 我就以為沒事了 | 第三方 library 的地雷——前面都做對了才踩到 |
| 我沒想到修完之後下游還有問題 | 前三個 Bug 都修好之後，執行路徑才真正打通，觸發了這個潛在問題 |

#### 5. 每個案例要有脈絡：發現 → 找因 → 修復

只說「問題是 X，改成 Y」不夠。讀者需要完整脈絡：

- **怎麼發現的**：什麼現象讓你意識到有問題
- **怎麼找到原因的**：一步步定位的過程
- **怎麼修的**：最終的解決方式

#### 6. 不要用特定產品/SDK 名稱

具體的產品名稱讓文章變成只適用於特定情境。著重描述問題的結構，而非特定技術。

| 修改前 | 修改後 |
|--------|--------|
| imin SDK 不驗證 width 總和 | POS 主機不驗證欄位寬度總和 |

例外：第三方 library 的名稱可以出現在技術細節區段（給自己未來查閱用）。

### 經驗分享文章檢查清單

1. 有沒有「你應該」「要記得」「教訓是」這類由上往下的用語？→ 改成描述自己做了什麼、發現了什麼
2. 有沒有情緒化的措辭（搞砸、偷懶、炸了、根本沒）？→ 改成陳述事實
3. 程式碼區塊外的函式名稱，不看程式碼的人能理解嗎？→ 用功能描述替代
4. 每個案例有完整的脈絡嗎？（怎麼發現 → 怎麼找因 → 怎麼修）
5. 有沒有把不是自己造成的問題寫成自己的錯？→ 用適當的角度描述
6. 有沒有出現特定產品/SDK 名稱？→ 除了技術細節區段外，用功能描述替代
