# Hook 階段平衡設計方法論

本方法論定義 Hook 系統在開發流程各階段的設計原則，防止 Hook 被動累積導致的摩擦力倒置。

> **核心主張**：從「錯誤發生才補 Hook」轉為「依階段特性主動設計 Hook」。前期階段主動設計、後期階段設定降級觀察期。

---

## 問題背景

### 被動防禦反射弧的結構性問題

Hook 系統常呈現以下反射弧：

```
錯誤發生 → 補 Hook 防護 → 觸發頻率上升 → 誤報累積 → 降級延後
```

此反射弧只對「錯誤立即可見」的階段有效，對「錯誤延遲顯現」的階段無效。結果：

- 後期執行階段（Phase 3b 實作）：Hook 高密度累積（錯誤立即可見）
- 前期決策階段（Proposal、Phase 1 規格）：Hook 長期缺席（錯誤延遲顯現）

### Hook 分佈倒置

依摩擦力管理方法論「開發流程階段的摩擦力曲線」，理想摩擦力應前期高、後期低。但 Hook 分佈卻前期低、後期高——這是摩擦力倒置的 Hook 層落地。

---

## 核心原則

### 原則 1：Hook 設計依階段特性主動規劃

每個新功能或新規則納入時，必須主動問以下三題：

| 問題 | 若答「是」 | 行動 |
|------|-----------|------|
| 這個錯誤會立即可見嗎？ | 是 → 錯誤反饋自然存在 | Hook 可後補（依測試結果設計） |
| 這個錯誤會延遲到後續 Phase 才顯現嗎？ | 是 → 錯誤反饋缺席 | **Hook 必須前置設計** |
| 這是決策點還是執行點？ | 決策點 → 錯誤放大成本高 | Hook 優先度提升 |

**核心**：錯誤延遲顯現的情境，必須在設計時就規劃 Hook，不能等錯誤發生才補。

### 原則 2：前期階段 Hook 先行

前期階段（Proposal、Phase 0、Phase 1）的新增工作，必須預設附帶 Hook 設計：

| 前期階段 | Hook 設計預設項 |
|---------|---------------|
| Proposal 建立 | 新增 PROP 時檢查分級、章節完備、ticket_refs 綁定 |
| Phase 0 SA | 前置審查衝突檢查報告存在驗證 |
| Phase 1 規格 | 規格文件存在性、多視角審查記錄驗證 |

不做為反模式：「先上線，錯誤出現再補 Hook」。

### 原則 3：後期階段 Hook 設降級觀察期

後期階段（Phase 3b 實作）的 Hook 從一開始就設定降級條件：

| 降級條件 | 觸發動作 |
|---------|---------|
| 連續 N 次（方法論 v2.0 建議 5 次）無錯 | 自動降級為提醒 |
| 連續 2 個 Wave 無錯 | 降級為非同步（PostToolUse 改 Stop） |
| 連續 3 個 Wave 無錯 | 進入廢除評估期 |

### 原則 4：Hook 本身也受摩擦力方法論約束

Hook 加入或降級時，必須依摩擦力管理方法論判斷：

- Hook 攔截的動作是哪個象限？
- Hook 本身的觸發頻率是否造成摩擦倒置？
- Hook 升級 / 降級是否觸發摩擦力重分配？

**反模式**：Hook 疊加 Hook（偵測誤報再加 Hook 的 Hook）。此為無限遞迴。

## Hook 階段分佈理想配置

### 階段 Hook 密度表

| 階段 | 現況密度 | 理想密度 | 設計原則 |
|------|---------|---------|---------|
| Proposal 評估 | 低（近零） | 中高 | 分級檢查、章節完備、狀態綁 ticket |
| Phase 0 SA | 低 | 中 | 衝突檢查報告驗證 |
| Phase 1 規格 | 低 | 中高 | 規格存在、多視角審查、AC 完備 |
| Phase 1.5 多視角 | 低 | 中 | 審查記錄驗證 |
| Phase 2 測試 | 中 | 中 | 測試檔案存在、命名規範 |
| Phase 3a 策略 | 中 | 中 | 虛擬碼/流程圖驗證 |
| Phase 3b 實作 | 高（75+ Hook） | 中低 | 核心防護保留 + 連續無錯降級 |
| Phase 4 重構 | 中高 | 中 | 多視角審查驗證 |

### 密度判定維度

| 維度 | 高密度條件 | 低密度條件 |
|------|-----------|-----------|
| 錯誤可逆性 | 不可逆 / 延遲顯現 | 可逆 / 即時反饋 |
| 錯誤放大範圍 | 跨 Phase / 跨版本 | 局部檔案 / 單一測試 |
| 決策 vs 執行 | 決策點 | 執行點 |
| 頻率 vs 嚴重 | 低頻高嚴重 | 高頻低嚴重 |

## Hook 設計決策樹

新增 Hook 或修改既有 Hook 前，依下列決策樹判斷：

```
新 Hook 設計需求
    |
    v
[階段判斷] 攔截的動作屬哪個階段？
    |
    +-- 前期（Proposal/Phase 0/Phase 1） → 進主動設計分支
    +-- 後期（Phase 3b/Phase 4） → 進被動防禦分支
    |
    v
[主動設計分支]
    |
    v
問 1：錯誤延遲顯現嗎？
    +-- 是 → Hook 必須前置，不可等錯誤發生
    +-- 否 → 依原則 1 判斷
    |
    v
問 2：決策點還是執行點？
    +-- 決策點 → 阻擋力建議 block
    +-- 執行點 → 阻擋力建議 warn
    |
    v
問 3：觸發頻率預估？
    +-- 高頻 → 考慮抽樣或條件化 matcher
    +-- 低頻 → 全量觸發
    |
    v
[輸出] 前期 Hook 設計規格（matcher + 阻擋力 + 頻率）

[被動防禦分支]
    |
    v
問 1：既有 Hook 能覆蓋嗎？
    +-- 是 → 不加新 Hook，調整既有
    +-- 否 → 進入設計
    |
    v
問 2：預設降級條件？
    +-- 連續 N 次無錯自動降級
    +-- 2 Wave 觀察期後評估移除
    |
    v
[輸出] 後期 Hook 設計規格（含降級條件）
```

## Hook 生命週期管理

### 生命週期階段

每個 Hook 必須明示所處階段：

| 階段 | 觸發行為 | 降級條件 |
|------|---------|---------|
| Active（活躍） | 正常攔截 | 連續 5 次無錯 → Observing |
| Observing（觀察中） | 正常攔截但統計觸發 | 2 Wave 無錯 → Deprecating |
| Deprecating（退役準備） | 降為提醒（warn → info） | 3 Wave 無錯 → Archived |
| Archived（封存） | 移除但保留歷史 | 回歸事件 → 恢復 Active |

### 降級追蹤機制

Hook 目錄下建議維護 `hook-lifecycle.yaml`：

```yaml
hooks:
  example-hook:
    stage: observing
    entered_stage_at: 2026-04-14
    consecutive_no_error: 3
    last_real_action: 2026-04-10
    downgrade_criteria:
      to_deprecating: "2 Wave 無錯"
      to_archived: "3 Wave 無錯"
```

### 降級執行流程

1. 自動統計：每次 Hook 觸發記錄 action / no-action
2. 週期檢查：每 Wave 結束盤點 Hook 統計
3. 降級決策：符合條件的 Hook 進入下一階段
4. 快速恢復：若觀察期間發生回歸，立即恢復 Active

## 反模式

### 反模式 1：Hook 疊加 Hook

偵測到 Hook 誤報，用另一個 Hook 去攔截誤報。這是無限遞迴。

**替代**：修正原 Hook 的 matcher 條件或降級。

### 反模式 2：前期 Hook 不設計，等錯誤發生再補

「先上線，錯誤出現再補 Hook」只對「錯誤立即可見」有效。前期階段錯誤延遲顯現，補 Hook 時 sunk cost 已支付。

**替代**：前期階段設計時同步規劃 Hook。

### 反模式 3：後期 Hook 無降級條件

Hook 設計時沒設降級條件，導致永久高頻觸發。

**替代**：所有 Hook 設計時明示降級條件 + 生命週期階段。

### 反模式 4：Hook 密度與摩擦力曲線相反

後期執行階段 Hook 密度高過前期決策階段，造成摩擦力倒置。

**替代**：Hook 密度設計參照摩擦力曲線（前期高、後期低）。

### 反模式 5：Hook 存在但無統計

Hook 長期運行但無觸發統計，無法評估 ROI。

**替代**：所有 Hook 必須有觸發日誌，定期統計 action 比例。

## 落地檢查清單

### 新增 Hook 時

- [ ] 明示 Hook 屬哪個階段（前期 / 後期）
- [ ] 若前期：已依階段特性主動設計
- [ ] 若後期：已設降級條件與觀察期
- [ ] 已規劃觸發日誌與統計機制
- [ ] Hook 本身符合摩擦力原則（不造成新倒置）

### 既有 Hook 盤點時

- [ ] 近 N 個 Wave 觸發次數與 action 比例
- [ ] 是否符合降級條件
- [ ] 是否與既有 Hook 功能重疊
- [ ] 是否有疊加 Hook 的反模式

### Hook 降級執行時

- [ ] 降級前記錄 action 歷史供回滾
- [ ] 設定觀察期（至少 2 Wave）
- [ ] 設定快速恢復機制
- [ ] 通知 basil-hook-architect 彙整統計
