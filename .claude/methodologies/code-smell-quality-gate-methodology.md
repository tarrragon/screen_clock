# Code Smell 品質閘門檢測方法論

## 核心概念

設計階段品質閘門：在 Ticket 進入 Phase 2 前強制執行 C1/C2/C3 檢測，修正成本降低 80%。

**三層檢測標準**：

| 檢測 | 目標 | 判定邏輯 |
|------|------|---------|
| **C1 God Ticket** | 防止範圍過大 | 任一超標 = 阻止 |
| **C2 Incomplete** | 確保元素齊全 | 任一缺失 = 阻止 |
| **C3 Ambiguous** | 確保職責明確 | 任一模糊 = 阻止 |

**檢測順序**：C1 → C3 → C2（複雜度 → 職責 → 完整性）

---

## 執行步驟

### Step 1: C1 God Ticket 檢測

| 指標 | 通過 | 警告 | 阻止 |
|------|------|------|------|
| 檔案數量 | 1-3 | 4-6 | > 10 |
| 層級跨度 | 1 | 2 | > 2 |
| 預估工時 | 2-4h | 4-8h | > 16h |

**失敗處理**：按層級拆分 → 按職責拆分 → 按功能模組拆分

### Step 2: C3 Ambiguous Responsibility 檢測

| 要素 | 通過 | 阻止 |
|------|------|------|
| 層級標示 | 標題含 `[Layer X]` | 缺少 |
| 職責描述 | 明確定義單一職責 | 模糊或跨層 |
| 檔案範圍 | 列出具體路徑 | 「相關檔案」 |
| 驗收限定 | 限定該層級 | 跨層驗收 |

**失敗處理**：明確層級 → 重寫職責 → 列出檔案 → 限定驗收

### Step 3: C2 Incomplete Ticket 檢測

| 要素 | 通過 | 阻止 |
|------|------|------|
| 驗收條件 | >= 3 個可量化項目 | 缺失或 < 3 |
| 測試規劃 | 明確檔案和項目 | 缺失 |
| 工作日誌 | 明確檔案名稱 | 缺失 |
| 參考文件 | >= 1 個連結 | 缺失 |

**失敗處理**：補充缺失元素

### Step 4: 提交審查

所有 Ticket 通過 C1/C2/C3 → 提交 PM 審查 → 批准後進入 Phase 2

---

## 檢查清單

### C1 God Ticket
- [ ] 檔案數量 <= 10
- [ ] 層級跨度 <= 2
- [ ] 預估工時 <= 16h

### C3 Ambiguous Responsibility
- [ ] 標題含 `[Layer X]`
- [ ] 目標章節定義單一職責
- [ ] 步驟列出具體檔案路徑
- [ ] 驗收條件限定該層級

### C2 Incomplete Ticket
- [ ] 驗收條件 >= 3 個
- [ ] 測試規劃明確
- [ ] 工作日誌規劃明確
- [ ] 參考文件 >= 1 個

### 阻斷機制
- [ ] 任何 Ticket 未通過 → 禁止進入 Phase 2
- [ ] 檢測失敗 → 立即修正 → 重新檢測
- [ ] 檢測過程記錄到工作日誌
