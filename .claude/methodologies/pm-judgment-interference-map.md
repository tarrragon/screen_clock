# PM 判斷干擾機制地圖

**版本**: v1.1.0
**建立日期**: 2026-04-24
**核心原則**: 判斷品質失效是多類系統性干擾，非單一事件；分層盤點與絆腳索防護

---

## 目的

本方法論為 PM（主線程）提供判斷品質自檢的**結構化地圖**。適用場景：

- 撰寫論述前的自檢（派發 prompt / commit message / AUQ / ticket body）
- 被用戶糾正後的根因分析（禁止自責式歸因，必須分層挖因）
- 失敗事件的學習累積（取代零散 PC 條目，提供統一分類）
- 規則 / PC 系統演進的上層框架

**邊界**：本方法論不處理**決策結果**的對錯（如派發選哪個 agent），只處理**決策生成過程**中的干擾機制。決策結果的對錯由領域知識 + WRAP 框架 + 多視角審查處理。

---

## 核心主張

### 主張 1：PM 判斷品質失效是多類系統性干擾

對應的**反面主張**（Consider the Opposite 已反駁）：「PM 失誤是特定事件，不需要系統性分類」。

反駁依據（W17-075 實測）：單一 session 內觸發至少 6 時點 × 4 類干擾；歷史 10 條 PC 皆可套入同一骨架。

### 主張 2：干擾因子有連鎖關係

觸發一類干擾會提高另一類觸發機率（例：CLI autopilot → 撞錯 → 不讀 help → 又猜變體；論述編造 → 被糾正 → 淺層歸因）。

### 主張 3：統一防護骨架適用所有干擾

所有 10+ 條歷史 PC 可套入三段結構：**觸發訊號 → 絆腳索 → 替代行為**。新發現的干擾因子應依此骨架記錄，避免零散 PC 無結構累積。

## 六層干擾分類

### 層 1：論述生成層

論述（文字輸出）在生成時的失效。

| 因子 | 觸發訊號 | 絆腳索 | 替代行為 |
|------|---------|--------|---------|
| 1.1 事後合理化編造機制 | 警戒術語出現（`working memory` / `race condition` / `context pollution`） | Q1-Q3 自檢：機制寫在哪？可引用嗎？事實句型會不會等價？ | 附引用或改為事實句型 |
| 1.2 自信度校準失效 | 技術術語未附規則 / 檔案 / memory 引用 | 輸出前引用檢查：能說出具體檔案路徑 / 規則 ID 嗎？ | 無引用就不說機制 |
| 1.3 機制句型偏好 | 傾向「會導致 X」「可能 Y」等預測性句型 | 句型改寫檢查：改為「依 Y 職責 / 規則 / 案例」是否等價？ | 事實句型替代 |
| 1.4 素材錯置抽象層級 | 引用了真實文件（PC / memory / SKILL），但所述層級（如 git index race 這種實作細節）與當前論述層級（如「working memory」這種認知抽象層）不符 | 素材層級檢查：文件所述層級 = 當前論述層級？（實作細節 ≠ 認知機制）| 引用時明示層級對應；若跨層則不引用，改為事實句型說明具體文件內容 |

**關聯 PC**：PC-111 R1/R2/R3/R5

### 層 2：歸因層

被糾正時的根因分析失效。

| 因子 | 觸發訊號 | 絆腳索 | 替代行為 |
|------|---------|--------|---------|
| 2.1 自責式屬性歸因 | 第一反應「我錯了」「判斷錯」「過度臆測」 | 四層挖因檢查：決策層錯了？論述層哪個句型？生成機制為何？防護是什麼？ | 分層結構化挖因 |

**關聯 PC**：PC-111 R4、PC-081 延伸

**禁用句型對照**：

| 禁用 | 替代 |
|------|------|
| 「我判斷錯誤」 | 「我在 X 情境下輸出了 Y 論述，該論述包含無事實支撐的機制 Z」 |
| 「我過度臆測」 | 「我跳過了某自檢，直接輸出機制句型」 |
| 「我承認錯誤」 | 「決策層對、論述層錯；生成機制層的根因是某因子；防護是某絆腳索」 |

### 層 3：決策路徑層

決策流程中跳過自檢的 autopilot。

| 因子 | 觸發訊號 | 絆腳索 | 替代行為 |
|------|---------|--------|---------|
| 3.1 CLI / 規則 autopilot | CLI 撞錯後立即重試或猜變體（非查 `--help` / 規則文件） | 強制「撞錯即 `--help`」規則 | 查文件後再試 |
| 3.2 Anchor（既有結論優先） | WRAP W 階段選項能一句話概括 | Consider the Opposite 強制 | 重定義問題後 W 重跑 |
| 3.3 規則失敗草率改規則 | 失敗第一反應「改規則」 | 先重試 2 次才改 | 挖根因再決定是改規則還是改行為 |
| 3.4 多步驟 perplexity 盲 | 多步驟計畫中所有中間步驟都預測成功 | WRAP R 階段基本率檢查 | 每步獨立驗證 |

**關聯 PC**：PC-066（autopilot）、PC-080（WRAP 跳框架）、PC-088（多步驟盲）、PC-105（CLI autopilot）、PC-106（草率改規則）

### 層 4：輸出污染層

輸出層面的被動污染，非論述本身的錯誤。

| 因子 | 觸發訊號 | 絆腳索 | 替代行為 |
|------|---------|--------|---------|
| 4.1 Charset 污染（簡體 / 日文漢字 / emoji） | AUQ payload 或對話含非繁體字符 | Hook 阻擋（askuserquestion-charset-guard） | 繁體自檢、工具修正 |
| 4.2 Git index.lock 忽視 | 串接多個 git 寫入指令 | Bash 規則 3 拆獨立呼叫 | 每 git 寫入獨立 Bash 呼叫 |

**關聯 PC**：PC-072、PC-084、memory feedback_git_index_lock_prevention

### 層 5：互動壓力層

用戶互動情境中的判斷失效。

| 因子 | 觸發訊號 | 絆腳索 | 替代行為 |
|------|---------|--------|---------|
| 5.1 用戶疲勞但 PM 繼續推 | 用戶回應字數驟降 / 急迫詞彙 / 跳過 AUQ 選項 | ai-communication-rules 規則 5 Layer 3 疲勞偵測 | 主動降權、提議 handoff |
| 5.2 Memory 引用過時 | 直接引用 memory 條目不驗證當下條件 | 引用前 `grep` / `Read` 驗證 | 先驗證後引用 |

**關聯 PC**：PC-081、ai-communication-rules 規則 5 Layer 3

### 層 6：WRAP 本身層

WRAP 框架自身的失效（元層級）。

| 因子 | 觸發訊號 | 絆腳索 | 替代行為 |
|------|---------|--------|---------|
| 6.1 WRAP autopilot（context 沉重時觸發率降） | Context 使用率 > 60% | decision-quality-guard-hook 自動偵測 | 強制外部觸發（Hook / 用戶提醒）|
| 6.2 A 階段跳框架測試 | 選項一句話可概括 | Consider the Opposite 強制 | 重新定義問題後 W 重跑 |

**關聯 PC**：PC-066（WRAP autopilot）、PC-080（A 階段跳框架）

## 使用流程

### 情境 A：撰寫論述前自檢

適用：派發 prompt、commit message、AUQ payload、ticket body。

```
1. 識別是否使用「技術機制性術語」（見層 1 警戒術語清單）
2. 套用 Q1-Q3 自檢
3. 無引用 → 改為事實句型
```

### 情境 B：被用戶糾正後的歸因

適用：用戶指出論述 / 決策問題後的分析。

```
1. 禁止第一反應「我錯了」/「我判斷錯」/「過度臆測」
2. 四層挖因：
   a. 決策層：決策本身對嗎？（通常對）
   b. 論述層：哪個句型出問題？
   c. 生成機制層：論述生成為何選那個句型？（對應哪個因子？）
   d. 防護層：下次具體自檢機制是什麼？
3. 若根因對應本地圖某因子，引用該因子編號（如「層 1 因子 1.1」）
4. 若新因子未在地圖中，追加至地圖
5. 素材來源層（當因子 1.4 觸發時必執行）：下挖具體素材來源 —— 哪個檔案路徑、哪條 memory 條目名稱、哪個 SKILL？該素材所述的抽象層級是什麼？與論述層級的差距是什麼？
```

### 情境 C：失敗事件的學習累積

適用：事後檢討、post-mortem、PC 建立前評估。

```
1. 定位事件對應的干擾因子（使用本地圖）
2. 判斷是否為既有因子的重複觸發：
   - 是 → 更新既有 PC 的「觸發案例」章節；不建新 PC
   - 否 → 評估是否為新因子；若是則追加至地圖並建新 PC
3. 避免零散 PC 無結構累積
```

### 情境 D：規則 / 工具演進的定位

適用：設計新 Hook / 規則 / Skill 時。

```
1. 定位該規則 / 工具目標防護的層級（1-6）
2. 檢查該層級既有絆腳索清單，避免重複
3. 新絆腳索需回填本地圖對應因子的「絆腳索」欄位
```

## 連鎖關係（已觀察）

| 連鎖起點 | 連鎖結果 | 機制 |
|---------|---------|------|
| 因子 3.1（CLI autopilot） | 因子 3.1 再次觸發 | 撞錯不學習，繼續猜變體 |
| 因子 1.1-1.3（論述編造） | 因子 2.1（淺層歸因） | 論述錯被糾正後，情緒保護驅動「我錯了」反應 |
| 因子 6.1（WRAP autopilot） | 因子 3.1-3.4 全線觸發 | WRAP 失效後，所有決策路徑層防護同時失效 |

**識別連鎖的觸發訊號**：同一 session 內某因子觸發 2+ 次，或兩因子先後觸發且有因果關係。

**連鎖觸發的應對**：不要只修復末端（如只改「我錯了」歸因），要回溯至連鎖起點（論述生成層）。

## 演進機制

### 新因子追加條件

- 觸發事件無法歸類於現有 6 層 13 因子
- 觸發頻率 > 1 次（單次觸發先記 memory feedback，第二次才加至地圖）
- 可定義「觸發訊號 → 絆腳索 → 替代行為」三段結構（無法定義則不是系統性干擾，改記 memory）

### 因子合併條件

- 兩因子的觸發訊號高度重疊（> 70%）
- 兩因子的絆腳索相同
- 合併後仍保留足夠區分度

### 層級重組條件

- 某層下因子數 > 6（考慮拆分）
- 某層下因子數 = 1 且無新增預期（考慮合併至相鄰層）

## 自測案例

**情境**：PM 派發兩個 agent 時論述寫「兩個 thyme 並發會 working memory 混淆」。

**自檢流程**：
1. 識別警戒術語：`working memory` ✓（層 1 因子 1.1）
2. Q1：這個機制寫在哪？→ 無引用
3. Q2：能說出檔案路徑？→ 否
4. Q3：事實句型等價？→「basil hook 職責 / thyme lib 職責」等價 ✓
5. 替代：改為事實句型「basil 是 hook 設計職責，thyme 是 lib 優化職責」

**若未自檢、被用戶糾正**：
1. 禁用「我錯了」反應（層 2 因子 2.1）
2. 四層挖因：
   - 決策層：派 basil + thyme 是對的（職責匹配）
   - 論述層：用了因子 1.1 警戒術語
   - 生成機制層：因子 1.1 + 1.2（自信度校準失效）
   - 防護層：Q1-Q3 自檢 + 引用檢查

---

### 自測案例 2：「working memory 混淆」推理鏈（因子 1.4 觸發範例）

**情境**：PM 在論述兩個 thyme 並發問題時引用了三個素材：PC-092（並行 thyme git index race）、memory `feedback_parallel_agent_arbitration`（subagent 各看局部）、`.claude/skills/agent-team/SKILL.md`（Team 共享 state）。論述輸出「兩個 thyme 並發會產生 working memory 混淆」。

**推理鏈拆解**：

| 步驟 | 素材 / 操作 | 所述抽象層級 | 問題 |
|------|------------|------------|------|
| 素材 1 | PC-092：git index.lock 並行競爭 | 實作層（git 指令層級） | 正確 |
| 素材 2 | feedback_parallel_agent_arbitration：subagent 各看局部 | 資訊可見性層（哪個 subagent 看到哪些 context） | 正確 |
| 素材 3 | agent-team SKILL：Team 共享 state | 架構設計層（Team 協作模式） | 正確 |
| 跨層泛化 | 三素材 → 統一歸納為「working memory 混淆」 | 認知機制層（AI 工作記憶概念） | 跨層：素材均為實作/資訊層，論述卻跳至認知機制層 |

**素材層級檢查（因子 1.4 絆腳索）**：

- PC-092 所述層級：git 指令實作層 ≠ 「working memory」認知機制層
- 各素材描述的是「哪個 subagent 發生什麼操作問題」，而非「working memory 是什麼機制」
- 結論：三素材均無法支撐「working memory 混淆」這個認知機制性論述

**替代行為**：改為事實句型「PC-092 記錄 thyme 並行時 git index.lock 競爭；subagent 各自看到局部 context，無法跨 agent 共享完整狀態」，不引入無素材支撐的認知機制術語。
