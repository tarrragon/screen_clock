# 並行評估方法論

本文件定義「並行多視角派發 → 彙整 → 決策」的原則層。涵蓋兩種用途：**品質掃描**（找出值得行動的問題，產出做/不做清單）與**理解型分析**（多角度交叉驗證根因，產出根因結論）。執行操作見 `.claude/skills/parallel-evaluation/SKILL.md`，報告完整模板見 `.claude/references/multi-perspective-report-templates.md`。

> **合併說明（1.0.0-W8-020.7）**：本方法論吸收原 multi-perspective-analysis-methodology 的獨特核心（視角分工、衝突處理策略、分析任務並行差異、決策樹整合）。原檔已刪除，引用統一重導至本檔。差異在「報告格式」而非「派發機制」——機制相同，故合併。

## 核心理念

> **單一視角容易遺漏問題，多視角並行可交叉驗證，提高決策品質；但每個改善建議都有成本，只做剛好夠用的事。**

### 最小充分行動原則（Minimum Sufficient Action）

並行評估的核心不是「找出所有問題」，而是「快速識別值得行動的問題」。每一個改善建議都有成本（變更風險、review 時間、認知負擔），只有當改善幅度明確大於成本時才值得執行。

| 單一視角的侷限 | 多視角並行如何補足 |
|---------------|------------------|
| 視角盲點：受限於專業領域，遺漏其他層面 | 不同視角同時掃描，互補覆蓋 |
| 效率瓶頸：序列執行時間成本高 | 並行派發，同時產出 |
| 決策品質：單一來源缺乏交叉驗證 | 多份報告交叉比對根因與風險 |

### 並行掃描 → 彙整 → 值得改才改

三步驟品質掃描模式：(1) 收集標的 → (2) 多視角同時掃描 → (3) 過濾後行動（只保留值得改的，其餘建 Ticket 追蹤）。

### 與既有工具的定位差異

| 工具 | 定位 | 產出 |
|------|------|------|
| /design-decision-framework | 方案比較（評估型） | 方案排名 |
| **並行評估（本方法論）** | **品質掃描（決策型）+ 理解型分析** | **做/不做清單 或 根因結論** |
| /bulk-evaluate | 批量處理（卸載型） | N 個子 Ticket 結論 |

關鍵差異：並行評估不比較方案，而是從多個視角同時掃描一個標的。品質掃描產出「值得行動」清單；理解型分析產出「根因結論 + 風險總評」。

### 常駐委員制度

每次並行評估除情境特定視角外，**必須**包含 linux（品質把關）作為常駐委員；情境 C/D/F/G 預設加入 basil-writing-critic（書面文字產出量最高的場景）。

```
每次 parallel-evaluation 執行:
  情境特定視角 (2-3 個) + 常駐委員 linux (1 個) [+ basil 視情境] = 3-4 個 Agent
```

**linux 常駐視角**（所有情境通用）：設計簡潔性（能更簡單嗎？）、問題真實性（解決真實問題還是假想問題？）、特例消除（有不必要的特例嗎？）、向後相容性（會破壞現有行為嗎？）。品質評分 Good taste / Acceptable / Garbage 直接對應 Worth-It Filter 改善幅度（Garbage = 高、Acceptable = 中、Good taste = 無發現）。

> 常駐委員加入規則、basil opt-out 機制（`--skip-basil`）、語言代理人加入規則：`.claude/skills/parallel-evaluation/SKILL.md`「常駐委員機制」。

## 適用場景與觸發條件

**符合任一條件即可考慮啟用**：認知負擔指數 > 10、風險等級 >= 高、安全敏感（認證/授權/敏感資料）、影響範圍 > 3 模組、根因不明確。

| 場景 | 用途類型 | 建議視角 | 報告消費者 |
|------|---------|---------|-----------|
| 測試失敗分析 | 理解型 | 程式碼、測試、架構、需求 | SA → PM |
| 架構決策 | 品質掃描 | 一致性、影響、簡潔性 | SA → PM |
| 需求/功能評估 | 品質掃描 | 重疊、契合、範圍 | SA → PM |
| 效能問題定位 | 理解型 | 程式碼、資料庫、網路、資源 | ginger → PM |
| 安全審查 | 品質掃描 | OWASP、認證、授權、資料 | security → PM |
| 重構決策 | 品質掃描 | 冗餘、耦合、複雜度 | cinnamon → PM |
| 結論審查 | 品質掃描 | 證據、替代方案、範圍 | SA → PM |

**不適用場景**：明顯的單點問題（原因明確）、簡單格式/筆誤、緊急 P0（先修復後分析）、認知負擔 < 5。這些情況直接派發對應代理人或單一代理人分析即可。

### 標準分析視角與代理人分工

| 視角 | 負責代理人 | 分析焦點 |
|------|-----------|---------|
| 程式碼視角 | parsley-flutter-developer / thyme-python-developer | 實作細節、程式碼品質、邏輯錯誤 |
| 測試視角 | sage-test-architect | 測試設計、覆蓋率、測試品質 |
| 架構視角 | saffron-system-analyst | 設計模式、依賴關係、架構一致性 |
| 需求視角 | lavender-interface-designer | 需求理解、規格對齊、功能完整性 |
| 效能視角 | ginger-performance-tuner | 效能瓶頸、資源使用、優化空間 |
| 安全視角 | clove-security-reviewer | 漏洞、認證授權、資料保護 |
| 品質把關（常駐） | linux | Good taste、特例消除、向後相容 |
| 文字品質（C/D/F/G 預設） | basil-writing-critic | 三明示、資訊優先序、禁用詞 |

## 通用三階段流程

### Phase 1: 收集標的（Gather Target）

確定掃描範圍和標的物（git diff 檔案清單 / 設計文件 / 分析結論 / 功能規格）。收集原則：標的範圍越精確掃描越有效；過大標的先拆分；明確列出「不在範圍內」的項目。

### Phase 2: 並行視角掃描（Parallel Lens Scan）

啟動 2-4 個 Agent，每個負責一個獨立且不重疊的視角，同時掃描標的。視角設計原則：獨立性（焦點不重疊）、可操作性（發現能轉化為行動）、數量控制（2-4 個為佳，超過 4 個噪音大於訊號）、Agent 匹配（每視角匹配最適 Agent）。所有視角同時啟動、互不依賴、各自獨立產出。

### Phase 3: 彙整與行動過濾（Aggregate & Filter）

收集所有視角發現，去重（多視角發現同一問題時合併，可信度提升），對每個發現套用 Worth-It Filter，**對每個延後項目執行 `ticket create` 取得 ID 填入報告**，產出最終行動清單。理解型分析則改產出彙整報告（交叉驗證根因、衝突決策、風險總評，見報告模板衛星檔）。

> **強制規則**：品質掃描報告的「延後追蹤」表格中，每一行 Ticket 欄必須填入實際 Ticket ID。空白 Ticket 欄 = 未追蹤 = 違反 quality-baseline 規則 5。

## Worth-It Filter（執行決策規則）

> **核心原則**：發現技術債和決定修復成本是兩個獨立決策。Worth-It Filter 只回答「現在修還是之後修」，不回答「要不要追蹤」——答案永遠是「要」。

**核心公式**：`行動價值 = 改善幅度 / (變更風險 + 實施成本)`。行動價值 > 門檻時值得立即執行；低於門檻延後，但仍必須建 Ticket 追蹤。

| 改善幅度 | 變更風險 | 執行決策 | 追蹤方式 |
|---------|---------|---------|---------|
| 高（消除 bug/安全漏洞） | 任何 | 立即執行 | 建 Ticket（P0/P1） |
| 高（顯著簡化） | 低 | 立即執行 | 建 Ticket（P1） |
| 高（顯著簡化） | 高 | 延後執行 | 建 Ticket（P1） |
| 中（可讀性/維護性） | 低 | 立即執行 | 建 Ticket（P2） |
| 中（可讀性/維護性） | 高 | 延後執行 | 建 Ticket（P2） |
| 低（風格偏好） | 任何 | 延後執行 | 建 Ticket（P2） |

**False Positive 處理**：需要超過 30 秒解釋「為什麼這值得**現在**改」就延後；多視角同時發現可信度更高；只有一個視角發現且改善幅度為「中」的項目降級為延後。

> 量化標準和案例：`.claude/skills/parallel-evaluation/references/worth-it-filter-details.md`

## 衝突處理策略（分類處理）

> **核心原則**：不同類型的衝突需要不同的解決策略。預設聯集會導致加法偏見——每增加一個審查視角，產出物只膨脹不收縮。

| 衝突類型 | 處理策略 | 舉證責任 |
|---------|---------|---------|
| 根因不一致 | 列出所有可能根因，建議逐一排查 | 各視角舉證 |
| 加法 vs 加法（都想新增） | 聯集（合併） | 無需額外舉證 |
| **加法 vs 減法（一方想保留、一方想刪除）** | **預設採減法，保留方需舉證** | **加法方（想保留的人）** |
| 減法 vs 減法（都想刪除） | 交集（刪最少的，保守刪除） | 無需額外舉證 |
| 優先級分歧 | 採用最高優先級 | 明確標註來源視角 |

**「加法 vs 減法」規則說明**：當審查者 A 說「刪除這個章節」而 B 說「保留」時，B 必須提出保留的具體理由（不是「可能有用」）。PM 禁止透過折衷合併（如「保留一半」）迴避二元決策。linux 對品味問題有否決權。

> **來源**：多視角發想結構性加法偏見分析的歷史教訓。

## 分析任務並行的安全特性

分析任務並行與實作任務並行有本質差異，安全檢查可放寬：

| 維度 | 實作任務並行 | 分析任務並行 |
|------|-------------|-------------|
| 檔案重疊 | **禁止**（會衝突） | **允許**（只讀分析） |
| 依賴關係 | 需明確標記 | 無依賴（各視角獨立） |
| 輸出衝突 | 會衝突 | 不衝突（各自產出報告） |
| 彙整需求 | 無 | **需要**彙整者整合報告 |

**並行安全檢查清單**：各視角分析範圍無重疊、各代理人可獨立完成（無需互等）、報告格式統一（便於彙整）、彙整者已確定（知道誰負責整合）。

## 情境速查與 Worth-It Filter 特化

> 情境快速選擇表、各視角詳細檢查項目、Agent 提示模板：`.claude/skills/parallel-evaluation/SKILL.md` + `references/lens-configurations.md` + `references/lens-prompts.md`

| 情境 | Worth-It Filter 特化 |
|------|---------------------|
| A: 程式碼審查 | 風格偏好延後追蹤；只立即行動於有具體改善的項目 |
| B: 重構評估 | 重構成本必須低於維護成本的 3 倍 |
| C: 架構評估 | 不一致性必須是實際問題（非美學偏好） |
| D: 功能評估 | 功能重疊超過 50% 時必須合併或重新定義 |
| E: 冗餘偵測 | 只處理跨模組的重複（同模組內由 Phase 4 處理） |
| F: 結論審查 | 閾值更嚴格——任一視角發現問題必須回到分析階段 |
| G: 系統設計 | 術語不一致必須修正；交叉引用缺失必須補齊；風格偏好延後追蹤 |

## 與決策樹的整合

| 決策樹層級 | 整合方式 |
|-----------|---------|
| 第負一層（並行化評估） | 任務拆分判斷後增加「分析並行化判斷」：可拆實作則並行派發實作；否則判斷是否需多視角分析，需要則並行派發分析代理人 → 彙整 → 派發修復 |
| 第五層（TDD Phase 4） | 情境 B 用於重構決策 |
| 第六層（事件回應） | incident-responder 初步評估後，若原因不明/複雜度高/風險高，啟用情境 F 審查分析結論；不通過則回到分析補充遺漏視角 |

### 觸發建議

| 場景 | 建議情境 |
|------|---------|
| Phase 3b 完成後 | 情境 A（程式碼審查） |
| Phase 4 開始前 | 情境 B（重構評估） |
| SA 前置審查 | 情境 C（架構評估） |
| Phase 1 完成後 | 情境 D（功能評估） |
| 版本規劃時 | 情境 E（冗餘偵測） |
| incident 分析後 | 情境 F（結論審查） |
| 規則/Skill/方法論變更後 | 情境 G（系統設計評估） |
