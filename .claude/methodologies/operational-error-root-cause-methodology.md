# 操作錯誤根因分析方法論

本方法論定義偵測操作失誤時的三層根因分析框架，強制分析「設計邊界/使用情境/說明充分性」，防止只修復表面症狀而讓同類問題反覆發生。

---

## 與現有方法論的區隔

| 文件 | 適用範圍 |
|------|---------|
| **本方法論** | 人的操作行為失誤（代理人/PM 做了錯誤的操作） |
| `systematic-debugging-methodology.md` | 程式碼層次的 unused 警告根因分析 |
| `incident-response.md` CLI 調查步驟 | 工具/CLI 回傳錯誤時的四步驟調查 |

**核心差異**：
- 程式碼 bug → `incident-response.md` 事件回應流程
- 工具報錯 → `incident-response.md` CLI 調查步驟
- **操作失誤（做了不該做的事、沒做該做的事）→ 本方法論**

---

## 什麼是操作失誤

操作失誤指「人（代理人或 PM）採取了錯誤操作方式」，而非工具或程式碼本身的問題。

**典型案例**：

| 案例 | 錯誤的操作行為 | 來源 |
|------|--------------|------|
| PC-005 | `ticket handoff` 失敗後，假設是邏輯問題而非語法錯誤，直接手動建檔繞過流程 | 未查語法就歸因 |
| PC-007 | 直接採信 Ticket 描述的數字/範圍，未獨立驗證 | 未重新確認假設 |
| IMP-011 | 基於欄位名稱假設其格式，未查閱生產端程式碼 | 假設驅動而非證據驅動 |

**辨識判斷**：操作失誤通常在 error-pattern 中歸類為 `process-compliance`，或描述中包含「假設」「直接」「跳過」等詞彙。

## 三層根因分析框架

當偵測到操作失誤時，**強制依序分析以下三層**，每層都必須回答。

### 第一層：設計邊界（Design Boundary）

> 設計是否清楚界定了「什麼情況下不能這樣做」？

**分析問題**（每個都必須回答）：

1. 失誤的操作是否在設計規範（rule/flow/lifecycle）中被明確禁止？
2. 規範是否清楚說明了「哪些邊界條件下不適用」？
3. 是否存在例外情況但未在規範中標注？

**目標**：判斷設計本身是否清楚，還是設計邊界本身有模糊地帶。

**結論類型**：
- 「設計清楚，邊界明確」→ 繼續第二層
- 「設計存在模糊地帶」→ 補充規範（補充 rule / 加入例外說明）

---

### 第二層：使用情境（Use Context）

> 在具體使用情境下，操作者是否有足夠資訊做出正確判斷？

**分析問題**（每個都必須回答）：

1. 操作者在當下的情境中，是否能看到足夠的警示信號（Hook 提醒、驗收條件、前置步驟）？
2. 失誤是發生在高壓/高速推進時，還是一般情況下也容易犯？
3. 這個操作路徑是否缺少「防呆機制」（如確認對話、強制查詢步驟）？

**目標**：判斷使用情境是否提供了足夠的決策支援，還是需要加強輔助機制。

**結論類型**：
- 「情境提供充分信號」→ 繼續第三層
- 「情境缺乏決策支援」→ 加強防呆（Hook 提醒、流程節點加確認步驟）

---

### 第三層：說明充分性（Documentation Adequacy）

> 操作者是否能輕易找到並理解正確做法？

**分析問題**（每個都必須回答）：

1. 正確做法是否在可查詢的文件中有清楚說明？
2. 反例（禁止行為）是否明確列出，且足夠具體（不只是「禁止假設」而是「禁止做 X，應改做 Y」）？
3. 說明的位置是否容易被找到（在決策點附近，而不是埋在深層文件中）？

**目標**：判斷文件品質是否能讓操作者在困惑時找到答案，還是需要補充說明。

**結論類型**：
- 「說明充分、位置正確」→ 問題在執行層面（需要強化 Hook 強制檢查）
- 「說明不足或難以找到」→ 補充文件說明或調整位置

---

## 觸發條件

以下情況必須啟動三層根因分析：

| 觸發情境 | 識別特徵 |
|---------|---------|
| 新增 error-pattern 屬於 `process-compliance` 類別 | PC-xxx 前綴 |
| 操作失誤導致需要手動修復或繞過標準流程 | 描述含「手動」「繞過」「直接」 |
| 同一類操作失誤在不同 session 中第二次發生 | error-pattern 更新而非新建 |
| incident-responder 分析結論為「操作層錯誤」而非「程式層錯誤」 | 分類結果為操作問題 |

## 分析模板

以下模板用於記錄三層分析結果（填寫到對應的 error-pattern 或 Ticket）：

```markdown
## 操作錯誤根因分析

**操作失誤描述**：[具體描述做了什麼錯誤操作]

### 第一層：設計邊界
- 設計規範是否明確禁止此操作？[是/否/部分]
- 規範是否涵蓋例外情況？[是/否]
- 模糊地帶識別：[描述，或填「無」]
- 補充規範行動：[需要改動的文件，或填「不需要」]

### 第二層：使用情境
- 情境中有無足夠警示信號？[是/否/不足]
- 高壓情境還是一般情況下的失誤？[高壓/一般]
- 缺少的防呆機制：[描述，或填「無需加強」]
- 改善行動：[Hook 或流程節點調整，或填「不需要」]

### 第三層：說明充分性
- 正確做法是否有清楚文件說明？[是/否/部分]
- 禁止行為是否具體列出（含正確替代做法）？[是/否]
- 說明位置是否容易被找到？[是/否]
- 補充文件行動：[需要修改的文件，或填「不需要」]

### 根因結論與行動
- 主要根因層：[第一層/第二層/第三層/複合]
- 防護行動：[建立 Ticket 的 ID 或具體行動]
```

---

## 分析結果的後續行動

| 根因層 | 後續行動 |
|--------|---------|
| 設計邊界模糊 | 修改對應 rule/flow 文件，補充邊界說明和例外情況 |
| 使用情境缺乏支援 | 建立或強化 Hook，在決策點加入確認提示 |
| 說明不充分 | 補充文件（在操作決策點附近），加入具體的禁止/正確對照 |
| 複合根因 | 建立多個追蹤 Ticket，分層改善 |

所有「後續行動」必須建立 Ticket 追蹤，不可只記錄在分析中而不追蹤。

## ANA 驗證共用函式正確性的範圍要求（PC-136 強制）

> **來源**：PC-136 — W17-179 ANA false negative 案例：ANA 驗證 stop hook L361 自身函式正確，誤判「整個問題已解決」，未追蹤 callees / 同名實作 / callers，導致 lib `handoff_utils.is_ticket_completed` 漏修，數週內從另一處重爆。
>
> **觸發時機**：ANA / 重啟調查驗證「函式 A 是否正確」時。

ANA 驗證函式正確性時，**範圍不可只到「直接被問的函式」本體**，必須追蹤呼叫鏈至少一層。

**強制驗證範圍清單**：

| 驗證對象 | 必查 | 動作 |
|---------|------|------|
| 函式 A 本體 | 是 | 讀 A 程式碼 + 驗證行為 |
| A 的 callees（A 呼叫的函式） | 至少一層 | 對每個 callee 確認是否也已正確修正 |
| A 的同名實作（grep `def A` / `function A`） | 全部 | grep 全 codebase，逐一比對是否同步修法 |
| A 的 callers | 抽樣 | 檢查是否假設一致 |

**ANA false negative 防護**：

| 反模式 | 正確做法 |
|-------|---------|
| 只驗證「直接被問的函式」 | 追蹤 callees 至少一層、同名實作全查 |
| 「修了一處」即結論「整個問題已解決」 | 在結論章節明示「已驗證 N 處同名實作 / M 處 callees」 |
| 假設 lib SSOT 必然被所有 caller 走 | grep all callers + 逐一驗證 import chain |

**驗證後必須在 ANA 結論記錄**：

- 已 grep 的關鍵字與命中位置數
- 已驗證的同名實作清單
- 已追蹤的 callees 清單（至少一層）
- 未驗證的 callers（抽樣或全量）說明

> 完整錯誤模式、ARCH-020 三次重爆軌跡、W17-179 案例：.claude/error-patterns/process-compliance/PC-136-structural-fix-incomplete-caller-scan.md
