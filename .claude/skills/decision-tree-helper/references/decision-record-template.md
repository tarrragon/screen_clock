# 決策記錄格式範本

使用本範本記錄每次重大決策，便於追蹤決策過程和驗收。

## 標準決策記錄

```markdown
## 決策記錄

### 任務資訊
- **任務描述**: [簡述任務目標]
- **提交時間**: [YYYY-MM-DD HH:MM:SS]
- **複雜度初步評估**: [低/中/高]

### 複雜度評估
| 指標 | 數值 | 等級 |
|------|------|------|
| 需修改檔案數 | [數] | [低/中/高] |
| 跨越架構層級數 | [數] | [低/中/高] |
| 依賴模組數 | [數] | [低/中/高] |
| 狀態追蹤數 | [數] | [低/中/高] |
| **認知負擔指數** | [分] | **[低/中/高]** |

### 決策路徑
1. 包含錯誤關鍵字？ [否 → 下一步] / [是 → incident-responder]
2. 包含不確定性詞彙？ [否 → 下一步] / [是 → 確認機制]
3. 複雜需求（3+ 代理人）？ [否 → 下一步] / [是 → 確認機制]
4. 訊息類型？ [問題 / 命令 / 其他]
5. 最終派發決策？ [...]

### 派發決策
- **建議代理人**: [...]
- **決策理由**: [簡述為何選擇此代理人]
- **預期產出**: [列舉期望的產出物]

### 拆分建議（如需要）
| 子任務 | 目標 | 代理人 | 執行順序 | 依賴關係 |
|--------|------|--------|---------|---------|
| Ticket 1 | [目標] | [代理人] | [順序] | [無/Ticket X] |
| Ticket 2 | [目標] | [代理人] | [順序] | [無/Ticket X] |

### 驗收標準
- [ ] 決策邏輯符合決策樹規則
- [ ] 派發建議明確可操作
- [ ] 若拆分，所有子任務有清晰的依賴和順序
- [ ] 工作日誌已更新

### 備註
[額外說明或特殊考量]
```

## 使用範例

### 例 1：低複雜度簡單修改

```markdown
## 決策記錄

### 任務資訊
- **任務描述**: 修複 README.md 中的拼寫錯誤
- **提交時間**: 2026-03-02 14:30:00
- **複雜度初步評估**: 低

### 複雜度評估
| 指標 | 數值 | 等級 |
|------|------|------|
| 需修改檔案數 | 1 | 低 |
| 跨越架構層級數 | 0 | 低 |
| 依賴模組數 | 0 | 低 |
| 狀態追蹤數 | 0 | 低 |
| **認知負擔指數** | 1 | **低** |

### 決策路徑
1. 包含錯誤關鍵字？ 否 → 下一步
2. 包含不確定性詞彙？ 否 → 下一步
3. 複雜需求（3+ 代理人）？ 否 → 下一步
4. 訊息類型？ 命令（文件修改）
5. 最終派發決策？ 直接執行（DOC 類 Ticket）

### 派發決策
- **建議代理人**: 主線程（可直接執行）
- **決策理由**: 認知負擔指數 = 1（遠低於 10），純文件修改無邏輯
- **預期產出**: 修正後的 README.md

### 驗收標準
- [x] 決策邏輯符合決策樹規則
- [x] 派發建議明確可操作
- [x] 無需拆分

### 備註
無
```

### 例 2：中複雜度需要拆分

```markdown
## 決策記錄

### 任務資訊
- **任務描述**: 實現用戶認證系統（JWT + 刷新令牌）
- **提交時間**: 2026-03-02 15:00:00
- **複雜度初步評估**: 高

### 複雜度評估
| 指標 | 數值 | 等級 |
|------|------|------|
| 需修改檔案數 | 12 | 高 |
| 跨越架構層級數 | 4 | 高 |
| 依賴模組數 | 5 | 高 |
| 狀態追蹤數 | 8 | 高 |
| **認知負擔指數** | 29 | **高** |

### 決策路徑
1. 包含錯誤關鍵字？ 否 → 下一步
2. 包含不確定性詞彙？ 否 → 下一步
3. 複雜需求（3+ 代理人）？ 是（涉及 SA、security-reviewer、parsley）→ 確認機制

### 派發決策
- **建議代理人**: 多代理人協作
  1. system-analyst - 架構審查
  2. security-reviewer - 安全評估
  3. 完整 TDD 流程
- **決策理由**: 認知負擔指數 29 > 10，必須拆分。涉及多個領域（架構、安全、實現），需多視角審查
- **預期產出**: 認證系統完整實現

### 拆分建議（按 TDD 階段 + 架構層）
| 子任務 | 目標 | 代理人 | 執行順序 | 依賴關係 |
|--------|------|--------|---------|---------|
| Ticket 1 | SA 審查 + 安全評估 | system-analyst + security-reviewer | 1 | 無 |
| Ticket 2 | 功能規格設計 | lavender-interface-designer | 2 | Ticket 1 |
| Ticket 3 | 測試案例設計 | sage-test-architect | 3 | Ticket 2 |
| Ticket 4 | Domain 層實現 | parsley-flutter-developer | 4 | Ticket 3 |
| Ticket 5 | Repository 層實現 | parsley-flutter-developer | 5 | Ticket 4 |
| Ticket 6 | UI 層實現 | parsley-flutter-developer | 6 | Ticket 5 |
| Ticket 7 | 重構評估 | cinnamon-refactor-owl | 7 | Ticket 6 |

### 驗收標準
- [x] 決策邏輯符合決策樹規則
- [x] 派發建議明確可操作
- [x] 拆分的所有子任務有清晰的依賴關係
- [x] 執行順序由 TDD 階段 + 架構層級確定

### 備註
- 安全相關功能需在 Phase 1 進行威脅建模
- JWT 實現需參考 OWASP 最佳實踐
- 應建立自動化安全測試
```

### 例 3：複雜需求的並行評估

```markdown
## 決策記錄

### 任務資訊
- **任務描述**: 評估 5 個 Feature 是否需要進行代碼風格統一重構
- **提交時間**: 2026-03-02 16:00:00
- **複雜度初步評估**: 中（評估工作）

### 複雜度評估
| 指標 | 數值 | 等級 |
|------|------|------|
| 需分析檔案數 | 50+ | 高 |
| 跨越架構層級數 | 3 | 中 |
| 涉及 Feature 數 | 5 | 中 |
| 決策維度 | 4（複雜性、收益、工期、風險） | 中 |
| **認知負擔指數** | 12 | **中偏高** |

### 決策路徑
1. 包含錯誤關鍵字？ 否 → 下一步
2. 包含不確定性詞彙？ 是（「是否需要」）→ 確認機制：確認優先級和評估範圍
3. 複雜需求（3+ 代理人）？ 是（SA、code-reviewer、cinnamon）→ 確認機制

### 派發決策
- **建議代理人**: parallel-evaluation（多視角掃描）
- **決策理由**: 認知負擔指數 12，複雜度高。需要多視角（架構一致性、程式碼品質、維護成本、工期）同時評估。使用 parallel-evaluation 不同視角並行掃描後，PM 統合結論
- **預期產出**: 評估報告 + 重構優先級建議

### 拆分建議（並行評估）
| 視角 | 負責代理人 | 評估標準 |
|------|-----------|--------|
| 架構一致性 | system-analyst | 各層設計風格是否統一 |
| 程式碼品質 | cinnamon-refactor-owl | 重複度、複雜度 |
| 維護成本 | system-engineer | 後續修改難度評估 |
| 工期影響 | PM | 重構對其他任務的影響 |

### 驗收標準
- [x] 決策邏輯符合決策樹規則
- [x] 已啟動 parallel-evaluation 多視角掃描
- [x] 各視角的評估獨立執行
- [x] 結論統合後由 PM 決定是否進行重構

### 備註
- 評估結果將作為後續版本規劃的輸入
- 若建議進行重構，應在 Phase 4 中實施
```

## 記錄最佳實踐

### DO（應該做）

- 在派發前記錄完整決策過程
- 對複雜度評估的每項指標給出具體數字
- 明確列出所有依賴關係
- 在備註中記錄特殊考量或已知的限制

### DON'T（不應該做）

- 模糊的複雜度評估（用「大概高」而非具體數字）
- 隱藏的依賴（「應該沒問題」）
- 不清晰的拆分原因（只說「拆分」而不說為什麼）
- 遺漏驗收標準
