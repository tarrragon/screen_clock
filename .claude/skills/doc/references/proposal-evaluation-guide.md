# 提案評估指南

> **核心目的**：在提案從 draft → discussing → confirmed 的過程中，嚴格審查提案的必要性和完整性。
> 審查不是橡皮圖章，而是主動挖掘用戶未思考的需求細節。

---

## 第一關：必要性評估（Should We Do This?）

在投入設計和實作前，先回答最根本的問題：**這個需求真的需要做嗎？**

### 1.1 替代方案排除矩陣

| 問題 | 若答「是」→ 處理方式 |
|------|---------------------|
| 能否用 exception/錯誤處理阻擋就好？ | 不開提案，直接在既有錯誤處理中加 guard clause |
| 能否用現有功能的組合達成？ | 不開提案，記錄使用方式到文件 |
| 能否用配置/參數調整解決？ | 不開提案，調整配置即可 |
| 是否只是「有了更好」但沒有實際使用場景？ | 否決，記錄理由到提案 withdrawn |
| 是否有明確的使用者痛點或業務驅動力？ | 若無 → 否決或降級 |

### 1.2 技術選型評估（涉及新技術/框架時）

若提案涉及引入新技術、框架或第三方依賴，額外評估：

| 面向 | 探問 |
|------|------|
| 成熟度 | 該技術是否穩定？社群活躍度？最近一次發布是什麼時候？ |
| 生態系統 | 套件/外掛支援是否足夠？遇到問題能否找到解答？ |
| 維護成本 | 團隊是否有能力維護？學習曲線多高？ |
| 鎖定風險 | 未來能否替換？是否會被單一廠商綁定？ |
| 授權合規 | 開源授權是否與專案相容（MIT/Apache/GPL）？ |
| 效能適配 | 是否滿足專案的效能需求？是否過度工程？ |

> 參考：Crosslake 六面向框架（成本、效能、安全、擴展、維護、策略對齊）

### 1.3 必要性判定結果

| 判定 | 後續動作 |
|------|---------|
| 必要 — 無替代方案可解決 | 進入第二關：完整性探問 |
| 可替代 — 有更簡單的解法 | 記錄替代方案，提案 withdrawn |
| 不明確 — 需要更多資訊 | 回問提案人，不進入下一關 |

## 第二關：UseCase 完整性探問（Is the Requirement Complete?）

> 用戶通常只有概念，沒想過完整流程。審查者的責任是**主動挖掘**用戶未考慮的面向。

### 2.1 通用探問清單

每個提案都必須經過以下維度的質問：

#### 時間維度

| 探問 | 目的 |
|------|------|
| 不同時間點的行為是否相同？ | 避免遺漏時間敏感邏輯 |
| 是否有頻率限制或冷卻時間？ | 避免遺漏防重複機制 |
| 資料是否有時效性？過期如何處理？ | 避免遺漏資料生命週期 |

#### 數量維度

| 探問 | 目的 |
|------|------|
| 0 筆時如何？1 筆時如何？100 筆時如何？ | 邊界條件 |
| 資料量超出預期時如何降級？ | 效能保護 |
| 是否需要分頁/批次處理？ | 大量資料策略 |

#### 權限維度

| 探問 | 目的 |
|------|------|
| 不同角色看到的資料是否相同？ | 權限控制 |
| 敏感資料是否需要遮蔽？ | 資料保護 |
| 未授權存取時如何處理？ | 安全邊界 |

#### 錯誤維度

| 探問 | 目的 |
|------|------|
| 網路斷線時如何？ | 離線策略 |
| 外部依賴異常時如何？ | 降級策略 |
| 資料格式不符預期時如何？ | 防禦性處理 |
| 操作中途失敗如何復原？ | 交易一致性 |

#### 使用者體驗維度

| 探問 | 目的 |
|------|------|
| 操作完成後使用者如何知道成功了？ | 回饋機制 |
| 等待時間長時使用者看到什麼？ | 載入狀態 |
| 操作可否取消或復原？ | 操作安全感 |

#### 資安維度

| 探問 | 目的 |
|------|------|
| 此功能是否需要認證/授權才能存取？ | 權限控制設計 |
| 是否需要密碼輸入或二次驗證？ | 身份確認機制 |
| 傳輸的資料是否需要加密（HTTPS/TLS）？ | 傳輸安全 |
| 儲存的資料是否需要加密（靜態加密）？ | 儲存安全 |
| 是否有暴力破解或重放攻擊的風險？需要防護？ | 攻擊防護 |
| 操作是否需要審計紀錄（誰在什麼時候做了什麼）？ | 稽核追蹤 |
| 第三方 API 金鑰/密鑰如何管理？是否暴露在前端？ | 機密管理 |

#### 可測試性維度

| 探問 | 目的 |
|------|------|
| 每個需求是否都有明確的「通過/失敗」判定標準？ | 避免模糊需求無法驗收 |
| 是否能用自動化測試驗證？還是必須人工確認？ | 測試策略規劃 |
| 需求的驗收條件是否量化（數字、時間、次數）？ | 避免主觀判斷 |

### 2.2 領域特定探問範例

> 以下為常見業務場景的具體探問，**審查時應根據提案內容選擇適用的探問**。

#### 範例 1：報表/匯出功能

- 不同日期區間的報表內容是否不同？
- 匯出的金額是否含稅？含手續費？
- 匯出格式是否需要支援多種？哪些欄位必填？
- 匯出的資料量上限？超過時的處理策略？
- 是否需要匯出歷史記錄或審計軌跡？

#### 範例 2：搜尋/篩選功能

- 搜尋是否區分全形半形？大小寫？
- 搜尋結果為零時顯示什麼？建議什麼？
- 篩選條件之間的邏輯是 AND 還是 OR？
- 搜尋效能在大量資料下是否可接受？
- 搜尋範圍包含哪些欄位？是否需要全文搜尋？

#### 範例 3：同步/跨設備功能

- 兩端同時修改同一筆資料時如何解決衝突？
- 同步失敗時本地資料如何保護？
- 離線期間的操作如何排隊？恢復後如何合併？
- 是否需要同步歷史/版本追蹤？
- 頻寬受限時的同步策略？

#### 範例 4：資料管理/儲存功能

- 資料備份頻率和恢復策略？
- 儲存空間不足時如何處理？
- 資料遷移（版本升級）的向後相容性？
- 敏感資料的加密和清除策略？
- 資料保留期限和自動清理策略？

#### 範例 5：使用者介面/互動功能

- 手機和桌面上的佈局是否不同？
- 載入中、空狀態、錯誤狀態各顯示什麼？
- 使用者連續快速點擊如何防護？
- 鍵盤操作和無障礙支援？
- 多語系/在地化需求？

#### 範例 6：資料遮蔽/隱私保護

- 前台和後台顯示的資料是否需要不同程度的遮蔽？
- 個人識別資訊（PII）如何處理？儲存時是否加密？
- 匯出/分享時敏感欄位是否自動遮蔽？
- 資料刪除請求如何處理？是否支援「被遺忘權」？
- 日誌/錯誤報告中是否會意外洩漏敏感資料？
- 第三方服務（分析、錯誤追蹤）是否會接收到敏感資料？

#### 範例 7：通知/提醒功能

- 通知頻率是否有上限？防止轟炸使用者？
- 是否需要靜音時段（如深夜不推送）？
- 通知管道有哪些（推播、Email、App 內、桌面）？是否可個別關閉？
- 通知內容是否需要因權限不同而差異化？
- 使用者未讀通知如何處理？是否有過期機制？
- 批量事件是否合併為單一通知（如「5 本書更新」而非 5 則通知）？

## 第三關：審查流程

### 3.1 審查方式

```
提案 draft
    │
    v
第一關：必要性評估
    │
    ├── 否決 → withdrawn（記錄理由）
    │
    v
第二關：完整性探問
    │
    ├── 不完整 → 回到提案人補充 → 再次審查
    │
    v
完整性確認
    │
    v
confirmed（開立 spec/usecase/ticket）
```

### 3.2 審查通過條件

提案必須同時滿足以下條件才能進入 confirmed：

| 條件 | 說明 |
|------|------|
| 必要性明確 | 無更簡單的替代方案 |
| 痛點真實 | 有具體的使用者場景或業務驅動 |
| UC 描述完整 | 經過完整性探問後，所有回答已納入 UC |
| 邊界條件已定義 | 0/1/N、正常/異常/邊界都有說明 |
| 範圍明確 | In Scope / Out of Scope 清楚 |

### 3.3 否決條件

符合以下任一條件即否決提案：

| 條件 | 理由 |
|------|------|
| 有更簡單的替代方案 | 過度工程（Over-Engineering） |
| 沒有明確的使用者痛點 | 需求虛構（Feature Creep） |
| 探問後發現範圍過大無法收斂 | 需拆分為多個獨立提案 |
| 提案人無法回答關鍵探問 | 需求不成熟，退回重新思考 |

### 3.4 反覆確認原則

> **審查不是一次性的問答，而是反覆確認直到看到需求全貌。**

| 原則 | 說明 |
|------|------|
| 不接受「之後再說」 | 邊界條件和異常處理必須在確認前定義 |
| 不接受「應該不會發生」 | 必須定義「萬一發生時」的處理方式 |
| 每輪探問後更新 UC | 將新發現的細節寫入 UC 文件 |
| 直到審查者無新問題可問 | 探問結束的標準 |

## 使用時機

| 時機 | 動作 |
|------|------|
| 新提案建立後 | 執行第一關 + 第二關 |
| 提案從 draft → discussing | 正式進入審查流程 |
| discussing → confirmed | 確認所有條件滿足 |
| 審查中發現新面向 | 更新提案的 UC 描述和範圍 |
