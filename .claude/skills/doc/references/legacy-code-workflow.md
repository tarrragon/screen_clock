# Legacy Code 接手處理工作流程

本文件定義接手既有程式碼（Legacy Code）時的標準化流程（前置步驟 + 步驟 0~6）。
適用於專案重啟、接手他人程式碼、或長期未維護的程式碼庫。

---

## 評估報告（必要）

每次執行本流程時，**必須**在 `docs/` 建立一份評估報告：

```bash
cp .claude/skills/doc/templates/legacy-assessment-report-template.md \
   docs/legacy-assessment-{date}.md
```

此報告是本流程的**核心交接文件**，功能：
- **進度追蹤**：記錄每個步驟的完成狀態、執行者、commit hash
- **產出物索引**：記錄每個步驟的產出物落地路徑
- **跨 session 延續**：中斷後可從報告恢復進度（不依賴對話 context）
- **最終交付**：流程完成後作為整體評估結論的載體

每完成一個步驟，**立即更新報告**對應區段。不要等到最後才回填。

---

## 追蹤整合（必要）

Legacy Code 流程的進度必須同時記錄在三個系統中。

> 通用同步原則：`.claude/references/multi-system-tracking-sync.md`

### Legacy 專屬要求

| 系統 | Legacy 場景 |
|------|------------|
| 核心文件 | `docs/legacy-assessment-{date}.md` — 評估報告 |
| Worklog | 新增「Legacy Code 流程」背景區段（啟動日期、報告路徑、步驟進度） |
| Ticket | 標題含「Legacy 步驟 N」，`why` 指向評估報告路徑 |

## 流程啟動時的回溯盤點（必要）

Legacy Code 流程可能在專案已有部分工作完成後才正式啟動。啟動時必須盤點既有工作，避免評估報告和實際進度脫節。

### 觸發條件

流程正式啟動時（建立評估報告時），若專案已存在以下任一情況：

| 情況 | 範例 |
|------|------|
| 已執行過測試驗證 | 區塊測試 Ticket 已完成 |
| 已有功能盤點 | spec/usecase 文件已存在 |
| 已修復部分測試 | 失敗測試已有修復 Ticket |

### 操作流程

1. 檢查 ticket 系統中已完成的相關 Ticket（測試執行、測試修復、功能盤點等）
2. 將既有工作的結果**回填到評估報告對應步驟**（標註為「回溯盤點」）
3. 在 worklog 中記錄回溯盤點事件
4. 調整 `current_step` 為實際進度

### 禁止行為

| 禁止行為 | 後果 |
|---------|------|
| 忽略既有工作，從零開始 | 重複勞動，浪費資源 |
| 既有工作不回填評估報告 | 報告與實際脫節，跨 session 恢復時誤判進度 |
| 回溯盤點不記錄在 worklog | 進度追蹤斷層 |

### 經驗來源

- **歷史案例** — W1 區塊測試在 Legacy 流程啟動前已完成逐 UC 驗證，但評估報告步驟 4 仍顯示「待執行」。跨 session 恢復時無法從任何追蹤文件判斷步驟 4 的實際進度

## 前置步驟：Worklog 結構初始化

### 目的

確保文件管理系統（worklog + ticket）可正常運作。Worklog 三層目錄結構是 ticket 系統的前提條件。

### 操作流程

執行 `/doc-flow worklog init` — 此指令會自動完成目錄建立、三層結構初始化和主 worklog 建立。

若專案存在舊版扁平結構（檔案直接散落在 `docs/work-logs/` 根目錄），需先手動遷移至三層結構再執行 init。

> 完整的 init 步驟和目錄規範：`.claude/skills/doc-flow/SKILL.md`（Worklog 管理章節）

### 為什麼是前置步驟

ticket 系統依賴 worklog 目錄結構來存放 ticket 檔案。如果目錄不存在，`/ticket create` 會失敗。所有後續步驟都會建立 ticket 來追蹤工作，因此必須在步驟 0 之前完成。

### 檢查清單

- [ ] `/doc-flow worklog init` 執行成功
- [ ] `/ticket create` 可正常建立 ticket

## 步驟 0：編譯與建置驗證

### 目的

確認專案能正常編譯和建置，排除基礎層級的阻礙。

### 操作流程

1. 執行依賴安裝指令，確認依賴安裝無誤
2. 執行建置指令，觀察編譯錯誤
3. 修復所有編譯阻礙問題
4. 確認建置能正常完成

> **常見指令參考**（依專案語言選用）：
> - Flutter/Dart：`flutter pub get` → `flutter build` / `dart analyze`
> - Node.js：`npm install` → `npm run build`
> - Go：`go mod download` → `go build ./...`
> - Python：`pip install -r requirements.txt` → `python -m compileall .`

### 常見問題類型

| 問題類型 | 範例 | 處理方式 |
|---------|------|---------|
| 過時引用 | 引用已刪除/重命名的檔案 | 更新引用路徑或移除無效引用 |
| 模組/套件宣告缺失 | 缺少必要的模組宣告或設定 | 補充缺失的宣告 |
| 建置腳本引用過時檔名 | 建置設定中的路徑與實際不符 | 更新建置設定 |
| 依賴版本衝突 | lock 檔案過時或依賴版本不相容 | 重新安裝依賴 |

### 經驗來源

- **v0.15.1**（Chrome Extension）— 檔案缺失、模組宣告缺失、建置驗證引用過時檔名

### 檢查清單

- [ ] Build 成功完成
- [ ] 無過時的檔案引用
- [ ] Build 產物與 manifest 一致

## 步驟 1：Spec 標準化與功能實作盤點

### 目的

建立程式碼與規格文件的對應關係，釐清「已做了什麼」和「還缺什麼」。

### 操作流程

#### 分支 A：有 spec 文件時

1. 將 spec 格式標準化，對齊 `docs/spec/{domain}/` 目錄結構
2. 逐一比對 spec 中的功能描述與程式碼實作
3. 標記每個功能的狀態：已實作 / 部分實作 / 未實作
4. 盤點常數和配置的完整性（是否有缺失的常數定義）

#### 分支 B：無 spec 文件時

1. 閱讀程式碼，理解各模組的職責和功能
2. 依 domain 劃分，為每個 domain 產生規格草案
3. 產出 subdomain 層級的功能清單
4. 標記可確認的功能和待確認的功能

### 盤點範圍

| 盤點項目 | 說明 | 產出 |
|---------|------|------|
| 功能實作狀態 | 每個 spec 功能是否已實作 | 功能狀態清單 |
| 常數完整性 | ErrorCodes、STORAGE_KEYS 等是否齊全 | 缺失常數清單 |
| Domain 覆蓋 | 所有 domain 是否都有 spec | Domain 覆蓋率 |
| 程式碼與 spec 偏差 | 實作與 spec 不一致之處 | 偏差清單 |

### 經驗來源

- **歷史案例** — ErrorCodes 系統分析，盤點錯誤碼定義與實際使用的差異
- **v0.15.2** — STORAGE_KEYS 發現 5 個常數缺失，導致儲存操作使用硬編碼字串

### 檢查清單

- [ ] 所有 spec 已遷移至標準結構
- [ ] 每個功能有明確的實作狀態標記
- [ ] 常數定義完整（無硬編碼替代品）
- [ ] 產出功能盤點報告

## 步驟 2：測試健康度評估

### 目的

評估測試套件的健康狀態，包含靜態審查和一次性執行分類，為步驟 4 的逐 UC 驗證提供優先級依據。

### 操作流程

本步驟分為兩個子任務，依序執行：

#### 2a. 靜態審查（純程式碼掃描，不執行測試）

使用 Grep/Glob 工具掃描，不執行任何測試：

1. **測試-UC 對應盤點**：掃描 `test/` 目錄結構，建立測試檔案與 UC 的對應表
2. **Skip 測試盤點**：搜尋該語言的 skip/pending 標記。常見範例：
   - Dart/Flutter：`skip: true`、`skip: '原因'`、`@Skip('原因')`
   - JavaScript：`test.skip`、`xit`、`xdescribe`、`.skip(`
   - Python：`@pytest.mark.skip`、`@unittest.skip`
   - Go：`t.Skip()`
3. **過時測試識別**：搜尋測試中引用的類別/函式名，確認是否仍存在於 `lib/`
4. **測試覆蓋差距**：比對步驟 1 的 UC 實作狀態與測試檔案數量

> **工具選擇**：優先使用 Grep（搜尋內容）和 Glob（搜尋檔案）。
> 大範圍掃描派發 Explore agent 處理，避免主線程 context 膨脹。

#### 2b. 失敗分類（一次性執行，摘要輸出）

執行一次完整測試，取得失敗清單並分類：

1. **執行方式**：使用摘要腳本或 `--reporter compact` + `tail` 限制輸出
2. **提取失敗清單**：僅取失敗測試的檔案路徑和錯誤類型
3. **失敗分類**：

| 分類 | 判斷標準 | 處理方式 |
|------|---------|---------|
| 編譯/靜態分析錯誤 | 編譯器或分析器報錯（未定義的類別、型別不符等） | 歸屬到對應的 in_progress Ticket |
| 邏輯失敗 | 斷言不符（Expected/Actual 不一致） | 步驟 4 逐 UC 修復 |
| 測試環境/配置問題 | timeout、mock/stub 配置缺失、DI 容器未設定 | 建立修復 Ticket |
| 測試框架特有失敗 | UI 測試建構失敗、非同步等待逾時等（見下方備註） | 修復測試 setup |
| 過時測試 | 引用已移除/重命名的 API、斷言已過時的行為 | 標記刪除或更新 |

> **測試框架特有失敗範例**（依專案語言選用）：
> - Flutter/Dart：`pumpWidget` 缺少 `MaterialApp` 包裹、`pumpAndSettle` 無限迴圈（動畫未結束）、`ProviderScope` 未配置、`MissingPluginException`（平台通道 mock 缺失）
> - React/Jest：`act()` 包裹缺失、`waitFor` 逾時、mock module 未設定
> - Go：`testify` mock 未滿足、`httptest` server 未關閉

4. **產出失敗分佈表**：按 UC/目錄分組統計

### 產出物

| 產出物 | 格式 | 用途 |
|--------|------|------|
| 測試-UC 對應表 | Markdown 表格 | 步驟 4 逐 UC 驗證的索引 |
| Skip 測試清單 | Markdown 表格 | 評估是否啟用或刪除 |
| 失敗分類表 | Markdown 表格（按 UC 分組） | 步驟 4 的優先級排序依據 |
| 測試健康度摘要 | 文字段落 | 步驟 3 提案輸入 |

### 經驗來源

- **v0.16.1** — 191 個 skip 測試逐批啟用（191 -> 147 -> 65），分批處理避免一次處理過多失敗
- **v0.31.1** — flutter test 完整輸出超過 4.6MB，必須使用摘要腳本或限制輸出

### 檢查清單

- [ ] 測試-UC 對應表已建立
- [ ] Skip 測試已列出並標記原因
- [ ] 失敗測試已分類（編譯/邏輯/環境/過時）
- [ ] 過時測試已識別
- [ ] 測試健康度摘要已撰寫

## 步驟 3：提案與標準流程

### 目的

將 spec 中未實現的部分或新需求，透過標準提案流程轉為可執行的 Ticket。

### 操作流程

1. 彙整步驟 1 和步驟 2 的發現
2. 釐清 spec 中未實現的功能，列為候選提案
3. 向開發者確認接下來要處理的優先順序
4. 使用 `/doc` Skill 建立提案文件
5. 提案走三關式審查（詳見 `proposal-evaluation-guide.md`）
6. 審查通過後，依序建立追蹤文件：
   - spec / usecase：`/doc create spec` 和 `/doc create usecase`
   - Ticket：`/ticket create --version {VERSION} --wave {N} --action "實作" --target "{目標}"`

### 經驗來源

- **v0.16.2** — 建立提案驅動需求追蹤機制（PROP-003~005、UseCase 遷移），確立從提案到 Ticket 的完整鏈路

### 檢查清單

- [ ] 步驟 1/2 的發現已彙整
- [ ] 提案已通過三關式審查
- [ ] spec 和 usecase 已建立
- [ ] Ticket 已開立

## 步驟 4：補充測試並逐 UC 驗證

### 目的

為缺少測試的程式碼補充基於 usecase 驗收標準的整合測試，逐個 UC 確認業務流程的完整性，採用「先局部再整體」策略。

> **核心原則**：全量測試的通過/失敗數只是數字，無法告訴你「哪些業務流程是斷的」。必須先逐 UC 補充整合測試確認端到端流程完整，才有意義跑全量測試。

### 策略：UC 整合測試優先於全量測試

```
錯誤順序：跑全量測試 → 看數字 → 逐個修 → 數字歸零
正確順序：逐 UC 補充整合測試 → 確認業務流程完整 → 最後跑全量測試
```

| 階段 | 做什麼 | 驗證什麼 |
|------|--------|---------|
| 先 | 逐 UC 依 AC 補充整合測試 | 業務流程是否端到端通過 |
| 中 | 修復被整合測試揭露的問題 | 程式碼邏輯和元件協作 |
| 後 | 全量測試 | 無交叉影響 |

### 操作流程

#### 4a. 逐 UC 補充整合測試

1. 閱讀 usecase 文件（步驟 3 產出），提取驗收條件（AC）
2. 檢查該 UC 是否已有整合測試，評估覆蓋率
3. 依 AC 設計缺失的整合測試（測試端到端業務流程，不是單一函式）
4. 建立 TDD Red Phase 測試（預期失敗）
5. 修復程式碼使測試通過（Green Phase）
6. 評估是否需要重構（Refactor Phase）

#### 4b. 逐 UC 驗證

1. 選擇一個 UseCase 的測試檔案
2. 僅執行該 UC 相關的測試（Unit + Widget + Integration）
3. 分析失敗原因，區分環境問題和程式碼問題
4. 修復後重新驗證
5. 確認通過後，**立即更新評估報告**的步驟 4 逐 UC 表格（UC 名稱、測試數、通過/失敗、結論、對應 Ticket）
6. 進入下一個 UC
7. 所有 UC 通過後，再執行一次完整測試確認無交叉影響

> **記錄時機**：每個 UC 驗證完成後立即記錄，不要等所有 UC 做完才回填。跨 session 恢復時，評估報告的逐 UC 表格就是進度的唯一 Source of Truth。

### 失敗分類與處理

```
執行該 UC 測試
    |
    +-- 全部通過 --> 下一個 UC
    |
    +-- 部分失敗
        |
        +-- 環境問題（mock 配置、依賴缺失）--> 修復環境，重新執行
        +-- 程式碼問題（邏輯錯誤）--> 修復程式碼，重新執行
        +-- 測試本身問題（過時斷言）--> 更新測試，重新執行
```

### 環境差異警示

| 層級 | 通過的意義 | 不代表 |
|------|----------|--------|
| 單元測試 | 函式邏輯正確 | 模組間協作正確 |
| 整合測試（mock） | 介面契約正確 | 實際平台行為正確 |
| E2E 測試 | 流程端到端正確 | 所有邊界情況已覆蓋 |
| 實機測試 | 目標平台可運作 | 其他平台也可運作 |

> **經驗教訓**：mock 環境通過的測試不代表實機能正常運作。平台特有問題（如瀏覽器的 var hoisting、行動裝置的權限限制、桌面應用的檔案系統差異）只在實機才會出現。

### 經驗來源

- **歷史案例** — 步驟 2 記載 259 個失敗，但後續修復已大幅降低失敗數。以「全量失敗數字」驅動修復策略是錯誤的，因為數字無法反映「哪些業務流程是斷的」。改為逐 UC 依 AC 補充整合測試後，才能真正確認系統健康度
- **歷史案例** — 多個測試差距修復 Ticket，基於 usecase AC 補充測試
- **v0.15.0** — 面對 512 個測試失敗（87.6% 失敗率），大規模失敗造成無法有效分析
- **v0.15.3~v0.16.0** — 測試全過但實機發現 popup var hoisting、SPA 渲染等待、選擇器失效等完全不同的問題

### 檢查清單

- [ ] 每個 AC 至少有一個測試
- [ ] 每個 UC 獨立執行測試通過
- [ ] 失敗原因已分類（環境/程式碼/測試）
- [ ] 最終完整測試通過
- [ ] 已識別需要實機驗證的項目

## 步驟 5：可觀測性審計

### 目的

確保系統在出錯時能被發現，消除靜默失敗和盲區。

### 操作流程

1. 檢查所有關鍵流程是否有日誌輸出
2. 偵測靜默失敗（流程執行但無日誌輸出的路徑）
3. 檢查 error handler 是否正確註冊（時序問題）
4. 識別平台特有陷阱
5. 驗證錯誤能被正確捕獲和報告

### 審計項目（通用）

| 審計項目 | 檢查內容 | 風險等級 |
|---------|---------|---------|
| 靜默失敗 | 例外處理區塊中只 return 不記錄 | 高 |
| Error handler 時序 | handler 是否在錯誤發生前已註冊 | 高 |
| 日誌覆蓋率 | 關鍵流程是否有啟動/完成/錯誤日誌 | 中 |
| 錯誤傳播 | 錯誤是否能從發生點傳播到使用者 | 中 |
| 非同步例外未捕獲 | 非同步操作的例外未被處理（見下方語言範例） | 高 |

### 語言特有的非同步例外審計

> 依專案語言選用對應的審計項目：

| 語言 | 審計項目 | 風險等級 |
|------|---------|---------|
| JavaScript | unhandled Promise rejection、async/await 未 try-catch、setTimeout 回調未捕獲 | 高 |
| Dart/Flutter | 未處理的 Future exception、Stream error 未監聽、Isolate 錯誤未傳播 | 高 |
| Go | goroutine panic 未 recover、channel 操作未處理 error | 高 |
| Python | 未 await 的 coroutine、threading 例外未捕獲 | 高 |

### 平台特有陷阱

> 依專案目標平台識別適用的陷阱。以下為常見範例：

| 平台 | 陷阱 | 說明 | 防護方式 |
|------|------|------|---------|
| Chrome Extension | var hoisting | 多 script 載入時變數衝突 | 使用 `let`/`const` |
| Chrome Extension | Service Worker 延遲註冊 | 啟動時錯誤未被捕獲 | 最早時機點註冊 handler |
| Flutter 行動端 | 權限對話框阻斷 | 權限拒絕後流程中斷 | 預先檢查權限狀態 |
| Flutter 行動端 | 平台通道例外 | Native 端拋出的例外未被 Dart 捕獲 | PlatformException 統一處理 |
| Web SPA | 渲染時序 | DOM 未就緒時操作元素 | 使用框架的生命週期鉤子 |

### 經驗來源

- **v0.15.3**（Chrome Extension）— 多 script 載入時變數衝突，導致 UI 無法正常顯示
- **v0.15.4**（Chrome Extension）— 提取流程靜默失敗、error handler 延遲註冊
- **v0.31.1**（Flutter）— export 模組 289 個靜態分析錯誤，部分為未處理的非同步例外
- **v0.31.1 W9**（Flutter）— 可觀測性審計發現大量靜默 catch、日誌工具不統一、缺乏分級輸出

### 常見可觀測性缺失模式

舊專案常見的可觀測性問題，依嚴重程度排序：

| 模式 | 症狀 | 影響 |
|------|------|------|
| 靜默 catch | `catch (e) {}` 或 `catch (e) { return null; }` | 錯誤完全不可見，問題無法定位 |
| 無全域錯誤處理 | 應用崩潰時無日誌輸出，只有系統層錯誤 | 生產環境問題無法重現 |
| 日誌工具不統一 | `print`、`debugPrint`、`console.log`、`Logger` 散落各處 | 無法統一過濾、搜尋或關閉日誌 |
| 無日誌分級 | 所有輸出使用同一級別（如全部 `print`） | 無法區分除錯、警告、錯誤，生產環境雜訊過大 |
| 可觀測性未隨規模擴展 | 初期 `print` 夠用，功能增加後無法追蹤跨模組流程 | 跨模組問題無法診斷，除錯成本隨規模指數增長 |

### 可觀測性成熟度矩陣

評估現有專案的可觀測性水準，決定改善策略：

| 等級 | 描述 | 特徵 | 行動 |
|------|------|------|------|
| L0 | 無可觀測性 | 無日誌、無錯誤處理、崩潰無記錄 | 從「最小可行」開始建立基礎 |
| L1 | 基礎 | 有 print/console.log、有部分 try-catch、無統一工具 | 統一日誌工具、補全域錯誤處理 |
| L2 | 結構化 | 統一日誌工具、全域錯誤處理、分級輸出 | 補關鍵流程三階段日誌（啟動/完成/錯誤） |
| L3 | 完整 | 三階段日誌覆蓋所有關鍵流程、可追蹤、可診斷 | 維護並持續改善 |

**判定方式**：逐項檢查特徵，符合該等級所有特徵才算達標。多數舊專案落在 L0-L1。

### 從零建立可觀測性（漸進式指引）

針對 L0-L1 等級的舊專案，分三步驟逐步提升：

**第一步：最小可行（L0 到 L1）**

目標：確保應用崩潰時有日誌可查。

- 建立全域錯誤處理（Flutter: `FlutterError.onError` + `PlatformDispatcher.onError`；JS: `window.onerror` + `unhandledrejection`）
- 選定統一日誌工具，取代散落的 `print`/`debugPrint`
- 所有現有 catch 區塊至少加上錯誤日誌（消除靜默 catch）

**第二步：關鍵路徑（L1 到 L2）**

目標：核心業務流程可追蹤。

- 識別 3-5 個核心業務流程（如登入、資料同步、付款）
- 每個流程加入三階段日誌：啟動（含輸入參數）、完成（含結果摘要）、錯誤（含例外詳情）
- 日誌分級：debug（開發用）、info（流程追蹤）、warning（異常但可恢復）、error（需要介入）

**第三步：全面覆蓋（L2 到 L3）**

目標：所有可能失敗的路徑都可診斷。

- 全量 catch 區塊審計：每個 catch 必須記錄或有正當理由不記錄（見 quality-baseline.md 規則 4）
- 非同步例外處理：依語言審計表（見上方「語言特有的非同步例外審計」）逐項確認
- 跨模組流程加入關聯 ID（correlation ID），支援分散式追蹤

> **經驗來源**：v0.31.1 W9 可觀測性審計 — 發現多數遺留模組停留在 L0-L1，靜默 catch 占例外處理區塊 60%+，全域錯誤處理存在時序問題。

### 檢查清單

- [ ] 無靜默失敗路徑（所有例外處理區塊有日誌）
- [ ] Error handler 在最早時機註冊
- [ ] 關鍵流程有啟動/完成/錯誤三階段日誌
- [ ] 平台特有陷阱已識別並加入防護
- [ ] 非同步例外已全部處理（依語言審計表逐項確認）

## 步驟 6：Roadmap 規劃與版本計畫

### 目的

將步驟 0-5 的所有發現，轉化為可執行的版本計畫。這是 legacy code 流程的最終產出——從「現在在哪」到「接下來要去哪」的銜接。

### 操作流程

1. **彙整所有缺口**：從步驟 1-5 的產出中，列出所有待處理項目：
   - 步驟 2：失敗測試、skip 測試
   - 步驟 3：已確認的提案
   - 步驟 4：尚未覆蓋的 UC 測試
   - 步驟 5：可觀測性不足、資安問題

2. **建立提案**（如尚未建立）：
   - 測試缺口 → 測試補強提案
   - 可觀測性不足 → 可觀測性提升提案
   - 資安問題 → 資安防護提案
   - 使用 `/doc` Skill 建立提案文件

3. **規劃版本**：依據缺口優先級，劃分中版本和小版本目標：

   | 優先級 | 版本範圍 | 目標 |
   |--------|---------|------|
   | P0 | 最近的中版本 | 失敗測試修復（品質底線） |
   | P1 | 下一個中版本 | 端到端功能驗證 |
   | P2 | 後續中版本 | 可觀測性/資安/打包 |

4. **建立版本 worklog**：
   - 每個中版本建立 `v{MAJOR}.{MINOR}-main.md`（目標、小版本規劃、成功標準）
   - 每個小版本建立 `v{VERSION}-main.md`（待執行項目、前情提要）

5. **建立父級 Ticket**（`/ticket create --version {VERSION} --wave 1 --action "實作" --target "{目標}"`）：
   - 每個小版本的核心任務建立父 Ticket
   - 實作細節在執行時再拆分子 Ticket
   - 父 Ticket 只定義 What 和 Why，不定義 How

6. **更新 todolist.yaml**（`/doc-flow todolist`）：將所有新版本加入追蹤

### 需求變更流程

Roadmap 建立後，若有新需求或設計變更，走相同模式：

```
新需求/設計變更
    |
    v
建立提案（/doc create proposal）
    |
    v
提案確認
    |
    v
決定插入哪個版本
    |
    +-- 插入既有版本 → 更新該版本 main worklog（`/doc-flow worklog`）+ 新增 Ticket（`/ticket create`）
    |
    +-- 需要新版本 → 建立新的中/小版本 worklog（`/doc-flow worklog init`）+ Ticket（`/ticket create`）
    |
    +-- 需要推遲既有計畫 → 使用 `/ticket migrate` 遷移 Ticket + 更新 worklog
```

### 追溯鏈

不需要獨立的 roadmap 文件。從提案到執行的追溯鏈：

```
提案 (proposals-tracking.yaml)
  → 版本計畫 (中版本 main worklog)
    → 小版本目標 (小版本 main worklog)
      → 執行 (Ticket)
```

### 經驗來源

- **v0.16.3**（Chrome Extension）— legacy code 流程完成後，規劃 v0.17~v0.19 路線圖，建立 7 個父級 Ticket 和 11 個 worklog

- [ ] 步驟 1-5 的所有缺口已列出
- [ ] 每個缺口有對應的提案或 Ticket
- [ ] 中版本和小版本目標已定義
- [ ] main worklog 已建立
- [ ] 父級 Ticket 已建立
- [ ] todolist.yaml 已更新

## 適用情境與排除情境

### 適用

| 情境 | 說明 |
|------|------|
| 專案重啟 | 長期暫停後重新啟動開發 |
| 接手他人程式碼 | 原作者已離開，需要接手維護 |
| 大版本升級前 | 升級前需要盤點現況 |
| 技術債清理 | 系統性處理累積的技術債務 |

### 排除

| 情境 | 說明 |
|------|------|
| 全新專案 | 從零開始的專案不需要 Legacy 流程 |
| 小幅修復 | 單一 bug fix 不需要完整流程 |
| 已有完整 spec 和測試的模組 | 直接走正常開發流程 |
