# Worth-It Filter 量化標準和案例

行動過濾的詳細判斷準則、量化指標和實際案例。本文件回答兩件事：(1) 「改善幅度 / 變更風險」如何量化判斷；(2) 決策矩陣的每一格如何映射到具體行動與追蹤方式。所有「延後執行」結果都必須綁定 Ticket，與 `.claude/rules/core/decision-trigger-binding.md` 規則 1 / 1.5（worklog / ticket 中的延後決策必須綁 ticket trigger）一致。

---

## 改善幅度定義

### 高改善幅度

觸發「必須執行」或「值得執行」。

| 類型 | 特徵 | 範例 |
|------|------|------|
| 安全/正確性 | 消除 bug 或安全漏洞 | 未驗證的使用者輸入、資源洩漏 |
| 顯著簡化 | 刪除 > 30% 的程式碼或消除整個抽象層 | 用既有 utility 替代 50 行自訂程式碼 |
| 消除重複 | 3+ 處重複合併為共用實作 | 跨模組的相同驗證邏輯提取為共用函式 |

### 中改善幅度

觸發「值得執行」（低風險）或「跳過」（高風險）。

| 類型 | 特徵 | 範例 |
|------|------|------|
| 可讀性 | 降低認知負擔指數 3+ 點 | 用 Guard Clause 替代深層巢狀 |
| 維護性 | 減少未來修改的檔案數 | 將散落的常數集中管理 |
| 一致性 | 統一 2+ 處不同的實作模式 | 錯誤處理模式統一 |

### 低改善幅度

延後執行，但仍必須建 Ticket 追蹤。

| 類型 | 特徵 | 範例 |
|------|------|------|
| 風格偏好 | 不影響可讀性的格式差異 | 變數命名的同義替換 |
| 微小改善 | 認知負擔改善 < 2 點 | 提取一個只用一次的區域變數 |
| 主觀判斷 | 無客觀衡量標準的「改善」 | 「我覺得這樣比較好」 |

## 變更風險定義

### 低變更風險

| 特徵 | 說明 |
|------|------|
| 修改範圍 | 1-2 個檔案 |
| 測試覆蓋 | 有完整測試覆蓋 |
| 向後相容 | 不影響公開介面 |
| 可回滾 | 容易還原 |

### 中變更風險

| 特徵 | 說明 |
|------|------|
| 修改範圍 | 3-5 個檔案 |
| 測試覆蓋 | 部分覆蓋，需補測試 |
| 向後相容 | 可能影響 1-2 個呼叫端 |
| 可回滾 | 需要一些工作才能還原 |

### 高變更風險

| 特徵 | 說明 |
|------|------|
| 修改範圍 | 6+ 個檔案或跨架構層 |
| 測試覆蓋 | 測試覆蓋不足 |
| 向後相容 | 影響多個模組的公開介面 |
| 可回滾 | 難以還原 |

## 決策矩陣（完整版）

**核心原則**：所有發現都必須追蹤。「執行決策」決定何時修，「追蹤」決定是否記錄——後者答案永遠是「是」。

**Why**：執行成本與追蹤成本是兩個獨立決策。將兩者綁在一起會讓「不值得立即執行」誤等同於「不值得記錄」，導致中低幅度發現系統性流失（quality-baseline 規則 5 的反模式）。

**Consequence**：缺漏追蹤的中幅度技術債會在版本疊加中累積，後期清理成本遠高於當下建 Ticket 的邊際成本。

**Action**：表格右欄的「建 Ticket」是延後決策的 trigger 綁定，依 `.claude/rules/core/decision-trigger-binding.md` 規則 1 / 1.5 執行；缺 Ticket 即等於無 trigger 延後（PC-093 反模式）。

| 改善幅度 | 變更風險 | 視角共識 | 執行決策 | 行動 | 追蹤 |
|---------|---------|---------|---------|------|------|
| 高 | 低 | 任何 | 立即執行 | 直接修復 | 建 Ticket（P1） |
| 高 | 中 | 2+ 視角 | 立即執行 | 直接修復 | 建 Ticket（P1） |
| 高 | 中 | 1 視角 | 立即執行 | 謹慎修復 | 建 Ticket（P1） |
| 高 | 高 | 任何 | 延後執行 | 規劃後修復 | 建 Ticket（P1） |
| 中 | 低 | 2+ 視角 | 立即執行 | 直接修復 | 建 Ticket（P2） |
| 中 | 低 | 1 視角 | 考慮 | 由 PM 判斷 | 建 Ticket（P2） |
| 中 | 中 | 任何 | 延後執行 | 後續版本處理 | 建 Ticket（P2） |
| 中 | 高 | 任何 | 延後執行 | 後續版本處理 | 建 Ticket（P2） |
| 低 | 任何 | 任何 | 延後執行 | 後續版本處理 | 建 Ticket（P2） |

**視角共識**：多個視角獨立發現同一問題時，可信度提升，允許更積極的行動。

## 情境特化規則

### 情境 A（程式碼審查）特化

- 風格偏好延後追蹤（由 lint 工具處理，但仍建 Ticket）
- 新增的重複程式碼（可用既有 utility）視為「高改善幅度」
- 測試程式碼的效率問題降級為「低改善幅度」

### 情境 B（重構評估）特化

- 重構成本估算：修改檔案數 x 平均每檔 15 分鐘
- 維護成本估算：每次修改需連帶修改的檔案數 x 預期修改頻率
- 只有重構成本 < 維護成本 x 3 時才值得

### 情境 F（結論審查）特化

- **閾值更嚴格**：結論錯誤的代價遠高於多花時間審查
- 任何視角發現「無佐證的結論」→ 強制回到分析
- 任何視角發現「遺漏的替代方案」→ 強制補充分析
- 只有三個視角都未發現問題時才通過

## 案例

### 案例 1: 程式碼審查 — 執行

**發現**: Reuse 視角發現新增的 `formatBookTitle()` 函式與 `lib/core/utils/string_utils.dart` 中的 `formatTitle()` 功能完全重複。

| 維度 | 判斷 |
|------|------|
| 改善幅度 | 高（消除重複，用既有 utility 替代 15 行新程式碼） |
| 變更風險 | 低（1 個檔案，只是替換呼叫） |
| 決策 | 執行 |

### 案例 2: 程式碼審查 — 跳過

**發現**: Quality 視角建議將 `isLoading` 變數重新命名為 `isDataLoading` 以更精確。

| 維度 | 判斷 |
|------|------|
| 改善幅度 | 低（風格偏好，上下文已經清楚） |
| 變更風險 | 低 |
| 決策 | 延後（建 Ticket P2 追蹤，不立即執行） |

### 案例 3: 重構評估 — 建 Ticket

**發現**: Coupling 視角發現 `BookDetailViewModel` 直接依賴 `BookRepository` 的具體實作，繞過了 UseCase 層。

| 維度 | 判斷 |
|------|------|
| 改善幅度 | 高（架構違規，影響測試可行性） |
| 變更風險 | 高（需要重新設計依賴注入，影響 4+ 檔案） |
| 決策 | 建 Ticket（高改善 + 高風險 = 規劃後執行） |

### 案例 4: 結論審查 — 回到分析

**發現**: Evidence 視角發現 incident-responder 結論「測試失敗是因為 API 回傳格式變更」，但未引用任何具體的 API response 或 diff。

| 維度 | 判斷 |
|------|------|
| 改善幅度 | N/A（情境 F 特殊規則：無佐證即強制回分析） |
| 決策 | 回到分析，要求提供具體的 API response 比對 |
