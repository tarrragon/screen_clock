# 視角 Agent 提示模板

各情境的 Agent prompt 模板，用於 Phase 2 並行派發。模板以「檢查焦點 + 編號清單」為骨架，PM 派發時替換 `{files}` / `{target}` / `{report}` / `{scope}` 佔位符即可。模板對應的視角設計（為何選此檢查項）見 `lens-configurations.md`。

> 視角定義和檢查項目: lens-configurations.md

---

## 情境 A: 程式碼變更審查

### Reuse 視角

```
檢查以下變更檔案，識別：
1. 可以用專案中既有 utility/helper 替代的新程式碼
2. 與其他模組功能重複的新增函式
3. 跨檔案的 copy-paste 片段

只報告有具體替代方案的發現，不報告風格偏好。
變更檔案：{files}
```

### Quality 視角

```
審查以下變更檔案的程式碼品質，聚焦：
1. 冗餘狀態管理
2. 抽象洩漏
3. 不必要的間接層
4. 命名與行為不一致

每個發現標註改善幅度（高/中/低）和變更風險（高/中/低）。
變更檔案：{files}
```

### Efficiency 視角

```
審查以下變更檔案的執行效率，聚焦：
1. 重複計算或可快取的值
2. 可並行但被序列化的操作
3. 未清理的資源（Stream、Timer、Subscription）
4. 不必要的 I/O

只報告有明確改善方案的效率問題。
變更檔案：{files}
```

## 情境 B: 重構決策評估

### Redundancy 視角

```
掃描以下模組/目錄，識別冗餘設計：
1. 功能重疊的模組或類別
2. 重複的資料轉換邏輯
3. 冗餘的中間層
4. 可合併的相似介面

標的：{target}
```

### Coupling 視角

```
分析以下模組的耦合度：
1. 跨層依賴（違反 Clean Architecture 方向）
2. 循環依賴
3. 高扇出模組（依賴 > 5 個其他模組）
4. 修改擴散風險（改一處要動幾處？）

標的：{target}
```

### Complexity 視角

```
識別以下模組的複雜度熱點：
1. 認知負擔指數高的函式（變數 + 分支 + 巢狀 + 依賴 > 10）
2. 過度工程的抽象（只有一個實作的介面、未使用的泛型參數）
3. 可簡化的實作（用 3 行能做的事寫了 30 行）

標的：{target}
```

## 情境 C: 架構方案評估

### basil 視角

```
審查以下架構評估文件的書面文字品質，聚焦：
1. 每個方案說明段落是否包含 Why（為何選此方案）、Consequence（不選的後果）、Action（下一步決策動作）三層
2. 各方案比較論述的資訊優先序（核心原則應先於具體示例）
3. 方案缺點描述是否有正向錨點（緩解方式或替代選項）
4. 是否有「通常來說」「按理應」「理想情況下」等條件模糊表述

報告格式：Critical（影響可執行性）/ Warning（降低清晰度）/ Info；每項附修正方向。
標的：{files}
```

## 情境 D: 功能設計評估

### basil 視角

```
審查以下功能設計規格的書面文字品質，聚焦：
1. 功能需求與設計說明是否包含 Why（為何需要此功能）、Consequence（不實作的影響）、Action（實作方向或下一步）三層
2. 功能描述的資訊優先序（核心功能定義應先於具體範例）
3. 功能限制或排除條件是否有正向說明（取代方案或可接受的簡化版本）
4. 是否有「希望讀者理解」「一般情況下」等未明示適用條件的描述

報告格式：Critical / Warning / Info；每項附修正方向。
標的：{files}
```

---

## 情境 F: 結論/報告審查

### Evidence 視角

```
審查以下分析報告的證據品質：
1. 每個結論是否引用了具體程式碼或日誌
2. 因果推斷是否有直接證據
3. 是否有未標註的推測

報告內容：{report}
```

### basil 視角

```
審查以下分析結論或報告的書面文字品質，聚焦：
1. 各章節（Problem Analysis / Solution / 結論段落）是否包含 Why（問題成因）、Consequence（不處理的後果）、Action（具體建議動作）三層
2. 根因與結論呈現的資訊優先序（結論應先於佐證細節）
3. 修復建議是否有正向說明（明示應做什麼，而非只列風險與禁止）
4. 是否有「理應如此」「推測可能」「一般情況」等未明示的結論性句型

報告格式：Critical / Warning / Info；每項附修正方向。
報告內容：{report}
```

## 情境 G: 系統設計評估

### Consistency 視角

```
檢查以下系統設計文件的一致性：
1. 交叉引用是否雙向完整
2. 術語在所有相關文件中是否一致
3. 不同文件的相同規則是否有矛盾
4. 相關文件列表是否遺漏

標的：{files}
相關文件範圍：{scope}
```

### Completeness 視角

```
檢查以下系統設計文件的完整性：
1. 覆蓋範圍是否有缺口（未處理的情境）
2. 必要的文件段落是否齊全
3. 正交工具是否有清楚的區別說明
4. 從各個入口點是否都能發現此文件

標的：{files}
```

### Cognitive Load 視角

```
評估以下系統設計文件的認知負擔：
1. 主文件大小是否合理（< 5,000 字）
2. 資訊分層是否符合 Progressive Disclosure
3. 是否有不必要的重複（應改為引用）
4. 表格和清單是否便於快速理解

標的：{files}
```

### basil 視角

```
審查以下系統設計文件的書面文字品質，聚焦：
1. 設計原則與規則段落是否包含 Why（為何有此設計）、Consequence（違反的後果）、Action（使用者應採取的動作）三層
2. 規則和規範的資訊優先序（核心原則應先於具體示例，示例先於邊界注意事項）
3. 設計限制或約束是否有正向說明（明示符合設計的做法，而非只列禁止行為）
4. 是否有「按理應」「假設讀者會注意到」「自然而然」等責任外推的表述

報告格式：Critical / Warning / Info；每項附修正方向。
標的：{files}
```

## 常駐委員：linux

```
以 Good Taste 原則審查以下變更：
1. 設計複雜度是否匹配問題複雜度？能更簡單嗎？
2. 這是在解決真實問題還是假想問題？
3. 有沒有特例可以透過更好的設計消除？
4. 會破壞任何現有功能或體驗嗎？

品質評分：Good taste / Acceptable / Garbage
如果 Garbage，直接指出最致命的問題。

標的：{files}
```
