# 對稱性偏誤檢查（Symmetry Bias Check）

雙向操作的設計判斷工具，用於 Stage 1（方案收集）階段避免「對稱性誘惑」誤導方案分布。

---

## 核心主張

雙向操作（push/pull、lock/unlock、create/destroy、publish/unpublish、subscribe/unsubscribe 等）的防呆設計不該因「兩端在語意上對稱」而強制方案等價，應依**副作用嚴重度**與**可逆性**差異化設計。

**Why**：人類認知傾向尋找對稱以降低工作記憶負擔，但工程系統的對稱性是描述語言的副產品，不是設計目標。同一對雙向操作的實際副作用嚴重度往往不對稱——寫公開狀態 vs 本地操作、永久污染歷史 vs 可備份還原、影響下游 vs 僅影響自己。

**Consequence**：
- **過度防護輕量端**：給副作用小的一端加重型守衛，提升複雜度與誤判風險（state 過時可能漏處理真實更新）
- **欠保護重型端**：誤以為兩端方案等價時，會把較弱方案套到重型端，留下真實風險（例如只防護 pull 但 push 仍會空 commit 進公開歷史）

**Action**：在 Stage 1 列方案時，對所有「雙向操作」題目先執行下方判別流程，再進入評分階段。

---

## 判別流程

| 步驟 | 動作 | 通過條件 |
|------|------|---------|
| 1. 列副作用清單 | 兩端各列出實際副作用（影響範圍 / 持久性 / 可見性 / 對下游影響） | 兩端清單已分別寫出 |
| 2. 量化可逆性差異 | 對「副作用發生後復原成本」打分（低 / 中 / 高） | 差異 ≥ 2 級即視為不對稱 |
| 3. 比對影響範圍 | 一端影響公開狀態 / 另一端僅影響本地？ | 答「是」即視為不對稱 |
| 4. 套用設計判斷 | 不對稱 → 方案差異化；對稱 → 可採對稱方案 | 結論寫入 Stage 1 方案來源欄位 |

**Why**：可逆性與影響範圍是兩個獨立軸——可逆性決定「出錯後能否復原」，影響範圍決定「出錯後傷及誰」。任一軸不對稱即足以拒絕對稱方案。

**Consequence**：跳過此流程直接套對稱方案的後果是規格層判斷錯誤，後續 Stage 3 評分階段再修正成本更高（已寫出多套方案，需回頭撤回部分方案）。

**Action**：判別結論納入 Stage 1 方案描述的「來源」欄位，便於 Stage 3 評分時對照。

---

## 反模式信號

對稱方案的提案理由若出現以下表述，必須先回答「副作用嚴重度差異」問題，再決定是否採用：

| 反模式信號 | 替代問句 |
|----------|---------|
| 「另一端做了所以這端也應該做」 | 兩端副作用嚴重度與可逆性是否同級？ |
| 「為了完整性」 | 完整性是指 API 表面對稱，還是內部行為對稱？兩者是否必須一致？ |
| 「對稱才容易記」 | 增加的記憶成本是否能用文件 / 命名替代？對稱錯誤的真實成本是多少？ |
| 「邊界一致」 | 邊界一致是用戶可見需求，還是設計者方便？用戶實際操作流程是否對稱？ |

**Why**：對稱性誘惑常以「美學論證」掩蓋工程判斷缺失。把美學論證翻譯回工程語言（副作用、可逆性、影響範圍）後，方案差異化的必要性即顯現。

**Consequence**：未經翻譯就接受美學論證會把對稱性升格為設計約束，限制後續方案空間，且後人接手時難以理解為何兩端用相同方案處理本質不同的問題。

**Action**：在 Stage 1 收方案時，對每個對稱方案套用上方替代問句檢驗；通不過即標記「對稱性誘惑」，提醒 Stage 3 評分時審慎對待。通過驗證的對稱方案應在 Stage 1「來源」欄位寫「已驗證副作用同級，採對稱方案合理」，讓後續評估能直接信任該方案。

## 案例：sync-push vs sync-pull 防呆設計

### 案例背景

`/sync-push`（推 .claude/ 變更到獨立 repo）與 `/sync-pull`（從獨立 repo 拉最新 .claude/）為一對雙向操作。初始設計直覺是「兩端做相同防呆」（hash early-exit + commit 後驗證），但實際分析顯示兩端不對稱。

### 副作用對照

| 評估維度 | sync-push | sync-pull |
|---------|----------|----------|
| 主要副作用 | 推 commit 到公開 repo（污染歷史） | 覆寫本地 .claude/（已有 backup 機制） |
| 可逆性 | 低（git 歷史永久，需 force-push 才能撤回） | 高（`_sync_with_backup` 自動建立 backup_dir） |
| 影響下游 | 是（公開 repo 訂閱者拉到空 commit） | 無（僅當前本地工作區） |
| 偵測時機 | 提交後（已污染） | 覆寫前（可預檢） |

副作用差異既已量化，方案應反映該差異而非追求對稱。

### 差異化方案

| 端點 | 方案 | 理由 |
|------|------|------|
| sync-push | hash + commit 訊息雙重訊號 early-exit | 副作用持久且影響下游，需多層 early-exit 防止空 commit |
| sync-pull | 順手清理累積 backup_dir | 副作用本地且可逆，無需重型 early-exit，附加價值在維護整潔 |

### 反映的判別結論

| 判別步驟 | 結論 |
|---------|------|
| 副作用清單 | push：公開 + 持久；pull：本地 + 可備份還原 |
| 可逆性差異 | 低 vs 高，差異 ≥ 2 級 → 不對稱 |
| 影響範圍 | 公開 vs 本地 → 不對稱 |
| 設計判斷 | 方案差異化 |

**Why**：本案例顯示「對稱命名（push/pull）+ 對稱目錄結構」會誘導設計者直覺採對稱方案，但 git 歷史的不可逆性 vs backup 機制的可逆性差異使對稱方案失當。

**Consequence**：若兩端皆採 `_sync_with_backup` 風格的可逆設計，push 端不會有 backup（公開 repo 不適用本地 backup），等同無防護；若兩端皆採重型 early-exit，pull 端增加複雜度但無實際收益。

**Action**：本案例可作為新一對雙向操作設計時的對照範本——先列出與 push/pull 對應的副作用維度，再判斷是否屬於相同模式。

## 與 Stage 1 的整合

本檢查在 Stage 1（方案收集）階段執行，**先於** Stage 2 評估維度定義。

| 階段 | 整合動作 |
|------|---------|
| Stage 1 收方案前 | 判別題目是否屬「雙向操作」；是 → 執行判別流程；否 → 跳過 |
| Stage 1 寫方案 | 對稱方案在「來源」欄位標註「對稱性誘惑風險」，並寫明已執行檢查的結論 |
| Stage 2 評估維度 | 將「可逆性」「副作用範圍」「對下游影響」納入維度，權重對應實際風險 |
| Stage 3 評分 | 對稱方案在「可逆性」維度給較低分（若已通過判別流程仍提案，需在評分理由說明） |

**Why**：把判別流程綁定到 Stage 1 確保所有後續評估都建立在已校準的方案集上，避免 Stage 5 才發現對稱方案不適用，重做評估。

**Consequence**：跳過此整合會讓對稱性誘惑通過 Stage 2-5 而未被偵測，最終決策可能採用實際不適用的對稱方案，落地後才發現副作用失控。

**Action**：Stage 1 收方案時若題目含雙向操作關鍵字（push/pull、lock/unlock、create/destroy 等），主動讀取本 reference 並完成判別流程後再進入 Stage 2。

---

## 適用範圍

| 適用 | 不適用 |
|------|--------|
| 雙向 CLI 命令（push/pull、import/export） | 單向操作（read-only、write-only） |
| Lifecycle 配對（create/destroy、open/close） | 純查詢類 API |
| 訊號配對（subscribe/unsubscribe、register/unregister） | 函式內部對稱結構（如 try/finally）—屬程式語法非設計題 |
| API 對稱方法（lock/unlock、acquire/release） | UI 對稱版面—屬視覺設計領域 |
