# 混合處理模式指南

大規模 i18n 技術債務的有效處理策略。

## 核心理念

**不同複雜度使用不同處理手段，最大化效率和品質**

## 複雜度分級

| 等級 | 描述 | 處理策略 | 範例 |
|------|------|---------|------|
| **A** | 簡單 UI 文字 | 腳本自動替換 | `Text('確認')` → `Text(l10n.confirm)` |
| **B** | 需評估內容 | 多代理人平行處理 | ViewModel 層訊息 |
| **C** | 需架構重構 | 建立獨立 Ticket | Domain 層硬編碼 |
| **D** | 測試檔案 | 逐案評估 | 測試輸出訊息 vs 驗證條件 |

## 等級判斷標準

### 等級 A - 簡單 UI 文字

判斷條件：
- 在 `presentation/` 或 `core/ui/` 目錄
- 直接在 Widget 中顯示的文字
- 已有對應 ARB 鍵值或可直接新增

處理方式：
- 使用批量替換腳本（見下文）
- 特別適合 Flutter 專案中的 UI 文字替換

### 等級 B - 需評估內容

判斷條件：
- ViewModel 層的訊息
- 需要判斷是 UI 文字還是業務邏輯
- 可能需要返回代碼讓 UI 處理

處理方式：
- 多代理人平行評估
- 決定各項目的最終歸屬（A/C/保留）
- 每個項目需要判斷：是否應該硬編碼 → 應該在哪層 → 應該怎麼翻譯

### 等級 C - 需架構重構

判斷條件：
- Domain/Model 層的中文字串
- 違反分層 i18n 規範
- 需要改為代碼/枚舉，由 UI 層翻譯

處理方式：
- 建立獨立 Atomic Ticket
- 按依賴順序排程
- 優先級：高（架構違規）

### 等級 D - 測試檔案

判斷條件：
- 測試輸出訊息：可使用 MockAppLocalizations
- 測試驗證條件：可能需要改用代碼比對
- 不應一概跳過

處理方式：
- 逐案評估
- 記錄評估結果（為什麼保留/為什麼替換）

## 批量替換腳本

針對等級 A，使用批量替換腳本：

```bash
# Dry run（預設，不實際修改）
uv run scripts/i18n_batch_replace.py --target lib/presentation

# 實際替換
uv run scripts/i18n_batch_replace.py --target lib/presentation --apply

# 生成報告
uv run scripts/i18n_batch_replace.py --target lib/presentation --report
```

腳本會自動處理常見映射：
- 按鈕文字：確認、取消、提交、搜尋、下一步、返回...
- 狀態文字：載入中、搜尋中、同步中、已完成、失敗...
- 連線狀態：已連線、離線中、未知狀態、連接中...
- 提示訊息：成功、錯誤、警告、資訊...

## 混合處理工作流程

完整的技術債務處理流程：

```text
1. 分析階段
   └─ uv run scripts/i18n_hardcode_checker.py --json
   └─ 將結果分類為 A/B/C/D 等級
   └─ 生成分類清單

2. 等級 A 處理
   ├─ 新增 ARB 鍵值（根據批量替換腳本映射）
   ├─ flutter gen-l10n
   └─ uv run scripts/i18n_batch_replace.py --apply

3. 等級 B 處理
   ├─ 多代理人平行評估
   └─ 決定各項目歸屬（A/C/保留）

4. 等級 C 處理
   ├─ 建立 Atomic Tickets
   └─ 按依賴順序排程

5. 等級 D 處理
   ├─ 逐案評估
   └─ 不跳過，記錄評估結果
```

## Ticket 派生策略

當發現等級 C（需架構重構）項目時：

| 情況 | 處理方式 | 優先級 |
|------|---------|--------|
| Domain 層包含 UI 文字 | 建立重構 Ticket | 高 |
| 默認參數硬編碼 | 建立策略 Ticket | 中 |
| 跨多檔案的模式 | 建立批量處理 Ticket | 高 |

## 實戰案例

### v0.25.1 i18n 技術債務處理

| 等級 | 數量 | 處理方式 | 對應 Ticket |
|------|------|---------|-----------|
| A | 35 處 | 腳本替換 | 範例案例 A |
| B | 90 處 | 待評估 | 範例案例 A |
| C | 947 處 | 架構重構 | 範例案例 B |
| C | 3 處 | 默認參數 | 範例案例 C |

**學習要點**：
- 架構層面的硬編碼（等級 C）佔比最高
- 批量替換腳本大幅降低等級 A 的處理成本
- 等級 B 的評估成本不可避免，但可並行處理

## 注意事項

1. **不要急於替換**：先分析、分類、再處理，避免無謂修改
2. **保留評估紀錄**：等級 B/D 項目需記錄評估結果，便於後續維護
3. **尊重架構分層**：Domain 層不應直接包含 UI 文字，違反 MVVM 分層
4. **測試檔案特殊**：測試中的硬編碼可能是有意為之（測試環境初始化、Mock 資料），需逐案分析
5. **保持增量更新**：大規模修改時分批進行，避免 diff 過大導致審查困難
