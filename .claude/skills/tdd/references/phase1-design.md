# Phase 1：功能設計指引

## Phase 目標

在寫任何程式碼之前，定義功能的完整規格、API 介面、驗收標準，並使用 SOLID 原則評估是否需要拆分。

**核心問題**：「這個功能的邊界在哪裡？如何與系統其他部分互動？」

---

## 產出（Layer 1）

Phase 1 完成後，必須產出：

- [ ] 功能規格文件（輸入/輸出/流程）
- [ ] API 介面定義（函式簽名、資料結構）
- [ ] 驗收標準清單（可量化、可驗證）
- [ ] SOLID 拆分分析報告（如需要拆分）
- [ ] 行為場景（Given-When-Then 格式）

---

## 功能規格設計

### 輸入定義

明確每個輸入的類型和限制：

```
輸入規格
├── 參數名稱
├── 資料類型
├── 是否必填
├── 合法範圍/約束
└── 預設值（如有）
```

### 輸出定義

明確每個輸出的類型和行為：

```
輸出規格
├── 成功情況：回傳值類型和結構
├── 失敗情況：錯誤類型和訊息
└── 副作用：狀態變更、事件觸發等
```

### 操作流程

定義正常操作流程（主要路徑）：

```
操作序列
1. 接收輸入
2. 驗證輸入合法性
3. 執行業務邏輯
4. 回傳結果
```

定義異常處理流程：

| 異常情況 | 處理方式 | 回傳訊息 |
|---------|---------|---------|
| 必填欄位為空 | 拒絕並回傳錯誤 | {錯誤描述} |
| 格式不合法 | 拒絕並回傳錯誤 | {錯誤描述} |
| 資源不存在 | 回傳 Not Found | {錯誤描述} |

## API 介面設計

### 介面定義原則

| 原則 | 說明 |
|------|------|
| 清晰命名 | 函式名稱以動詞開頭，描述「做什麼」 |
| 最小介面 | 只暴露必要的方法，遵循 ISP |
| 高內聚低耦合 | 相關功能集中，不相關功能分離 |
| 依賴抽象 | 依賴介面而非具體實作，遵循 DIP |

### 函式簽名格式

```
函式名稱(
  參數名: 類型,
  參數名: 類型
) -> 回傳類型 | 錯誤類型
```

### 資料結構定義

```
資料結構名稱 {
  欄位名: 類型  // 說明
  欄位名: 類型  // 說明（可選）
}
```

## SOLID 原則拆分分析

使用 `/tdd split` 執行完整 SOLID 分析。以下為快速參考：

### SOLID 檢查清單

| 原則 | 核心問題 | 拆分信號 |
|------|---------|---------|
| SRP | 這個功能有幾個獨立的修改原因？ | 需要用「和」連接描述 |
| OCP | 未來擴展需要修改現有程式碼嗎？ | 需要 switch/case 或 if/else 分岐 |
| LSP | 繼承關係正確嗎？ | 子類別有需要覆寫的方法 |
| ISP | 介面是否強迫實作不需要的方法？ | 實作類別有空方法 |
| DIP | 依賴的是抽象還是具體實作？ | 直接 import 具體類別 |

### 拆分決策流程

```
SOLID 分析結果
    |
    +-- 多個獨立職責 → 拆分為多個任務
    |
    +-- 需要介面抽象 → 新增介面定義任務
    |
    +-- 有依賴關係 → 確定執行順序
    |
    v
產出拆分清單（見 SKILL.md `/tdd split` 格式）
```

### 拆分範例（書籍搜尋功能）

**原始需求**：「實作書籍搜尋功能」

**SOLID 分析**：

| 原則 | 結果 |
|------|------|
| SRP | 識別 3 個職責：查詢建立、搜尋執行、結果呈現 |
| OCP | 搜尋來源可能多樣（本地、API），需要抽象 |
| DIP | UseCase 不應依賴具體 Repository |

**拆分結果**：

| 編號 | 描述 | 架構層 | 執行批次 | 依賴 |
|------|------|--------|---------|------|
| A | SearchQuery 值物件 | Domain | 第一批 | 無 |
| B | SearchResult 實體 | Domain | 第一批 | 無 |
| C | ISearchRepository 介面 | Domain | 第一批 | 無 |
| D | SearchBooksUseCase | Application | 第二批 | A、B、C |
| E | SearchRepository 實作 | Infrastructure | 第二批 | C |
| F | SearchWidget | Presentation | 第三批 | D |

## 行為場景（Given-When-Then）

Phase 1 必須提取行為場景，作為 Phase 2 測試設計的輸入。

### 格式

```
場景 {編號}：{業務流程名稱} - {情境類型}
Given: {前置條件}
When: {使用者操作}
Then: {預期結果}
```

### 情境類型覆蓋

| 類型 | 說明 | 必要性 |
|------|------|--------|
| 正常流程 | 標準成功路徑 | 必要 |
| 異常流程 | 輸入不合法、資源不存在等 | 必要 |
| 邊界條件 | 極端輸入、系統限制 | 必要 |

### 範例（書籍搜尋）

```
場景 1：書籍搜尋 - 成功
Given: 使用者已登入，系統有 10 本書
When: 使用者輸入關鍵字「Dart」
Then: 回傳包含「Dart」的書籍列表，按相關性排序

場景 2：書籍搜尋 - 空關鍵字
Given: 使用者已登入
When: 使用者提交空白搜尋關鍵字
Then: 回傳錯誤訊息「搜尋關鍵字不能為空」

場景 3：書籍搜尋 - 無結果
Given: 使用者已登入，系統無匹配書籍
When: 使用者輸入關鍵字「XYZ_不存在」
Then: 回傳空列表，顯示「找不到相關書籍」
```

## 驗收標準定義

驗收標準必須**可量化、可驗證**，禁止使用模糊描述。

| 禁止（抽象） | 正確（可驗證） |
|------------|-------------|
| 「提升使用者體驗」 | 「搜尋結果在 500ms 內回傳」 |
| 「提高穩定性」 | 「輸入空關鍵字回傳錯誤碼 400」 |
| 「支援多種輸入」 | 「支援中英文關鍵字，長度 1-100 字元」 |

### 驗收標準格式

```markdown
驗收標準：
- [ ] {具體可驗證的條件 1}
- [ ] {具體可驗證的條件 2}
- [ ] {具體可驗證的條件 3}
```
