# Phase 0：架構前置審查規則與指引

## Phase 目標

在開始功能設計前，確認新功能與現有系統的一致性，避免重複造輪子，確保需求完整。

**核心問題**：「這個功能是否適合現在的系統架構？有沒有類似功能已存在？」

---

## 產出（Layer 1）

Phase 0 完成後，必須產出：

- [ ] 系統一致性評估報告
- [ ] 重複實作檢查結果
- [ ] 需求文件完整性確認
- [ ] 審查結論（通過 / 需補充 / 不建議）

---

## 觸發條件

以下任一條件成立，必須執行 Phase 0：

| 條件 | 說明 |
|------|------|
| 新功能需求 | 非 Bug 修復、非小型調整 |
| 架構變更 | 影響系統架構層級 |
| 影響 3+ 模組 | 跨多個系統模組 |

以下情況可豁免 Phase 0：

| 豁免條件 | 說明 |
|---------|------|
| 小型 Bug 修復 | 影響範圍明確，不影響架構 |
| 純文件更新 | 不涉及程式碼 |
| 單一函式修改 | 影響範圍限於一個函式 |

---

## 審查項目

### 1. 系統一致性檢查

確認功能設計與現有架構一致：

| 檢查項目 | 評估方式 |
|---------|---------|
| 架構層次符合 | 功能歸屬的架構層（Domain / Application / Infrastructure / Presentation）是否正確 |
| 設計原則遵循 | 是否符合系統採用的設計原則（如 Clean Architecture、DDD） |
| 命名規範一致 | 實體、介面、服務的命名是否與現有命名慣例一致 |
| 依賴方向正確 | 依賴關係是否遵循架構規定的方向 |

**評估方式**：

- 閱讀現有相似功能的實作
- 確認架構層次邊界
- 確認命名慣例

### 2. 重複實作檢查

確認不重複造輪子：

| 檢查項目 | 問題 |
|---------|------|
| 類似功能 | 現有系統中是否有類似功能？ |
| 可重用元件 | 是否有現有元件可以直接使用或稍作修改後使用？ |
| 介面契約 | 是否已有適合的介面定義可以實作？ |

**發現可重用元件時**：
1. 評估現有元件是否能直接滿足需求
2. 評估修改現有元件 vs 新建元件的成本
3. 記錄決策理由

### 3. 需求文件完整性

確認需求有足夠資訊推進設計：

| 文件項目 | 必要性 | 說明 |
|---------|--------|------|
| 功能需求描述 | 必要 | 功能的核心目標 |
| 使用情境 | 必要 | 誰在什麼情況下使用 |
| 驗收標準 | 必要 | 如何判斷功能完成 |
| 邊界條件 | 建議 | 異常情況和限制 |
| 效能要求 | 視情況 | 有效能需求時必須明確 |

## 審查結論格式

```markdown
## Phase 0 架構審查報告

### 功能描述
{功能一句話描述}

### 系統一致性
- 架構層次：{評估結果}
- 設計原則：{評估結果}
- 命名規範：{評估結果}
- 依賴方向：{評估結果}

### 重複實作檢查
- 類似功能：{有/無，若有則描述}
- 可重用元件：{有/無，若有則描述}
- 決策：{新建 / 重用現有 / 修改現有}

### 需求完整性
- 功能需求：{完整/不完整}
- 使用情境：{明確/模糊}
- 驗收標準：{明確/缺少}
- 缺失項目：{列出缺失}

### 結論
**判定**：通過 / 需補充 / 不建議

**理由**：{說明}

**下一步**：
- 通過 -> 進入 Phase 1 功能設計
- 需補充 -> {列出需補充的項目}，完成後重新審查
- 不建議 -> {說明理由}，升級決策
```

## 轉換條件

### 進入 Phase 0 的條件

- 功能需求已描述（至少一句話）
- 觸發條件評估已完成

### 退出 Phase 0 的條件（進入 Phase 1）

- [ ] 系統一致性確認完成
- [ ] 無重複實作（或已決定重用策略）
- [ ] 需求文件完整性確認
- [ ] 審查結論為「通過」

> **框架整合**：審查由架構分析專業角色執行，結論寫入 Ticket Execution Log。審查通過後執行 `/ticket track complete` 並推進到 Phase 1。

---

## Layer 1/2 功能點分類

Phase 0 的功能點在 Layer 1（通用）和 Layer 2（框架特定）之間的歸屬：

| 功能點 | Layer | 歸屬理由 |
|--------|-------|---------|
| 審查觸發條件（新功能、架構變更、3+ 模組影響） | Layer 1 | 純邏輯判斷，不依賴系統 |
| 系統一致性檢查（架構一致、設計原則、命名衝突） | Layer 1 | 通用設計審查原則 |
| 重複實作檢查（類似功能、可重用元件） | Layer 1 | 通用架構分析方法論 |
| 需求文件完整性（需求書、用例、驗收標準） | Layer 1 | 通用需求工程實踐 |
| SA 派發和審核流程 | Layer 2 | 依賴代理人定義和 PM 決策樹 |

> Layer 1/2 設計原則的完整說明見 SKILL.md「Layer 1 / Layer 2 設計原則」章節。
