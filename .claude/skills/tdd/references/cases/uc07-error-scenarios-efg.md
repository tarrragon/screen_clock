# UC-07 錯誤場景 E/F/G：遷移失敗、Tag 不一致、配額不足

> **背景**：tag-based model 新增了三種過去不存在的錯誤場景 — 遷移中途斷電（E）、tag 引用指向已刪除的 category（F）、Chrome Storage 配額不足（G）。規格只定義了正常流程，這三種失敗場景沒有恢復策略。
> 對應 Phase：Phase 1（功能規格設計）
> 對應 Decision Question：Q7（錯誤恢復策略與使用者介面一致性）

---

## 問題概述

Tag-based Model（PROP-007）新增三個錯誤場景（E/F/G），涉及資料遷移、Tag 引用一致性和 Chrome Storage 配額。三個場景各有不同的恢復策略，但共用相似的使用者介面模式（訊息 + 操作選項），規格需確保策略選擇一致且使用者體驗連貫。

---

## 三個具體場景

### 場景 E：v1 → v2 資料遷移失敗

**觸發條件**：從舊版升級時，`isNew`/`isFinished` 布林組合無法對映到 6 種 `readingStatus`。

**邊界條件**：

| 條件 | 預期行為 | 測試重點 |
|------|---------|---------|
| isNew=undefined, isFinished=undefined | 對映為 unread | 缺失值預設處理 |
| isNew=true, isFinished=true | 對映為 finished（異常組合） | 衝突值優先級 |
| 遷移中途 Chrome 被關閉 | 已遷移的保留，未遷移的下次啟動繼續 | 冪等性 |
| 500 本書遷移，第 250 本失敗 | 前 249 本已遷移，第 250 本保留原始格式 | 部分成功隔離 |
| progress 欄位缺失或非數字 | 預設為 0，狀態為 unread | 髒資料容錯 |

**使用者訊息設計**：

- 進度顯示：「遷移進度：{已完成}/{總計}」
- 失敗訊息：「{N} 本書籍資料遷移失敗，已保留原始格式」
- 操作選項：「重試遷移」「手動修正」「略過」

**恢復策略**：孤立策略（成功的保留，失敗的隔離）

### 場景 F：Tag 資料不一致

**觸發條件**：tag 的 `categoryId` 引用了不存在的 `tag_category`，或 tag 資料結構損壞。

**邊界條件**：

| 條件 | 預期行為 | 測試重點 |
|------|---------|---------|
| tag.categoryId 指向不存在的 category | 自動重建 category（name 標記為「[已恢復]」） | 自動修復正確性 |
| 多個 tag 指向同一個不存在的 category | 只重建一次 category，所有 tag 指向同一個 | 去重修復 |
| tag 的必要欄位（id/name）缺失 | 移除該 tag，從所有書籍的 tagIds 中清除引用 | 級聯清理 |
| book.tagIds 包含不存在的 tag ID | 從 tagIds 中移除無效 ID | 單向清理 |
| 修復過程中 Chrome Storage 寫入失敗 | 回滾修復，保持修復前狀態 | 修復原子性 |

**修復報告結構**：

```
修復報告：
- 重建 {N} 個遺失的 tag_category
- 移除 {M} 個無效 tag
- 清理 {K} 本書籍中的無效 tag 引用
```

**恢復策略**：自動修復 + 報告（非靜默修復）

### 場景 G：Chrome Storage 配額不足

**觸發條件**：tag 資料導致 Chrome Storage 接近 5MB（chrome.storage.local）限制。

**邊界條件**：

| 條件 | 預期行為 | 測試重點 |
|------|---------|---------|
| 使用量 > 80% | 顯示警告，不阻止操作 | 閾值觸發 |
| 使用量 > 95% | 顯示強烈警告 + 清理建議 | 緊急狀態 UI |
| 新增 tag 會超過限制 | 阻止操作，顯示「空間不足」 | 前置檢查 |
| tag 數量 > 1000 | 建議清理未使用的 tag | 數量閾值 |
| 清理後仍不足 | 建議匯出資料後重置 | 最終手段 |

**清理建議優先級**：

1. 移除未被任何書籍引用的 tag（孤立 tag）
2. 移除空的 tag_category（無 tag 的類別）
3. 匯出資料後清空並重新匯入

**恢復策略**：預防中止策略（前置配額檢查）

## 跨場景一致性要求

### 使用者訊息格式

三個場景的訊息需遵循統一格式：

| 元素 | 格式 | 範例 |
|------|------|------|
| 標題 | 動詞 + 名詞 | 「遷移失敗」「資料修復」「空間不足」 |
| 描述 | 具體數字 + 影響說明 | 「3 本書籍遷移失敗」 |
| 操作按鈕 | 動詞片語，最多 3 個選項 | 「重試」「手動修正」「略過」 |

### 手動操作選項

每個場景都必須提供至少一個手動操作選項，使用者不應被迫接受自動處理結果。

## 測試設計建議

### 場景 E 測試重點

1. 所有 v1 布林組合到 v2 狀態的對映覆蓋（6 種有效組合 + 缺失值）
2. 部分成功場景的隔離正確性
3. 遷移冪等性（重複執行不會破壞已遷移的資料）
4. 進度回報的準確性

### 場景 F 測試重點

1. 自動修復的完整性（修復後 tag 結構完全一致）
2. 修復報告的準確性（數字與實際操作一致）
3. 修復失敗的回滾正確性
4. 連環不一致（category 不存在 + tag 引用該 category + book 引用該 tag）

### 場景 G 測試重點

1. 閾值觸發的準確性（80%/95% 界線）
2. 前置檢查的時機（必須在寫入之前）
3. 清理建議的排序正確性
4. 孤立 tag 識別的準確性

## 風險等級與影響範圍

| 場景 | 風險等級 | 影響範圍 | 資料損失風險 |
|------|---------|---------|------------|
| E（遷移失敗） | 高 | 所有升級使用者 | 中（保留原始資料可降低） |
| F（Tag 不一致） | 中 | tag 功能使用者 | 低（自動修復 + 報告） |
| G（配額不足） | 中 | 重度 tag 使用者 | 低（前置阻止寫入） |

---

## 教訓

1. **錯誤場景的使用者介面不是附屬品** -- 訊息格式、操作選項、進度顯示都需要在規格階段定義，否則實作者各自發明不同的 UI 模式
2. **自動修復必須搭配報告** -- 場景 F 的自動修復如果靜默執行，使用者不知道發生了什麼，無法驗證修復結果
3. **配額檢查必須是前置檢查** -- 場景 G 重複了 storage-write-failure-handling 的教訓：先寫後查會導致不一致狀態
