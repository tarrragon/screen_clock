# Phase 0 SA 審查：Tag-based Book Model 重構（PROP-007）

> **背景**：從分類系統（category + parentId）改為標籤系統（tag-based）是跨 7 個 Use Case 的大型重構。Phase 0 SA 審查需要確認：哪些模組受影響、是否有重複造輪子、既有測試是否需要全面重寫。
> 對應 Phase：Phase 0（架構前置審查）

---

## 問題概述

PROP-007 是大規模的資料模型重構，影響 Book Schema（v2 -> v3）、新增 tag_categories/tags Storage、引入 6 種閱讀狀態、影響全部 7 個 UC。Phase 0 審查發現四項需補充的設計缺口，若跳過審查直接進入實作，預估會產生 100+ 行重複程式碼和資料遷移失敗風險。

---

## 變更範圍

| 項目 | 影響 |
|------|------|
| Book Schema | v2 -> v3（新增 tagIds、readingStatus） |
| 新增 Storage | tag_categories、tags 兩個獨立儲存 |
| 閱讀狀態 | 6 種（未讀/在讀/已讀/想讀/暫停/棄讀） |
| 影響 UC | 全部 7 個 UC 需要適配 |
| 跨越子域 | 3 個（Book、Tag、Reading Status） |

---

## 審查發現

### 1. 系統一致性：架構符合但跨域範圍大

架構設計符合 DDD + Event-Driven 模式，但跨越 3 個子域的變更增加了一致性風險。任何一個子域的 Schema 變更都需要其他兩個子域同步調整。

### 2. 重複實作風險

| 領域 | 相似度 | 風險說明 |
|------|--------|---------|
| 驗證框架 | 70% | BookSchemaV3 和 TagSchema 的驗證邏輯結構高度相似 |
| 狀態轉換 | 90% | readingStatus 的 v2->v3 映射和 v1->v2 映射邏輯幾乎相同 |
| ID 生成 | 60% | tag_categories 和 tags 的 ID 生成策略類似 |

### 3. 需求完整性缺失

四項關鍵缺失：

| 缺失項目 | 影響 |
|---------|------|
| v1->v2 欄位映射表 | 開發者需自行推斷欄位對應，容易遺漏 |
| ID 唯一性保證 | 批量建立場景下 Date.now() 碰撞 |
| 狀態轉換邊界條件 | 無效狀態值的處理未定義 |
| 可觀測性需求 | 遷移失敗時的日誌和告警策略未定義 |

### 4. 審查結論

需補充四項才可進入 Phase 1：

1. **共用策略**：定義驗證框架、狀態轉換、ID 生成的共用方案，避免重複實作
2. **映射表**：建立完整的 v2->v3 欄位映射表，包含型別變化和遷移邏輯
3. **邊界條件**：定義狀態轉換、ID 碰撞、配額超限的錯誤處理
4. **非功能需求**：定義遷移的效能目標、回滾策略、可觀測性要求

---

## 根因

PROP-007 的提案文件聚焦於功能設計（新的資料模型和標籤系統），未從系統工程角度評估與現有模組的交互影響。提案越大，Phase 0 審查越重要。

---

## 若跳過 Phase 0 的預估影響

| 影響 | 估計 |
|------|------|
| 重複程式碼 | 100+ 行（驗證框架 + 狀態轉換） |
| 資料遷移失敗風險 | 高（缺少映射表和回滾策略） |
| ID 碰撞問題 | 批量操作場景必然觸發 |
| 返工成本 | Phase 3 實作後才發現，需回到 Phase 1 重新設計 |

---

## 教訓

1. **變更跨越 3+ 子域時，Phase 0 不可省略** -- 跨域變更的一致性風險無法在單一模組的 Phase 1 中發現
2. **提案越大，非功能需求缺失越嚴重** -- 大型提案的撰寫者傾向聚焦功能設計，容易忽略遷移、效能、回滾
3. **重複實作的識別只能在系統層級完成** -- 模組級的 Phase 1 設計無法看到跨模組的相似度
