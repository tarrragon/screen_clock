# 批次書籍寫入失敗：孤立策略、回滾與預防中止的權衡

> **背景**：批量寫入 100+ 本書到 Chrome Storage 時，第 50 本失敗了 — 是回滾前 49 本、忽略失敗繼續、還是中止整個操作？4 個模組各自面臨相同問題，但規格未定義失敗策略。
> 對應 Phase：Phase 1（功能規格設計）
> 對應 Decision Question：Q6b（寫入失敗處理策略）

---

## 問題概述

四個模組各自處理寫入失敗，但策略不一致且測試不足。規格只定義了成功路徑的行為，未明確失敗場景的處理策略選擇依據。

---

## 四個具體問題

### 1. 批次匯入只測成功路徑

ValidationBatchProcessor 處理部分失敗時，即使有項目驗證失敗，整體 status 仍標記為 `completed`。呼叫端無法區分「全部成功」和「部分成功」。

### 2. 配額檢查在儲存之後

ChromeStorageAdapter 先執行寫入（可能 7MB），寫入完成後才檢查配額。若超過 Chrome Storage 的 10MB 限制，資料已經寫入但狀態不一致。

### 3. 備份無自動回滾

SchemaMigrationService 在遷移前建立備份，但遷移失敗時不會自動回滾到備份版本。需要手動觸發回滾，失敗窗口期內資料處於不一致狀態。

### 4. 部分同步失敗未測試

SynchronizationOrchestrator 的 `partialSync` 在部分裝置同步失敗時設定失敗標記，但這條路徑沒有對應的測試案例。

---

## 三種策略對照

| 策略 | 適用場景 | 核心邏輯 | 本專案對應 |
|------|---------|---------|-----------|
| 回滾 | 資料一致性最高優先 | 失敗時還原到操作前狀態 | SchemaMigrationService（遷移） |
| 孤立 | 容忍部分失敗 | 成功的保留，失敗的列出 | ValidationBatchProcessor（匯入） |
| 預防中止 | 無法回滾的環境 | 操作前檢查，不滿足則不執行 | ChromeStorageAdapter（配額） |

### 策略選擇判斷流程

```
寫入失敗發生
  |
  +-- 操作可回滾？
  |     +-- 是 -> 資料一致性要求高？
  |     |         +-- 是 -> 回滾策略
  |     |         +-- 否 -> 孤立策略（列出失敗項）
  |     +-- 否 -> 預防中止策略（前置檢查）
```

---

## 根因

規格定義了各模組的 CRUD 操作，但未回答「寫入失敗時怎麼辦」。四個模組的開發者各自決定策略，導致：

- 策略選擇不一致（有的回滾、有的靜默忽略）
- 失敗路徑測試缺失（只測成功路徑）
- 前置檢查順序錯誤（先寫後查）

---

## Decision Question Q6b 回答

> 「策略取決於場景。資料一致性高（遷移）-> 回滾；容忍部分失敗（匯入）-> 孤立但必須列出失敗項；無法回滾（Chrome Storage）-> 前置檢查。」

---

## 教訓

1. **規格必須為每個寫入操作定義失敗策略** -- 不能讓實作者自行決定，否則同一系統內策略不一致
2. **前置檢查的順序是設計決策** -- 配額檢查必須在寫入之前，這屬於規格層級的流程定義
3. **「部分成功」是一個獨立狀態** -- 不能歸類為 completed 或 failed，必須有明確的 status 值和後續處理流程
