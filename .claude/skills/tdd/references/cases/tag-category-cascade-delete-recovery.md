# tag_category 級聯刪除失敗恢復：三層級聯的不一致狀態與回滾

> **背景**：使用者刪除一個 tag_category 時，需級聯刪除底下的 tags，並移除所有書籍對這些 tags 的引用。三層級聯操作中，若第二層成功、第三層失敗，書籍仍引用著已不存在的 tags — 規格未定義這種不一致狀態的恢復策略。
> 對應 Phase：Phase 1（功能規格設計）
> 對應 Decision Question：Q10（級聯刪除的原子性保證與失敗恢復）

---

## 問題概述

刪除 `tag_category` 時，需要執行三層級聯操作：(1) 刪除 category 本身、(2) 刪除該 category 下的所有 tag、(3) 從所有書籍的 `tagIds` 中移除被刪除的 tag ID。Chrome Storage 不支援 transaction，任何一步失敗都會產生不一致狀態。

---

## 三層級聯操作

```
步驟 1: 刪除 tag_category
    ↓
步驟 2: 刪除該 category 下的所有 tag（可能多個）
    ↓
步驟 3: 從所有書籍的 tagIds 中移除已刪除的 tag ID（可能影響多本書）
```

### 操作規模估算

| 步驟 | 操作對象 | 最大規模 |
|------|---------|---------|
| 步驟 1 | 1 個 category | 1 次 Storage 寫入 |
| 步驟 2 | N 個 tag（預估 5-50 個/category） | 1 次 Storage 寫入（整個 tags 陣列） |
| 步驟 3 | M 本書（預估 10-500 本引用） | 1 次 Storage 寫入（整個 books 陣列） |

---

## 失敗場景分析

### 場景 A：步驟 1 成功，步驟 2 失敗

| 狀態 | 結果 |
|------|------|
| tag_categories | category 已刪除 |
| tags | tag 仍存在，但 categoryId 指向不存在的 category |
| books.tagIds | 仍引用存在的 tag |

**不一致類型**：孤立 tag（tag 的 category 不存在）。
**使用者感知**：tag 仍可使用但無法按 category 分類瀏覽。

### 場景 B：步驟 1-2 成功，步驟 3 失敗

| 狀態 | 結果 |
|------|------|
| tag_categories | category 已刪除 |
| tags | tag 已刪除 |
| books.tagIds | 仍引用已刪除的 tag ID |

**不一致類型**：幽靈引用（book.tagIds 包含不存在的 tag ID）。
**使用者感知**：書籍的 tag 欄位顯示異常（引用了不存在的 tag）。

### 場景 C：步驟 2 部分成功

步驟 2 需要刪除多個 tag，若中途 Storage 寫入失敗：

| 狀態 | 結果 |
|------|------|
| tags | 部分 tag 已刪除，部分仍存在 |

**不一致類型**：部分級聯（同一 category 下的 tag 狀態不一致）。

### 場景 D：步驟 3 部分成功

步驟 3 需要更新多本書的 tagIds：

| 狀態 | 結果 |
|------|------|
| books | 部分書籍的 tagIds 已清理，部分仍包含已刪除的 tag ID |

**不一致類型**：部分清理（不同書籍的 tagIds 一致性不同）。

---

## 恢復策略選項

### 選項 1：事務日誌 + 回滾（推薦）

**流程**：

```
1. 記錄事務日誌：即將執行的操作清單
   { action: 'delete_category', categoryId: 'cat_001',
     affectedTags: ['tag_001', 'tag_002'],
     affectedBooks: ['book_001', 'book_002'] }

2. 執行步驟 1 → 2 → 3

3. 任何步驟失敗：
   a. 讀取事務日誌
   b. 回滾已完成的步驟（反向操作）
   c. 清除事務日誌

4. 全部成功：清除事務日誌
```

**邊界條件**：

| 條件 | 處理 |
|------|------|
| 事務日誌寫入失敗 | 不開始級聯操作 |
| 回滾過程中再次失敗 | 保留事務日誌，下次啟動時重試回滾 |
| 事務日誌存在但操作已完成 | 冪等檢查，跳過已完成的回滾步驟 |

### 選項 2：標記刪除 + 背景清理

**流程**：

```
1. 將 category 標記為 deleted（而非實際刪除）
2. 背景任務逐步清理 tag 和 book 引用
3. 全部清理完成後，實際刪除 category
```

**優點**：無需回滾，最終一致性。
**缺點**：使用者可能看到「已刪除」的 category 暫時仍存在。

### 選項 3：先收集再批次刪除

**流程**：

```
1. 收集所有需要修改的資料（category、tags、affected books）
2. 在記憶體中完成所有修改
3. 一次性寫入所有變更（盡量減少 Storage 寫入次數）
```

**優點**：最少的中間不一致狀態。
**缺點**：大量資料時記憶體佔用高；單次寫入仍可能失敗。

## 孤立資料處理

### 孤立 tag 的定義與處理

| 類型 | 定義 | 處理方式 |
|------|------|---------|
| 孤立 tag（無 category） | tag.categoryId 指向不存在的 category | 自動修復：建立預設 category 或移入「未分類」 |
| 孤立 tag（無引用） | tag 存在但無任何 book 引用 | 保留（使用者可能稍後使用）或建議清理 |
| 幽靈引用 | book.tagIds 包含不存在的 tag ID | 自動清理：從 tagIds 中移除 |

### 一致性檢查

啟動時或定期執行：

```
對所有 tag：
  if tag.categoryId 不在 tag_categories 中 → 記錄為孤立 tag

對所有 book.tagIds 中的每個 tagId：
  if tagId 不在 tags 中 → 記錄為幽靈引用

產出一致性報告
```

## 測試設計建議

### 正常路徑

1. 刪除含 1 個 tag 的 category → 驗證三層全部清理
2. 刪除含 10 個 tag 的 category → 驗證批次 tag 刪除
3. 刪除被 50 本書引用的 tag 的 category → 驗證大量 book 更新

### 失敗路徑

| 測試 | 注入的失敗 | 預期結果 |
|------|-----------|---------|
| 步驟 2 失敗 | mock Storage.set 在 tags 寫入時拋錯 | category 回滾（重新出現），tags 不變 |
| 步驟 3 失敗 | mock Storage.set 在 books 寫入時拋錯 | category 回滾，tags 回滾，books 不變 |
| 回滾失敗 | mock Storage.set 在回滾時拋錯 | 事務日誌保留，下次啟動時重試 |

### 冪等性測試

1. 同一 category 連續刪除兩次 → 第二次不報錯（已經不存在）
2. 事務日誌存在時重新執行 → 先完成回滾再執行新操作
3. 刪除空 category（無 tag） → 只需步驟 1，不觸發步驟 2-3

### 邊界條件

| 條件 | 測試重點 |
|------|---------|
| category 下 tag 數量為 0 | 步驟 2 跳過，直接步驟 3（books 可能仍有殘留引用） |
| 被刪除的 tag 同時屬於其他 category | 不應發生（tag 只屬於一個 category），驗證約束 |
| 刪除操作進行時另一個寫入同時發生 | Chrome Storage 無鎖機制，可能產生覆蓋 |
| category 被刪除後立即建立同名新 category | 新 category 應有新 ID，不復用舊 ID |

## 與 storage-write-failure-handling 的關聯

本案例的核心問題（級聯寫入的部分失敗）是 storage-write-failure-handling 案例的延伸。差異在於：

| 面向 | storage-write-failure-handling | 本案例 |
|------|-------------------------------|--------|
| 失敗粒度 | 單一操作 | 三步驟級聯 |
| 回滾複雜度 | 簡單（還原一個 key） | 複雜（多個 key 的反向操作） |
| 一致性要求 | 單一 Storage key | 跨三個 Storage key |
| 策略選擇 | 按場景選策略 | 必須用回滾或標記刪除 |

## 風險等級與影響範圍

| 風險點 | 風險等級 | 影響範圍 |
|-------|---------|---------|
| 步驟 2 失敗未回滾步驟 1 | 高 | 產生孤立 tag，tag 功能異常 |
| 步驟 3 失敗未回滾步驟 1-2 | 高 | 產生幽靈引用，書籍顯示異常 |
| 事務日誌寫入失敗 | 中 | 無法啟動級聯操作（安全失敗） |
| 回滾過程再次失敗 | 高 | 資料不一致且無法自動恢復 |

---

## 教訓

1. **Chrome Storage 不是資料庫** -- 沒有 transaction 支援，級聯操作必須自行實作事務日誌或標記刪除機制
2. **三層級聯 = 三個失敗點** -- 每增加一層，失敗場景組合數倍增，回滾邏輯也相應複雜化
3. **孤立資料是必然產物** -- 在無 transaction 的環境中，不一致狀態不可完全避免，必須設計定期的一致性檢查和修復機制
4. **回滾的回滾是最難的問題** -- 事務日誌本身也可能寫入失敗，需要定義「最壞情況下的行為」
