# Chrome Storage API 效能延遲與批量操作瓶頸

> **背景**：規格只寫「操作要快」「使用者體驗流暢」，未定義具體數值。實作者各自決定批次大小和延遲間隔，導致不同模組的效能假設不一致，整合時才發現 Chrome Storage API 的讀寫延遲遠超預期。
> 對應 Phase：Phase 1（功能規格設計）
> 對應 Decision Question：Q6a（效能瓶頸與最慢操作路徑）

---

## 問題概述

功能規格定義效能需求時，只描述「操作要快」、「使用者體驗流暢」，未定義具體數值目標。實作者自行決定批次大小和延遲間隔，導致不同模組的效能假設不一致，直到整合測試才發現瓶頸。

---

## 效能目標與實際結果

規格最終補充的量化目標和實際測量結果：

| 操作 | 目標 | 實際結果 | 狀態 |
|------|------|---------|------|
| Popup 載入 | 1000ms | < 500ms | 通過 |
| Overview 頁面 | 未定義 | 1.8s | 無基準可判斷 |
| 搜尋 | 200ms | 180ms | 通過 |
| 匯出 | 5000ms | 3.2s | 通過 |

### 提取操作（依書籍數量）

| 規模 | 目標 | 說明 |
|------|------|------|
| 小量（5 本） | < 5s | 基本使用場景 |
| 中量（50 本） | < 15s | 一般使用者書庫 |
| 大量（100+ 本） | < 30s | 重度使用者，最慢路徑 |

---

## 瓶頸分析

大量書籍提取（100+ 本）是最慢操作，瓶頸在兩個環節：

### 1. Chrome Storage 序列化延遲

Chrome Storage API 每次讀寫都需要 JSON 序列化/反序列化。批量操作的關鍵參數：

| 參數 | 值 | 用途 |
|------|-----|------|
| 批次大小 | 1000-10000 項目 | 單次 API 呼叫處理量 |
| 批次間延遲 | 50ms | 避免阻塞 UI 主線程 |
| 重試延遲 | 1000ms 指數退避 | API 暫時性失敗 |
| 操作超時 | 30s | 防止無限等待 |

### 2. 記憶體限制

| 限制 | 值 |
|------|-----|
| 記憶體上限 | 50MB |
| 警告閾值 | 40MB |

超過警告閾值時需觸發分頁載入或資料壓縮，否則 Chrome 可能終止 Service Worker。

---

## 根因

規格只描述了功能正確性（「能提取書籍資料」），未定義非功能性需求的具體數值。開發者缺乏效能基準，無法在實作階段做出正確的批次大小和延遲間隔決策。

---

## Decision Question Q6a 回答

> 「最慢操作是大量書籍提取（100+ 本，< 30s），瓶頸在 Chrome Storage 序列化和網路 I/O。批次操作需設定合理批次大小（1000）和延遲間隔（50ms），避免阻塞 UI。」

---

## 教訓

1. **規格應定義效能目標數值，而非只描述「要快」** -- 沒有數值就沒有驗收標準，實作者無從判斷是否達標
2. **批次操作的參數（大小、延遲、超時）屬於規格範疇** -- 這些不是實作細節，而是影響使用者體驗的設計決策
3. **記憶體限制是 Chrome Extension 的硬性約束** -- 必須在規格階段就定義記憶體預算和超限策略
