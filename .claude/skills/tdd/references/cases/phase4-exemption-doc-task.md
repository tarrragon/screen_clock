# Phase 4 豁免判斷：DOC 任務豁免條件的邊界確認

> **背景**：一個標記為 DOC 的任務（TDD references 目錄整理），實際修改了 23 個檔案、淨減少 1382 行程式碼。依現有豁免條件「純文件任務」可跳過 Phase 4 — 但這個「文件任務」的影響範圍比很多 IMP 任務都大。
> 對應 Phase：Phase 4（重構評估）

---

## 問題概述

某 DOC 任務標籤為純文件任務，依豁免條件應可跳過 Phase 4a/4c。但實際修改了 23 個檔案、淨減少 1382 行，涉及結構重組和檔案遷移。DOC 標籤不等於小範圍，當前豁免條件存在漏洞。

---

## 實際影響

| 指標 | 數值 |
|------|------|
| 修改檔案數 | 23 |
| 淨減少行數 | 1382 |
| 涉及操作 | 檔案搬移、目錄重組、引用路徑更新、索引重建 |
| 任務標籤 | DOC |

---

## 豁免條件逐項評估

當前豁免條件為「任一符合即可簡化」：

| 豁免條件 | 是否符合 | 說明 |
|---------|----------------|------|
| 修改 <= 2 個檔案 | 不符合 | 修改 23 個檔案 |
| 純文件任務 | 符合 | 不涉及程式碼檔案 |
| 任務範圍單純 | 不符合 | 跨多個目錄、涉及結構遷移 |

由於「任一符合即可」，該任務因為「純文件任務」而獲得豁免。但 23 個檔案的結構重組，其風險等同程式碼重構：

- 引用路徑可能斷裂
- 索引可能遺漏新增或刪除的檔案
- 目錄結構變更影響其他文件的相對路徑引用

---

## 根因

豁免條件的邏輯運算子（OR）過於寬鬆。「純文件任務」在設計時假設文件修改的風險低於程式碼修改，但大規模文件結構重組的風險實際上接近程式碼重構。

---

## 改善建議

### 1. 改為全部符合才豁免（AND 邏輯）

```
豁免條件（全部符合）：
  - 修改 <= 2 個檔案
  - 純文件任務（不涉及程式碼）
  - 任務範圍單純（單一目錄、無遷移操作）
```

### 2. 新增排除條件

即使符合上述三項，以下情況仍不豁免：

| 排除條件 | 說明 |
|---------|------|
| 影響模組數 >= 3 | 跨模組結構變更風險高 |
| 包含遷移或刪除操作 | 不可逆操作需要審查 |
| 淨變更超過 500 行 | 大規模變更需要品質確認 |

---

## 教訓

1. **DOC 標籤不等於低風險** -- 大規模文件結構重組的風險等同程式碼重構，需要同等級的品質審查
2. **豁免條件應使用 AND 邏輯** -- OR 邏輯容易被單一寬鬆條件繞過
3. **量化指標比類別標籤更可靠** -- 「修改檔案數」和「變更行數」比「DOC/IMP」標籤更能反映任務的實際複雜度
