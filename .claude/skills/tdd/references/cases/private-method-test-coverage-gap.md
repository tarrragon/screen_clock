# 私有方法測試覆蓋缺口：合併邏輯和快取鍵生成

> **背景**：新建的私有方法（書籍合併的 key 生成、快取鍵計算）只靠上層公開方法間接覆蓋，沒有獨立的單元測試。當 key 生成邏輯讀錯欄位名（author vs authors），因測試資料碰巧同時有兩個欄位而未被偵測到。
> 對應 Phase：Phase 2（測試設計）
> 對應 Decision Question：Q8（私有方法測試覆蓋）

---

## 問題概述

6 個私有方法承擔關鍵業務邏輯，但測試只覆蓋公開入口方法 `_handleDuplicateBooks`，未對中間步驟的邊界條件設置獨立斷言。這些方法在 commit b7cc6c8（2026-03-30）引入。

---

## 風險最高的兩個方法

### `_mergeReadingStatus`

3 層條件邏輯判斷哪個閱讀狀態應該保留：

```
if (statusA 有優先級 && statusB 有優先級)
  -> 比較優先級數值
  -> if (優先級相同)
    -> 比較最後更新時間
    -> if (時間也相同)
      -> 保留第一本的狀態
```

未測試的邊界條件：
- 兩者優先級相同、時間也相同
- 其中一個狀態為 null 或未定義
- 狀態值不在預期的 enum 範圍內

### `_mergeProgress`

數值範圍驗證邏輯：

- 進度值必須在 0-100 之間
- 取較大值作為合併結果
- NaN 和 undefined 的處理

未測試的邊界條件：
- 進度值為負數
- 進度值超過 100
- 一方為 NaN、另一方為有效值

---

## 根因

測試設計者從公開 API 入口（`_handleDuplicateBooks`）設計測試，驗證最終輸出正確。但合併過程中的中間步驟沒有獨立的 expect 斷言，導致：

- 邊界條件在正常輸入下碰巧不觸發
- 中間步驟的錯誤被後續步驟掩蓋
- 快取鍵生成完全沒有被呼叫到的測試路徑

---

## Decision Question Q8 回答

> 「6 個私有方法承擔關鍵業務邏輯但無獨立 expect 驗證。測試從入口 `_handleDuplicateBooks` 間接覆蓋，但中間步驟的邊界條件未被斷言。」

---

## 教訓

1. **私有方法承擔業務邏輯時，測試設計必須為其邊界條件設置斷言** -- 即使不直接測試私有方法，也要透過輸入構造觸發邊界條件並驗證輸出
2. **多層條件邏輯是測試缺口的高風險區** -- 3 層條件的 `_mergeReadingStatus` 至少需要 8 個測試案例才能覆蓋主要路徑
3. **快取鍵生成看似簡單但影響全域** -- 鍵碰撞會導致錯誤的快取命中，必須測試不同輸入產生不同鍵
