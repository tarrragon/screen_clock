# 可攜式 TDD 流程設計邊界

本文件定義 TDD 流程的 Layer 1（通用/可攜式）和 Layer 2（框架特定）邊界，用於指導 `/tdd` SKILL 的設計。

> **目標**：將本專案的 TDD 流程標準化為可攜式工具，讓其他工程師在別的專案也能直接使用。

---

## 設計邊界總覽

```
TDD 流程架構
├── Layer 1（通用知識，任何專案可用）
│   ├── Phase 定義和目標
│   ├── 階段轉換條件
│   ├── TDD 原則和模式
│   ├── 品質基準
│   └── 任務類型豁免規則
│
└── Layer 2（框架特定，依賴本專案基礎設施）
    ├── Ticket 系統整合
    ├── Hook 系統觸發
    ├── Agent 派發規則
    ├── Commit 管理
    ├── 決策樹路由
    └── 狀態管理 API
```

## Phase 0-4 功能點分類表

### Phase 0：SA 前置審查

| 功能點 | 描述 | Layer | 歸屬理由 |
|--------|------|-------|---------|
| 審查觸發條件 | 新功能、架構變更、3+ 模組影響 | Layer 1 | 純邏輯判斷，不依賴系統 |
| 系統一致性檢查 | 與架構一致性、設計原則遵循、命名衝突 | Layer 1 | 通用設計審查原則 |
| 重複實作檢查 | 無類似功能、無可重用元件 | Layer 1 | 通用架構分析方法論 |
| 需求文件完整性 | 需求書、用例、驗收標準 | Layer 1 | 通用需求工程實踐 |
| SA 派發和審核流程 | 派發給 agent + 審核結論路由 | Layer 2 | 依賴 agent 定義和 PM 決策樹 |

### Phase 1：功能設計

| 功能點 | 描述 | Layer | 歸屬理由 |
|--------|------|-------|---------|
| 設計產出定義 | 功能規格、API 定義、驗收標準 | Layer 1 | 標準設計工程產出 |
| API 設計原則 | 清晰介面、低耦合、高內聚 | Layer 1 | 通用 API 設計原則 |
| SOLID 檢查清單 | SRP、OCP、LSP、ISP、DIP | Layer 1 | 通用設計原則 |
| 功能拆分策略 | 按架構層、按模組、按操作類型 | Layer 1 | 通用拆分方法論 |
| 代理人派發 | 派發給 design agent | Layer 2 | 依賴 agent 定義 |
| 轉換到 Phase 2 | 功能規格完成 | Layer 1 | 通用轉換標準 |
| 代理人自治 commit | 自行 git commit 和 Ticket 更新 | Layer 2 | 依賴 ticket 系統 |

### Phase 2：測試設計

| 功能點 | 描述 | Layer | 歸屬理由 |
|--------|------|-------|---------|
| 測試策略設計 | 單元、整合、端到端、驗收測試 | Layer 1 | 標準測試金字塔 |
| BDD / GWT 格式 | Given-When-Then 測試用例 | Layer 1 | 通用 BDD 方法論 |
| 測試案例設計 | 邊界、正常、異常、性能 | Layer 1 | 標準測試設計方法 |
| 檔案結構規劃 | 測試檔案位置、命名規範 | Layer 1 | 通用檔案組織原則 |
| 代理人派發 | 派發給 test-architect agent | Layer 2 | 依賴 agent 定義 |
| 轉換到 Phase 3a | 測試案例設計完成 | Layer 1 | 通用轉換標準 |

### Phase 3a：策略規劃

| 功能點 | 描述 | Layer | 歸屬理由 |
|--------|------|-------|---------|
| 實作策略設計 | 實作順序、模組分割、虛擬碼 | Layer 1 | 通用實作規劃方法 |
| 認知負擔評估 | 複雜度指數、拆分判斷 | Layer 1 | 通用認知負擔設計原則 |
| 技術債務評估 | 預期產生的技術債務識別 | Layer 1 | 通用技術債務管理 |
| 代理人派發 | 派發給 strategy agent | Layer 2 | 依賴 agent 定義 |
| 轉換到 3b 拆分評估 | 策略文件完成 | Layer 1 | 通用轉換標準 |
| 3b 拆分評估邏輯 | 基於檔案數、認知指數、跨層級判斷 | Layer 1 | 通用複雜度評估 |
| 拆分豁免條件 | 修改 ≤5 檔案、單一模組、低複雜度 | Layer 1 | 通用複雜度評估 |

### Phase 3b：實作執行

| 功能點 | 描述 | Layer | 歸屬理由 |
|--------|------|-------|---------|
| 程式碼實作 | 按 Phase 1/2 規格實作 | Layer 1 | 標準實作活動 |
| 測試實現 | 將測試案例轉為可執行測試 | Layer 1 | 標準測試實現方法 |
| 品質檢查 | 程式碼品質、測試通過率、文件 | Layer 1 | 通用品質基準 |
| 代理人派發 | 派發給 implementation agent | Layer 2 | 依賴 agent 定義 |
| 3b 拆分執行 | 建立子任務、並行派發 | Layer 2 | 依賴 ticket 系統和並行框架 |
| 代理人自治 commit | 自行提交和 Ticket 更新 | Layer 2 | 依賴 ticket 系統 |
| 轉換到 Phase 4 | 測試全部通過 | Layer 1 | 通用轉換標準 |
| Phase 4 豁免判斷 | 修改 ≤2 檔案、DOC、單一模組 | Layer 1 | 通用複雜度評估 |

### Phase 4a：多視角分析

| 功能點 | 描述 | Layer | 歸屬理由 |
|--------|------|-------|---------|
| 分析維度 | 冗餘、耦合、複雜度、設計品質 | Layer 1 | 標準程式碼品質分析維度 |
| 多視角分析方法 | 多維度同時評估、交叉驗證 | Layer 1 | 通用分析方法論 |
| 分析報告產出 | 重構建議、優先級、方案 | Layer 1 | 標準分析報告格式 |
| 代理人派發 | 派發給 multi-perspective analysis | Layer 2 | 依賴本專案工具 |
| Phase 4a 豁免 | 簡化條件下跳過 4a | Layer 1 | 通用複雜度評估 |

### Phase 4b：重構執行

| 功能點 | 描述 | Layer | 歸屬理由 |
|--------|------|-------|---------|
| 重構實施 | 按 4a 報告改進程式碼品質 | Layer 1 | 標準重構活動 |
| 技術債務記錄 | 發現的技術債務記錄 | Layer 1 | 通用技術債務追蹤 |
| 代理人派發 | 派發給 refactor agent | Layer 2 | 依賴 agent 定義 |
| PM 管理 commit | PM 執行提交 | Layer 2 | 依賴 PM 角色定義 |
| 轉換到 Phase 4c | 重構執行完成 | Layer 1 | 通用轉換標準 |

### Phase 4c：多視角再審核

| 審核報告產出 | 確認重構效果、遺留問題 | Layer 1 | 標準審核報告格式 |
| 轉換到完成 | 審核報告完成 | Layer 1 | 通用轉換標準 |

## 可攜式約束規則（Layer 1 不可引用）

| 禁止項 | 說明 | 替代方式 |
|--------|------|---------|
| `/ticket` CLI | 如 `/ticket create`, `/ticket track` | 「任務系統」「狀態管理」 |
| 具體 agent 名稱 | lavender, parsley, sage 等 | 「設計者」「實作者」「測試者」 |
| `.claude/hooks/` 系統 | 檢查點自動化 | 「驗證機制」「檢查點」 |
| `decision-tree` 路由 | PM 決策邏輯 | 「階段轉換」「路由決策」 |
| `/parallel-evaluation` | 本專案多視角工具 | 「多維度分析」「交叉審查」 |
| 本專案路徑 | `.claude/`, `docs/` 等 | 「規則目錄」「工作目錄」 |
| Wave、Patch 概念 | 本專案版本語義 | 「執行批次」「版本」 |

## Layer 2 整合點

**Layer 1**：任務轉換條件
**Layer 2**：`/ticket track complete` 實現

### 整合點 2：Agent 派發

**Layer 1**：「Phase 1 由設計者執行」

### 整合點 3：自治提交（Phase 1-3）

**Layer 1**：「完成後自行提交」

### 整合點 4：多視角分析（Phase 4a/4c）

**Layer 1**：「多維度交叉審查」

## 邊界模糊功能點

### 1. Phase 0 要不要納入？

**現況**：本專案包含 Phase 0
**建議**：Layer 1 包含架構審查原則，Layer 2 可選擇是否納入派發流程

### 2. 3b 拆分評估時機

**現況**：Phase 3a 完成後進行
**建議**：Layer 1 提供拆分決策邏輯，Layer 2 定義時機

### 3. Commit 管理策略

**現況**：Phase 1-3 代理人自治，Phase 3b+ PM 管理
**建議**：Layer 1 提供通用規範，Layer 2 實現角色差異

### 4. Phase 4 豁免條件

**現況**：修改 ≤2 檔案、DOC、單一模組
**建議**：Layer 1 提供判斷標準和決策樹，Layer 2 實施豁免
