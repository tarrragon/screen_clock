# Phase 3：實作指引（3a 策略規劃 + 3b 執行）

## Phase 目標

Phase 3 分為兩個子階段：

- **Phase 3a**：策略規劃，設計實作順序、識別技術風險、評估認知負擔
- **3b 拆分評估**：決定 Phase 3b 是否需要拆分為並行子任務
- **Phase 3b**：依規格實作程式碼，讓所有測試通過

**核心問題**：「如何高效實作？是否需要拆分讓多人並行？」

---

## Phase 3a：策略規劃

### 產出（Layer 1）

Phase 3a 完成後，必須產出：

- [ ] 實作策略文件（實作順序、模組分割）
- [ ] 虛擬碼或設計草稿（關鍵邏輯）
- [ ] 認知負擔評估（是否需要 3b 拆分）
- [ ] 技術債務預評（預期產生的債務）

### 實作策略設計

#### 實作順序原則

遵循由底層到上層的實作順序（依賴方向）：

```
Layer 1（Domain / 業務邏輯）
    ↓
Layer 2（Application / 應用服務）
    ↓
Layer 3（Infrastructure / 資料存取）
    ↓
Layer 4（Presentation / UI）
```

**例外**：使用 TDD 時，先寫測試再實作，從需求角度思考介面設計。

#### 模組分割評估

列出所有需要修改或新增的模組：

| 模組 | 類型（新建/修改） | 架構層 | 估計複雜度 |
|------|---------------|--------|----------|
| {模組名} | 新建 | Domain | 低 |
| {模組名} | 修改 | Application | 中 |

#### 關鍵邏輯虛擬碼

對於複雜的業務邏輯，用虛擬碼說明核心算法或流程：

```
函式：{功能名稱}(輸入)
  驗證輸入合法性
  如果不合法 → 回傳錯誤

  執行主要業務邏輯
    步驟 1：{描述}
    步驟 2：{描述}

  回傳結果
```

### 認知負擔評估

計算認知負擔指數：

```
認知負擔指數 = 變數數 + 分支數 + 巢狀深度 + 依賴數
```

| 指數 | 評估 | 建議行動 |
|------|------|---------|
| 1-5 | 優良 | 不拆分 |
| 6-10 | 可接受 | 不拆分 |
| 11-15 | 需重構 | 建議拆分 3b |
| > 15 | 必須重構 | 必須拆分 3b |

|---------|------|---------|
| 暫時性繞過 | {描述} | 中 |
| 設計簡化 | {描述} | 低 |
| 待改進 | {描述} | 低 |

> **框架整合**：Phase 3a 由策略規劃者角色執行，完成後自行提交（git commit），更新任務狀態，回報：「Phase 3a 完成，策略文件已提交，請進行 3b 拆分評估」。

## 3b 拆分評估

Phase 3a 完成後，必須評估 Phase 3b 是否需要拆分。

### 任務粒度 — 引導式拆分 + 安全網閾值

> 完整規則：[task-granularity-rules.md](../task-granularity-rules.md)

**主要機制**：Phase 1 的行為單元清單決定 Phase 2/3 的 Ticket 粒度。每個行為單元 = 1 個測試 Ticket = 1 個實作 Ticket。如果 Phase 1 拆分充分，Ticket 粒度自然合適。

**安全網閾值**（單一 Ticket 粒度檢查 — 超標時回到 Phase 1 重新檢視行為單元拆分）：

| 指標 | 上限 | 超標意味著 |
|------|------|-----------|
| 驗收關注點數 | 2 個 | 可能混入多個行為單元（正常路徑 + 邊界條件算同一關注點） |
| 修改檔案數 | 2 個 | 可能跨越模組邊界（測試檔案不計入） |
| 關注點數 | 1 個 | 需要「和」或「+」描述 = 不是一個單元 |
| 預估執行時間 | 10 分鐘 | 行為單元可能還能再拆 |

> **安全網 vs 3b 拆分是不同層級的判斷**：安全網閾值檢查「單一 Ticket 是否只含一個行為單元」（粒度），3b 拆分觸發判斷「整個 Phase 3b 是否需要拆為多個並行子任務」（並行度）。3-5 檔案的 Ticket：安全網超標（回 Phase 1 確認行為單元是否過大）但不一定需要 3b 拆分（可能是同一關注點跨多檔案）。

**粒度超標時的回環路徑**：安全網超標 → 建立「補充行為單元識別」修正 Ticket → 回到 [Phase 1 行為單元識別章節](../phase1/rules.md)（「行為單元識別」段落）重新拆解 → 產出更細的行為單元清單 → 據此重建 Phase 2/3 Ticket。

### 3b 拆分觸發條件（整體 Phase 3b 並行度判斷，任一符合）

| 指標 | 閾值 | 說明 |
|------|------|------|
| 預期修改檔案數 | > 5 個 | 修改範圍過大 |
| 認知負擔指數 | > 10 | 複雜度過高 |
| 跨架構層級數 | > 2 層 | 跨層影響複雜 |
| 獨立模組數 | > 1 且無交叉依賴 | 可獨立並行 |

### 拆分豁免條件（任一符合即不拆分）

| 條件 | 說明 |
|------|------|
| 修改 <= 5 個檔案 | 影響範圍有限 |
| 單一模組內修改 | 無跨模組衝突風險 |
| 修改目的單一明確 | 低複雜度 |

### Decision Questions — 並行派發前評估

> 觸發條件：3b 拆分評估決定拆分，即將派發 2+ 代理人時。

**Q12**：Phase 1 的共用策略結論是什麼？是否已建立共用模組？若未建立，先建再派發。

- 行動路徑：有共用需求 → 先建 shared module，各子任務 import。無共用需求 → 直接並行。

**Q_new5**：各 worktree 子任務之間是否有共用的常數/版本號/配置？若有，定義在哪個共用位置？

- 行動路徑：有共用常數 → 定義在共用模組並匯出，各子任務 import。

**正面範例** [v0.17.0 P1-1/2]：
BookSchemaV2 和 TagSchema 分別在獨立 worktree 實作，各自建立驗證框架，重複率 20%。
防護：Phase 3a 先建 ValidationEngine，兩個 worktree 都 import。

**反面範例**：
兩個子任務完全不同領域（UI 元件 vs API 處理器），無共用需求，直接並行。

**邊界範例** [v0.17.0 P2-4]：
v1-to-v2.js 和 tag-storage-adapter.js 都硬編碼 "3.0.0"。應由 BookSchemaV2 匯出 SCHEMA_VERSION，各模組 import。

### 拆分後的並行安全檢查

若決定拆分，必須確認：

| 檢查項目 | 說明 |
|---------|------|
| 檔案無重疊 | 各子任務修改的檔案集合無交集 |
| 測試無衝突 | 各子任務的測試可獨立執行 |
| 依賴無循環 | 子任務之間無先後依賴關係 |
| 共用元件已就緒 | Q12/Q_new5 確認的共用模組已建立並可 import |

> **框架整合**：拆分時建立子任務（`/ticket create --parent {3b_ticket_id}`），每個子任務指定 `where.files`（無交集），然後並行派發執行者。

## Phase 3b：實作執行

### 代理人派發模板

```
執行前必須 Read：
- .claude/skills/tdd/references/phase3/rules.md
- 特別注意 Decision Questions 區段（必答）

完成前必須：
- 產出文件的必填欄位全部填寫
- 執行清單全部執行（不只是打勾）
```

### 產出（Layer 1）

Phase 3b 完成後，必須產出：

- [ ] 實作程式碼（符合 Phase 1 API 規格）
- [ ] 通過率 100% 的測試執行結果
- [ ] 程式碼品質自評報告

### 實作原則

#### 測試驅動實作（Red-Green-Refactor）

```
1. Red：執行測試，確認失敗（測試存在）
2. Green：寫最少的程式碼讓測試通過
3. Refactor：重構程式碼，保持測試通過
```

**每個測試案例**逐一按此循環執行。

#### 程式碼品質基準

| 指標 | 要求 |
|------|------|
| 函式長度 | <= 30 行（理想 5-10 行） |
| 巢狀深度 | <= 3 層 |
| 參數數量 | <= 3 個 |
| 認知負擔指數 | < 10 |
| 測試通過率 | 100% |

#### 禁止事項

| 禁止 | 說明 |
|------|------|
| 魔法數字 | 使用具名常數 |
| 硬編碼字串 | 使用常數或配置 |
| 深層巢狀 | 使用 Guard Clause 提前返回 |
| 未處理例外 | 所有例外必須有明確處理 |

### 測試通過率驗證

實作完成後必須執行完整測試套件，確認：

1. 所有新增測試通過
2. 所有現有測試仍然通過（無回歸）
3. 整體測試通過率 100%

如有測試失敗，不可跳過，必須找出根本原因並修復。

### Phase 3b 完成前執行清單

> 觸發條件：Phase 3b 實作完成，準備提交前。此清單為「執行清單」，每一項必須實際執行（不只是打勾確認）。

- [ ] 執行 `npm run lint` 並確認 0 errors
- [ ] 執行對應的測試套件並確認全部通過
- [ ] 新建私有方法：參數名稱是否對應規格中的欄位定義？逐一比對
- [ ] **Q_new2**：新增模組的每個 import 是否都在程式碼中被使用？列舉確認

**正面範例** [v0.17.0 P0-2]：
BookSchemaV2.js 第 12 行 `require('src/core/errors/ErrorCodes')` — 未使用，且 bare specifier 會阻斷 build。
防護：完成前執行 `npm run lint` → ESLint no-unused-vars 攔截。

**正面範例** [v0.17.0 P0-1]：
`_buildTitleAuthorKey` 讀取 `book.author` 但規格定義 `authors`（陣列） → 參數未對應規格。
防護：自我驗證清單「參數對應規格欄位」→ 發現不一致。

### 程式碼品質自評

完成實作後，對照品質基準自評：

```markdown
## Phase 3b 品質自評

### 測試結果
- 新增測試：{通過數} / {總數}
- 現有測試：{通過數} / {總數}
- 整體通過率：100% / 未達標

### 程式碼品質
- 最長函式：{行數} 行（限 30 行）
- 最深巢狀：{層數} 層（限 3 層）
- 認知負擔最高：{指數}（限 10）

### 已知問題
- {問題描述}（優先級：高/中/低）
```

## 轉換條件

### 進入 Phase 3a 的條件

- Phase 2 測試案例設計完成
- GWT 格式測試規格化完成

### 退出 Phase 3a 的條件（進行 3b 拆分評估）

- [ ] 實作策略文件完成
- [ ] 認知負擔評估完成
- [ ] 技術債務預評完成

### 退出 3b 拆分評估（進入 Phase 3b）

- [ ] 拆分或豁免決策已做出
- [ ] 若拆分：子任務已定義，檔案無重疊確認完成
- [ ] 若拆分：Q12/Q_new5 已回答，共用元件已就緒

### 退出 Phase 3b 的條件（進入 Phase 4）

- [ ] 所有測試通過（100%）
- [ ] 程式碼品質基準達標
- [ ] 品質自評報告完成
- [ ] 完成前執行清單全部通過

> **框架整合**：Phase 3b 完成後由管理者執行 git commit，更新任務狀態，然後依 Phase 4 豁免評估決定下一步。
