# Phase 3：實作指引（3a 策略規劃 + 3b 執行）

## Phase 目標

Phase 3 分為兩個子階段：

- **Phase 3a**：策略規劃，設計實作順序、識別技術風險、評估認知負擔
- **3b 拆分評估**：決定 Phase 3b 是否需要拆分為並行子任務
- **Phase 3b**：依規格實作程式碼，讓所有測試通過

**核心問題**：「如何高效實作？是否需要拆分讓多人並行？」

---

## Phase 3a：策略規劃

### 產出（Layer 1）

Phase 3a 完成後，必須產出：

- [ ] 實作策略文件（實作順序、模組分割）
- [ ] 虛擬碼或設計草稿（關鍵邏輯）
- [ ] 認知負擔評估（是否需要 3b 拆分）
- [ ] 技術債務預評（預期產生的債務）

### 實作策略設計

#### 實作順序原則

遵循由底層到上層的實作順序（依賴方向）：

```
Layer 1（Domain / 業務邏輯）
    ↓
Layer 2（Application / 應用服務）
    ↓
Layer 3（Infrastructure / 資料存取）
    ↓
Layer 4（Presentation / UI）
```

**例外**：使用 TDD 時，先寫測試再實作，從需求角度思考介面設計。

#### 模組分割評估

列出所有需要修改或新增的模組：

| 模組 | 類型（新建/修改） | 架構層 | 估計複雜度 |
|------|---------------|--------|----------|
| {模組名} | 新建 | Domain | 低 |
| {模組名} | 修改 | Application | 中 |

#### 關鍵邏輯虛擬碼

對於複雜的業務邏輯，用虛擬碼說明核心算法或流程：

```
函式：{功能名稱}(輸入)
  驗證輸入合法性
  如果不合法 → 回傳錯誤

  執行主要業務邏輯
    步驟 1：{描述}
    步驟 2：{描述}

  回傳結果
```

### 認知負擔評估

計算認知負擔指數：

```
認知負擔指數 = 變數數 + 分支數 + 巢狀深度 + 依賴數
```

| 指數 | 評估 | 建議行動 |
|------|------|---------|
| 1-5 | 優良 | 不拆分 |
| 6-10 | 可接受 | 不拆分 |
| 11-15 | 需重構 | 建議拆分 3b |
| > 15 | 必須重構 | 必須拆分 3b |

### 技術債務預評

識別實作過程中可能產生的技術債務：

| 債務類型 | 說明 | 風險等級 |
|---------|------|---------|
| 暫時性繞過 | {描述} | 中 |
| 設計簡化 | {描述} | 低 |
| 待改進 | {描述} | 低 |

> **框架整合**：Phase 3a 由策略規劃者角色執行，完成後自行提交（git commit），更新任務狀態，回報：「Phase 3a 完成，策略文件已提交，請進行 3b 拆分評估」。

## 3b 拆分評估

Phase 3a 完成後，必須評估 Phase 3b 是否需要拆分。

### 拆分觸發條件（任一符合）

| 指標 | 閾值 | 說明 |
|------|------|------|
| 預期修改檔案數 | > 5 個 | 修改範圍過大 |
| 認知負擔指數 | > 10 | 複雜度過高 |
| 跨架構層級數 | > 2 層 | 跨層影響複雜 |
| 獨立模組數 | > 1 且無交叉依賴 | 可獨立並行 |

### 拆分豁免條件（任一符合即不拆分）

| 條件 | 說明 |
|------|------|
| 修改 <= 5 個檔案 | 影響範圍有限 |
| 單一模組內修改 | 無跨模組衝突風險 |
| 修改目的單一明確 | 低複雜度 |

### 拆分後的並行安全檢查

若決定拆分，必須確認：

| 檢查項目 | 說明 |
|---------|------|
| 檔案無重疊 | 各子任務修改的檔案集合無交集 |
| 測試無衝突 | 各子任務的測試可獨立執行 |
| 依賴無循環 | 子任務之間無先後依賴關係 |

> **框架整合**：拆分時建立子任務（`/ticket create --parent {3b_ticket_id}`），每個子任務指定 `where.files`（無交集），然後並行派發執行者。

## Phase 3b：實作執行

### 產出（Layer 1）

Phase 3b 完成後，必須產出：

- [ ] 實作程式碼（符合 Phase 1 API 規格）
- [ ] 通過率 100% 的測試執行結果
- [ ] 程式碼品質自評報告

### 實作原則

#### 測試驅動實作（Red-Green-Refactor）

```
1. Red：執行測試，確認失敗（測試存在）
2. Green：寫最少的程式碼讓測試通過
3. Refactor：重構程式碼，保持測試通過
```

**每個測試案例**逐一按此循環執行。

#### 程式碼品質基準

| 指標 | 要求 |
|------|------|
| 函式長度 | <= 30 行（理想 5-10 行） |
| 巢狀深度 | <= 3 層 |
| 參數數量 | <= 3 個 |
| 認知負擔指數 | < 10 |
| 測試通過率 | 100% |

#### 禁止事項

| 禁止 | 說明 |
|------|------|
| 魔法數字 | 使用具名常數 |
| 硬編碼字串 | 使用常數或配置 |
| 深層巢狀 | 使用 Guard Clause 提前返回 |
| 未處理例外 | 所有例外必須有明確處理 |

### 測試通過率驗證

實作完成後必須執行完整測試套件，確認：

1. 所有新增測試通過
2. 所有現有測試仍然通過（無回歸）
3. 整體測試通過率 100%

如有測試失敗，不可跳過，必須找出根本原因並修復。

### 程式碼品質自評

完成實作後，對照品質基準自評：

```markdown
## Phase 3b 品質自評

### 測試結果
- 新增測試：{通過數} / {總數}
- 現有測試：{通過數} / {總數}
- 整體通過率：100% / 未達標

### 程式碼品質
- 最長函式：{行數} 行（限 30 行）
- 最深巢狀：{層數} 層（限 3 層）
- 認知負擔最高：{指數}（限 10）

### 已知問題
- {問題描述}（優先級：高/中/低）
```

## 轉換條件

### 進入 Phase 3a 的條件

- Phase 2 測試案例設計完成
- GWT 格式測試規格化完成

### 退出 Phase 3a 的條件（進行 3b 拆分評估）

- [ ] 實作策略文件完成
- [ ] 認知負擔評估完成
- [ ] 技術債務預評完成

### 退出 3b 拆分評估（進入 Phase 3b）

- [ ] 拆分或豁免決策已做出
- [ ] 若拆分：子任務已定義，檔案無重疊確認完成

### 退出 Phase 3b 的條件（進入 Phase 4）

- [ ] 所有測試通過（100%）
- [ ] 程式碼品質基準達標
- [ ] 品質自評報告完成

> **框架整合**：Phase 3b 完成後由管理者執行 git commit，更新任務狀態，然後依 Phase 4 豁免評估決定下一步。
