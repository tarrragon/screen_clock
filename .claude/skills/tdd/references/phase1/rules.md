# Phase 1：功能設計規則與決策指引

> 本檔案整合 Phase 1 所有規則與指引，包含功能規格設計、SOLID 拆分分析、Decision Questions、拆分工具和範本。

## Phase 目標

在寫任何程式碼之前，定義功能的完整規格、API 介面、驗收標準，並使用 SOLID 原則評估是否需要拆分。

**核心問題**：「這個功能的邊界在哪裡？如何與系統其他部分互動？」

---

## 產出（Layer 1）

Phase 1 完成後，必須產出：

- [ ] 功能規格文件（輸入/輸出/流程）
- [ ] API 介面定義（函式簽名、資料結構）
- [ ] 驗收標準清單（可量化、可驗證）
- [ ] SOLID 拆分分析報告（如需要拆分）
- [ ] 行為場景（Given-When-Then 格式）
- [ ] **行為單元清單**（每個 Use Case 分解為獨立可測試的單元，含邊界條件歸屬）
- [ ] **跨模組共用決策**（Q1-Q2 的答案，規格涉及 2+ 模組時必填）
- [ ] **非功能性需求定義**（Q5-Q6b 的答案，所有功能規格必填）

> **行為單元的品質決定後續所有 Phase 的粒度**。如果 Phase 1 只產出大 Use Case 而不分解行為單元，Phase 2/3 的 Ticket 必然過大。詳見 [任務粒度規則](../task-granularity-rules.md)。

## 功能規格設計

### 輸入定義

明確每個輸入的類型和限制：

```
輸入規格
  參數名稱
  資料類型
  是否必填
  合法範圍/約束
  預設值（如有）
```

### 輸出定義

明確每個輸出的類型和行為：

```
輸出規格
  成功情況：回傳值類型和結構
  失敗情況：錯誤類型和訊息
  副作用：狀態變更、事件觸發等
```

### 操作流程

定義正常操作流程（主要路徑）：

```
操作序列
1. 接收輸入
2. 驗證輸入合法性
3. 執行業務邏輯
4. 回傳結果
```

定義異常處理流程：

| 異常情況 | 處理方式 | 回傳訊息 |
|---------|---------|---------|
| 必填欄位為空 | 拒絕並回傳錯誤 | {錯誤描述} |
| 格式不合法 | 拒絕並回傳錯誤 | {錯誤描述} |
| 資源不存在 | 回傳 Not Found | {錯誤描述} |

## API 介面設計

### 介面定義原則

| 原則 | 說明 |
|------|------|
| 清晰命名 | 函式名稱以動詞開頭，描述「做什麼」 |
| 最小介面 | 只暴露必要的方法，遵循 ISP |
| 高內聚低耦合 | 相關功能集中，不相關功能分離 |
| 依賴抽象 | 依賴介面而非具體實作，遵循 DIP |

### 函式簽名格式

```
函式名稱(
  參數名: 類型,
  參數名: 類型
) -> 回傳類型 | 錯誤類型
```

### 資料結構定義

```
資料結構名稱 {
  欄位名: 類型  // 說明
  欄位名: 類型  // 說明（可選）
}
```

## 行為單元識別（Phase 1 核心產出）

> **與 SOLID 拆分的關係**：兩者是先後步驟，不是替代品。
> 1. 先做 SOLID 分析 → 產出模組/職責邊界（架構層級拆分）
> 2. 再在每個職責內識別行為單元 → 產出 GWT 場景（測試層級拆分）
>
> SOLID 回答「這個功能應該拆成幾個模組？」，行為單元回答「每個模組裡有哪些可獨立測試的行為？」。衝突時以行為單元為準（因為 Ticket 粒度由可測試性決定）。

### 操作步驟

1. **列出 Use Case 包含的動作**：這個功能讓使用者做哪些事？
2. **對每個動作問**：「能否用一個 Given-When-Then 場景完整描述？」
   - 能 → 這是一個行為單元
   - 不能（需要多個 GWT）→ 繼續拆解
3. **歸屬邊界條件**：每個行為單元在什麼情況下走不同路徑？
   - 邊界條件歸屬到對應的行為單元，不獨立成 Ticket
4. **識別整合場景**：哪些行為單元之間需要串連驗證？

### 引導問題

| 問題 | 判斷 |
|------|------|
| 這個 UC 包含幾個獨立可測試的行為？ | > 1 → 拆解為行為單元 |
| 每個行為單元能用 1 個 GWT 場景描述嗎？ | 否 → 繼續拆解 |
| 邊界條件歸屬到哪個行為單元？ | 說不清 → 單元定義不夠清楚 |
| 有沒有「無主」邊界條件？ | 有 → 回頭補充行為單元 |

### 產出格式

| 行為單元 | Given-When-Then | 邊界條件 | 獨立可測？ |
|---------|----------------|---------|-----------|
| {單元名} | Given X → When Y → Then Z | {邊界 1, 邊界 2} | 是/否 |

**每個行為單元 = 1 個測試 Ticket = 1 個實作 Ticket**。完整的粒度規則和反模式目錄見 [task-granularity-rules.md](../task-granularity-rules.md)。

## SOLID 原則拆分分析

使用 `/tdd split` 執行完整 SOLID 分析。以下為快速參考：

### SOLID 檢查清單

| 原則 | 核心問題 | 拆分信號 |
|------|---------|---------|
| SRP | 這個功能有幾個獨立的修改原因？ | 需要用「和」連接描述 |
| OCP | 未來擴展需要修改現有程式碼嗎？ | 需要 switch/case 或 if/else 分岐 |
| LSP | 繼承關係正確嗎？ | 子類別有需要覆寫的方法 |
| ISP | 介面是否強迫實作不需要的方法？ | 實作類別有空方法 |
| DIP | 依賴的是抽象還是具體實作？ | 直接 import 具體類別 |

### 拆分決策流程

```
SOLID 分析結果
    |
    +-- 多個獨立職責 -> 拆分為多個任務
    |
    +-- 需要介面抽象 -> 新增介面定義任務
    |
    +-- 有依賴關係 -> 確定執行順序
    |
    v
產出拆分清單（見 SKILL.md `/tdd split` 格式）
```

### 版本號分配規則

拆分後的子任務需要合理的版本/批次分配，確保依賴關係正確。

| 情況 | 版本/批次分配 | 範例 |
|------|-------------|------|
| 無依賴任務 | 同一批次（可並行） | Domain Entities -> 第一批 |
| 依賴前一批 | 下一個批次 | UseCases -> 第二批 |
| 依賴多批次 | 最後依賴的下一批 | Widget -> 第三批 |

**並行判斷**：

| 條件 | 可並行 |
|------|--------|
| 同一批次 | 是 |
| 無 blockedBy | 是 |
| 不同架構層但無依賴 | 是 |
| 有 blockedBy | 否 |

**架構層級執行順序**：

| 架構層 | 執行順序 | 說明 |
|--------|---------|------|
| Domain | 第一批 | Entity、Value Object、介面定義（無依賴） |
| Application | 第二批 | UseCase（依賴 Domain） |
| Infrastructure | 第二批 | Repository 實作（依賴 Domain 介面） |
| Presentation | 第三批 | Widget、Controller（依賴 Application） |

> **框架整合**：本專案使用 Patch 版本號（如 v0.29.1 -> v0.29.2）對應批次，由 `/ticket create` 管理。

## 驗收標準定義

驗收標準必須**可量化、可驗證**，禁止使用模糊描述。

| 禁止（抽象） | 正確（可驗證） |
|------------|-------------|
| 「提升使用者體驗」 | 「搜尋結果在 500ms 內回傳」 |
| 「提高穩定性」 | 「輸入空關鍵字回傳錯誤碼 400」 |
| 「支援多種輸入」 | 「支援中英文關鍵字，長度 1-100 字元」 |

### 驗收標準格式

```markdown
驗收標準：
- [ ] {具體可驗證的條件 1}
- [ ] {具體可驗證的條件 2}
- [ ] {具體可驗證的條件 3}
```

## Decision Questions

> 規格撰寫者在 Phase 1 必須回答的結構化問題。每組問題有明確的觸發條件和行動路徑。
> 真實案例見 `cases/` 目錄。

### 組 1：跨模組共用策略

**觸發條件**：規格涉及 2+ 模組時必答。

#### Q1: 多個模組是否有相似的驗證邏輯/資料結構/狀態轉換？

**行動路徑**：列出所有模組的驗證邏輯和資料結構，逐一比對相似度。

**範例**：
- [v0.17.0] 正面：BookSchemaV2 和 TagSchema 都驗證字串長度和格式 -> 應共用驗證引擎
- [v0.17.0] 反面：BookSchema 驗證 title（1-200 字）、BlogSchema 驗證 content（任意長度含 Markdown）-> 語義不同，不應共用

<!-- 擴充：Phase 4 檢討時追加真實案例 -->

#### Q2: 若有相似邏輯，選擇共用或各自實作？

**觸發條件**：Q1 回答「是」時必答。
**行動路徑**：填寫以下三項判斷依據，相似度 > 50% 且未來同步變更 -> 定義共用元件。

| 判斷項目 | 填寫內容 |
|---------|---------|
| (a) 相似度 | 行數/邏輯步驟的重疊比例 |
| (b) 未來變更是否同步 | 修改 A 時是否必然要改 B |
| (c) 若各自實作，重複行數估計 | 具體數字 |

**範例**：
- [v0.17.0] 正面：驗證框架相似度 70%，修改驗證規則時兩邊必須同步 -> 定義共用驗證引擎
- [v0.17.0] 反面：Book 和 Tag 的 CRUD 操作語義不同，各自實作更清晰

### 組 2：ID 欄位設計

**觸發條件**：資料結構含 ID 欄位時必答。

#### Q3: ID 生成涉及時序嗎？快速連續操作是否可能碰撞？

**觸發條件**：資料結構包含以 timestamp 為基礎的 ID 時必答。
**行動路徑**：評估批量操作場景（如迴圈建立、匯入）下的碰撞風險。

**範例**：
- [v0.17.0] 正面：cat_{Date.now()} 在批量建立 tag 時碰撞 -> 應改為 cat_{timestamp}_{random6}
- [v0.17.0] 邊界：單次操作且使用者手動觸發 -> timestamp 足夠，無需 UUID

<!-- 擴充：Phase 4 檢討時追加真實案例 -->

#### Q4: 唯一性策略選擇

**觸發條件**：Q3 回答有碰撞可能時必答。
**行動路徑**：從以下選項中選擇並記錄理由。

| 策略 | 適用場景 | 碰撞風險 |
|------|---------|---------|
| timestamp+random | 可讀性需求高、碰撞機率低 | 極低（加隨機後綴） |
| UUID v4 | 分散式系統、無需可讀性 | 接近零 |
| 序列號 | 單一來源、需排序 | 無（但需管理序列狀態） |

### 組 3：非功能性需求

**觸發條件**：所有功能規格必答。

#### Q5: 出錯時操作者如何知道？

**觸發條件**：功能包含 catch/error handling 時必答。
**行動路徑**：選擇通知機制（日誌/通知/靜默），若選擇日誌，定義等級和包含欄位。

**範例**：
- [v0.17.0] 正面：tag-storage-adapter 所有 catch 區塊零日誌 -> 規格應定義「CRUD 操作的 catch 必須記錄 warning 日誌」
- [v0.17.0] 反面：純讀取操作的快取 miss -> 不需要日誌，靜默即可

<!-- 擴充：Phase 4 檢討時追加真實案例 -->

#### Q6a: 本功能最慢的操作是什麼？預期延遲多少？

**觸發條件**：功能涉及 I/O 操作（網路/儲存/檔案）時必答。
**行動路徑**：識別瓶頸操作，定義可接受延遲上限。

<!-- 擴充：Phase 4 檢討時追加真實案例 -->

#### Q6b: 寫入中途失敗，已寫入資料如何處理？

**觸發條件**：功能涉及多步驟寫入操作時必答。
**行動路徑**：從以下選項中選擇並記錄理由。

| 策略 | 適用場景 | 複雜度 |
|------|---------|--------|
| 回滾 | 資料一致性要求高 | 高（需快照/事務） |
| 孤立 | 可容忍部分寫入、後續清理 | 低 |
| 重試 | 暫時性失敗為主因 | 中（需冪等設計） |

### 補充：欄位命名一致性

**觸發條件**：涉及 schema 版本遷移時必答。

#### Q_new1: 哪些欄位名稱在不同版本間有變化？

**觸發條件**：規格定義新版本 schema 且需與舊版本相容時必答。
**行動路徑**：建立完整映射表，確保遷移程式碼和消費端程式碼使用一致的欄位名稱。

| 舊版本欄位 | 新版本欄位 | 型別變化 | 遷移邏輯 |
|-----------|-----------|---------|---------|
| {v1 欄位} | {v2 欄位} | {型別差異} | {轉換方式} |

**範例**：
- [v0.17.0] 正面：v1 用 author（字串），v2 用 authors（陣列）-> 規格應有明確映射表，防止實作混用

## 轉換條件

### 進入 Phase 1 的條件

- Phase 0 審查通過（或合法豁免）
- 功能需求已有基本描述

### 退出 Phase 1 的條件（進入 Phase 2）

- [ ] 功能規格文件完整（輸入/輸出/流程）
- [ ] API 介面定義完成（函式簽名、資料結構）
- [ ] 驗收標準明確且可量化
- [ ] SOLID 分析完成（如適用）
- [ ] 行為場景（GWT）已提取
- [ ] **行為單元清單已產出**（每個單元可用單一 GWT 場景完整描述）
- [ ] **每個行為單元的邊界條件已歸屬**（不存在「無主」邊界條件）
- [ ] Decision Questions 已回答（依觸發條件判斷哪些必答）
- [ ] 跨模組共用決策已記錄（涉及 2+ 模組時）
- [ ] 非功能性需求已定義（所有功能規格）

> **框架整合**：Phase 1 由設計者角色執行，完成後自行提交（git commit），更新任務狀態，回報執行者：「Phase 1 完成，已提交」。
