# 決策呈現：選項 + 推薦 + 開放修改

> **角色**：本卡是 `requirement-protocol` 的支撐型原則（principle）、被 SKILL.md「相關抽象層原則」段與 reference `decision-dialogue.md`（步驟 5「呈現格式」維度）引用、是支柱 1「可見性確認」與原則 1「可決定 vs 該確認的邊界」在「決策呈現格式」面向的展開。
>
> **何時讀**：當你要把決策交給使用者選、或寫到「你覺得呢？」「你想怎麼做？」「ABCDE 你選哪個？」這類開放問時。讀本卡用三層格式（選項 + 適配性 + 推薦 + 開放修改）取代開放問。

---

## 結論

把決策交給使用者時、用三層格式：**選項列表 + 每選項適配性 + 標出推薦 + 「想改？」開放**。不要用「你覺得呢？」「你想怎麼做？」這類開放問。

開放問看似尊重、實際把「整理問題」的成本完全丟給使用者；推薦看似越權、實際讓使用者用「同意 / 反對」這個低成本動作完成決策。

---

## 為什麼開放問是反模式

開放問（"你想怎麼做？"）預設使用者**已經完成**這幾步：

1. 知道有哪些選項
2. 每個選項的成本與風險
3. 哪個選項最適合這個情境
4. 願意在當下重新整理思緒、把答案寫成完整指令

如果使用者已經做完這四步、他不會找你來問 — 他會直接下指令。會走到「需要決策」這個動作、通常是因為**手上沒有完整 1-3**、需要對方協助整理。

開放問 = 把「整理選項」這件事再丟回去、讓使用者重新做一遍你已經做過的功課。

---

## 三層格式的展開

### Layer 1：選項列表

把所有合理選項列出（包含「不做」這個選項、如果合理）。每個選項一行、不超過 6 個 — 超過代表還沒篩選完、不該丟給使用者。

### Layer 2：適配性 / 取捨

每個選項配一句「為什麼適合 / 不適合」 — 維度可以是成本、風險、相容性、複雜度等、依任務選 1-2 個維度。

### Layer 3：標推薦 + 開放修改

明確說「我推薦 X、因為 Y」、然後加一句「想改成其他選項或調整、跟我說」。推薦不是越權、是**把判斷攤開供質疑** — 使用者反對時知道反對什麼、同意時知道同意了什麼。

---

## 完整模板

```text
我看到三個方向：

| 選項 | 適配性 | 取捨 |
|---|---|---|
| A 用 X 庫 | 既有依賴、易維護 | 功能受限、要小妥協 |
| B 自己寫 | 完全控制 | 維護成本、邊界 case 要寫測試 |
| C 不做 | 0 成本 | 使用者繼續手動 |

我會選 **A**、因為這個 feature 不是核心、用既有依賴的維護負擔最低。想改成 B 或補充 C 沒問題、跟我說。
```

關鍵：使用者只需要回 "好"（同意）或 "改 B、原因是 ⋯⋯"（反對 + 提供新訊息）— 不必重新整理整個問題空間。

## 反模式對照

| 反模式                         | 為什麼不好                                   | 修法                                   |
| ------------------------------ | -------------------------------------------- | -------------------------------------- |
| "你想怎麼做？"                 | 把整個決策空間丟回去                         | 列選項、給推薦                         |
| "我可以選 A 或 B、你選哪個？"  | 沒講 A/B 差異、使用者要自己 reverse engineer | 補上每選項的適配性                     |
| "我建議 A"（沒講為什麼）       | 推薦不可質疑、使用者只能盲信或盲拒           | 補 reason、讓推薦可質疑                |
| "我覺得 A 比較好、不過 B 也行" | 推薦不夠明確、使用者不知道你真的傾向哪邊     | 標明推薦、別騎牆                       |
| 給 10+ 選項                    | 篩選沒做完、認知超載                         | 自己先篩到 ≤ 5、剩下歸為「其他可討論」 |
| "你授權我做嗎？"               | Yes/No 二選、缺中間態                        | 給選項表、讓使用者選擇執行範圍         |

## 何時可以例外（純開放問是合理的）

| 情境                                     | 為什麼開放問合理                        |
| ---------------------------------------- | --------------------------------------- |
| 探索期、選項本身還沒成形                 | 還沒到「該選」的階段、需要先 brainstorm |
| 使用者明確說「我來決定方向、你別給意見」 | 推薦會干擾                              |
| 主觀偏好題（命名、配色、文字風格）       | 沒「客觀適配性」可比較                  |
| 使用者已給出完整偏好、執行細節純客戶化   | 不需要推薦、純執行                      |

四類共同點：**整理選項這個工作不需要做、或不該由你做**。其他情境都該套三層格式。

## 判讀徵兆

| 訊號                                  | 該做的事                                               |
| ------------------------------------- | ------------------------------------------------------ |
| 寫到「你想怎麼做？」                  | 改成三層格式、把選項與推薦攤開                         |
| 推薦的 reason 寫成「我覺得 A 比較好」 | 補實質判準（成本 / 風險 / 相容性）                     |
| 一次列 7+ 選項                        | 自己沒篩夠、再過濾一遍                                 |
| 使用者回 "都可以、你決定"             | 推薦不夠明確、改成「我做 X、除非你反對」               |
| 推薦後使用者每次都同意                | 推薦變單純的「你決定」、檢查使用者是否還有實質參與空間 |
| 使用者回「為什麼？」追問 reason       | reason 寫不夠清楚、補強                                |

**核心**：決策呈現的目標不是「讓使用者參與」、是「讓使用者用最低成本參與」。低成本來自選項已經整理好、推薦已經攤開、只需要說「同意」或「改 X」。

## 與其他原則的串連

- 篩選指令的澄清三問：是「使用者下了模糊指令、你列三問澄清」、本卡是「你列了選項、用三層格式呈現」— 同一條協議的兩端 — 詳見 [`filter-instruction-clarification.md`](./filter-instruction-clarification.md)
- Filter × Source 五策略選擇矩陣：「五策略 × 適配性表」就是本卡 Layer 1+2 的具體展現 — 詳見 [`filter-source-composition-strategies.md`](./filter-source-composition-strategies.md)
- 寫作便利度跟意圖對齊反相關：開放問是「容易寫」的格式（少打字）、跟使用者意圖對齊（不重做功課）反相關 — 詳見 [`ease-of-writing-vs-intent-alignment.md`](./ease-of-writing-vs-intent-alignment.md)
- 高 ROI 無外部觸發：「列選項 + 標推薦」是高 ROI 但無觸發的工作（多打字、慢）、需要協議結構強制 — 詳見 [`external-trigger-for-high-roi-work.md`](./external-trigger-for-high-roi-work.md)
- 決策對話的五維度：本卡是「呈現格式」維度的展開 — 開放問 vs 結構表 + 推薦 — 詳見 [`decision-dialogue-dimensions.md`](./decision-dialogue-dimensions.md)
