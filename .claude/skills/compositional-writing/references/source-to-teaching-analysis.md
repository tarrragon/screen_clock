# Source to Teaching Analysis

本 reference 為「把外部分析文章或高密度研究材料，轉成教學型分析文章」情境。適用於分析師文章、投資人備忘錄、產業評論、研究摘要、AI 轉寫稿。目標是把 source 轉成讀者可重用的判讀框架，不是只做摘要、翻譯或語氣改寫。

> **自包含聲明**：本文件不依賴其他 reference。讀完本文件即可獨立執行外部分析材料到教學文章的轉換。

---

## 何時參閱本文件

| 觸發情境                               | 動作                                        |
| -------------------------------------- | ------------------------------------------- |
| 要把外部分析師文章改寫成自己的教學文章 | 跑完整 source-to-teaching pass              |
| AI 改寫稿讀起來流暢、但像原文摘要      | 跑 deliverable pass                         |
| 文章術語密度像寫給同業、目標讀者跟不上 | 跑 reader-level pass                        |
| 來源文章的觀點與本文結論混在一起       | 跑 source layering pass                     |
| 文章已套決策框架、但正文像分析過程記錄 | 跑 title promise 與 teaching narrative pass |

不適用：純翻譯逐句校稿、學術論文正式摘要、內部 analyst-to-analyst 報告。這些產物可以保留原文的 reader contract 或 process metadata。

---

## 為什麼外部分析改寫需要獨立指引

外部分析文章同時帶有三種東西：事實材料、原作者 frame、原作者 reader contract。AI 或人類若只做風格轉換，會把三者一起搬進新文章。結果是文字符合目標風格，但結論仍沿用原作者視角，讀者仍需要原領域背景，文章也缺少可遷移的教學框架。

教學型分析文章要做的是「source transformation」：保留可驗證事實，辨識原作者判讀，重建適合目標讀者的推導，最後交付可重用的判斷問題與預警訊號。

---

## 五個 Pass

### Pass 1：Source Layering

Source layering 是把材料拆成事實、原作者判讀、本文推導。每句材料進正文前先問：

| 問題                                 | 層級       |
| ------------------------------------ | ---------- |
| 這件事有公開來源可驗證嗎？           | 事實       |
| 這是不是原作者或市場敘事的解釋？     | 原作者判讀 |
| 這是不是本文從多個材料合成出的框架？ | 本文推導   |

寫法規則：

- 事實放在背景或事件段。
- 原作者判讀只當 hypothesis prior，不直接當本文結論。
- 本文推導要明確標成本文整理出的框架或判斷問題。

反例：

```markdown
這代表基礎設施廠商正在垂直整合 data pipeline。
```

問題：若「垂直整合」是本文推導，這句把推導寫成事實。

正例：

```markdown
本文把這個收購視為「基礎設施廠商往 data pipeline 延伸」的訊號。這是本文判讀，不是事件本身。
```

### Pass 2：Reader Level

Reader-level pass 是把原文 reader contract 改成目標讀者 contract。先辨識原文寫給誰，再決定是否降一級。

| 原文類型     | 常見 reader contract   | 轉換策略                 |
| ------------ | ---------------------- | ------------------------ |
| 大眾新聞     | 只需要知道事件         | 補機制與可遷移框架       |
| 投資人備忘錄 | 預設懂市場 shorthand   | 降術語密度、補因果鏈     |
| 產業內部文   | 預設懂供應鏈與競爭格局 | 補背景角色與資料形狀     |
| 教材型文章   | 已經有教學推導         | 檢查是否符合本站段落責任 |

檢查法：

- 三句內連續三個以上跨領域術語，拆段或補卡。
- 一句話跨三個以上因果步，拆成多段。
- 讀者能背結論但說不出推導，補機制、公式、數字或例子。

### Pass 3：Internal Analysis

Internal analysis 是作者背後的 hypothesis 探索，不等於文章章節。可以用 WRAP 或其他決策框架做內部檢查：

- 要回答什麼問題？
- 手上 evidence 夠不夠？
- 有哪些本質不同的合理解釋？
- 每個解釋的 evidence weight 是什麼？
- 哪些觀察會推翻本文判讀？

這些問題要做完，但不直接當正文標題。正文標題描述讀者會學到什麼，不描述作者正在執行哪個分析步驟。

### Pass 4：Title Promise

Title promise 是標題對讀者做的合約。跑完內部分析後，列出文章每段是否服務標題承諾。

| 檢查項                                 | 行動                           |
| -------------------------------------- | ------------------------------ |
| 某段超過 20% 篇幅但不在標題承諾內      | 壓縮成背景句、移到別篇、或刪除 |
| 來源 prior 比正文機制更長              | 縮減 prior，讓主體回到本文承諾 |
| 標題承諾的是框架、正文卻在解釋作者動機 | 改寫主體或重命名標題           |

Title promise pass 保護讀者不被作者的分析過程帶離主題。

### Pass 5：Deliverable

Deliverable pass 確認文章交付的是可遷移框架，而不是原文摘要。結尾至少回答：

- 下次遇到同類事件要看哪些訊號？
- 哪些機制若成立，本文判讀才成立？
- 哪些觀察會削弱或推翻本文判讀？
- 讀者下一步該看哪個概念、卡片或案例？

常見責任順序：

```text
事件本身 → 結構性機制 → 長期影響 → 預警訊號 → 可遷移框架
```

這不是固定模板，而是責任清單。不同文章可以調整順序，但不能缺少可遷移框架與失效條件。

## 自檢清單

- [ ] 每個來源句子都能標成事實、原作者判讀、本文推導之一。
- [ ] 原作者 frame 沒有被寫成事件事實。
- [ ] 目標讀者不需要原文 reader contract 也能理解主體推導。
- [ ] 三句內沒有未解釋的連續跨領域術語。
- [ ] 內部分析框架沒有外露成正文標題。
- [ ] 正文段落都服務 title promise。
- [ ] 結尾提供可遷移框架、預警訊號與下一步路由。

## 反模式速查

| 反模式                          | 修法                              |
| ------------------------------- | --------------------------------- |
| 只要求 AI「改成我們的風格」     | 改成「抽出可遷移判讀框架」        |
| 把原作者分類直接當本文標題      | 標出這是原作者 frame 或本文 frame |
| 引用機構名但沒有具體來源        | 刪具名 attribution 或重新查證     |
| 章節標題是分析 process          | 改成教學標題                      |
| 文章結尾只有事件結論            | 補下次遇到同類事件的判斷問題      |
| 讀者需要懂原領域 shorthand 才懂 | 降術語密度，補 bridge sentence    |

**核心原則**：外部分析文章改寫的核心是 source transformation。保留事實、分離原作者判讀、重建目標讀者可理解的推導，最後交付可遷移框架。
