# 教材用中性 register：陳述概念、不喊話 / 不誇飾 / 不必然

> **角色**：本卡是 `compositional-writing` 的支撐型原則（principle）、被 SKILL.md 原則三（機會成本語氣）+ `references/writing-articles.md` multi-pass review 輪 8（keyword bank）引用。
>
> **何時讀**：寫技術教材 / 設計檢討時、用本卡判斷字句層的 register（語氣立場）跟 framing（框架）是否偏離「中性陳述概念」——尤其在 multi-pass review 輪 8 跑 keyword bank 比對。

---

## 論述基礎與限制

本卡的論述基於 **1 個 case**（一篇技術教材 review 一次抓到三種違反）抽出、建卡判準是教學需求（這三類在技術寫作、尤其 LLM 生成內容裡是高發 pattern）。具體限制：

- **keyword bank 是 starting set、不窮舉**：列出的詞是該 case 出現過的、新形式持續擴充。
- **三類靠「對象」區分、邊界有時模糊**：喊話對象是讀者、誇飾對象是方案、必然對象是設計選擇 —— 某些句子可能跨類、需個案判讀。
- **修補有效性未獨立驗證**：改成中性 register 後讀起來更聚焦、但跟讀者實際理解的相關性沒做使用者測試。

讀者使用本卡時、先判斷段落是否屬於「概念建立 / 判斷工具型」——是 → 套用；否（hook / narrative / 復盤）→ 放寬。

---

## 核心原則

**教材的 register 是中性陳述概念：說明「這是什麼、為什麼這樣、在什麼條件下成立」，不對讀者喊話、不替方案打分數、不把設計選擇講成自然法則。** 三類違反共享同一個失敗 —— 把「需要依據 / 條件的內容」包裝成「不需要依據的形式」（喊話偽裝成親近、誇飾偽裝成事實、必然偽裝成天性），躲過讀者跟 reviewer 的審查。

| 違反類型   | 喊話 / 誇飾 / 必然的對象 | 句型                                              | 中性改法                                    |
| ---------- | ------------------------ | ------------------------------------------------- | ------------------------------------------- |
| 對讀者喊話 | 讀者（管理情緒 / 節奏）  | 「很多人卡在」「你天天寫的 X」「先讀懂 / 別搞混」 | 直接陳述概念 / 描述性名詞標題               |
| 自評誇飾   | 方案（評價品質）         | 「X 是教科書級的適配」「堪稱經典」「完美契合」    | 換成機制 / 條件（X 為什麼適配）             |
| 必然性框架 | 設計選擇（絕對化）       | 「X 天生就是 Y」「本質就是」「必然」              | 還原條件性（選了某前提後 X 才以此形式成立） |

三類都比「命令式絕對」（「應該做 X」）更隱形：命令式讀者聽得出是主張、會質疑；這三類偽裝成親近 / 事實 / 天性、讀者直接接受。

---

## 三類的具體判讀

### 對讀者喊話

教材的責任是把概念講清楚、不是管理讀者的情緒或閱讀節奏。三種形式：

- **安撫 / 訴諸群體**：「很多人卡在 X」—— 讓讀者覺得不孤單、無資訊量。
- **第二人稱代入**：「你天天寫的 `int count`」—— 把讀者拉進句子、預設讀者日常。
- **祈使控制閱讀**：標題「先讀懂 X」「別搞混」—— 指揮閱讀順序 / 動作。

修法是換中性指稱（常見的 / 一般而言）或描述性名詞標題（「簽章的型別與名字拆解」取代「先讀懂這個簽章」）。

**邊界**：hook / 引言 / narrative 開場的輕度第二人稱（「假設你在維護一個…」）能幫讀者進入情境、不一律禁。protocol 型文件（操作步驟 / checklist）的祈使是合理 register。

### 自評誇飾

品質 verdict（教科書級 / 堪稱經典 / 完美 / 漂亮地）傳遞作者滿意度而非概念、且會**頂替技術理由** —— 寫了「X 是教科書級的適配」就佔掉了「X 為什麼適配」的位置。修法是把 verdict 換成具體機制 / 條件。

**判別**：刪掉誇飾詞後、適配理由若仍完整 → 純贅字、直接刪；若只剩 verdict 沒有理由 → 理由根本沒寫、補機制（不是補誇飾）。

### 必然性框架

「天生 / 本質就是 / 必然 / 唯一」把設計選擇講成自然法則、抹掉設計能動性。讀者拿到沒有條件的規則、不知邊界（換情境就不適用）。高發訊號是**局部牴觸作者自己在別處的條件性立場**（通篇講「看條件」、唯獨一句講「天生」）。

**判別線**：這個必然有沒有上游設計選擇當前提？有（「不可變模型下、更新是 current → next」、前提=選了不可變）= 條件性、要講前提；無（「任何雜湊函式都有碰撞」）= 真必然、可斷言。物理 / 法律 / 數學 / 合規事實可用必然語氣。

## keyword bank（輪 8 比對）

```text
對讀者喊話：
  很多人 / 大家 / 不少人 / 你天天 / 你會 / 你可能 / 先讀懂 / 先釐清 / 別搞混 / 別被
自評誇飾：
  教科書級 / 堪稱 / 可謂 / 完美 / 經典 / 範本級 / 大師級 / 漂亮地 / 優雅地 / 最佳實踐 / best practice
必然性框架：
  天生 / 與生俱來 / 本質就是 / 本來就是 / 必然 / 唯一 / 註定 / 理所當然
```

**命中是候選、不是判決**：grep 命中後仍要一個語意判定步驟——這個命中是「概念建立段的違規」、還是「合規的 hook / narrative / 反例對照 / 真必然」。偵測（bank）跟判定（語意）是兩個認知步驟、reviewer 容易把違規合理化放行。判定層的完整論述見 [multi-pass-review-frame-granularity](multi-pass-review-frame-granularity.md)。
