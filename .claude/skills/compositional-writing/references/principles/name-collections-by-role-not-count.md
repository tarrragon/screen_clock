# 集合命名用角色、不內嵌數量

> **角色**：本卡是 `compositional-writing` 的支撐型原則（principle）、被 SKILL.md 原則 2（索引建立）與 `references/principles/reference-by-semantic-title-not-number.md` 引用、是後者的命名端 sibling。
>
> **何時讀**：命名一個集合型結構（問題清單、階段序列、原則組、支柱組）時；或集合成員增減、發現名稱裡的數字失真時。

---

## 核心原則

**集合的名稱只承載角色與層級、把成員數量留給清單自己呈現。**「核心問題」「次要問題」「撞牆階段」「防護底線」是穩定名稱；「核心七問」「成長六階段」「四大支柱」把當前成員數烤進名字 — 數量是集合成員的 derivation、不是集合的語意身分。加一問、七就變八、名稱先失真、所有複製過這個名稱的地方（標題、引用、索引、目錄、對話）跟著過期。

區分核心問題跟次要問題、靠「核心 / 次要」這組角色詞就完成了。數字的記憶掛鉤價值（「五原則我只想起四個」）真實存在、但只有凍結集合收割得到 — 活集合下它被漂移風險壓過、路由本身角色詞已經完成。

---

## 為什麼比編號引用更深一層

「引用錨在語意標題」的前提是標題是這個單位的 fact。count-bearing 標題打破前提 — 它一半是語意（核心問題）、一半是 derivation（七）。成員一變、兩條路都有代價：

| 成員變動後的選擇 | 代價                                                          |
| ---------------- | ------------------------------------------------------------- |
| 全面改名         | 名稱出現過的每一處都要修、等同一次結構重排                    |
| 保留舊名         | 名實不符常態化 — 讀者數出是八問、開始懷疑文件其他部分的可信度 |

且名稱是知識系統裡被複製最多次的字串、缺陷隨每次引用繁殖 — 編號錯位寄生在引用句、數量失真寄生在權威命名層本身。

實際 case：同一份多階段協議文件、初版「四大支柱」在第二版需求調整後變六個 — 被迫全面改名成「六大支柱」、改完的新名字仍內嵌數量、下次支柱增減原樣重演。同一文件的「核心七問」在加問題時會走上同一條路。

---

## 修法

1. **命名時抽掉數字、看角色詞夠不夠分層**：幾乎都夠（核心 / 次要、必展開 / 觸發展開、底線 / 可延後）。
2. **數量讓清單自己呈現**：讀者需要知道數量的時刻、就是看到清單的時刻、清單天然自帶數量。
3. **成員增減的 commit 檢查名稱誠實度**：需要改名的話、這次順便把數量抽掉、別讓下次再來一遍。
4. **掃描候選**：`rg "[一二三四五六七八九十0-9]+ ?(大|問|階段|支柱|原則|步驟|件事|個維度)" --type md` — 命中後用「fact 還是 derivation」判別。

## 邊界：三種數字可以留

1. **外部凍結的品牌名**：SOLID 五原則、OWASP Top 10、WRAP 四步驟 — 數量由發布方凍結、是名稱 fact 的一部分。
2. **數字是概念內容本身**：「兩次門檻」的二是規則的閾值、不是成員數 — 改了二就是改了概念、該留。
3. **緊鄰清單的行內計數**：「確認三件事：」後面直接跟三條列 — 數字跟清單同視野、漂移在編輯當下可見。風險低、但清單就在下面時「確認下列事項：」更省。
4. **內部宣告凍結的集合**：明文宣告凍結的內部方法論名可帶數字（當內部品牌用）— 代價比凍結編號更嚴、成員連加都不能加、宣告前先確認集合真的不會再長。
