# 任務啟動快速 WRAP（Claim Quick WRAP）

任務啟動快速 WRAP（Claim Quick WRAP）的責任是在開始一個任務前，用 1-2 分鐘檢查選項、機會成本與失敗防護。它提供任務啟動前的最低品質閘門；高風險任務接著進入完整 WRAP。

---

## 使用時機

| 情境                 | 使用方式                                             |
| -------------------- | ---------------------------------------------------- |
| 接手小型實作任務     | 跑 W/A/P 三問即可開始                                |
| 接手文件或規則任務   | 跑 W/A/P，並補一個來源或既有規則檢查                 |
| 接手分析或根因任務   | 快速 WRAP（Quick WRAP）只做啟動檢查，接著跑完整 WRAP |
| 三問中出現高風險答案 | 暫停任務，改跑完整 WRAP                              |

---

## 三問模板

### W：還有什麼路徑

**問題**：除了我第一個想到的做法，還有哪兩個可行路徑？

**最低回答格式**：

```text
- 首選路徑：...
- 替代路徑 A：...
- 替代路徑 B：...
- 放棄理由：...
```

**通過標準**：至少有兩個替代路徑，且差異落在假設或交付方式上，避免只做同一路徑的參數微調。

### A：會擠壓什麼

**問題**：執行這個任務會擠壓哪個更重要的目標或工作？

**最低回答格式**：

```text
- 預估投入：...
- 擠壓目標：...
- 延後代價：...
- 現在做的理由：...
```

**通過標準**：回答要指向具體目標、里程碑或使用者價值，避免只寫「影響進度」。

### P：最可能怎麼失敗

**問題**：如果這個任務完成後仍然失敗，最可能的三個原因是什麼？

**最低回答格式**：

```text
- 失敗原因 1：...；防護：...
- 失敗原因 2：...；防護：...
- 失敗原因 3：...；防護：...
```

**通過標準**：每個失敗原因都有對應防護；任一原因機率高於 50% 時，先調整方案再開始。

## 任務類型差異

| 任務類型 | W 重點                                   | A 重點                       | P 重點                                 |
| -------- | ---------------------------------------- | ---------------------------- | -------------------------------------- |
| 實作任務 | 比較新增、改造既有、純文件或零工具方案   | 檢查是否擠壓核心功能或修復   | 防止測試不足、範圍外擴張、回退成本過高 |
| 文件任務 | 比較新增文章、補既有文章、補索引或補卡片 | 檢查是否擠壓更高價值章節     | 防止重複、連結失效、抽象層錯置         |
| 分析任務 | 比較多個根因假設                         | 檢查是否擠壓可直接驗證的工作 | 防止過早收斂、缺來源、缺反向驗證       |
| 規則任務 | 比較改規則、改流程、加提示或加檢查       | 檢查是否增加長期摩擦         | 防止規則自相矛盾或過度限制使用者       |

## 升級完整 WRAP 的訊號

| 訊號                                     | 升級原因                                     |
| ---------------------------------------- | -------------------------------------------- |
| 只有一個選項                             | W 階段選項品質不足                           |
| 替代方案都指向同一假設                   | 偽擴增選項（pseudo-Widen），需要重新定義問題 |
| 機會成本大於任務價值                     | A 階段需要重新排序                           |
| 最可能失敗原因高於 50%                   | P 階段已顯示方案脆弱                         |
| 任務屬於根因分析、提案評估、重大架構決策 | 快速 WRAP（Quick WRAP）資訊量不足            |

## 常見反模式

| 反模式           | 表現                                   | 修正方式                                            |
| ---------------- | -------------------------------------- | --------------------------------------------------- |
| 儀式化填答       | 每題只寫一句空話                       | 寫出具體路徑、成本與防護                            |
| 假多選項         | 三個選項都是同一方案變體               | 回到假設層級重列選項                                |
| 機會成本空白     | 寫「無擠壓」但任務會占用核心時間       | 指出被延後的具體目標                                |
| 防護不可驗證     | 防護寫「小心」或「注意」               | 改成測試、檢查、回退或限制範圍                      |
| 分析任務只跑三問 | 用快速 WRAP（Quick WRAP）取代完整 WRAP | 進入完整 WRAP，補現實檢驗（Reality Test）與反向驗證 |

## 可複製提示

```text
開始前先跑快速 WRAP（Quick WRAP）：

W：除了第一個方案，還有哪兩條可行路徑？為什麼選現在這條？
A：這會擠壓哪個更重要的目標？現在做的理由是什麼？
P：如果完成後仍失敗，最可能的三個原因是什麼？各自的防護是什麼？
```
