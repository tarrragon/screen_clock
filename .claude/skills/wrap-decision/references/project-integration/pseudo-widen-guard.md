# 偽 Widen vs 真 Widen — 本專案落地

本文件為 WRAP 的 W 階段在**分析類任務**（特別是 Bug 根因調查、代理人失敗歸因）的強化補丁。源自 PC-063 W5-031 案例：Widen 列出 4 個候選方案全部脫靶，暴露出「選項多元性」的假象。

> **SKILL.md 留抽象原則（假設層級多元性），本檔放本專案完整防護機制 + PC-063 案例**。ANA 類型 Ticket 強制執行本檔規格。

---

## 核心警示

**Widen 出多個方案 ≠ 真正擴增了選項空間。**

選項要在「**假設根因**」層級多元，而不只是「**實作手段**」層級多元。

## 偽 Widen vs 真 Widen 判別

| 維度 | 偽 Widen | 真 Widen |
|------|---------|---------|
| 選項差異層級 | 實作手段不同（哨兵值 vs gitignore vs hook） | 假設根因不同（mock 漏 vs 環境隔離缺失 vs 上游邏輯錯） |
| 共同前提 | 全部接受同一假設根因 | 列出多種可能根因，每個都有對應方案 |
| Reality Test | 跳過 / 流於形式 | 強制執行，每個假設都需證據 |
| 結果 | 候選方案脫靶率高（PC-063: 4/4 脫靶） | 至少有一個方案命中真因 |

## PC-063 案例（W5-031）

**情境**：代理人派發連續失敗，PM 啟動 WRAP 分析。

**Widen 結果**（4 個候選方案）：
- A：用哨兵值標記版本字面值
- B：加 gitignore 排除版本字面值
- C：寫 pre-commit hook 檢查版本字面值
- D：修改 CI 檢查版本字面值

**表面看起來**：方案多元（哨兵/gitignore/hook/CI，4 種不同實作手段）

**實際上**：4 個方案**全部接受同一個未驗證的假設根因**：「問題是版本字面值造成的」。

**重現實驗後發現**：真正的根因完全不是版本字面值。後續衍生的方案 F（mock 漏驗證）/H（環境隔離缺失）才命中。

**脫靶率**：**4/4 = 100%**。

**教訓**：Widen 的多元性必須在**假設層級**驗證，不是在實作層級。

## 質疑假設步驟引導

進入 Widen 「爬梯子法」前，**先做假設層級盤點**：

### Step 1：列出當前接受的根因假設

明確寫出：「我假設問題是 **X** 造成的」。

如果連一句話都寫不出來，表示根因假設是**隱性**的 — 這是偽 Widen 最常見的起點（「我沒假設什麼啊」= 沒意識到自己已經假設了）。

### Step 2：三層質疑

#### Level 1：質疑根因

> 「除了 X 還有什麼可能？」

列 2-3 個替代根因。若列不出來：
- 可能是資料不足 → 回到 Step 0 資料充足度檢查
- 可能是確認偏誤 → 觸發 Consider the Opposite

#### Level 2：質疑問題框架

> 「真正該問的問題是 X 嗎？還是其實是 Y？」

這層質疑與 Consider the Opposite 銜接。本專案範例：
- 表面問題：「tool call 閾值該設多少？」
- 真正問題：「任務的 SRP 品質是否足夠？」

如果第一個問題是錯的框架，所有後續 Widen 都是在錯誤的地圖上找路。

#### Level 3：質疑場景

> 「這個問題在什麼條件下不會發生？」

反向找邊界，可能揭露隱藏變因。例如：
- 如果「連續失敗」不發生時 → 代理人有什麼特徵？
- 這些特徵揭露的隱藏變因可能就是根因。

### Step 3：Reality Test 驗證

對每個假設根因最低做一次**重現實驗**。

重現實驗規格見 `.claude/pm-rules/incident-response.md`「Reality Test 閘門」章節。

### Step 4：基於驗證後的根因列方案

每個 Widen 出的方案對應一個「被驗證」或「暫未否決」的根因；**禁止全部對應同一假設**。

## 偽 Widen 警告信號

| 信號 | 含義 | 應對 |
|------|------|------|
| 4 個方案都改同一檔案類型 | 可能是同假設變體 | 質疑根因，嘗試找出指向其他位置的方案 |
| 方案描述圍繞同一動詞（全是「清理」/「禁止」/「遮罩」） | 框架太窄 | Consider the Opposite，列出反向動詞 |
| 沒有任何方案改動「我以為的根因」之外的位置 | 未質疑根因 | 強制執行 Level 1 質疑 |
| 列方案時跳過 Reality Test | 必然產生偽 Widen | 退回 Reality Test 閘門，先收集證據 |
| 方案間「互斥」感很強（必須二選一） | 假設根因唯一 | 真正多元的根因會產生「互補」方案 |

## 執行時機

**強制執行**：
- 所有 ANA 類型 Ticket（簡化三問不足以保證分析品質）
- Bug 根因調查（疑難雜症、連續失敗、重複修復）
- 代理人失敗歸因

**建議執行**：
- IMP ticket 的 Solution 設計階段（若方案選擇有 3+ 候選）
- 重大架構決策

**可略過**：
- 資訊查詢
- DOC 類型 ticket 的「有無類似文件」W 檢查
