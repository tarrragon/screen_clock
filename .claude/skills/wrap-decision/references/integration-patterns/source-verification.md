# Source Verification Pattern

來源核對模式的責任是防止清單類答案被整批信任。它要求逐項對照來源，並把找不到來源的項目標成候選幻覺。

---

## 適用情境

| 情境                               | 風險                   |
| ---------------------------------- | ---------------------- |
| LLM 列出 API、欄位、規則或功能清單 | 可能補齊不存在的項目   |
| Agent 回報某套工具的能力           | 可能混入相似工具的功能 |
| 文章引用外部標準或框架             | 可能把推論寫成來源事實 |
| 規則設計依賴既有設定               | 可能讀到過期或錯誤記憶 |

---

## 核對流程

1. 找到權威來源：官方文件、本機 schema、原始設定、標準文件或實際程式碼。
2. 拆成逐項清單：每個欄位、能力、規則或聲明各自核對。
3. 標記來源狀態：confirmed、inferred、missing、conflict。
4. 對 missing 與 conflict 項目重新查證或移除。
5. 在輸出中標示哪些是來源事實，哪些是分析推論。

---

## 來源狀態表

| 狀態      | 意義                   | 可否作為結論       |
| --------- | ---------------------- | ------------------ |
| confirmed | 來源明確支持           | 可以               |
| inferred  | 來源間接支持，需要推論 | 可以，但需標示推論 |
| missing   | 找不到來源             | 不作為結論         |
| conflict  | 來源互相矛盾           | 先釐清版本或範圍   |

---

## 反模式

| 反模式                 | 修正                               |
| ---------------------- | ---------------------------------- |
| 找到一個來源就整批信任 | 每一項逐項核對                     |
| 把相似工具功能混用     | 只引用當前工具來源                 |
| 把最佳實務寫成標準要求 | 區分 recommendation 與 requirement |
| 把推論包裝成引用       | 明確標示「由來源推論」             |
