# 以價值 / 容量 / 優先級為決策依據（規則 6 主文）

> **載入方式**：按需讀取。觸發時機詳見 `ai-communication-rules.md` 規則 6 摘要與「何時讀本檔」指引。
> **本檔範圍**：完整 Why / Consequence / 替代句型表 / hotpath 對照表 / 與規則 5 的雙向銜接。

---

## 副標 / grep 訊號層

> 禁止以估時為決策依據（保留供 hook 偵測估時話術用）

---

## Why

估時是可量化的誘餌，容易取代「是否應該做」的本質判斷。AI 輔助開發情境下估時準確度通常偏低（token 預算波動大、推理深度難預測）；以估時驅動決策接近於以雜訊驅動。估時框架還會讓代理人傾向「討好式選擇快的路徑」，遮蔽正確但慢的解法（規則 5 §5.6 機制 4 反諂媚設計的變體）。

## Consequence

以估時驅動決策會導致下列退化模式：

1. 跳過多視角審查（理由：「太耗時」）
2. 選簡化版而非正確版（理由：「先做能做的」）
3. 把「做不到」偽裝成「時間不夠」（避免承認能力或規格缺口）
4. 代理人輸出品質隨估時壓力下降（趕工 → 不檢驗 → 出錯）

## Action：替代句型表

禁止在決策陳述中出現以下框架，改用替代句型：

| 禁止句型 | 替代句型 |
|---------|---------|
| 「預估花太久所以跳過 X」 | 「X 的價值是 Y；若超出 Wave 容量則建 Ticket 延後」 |
| 「這樣會花太多 token」 | 「此方案需要 Z 資源；若超出預算則拆 Wave」 |
| 「先做快的，慢的之後補」 | 「先做正確的；若有容量約束則依優先級排序」 |
| 「時間緊迫不跑多視角」 | 依摩擦力方法論判斷階段，前期階段（Proposal / Phase 0 / 1）無豁免 |

**邊界說明**：估時的合理用途包含（1）資源規劃輸入；（2）Wave 容量檢查；（3）向用戶報告進度。本規則約束的場景限於估時進入「做 vs 不做」「做 A vs 做 B」的決策邏輯——此時應以價值 / 容量 / 優先級替代。

## Hotpath 補強對照表（防止規則 6 在高頻決策場景被變相繞過）

| Hotpath 場景 | 違規訊號 | 正確處理 |
|-------------|---------|---------|
| Wave 容量判斷 | 「Wave 容量不足要延後」未量化 | 容量 = token 預算 + ticket 優先級，非時間；用 `ticket track runqueue` 與 priority 排序判斷 |
| 代理人遇 context 上限 | 「token 不足所以跳過」 | 明示為「資源不足」，觸發 WRAP（Widen 找替代解法 / Attain distance 重評優先級）而非跳過 |
| PM 派發優先級排序 | 用戶問「哪個先做」時回答「短任務先做」 | 依 `blockedBy` 與 Wave 策略排序，不依估時 |

## 與規則 5 的銜接

規則 5 §5.6 機制 4 禁止「最佳實踐 / 業界標準 / Recommended」等諂媚框架；本規則禁止「估時過長」作為諂媚用戶的捷徑。兩者同為 receiver-end 前提查驗的延伸——前者保護用戶免於 Claude 用權威話術引導，後者保護用戶免於 Claude 用時間話術引導。

**雙向交叉引用**：

- 規則 5 觸發情境表已加註「§5.6 機制 4 反諂媚設計（亦覆蓋估時諂媚，詳規則 6）」與新列「準備用估時 / 太久 / token 不夠當決策依據」指向規則 6
- 修訂規則 5 §5.6 機制 4（位於 `power-asymmetry-rules.md`）時應在末段加註「此設計亦覆蓋估時諂媚（詳本檔規則 6）」，保持雙向因果鏈可追蹤
