# 版本決策案例研究

本文件包含版本推進決策的實際案例和錯誤案例分析，供學習和參考。

> 核心規則：@.claude/pm-rules/version-progression.md

## 實際案例驗證

使用 Q1-Q4 判斷流程來驗證實際決策場景：

| 案例 | Q1 同主題？ | Q2 獨立發布？ | 決策 |
|------|-----------|-------------|------|
| 0.31.0 W4 驗證 | 是（Ticket 系統） | - | Wave（W4） |
| UC-02~09 區塊測試 | 否（UC 測試 vs Ticket 系統） | 是 | 新 Patch（0.31.1） |
| W4 中發現的 Hook 缺陷 | 是（Ticket 系統完善） | - | Wave（W5） |
| 用戶要求新 UC 功能 | 否 | 是 | 新 Minor |

## 錯誤案例分析

### 案例 1：Plan-to-Ticket 機制

**錯誤判斷**：
```
用戶: 實作 Plan-to-Ticket 機制

Claude: "Plan-to-Ticket 功能是一個全新的功能，應該使用 v0.32.0"
```

**正確判斷**：
```
用戶: 實作 Plan-to-Ticket 機制

Claude: "這是當前版本開發過程中發現的流程缺口，
        應該在當前版本後續 Wave 處理"
```

**分析**：
- Plan-to-Ticket 是開發 Ticket 系統時發現的缺口
- 屬於「流程缺口」類型
- 應在當前版本新增 Wave 處理
- 對應決策樹：**二元判斷樹 → 開發過程中發現 → 流程缺口**

---

### 案例 2：新增驗證 Hook

**錯誤判斷**：
```
發現 Ticket ID 格式可能錯誤

錯誤: "需要建立新版本來處理這個問題"
```

**正確判斷**：
```
發現 Ticket ID 格式可能錯誤

正確: "這是當前版本的工具改善，在 W4 或 W5 處理"
```

**分析**：
- 新增 Hook 是為了改善開發效率
- 屬於「工具改善」類型
- 應在當前版本新增 Wave 處理
- 對應決策樹：**二元判斷樹 → 開發過程中發現 → 工具改善**
- **強制規則 2**：工具改善不推進版本

---

### 案例 3：版本偵測誤判

**錯誤判斷**：
```
v0.31.1 目錄已存在 → get_current_version() 回傳 v0.31.1
→ 新 Ticket 自動歸入 0.31.1（錯誤）
```

**正確判斷**：
```
todolist.yaml 中 0.31.0 標記為 active
→ get_current_version() 回傳 v0.31.0（正確）
→ 新 Ticket 歸入 0.31.0
```

**分析**：
- 版本目錄存在不代表該版本是活躍的
- 活躍版本應由 `todolist.yaml` 的 `status` 欄位決定
- 目錄掃描僅作為 fallback
- 對應規則：**強制規則 4**：活躍版本由 todolist.yaml 決定

## 決策樹應用示例

### 示例 1：用戶提出新功能 - 新的 UC 實作

**流程**：
```
用戶: "能否新增 UC-15 的用例"

判斷: 二元判斷樹
    開發過程中發現？ NO
    → 用戶主動提出新需求？ YES
    → 進入 Q1-Q4 判斷

[Q1] 和當前版本開發主題相同？ NO（Ticket 系統 vs 新 UC）
[Q2] 完成後能獨立發布？ YES
→ 新 Patch 版本（0.31.1）

[Q4] 達成用戶可感知的功能里程碑？
    如果完成多個 UC → 新 Minor（0.32.0）
    如果只是單一 UC → 繼續 Patch 系列（0.31.1, 0.31.2...）
```

### 示例 2：開發中發現技術債務

**流程**：
```
Phase 3b 中發現: "程式碼重複度高，應該重構"

判斷: 二元判斷樹
    開發過程中發現？ YES
    → 技術債務類型
    → 當前版本 TD Ticket

決策: 記錄為技術債務，根據風險等級
    - 高風險 → 下個 Patch 版本處理
    - 中風險 → 下個 Minor 版本處理
    - 低風險 → 技術債務清理版本處理
```

### 示例 3：發現流程缺口

**流程**：
```
執行 Ticket 時發現: "Plan 檔案無法自動轉換為 Ticket"

判斷: 二元判斷樹
    開發過程中發現？ YES
    → 流程缺口
    → 當前版本新增 Wave

決策: 在 0.31.0 新增 W5 處理 Plan-to-Ticket 機制
```

## 學習要點

### 1. 優先認知：開發衍生 vs 新需求

所有版本決策的第一步是區分：

| 類型 | 判定 | 處理 |
|------|------|------|
| 開發衍生 | 在開發過程中發現 | 當前版本處理 |
| 新需求 | 用戶主動提出 | 使用 Q1-Q4 判斷 |

### 2. Q1-Q4 判斷的本質

每個問題都是在問：「這組工作在語義上屬於什麼層級？」

- **Q1 - 同主題**：屬於當前版本的進一步完善（Wave）
- **Q2 - 獨立發布**：是否能獨立為新版本（Patch）
- **Q3 - 依賴關係**：是否有前置依賴（Wave vs Patch）
- **Q4 - 用戶感知**：是否達到用戶可感知的功能里程碑（Minor）

### 3. 強制規則的應用

四條強制規則是版本推進的底線：

1. 開發衍生問題不推進版本
2. 工具改善不推進版本
3. 版本推進需要語義理由
4. 活躍版本由 todolist.yaml 決定

### 4. 常見誤判情況

| 誤判 | 原因 | 改正 |
|------|------|------|
| 將流程缺口當新功能 | 混淆了開發衍生和新需求 | 使用二元判斷樹 |
| 將工具改善當新版本 | 忽視強制規則 2 | 在當前版本新增 Wave |
| 使用目錄掃描決定版本 | 誤解活躍版本定義 | 查看 todolist.yaml 的 status |
| 版本號隨意變更 | 缺乏語義支撐 | 執行完整 Q1-Q4 判斷 |
