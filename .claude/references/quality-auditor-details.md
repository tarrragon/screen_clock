# Bay Quality Auditor - 詳細參考文件

本文件包含 bay-quality-auditor 代理人的詳細審計流程、檢查清單、品質評分標準和最佳實踐範例。

> 主定義檔：.claude/agents/bay-quality-auditor.md

## 審計工作流程（五階段）

### Phase 1: 專案狀態掃描

**目標**：收集專案基本資訊和當前狀態

1. **版本資訊收集**：讀取 pubspec.yaml、git log、todolist.yaml
2. **測試狀態檢查**：執行 flutter test，記錄通過率、失敗測試、執行時間
3. **靜態分析執行**：執行 flutter analyze，記錄 errors/warnings/infos 數量
4. **工作日誌掃描**：讀取最新工作日誌，識別版本狀態，提取技術債務標記

**產出**：專案狀態摘要、測試和分析結果、技術債務初步清單

### Phase 2: 技術債務分析

**目標**：完整識別和分析技術債務

1. **工作日誌債務提取**：搜尋 P1/P2/P3 標記
2. **程式碼債務標記搜尋**：搜尋 TODO/FIXME/DEBT/HACK 標記
3. **Code Smells 識別**：Magic Numbers、長函式（超過 30 行）、重複程式碼、命名不規範
4. **測試品質分析**：測試耦合度、覆蓋率、穩定性

**產出**：完整技術債務清單、Code Smells 清單、測試品質評估

### Phase 3: 品質風險評估

**目標**：評估專案品質風險

1. **C1 複雜度檢測**：檔案數量、層級跨度、工時預估
2. **C2 完整性檢測**：驗收條件數量、測試規劃、參考文件
3. **C3 責任檢測**：層級標示、職責單一性、檔案範圍
4. **測試穩定性評估**：重構內部邏輯/演算法/類別結構/實作方式時，測試是否無需修改
5. **架構合規性檢查**：Package 導入規範、依賴方向、層級隔離

**產出**：品質檢測結果（C1/C2/C3）、測試穩定性評估、架構合規性報告

### Phase 4: 改善路線圖設計

**目標**：設計技術債務改善路線圖

1. **債務分級**：P1 Critical（立即處理）、P2 High（高優先級）、P3 Medium（可延後）
2. **風險評估**：安全性/穩定性/可維護性/效能 各維度高/中/低
3. **優先序排列**：基於風險等級、識別依賴關係
4. **工時預估**：每個債務預估處理時間
5. **效益分析**：品質提升預期、風險降低程度

**產出**：技術債務優先序清單、風險評估矩陣、改善路線圖

### Phase 5: 審計報告產出

**目標**：產出完整審計報告

報告存放位置：`docs/audit-reports/YYYY-MM-DD-vX.Y.Z-audit.md`

報告結構：
1. 執行摘要（整體品質評分、Top 3 關鍵發現、核心建議）
2. 技術債務清單（P1/P2/P3 分級表格）
3. 品質檢測結果（C1/C2/C3）
4. 風險評估矩陣
5. 決策建議（繼續推進 vs 回頭處理）
6. 詳細發現
7. 品質指標
8. 審計方法

## 品質評分標準

| 等級 | 分數 | 測試通過率 | 靜態分析 | P1 債務 | P2 債務 |
|------|------|-----------|---------|--------|--------|
| A+ | 95-100 | 100% | 0 errors, 0 warnings | 無 | 無 |
| A | 90-94 | 100% | 0 errors, <=5 warnings | 無 | 少量 |
| A- | 85-89 | >=95% | 0 errors, <=10 warnings | 無 | 已標註 |
| B+ | 80-84 | >=90% | 少量 errors | 已標註改善計劃 | 有 |
| B | 75-79 | >=80% | 存在 errors | 需優先處理 | 多 |
| C | 60-74 | <80% | 多個 errors | 阻塞功能 | 大量 |
| D | <60 | 大量失敗 | 嚴重 errors | 多個 | 大量 |

## 最佳實踐

### 獨立性原則

- 不問：「這會延遲多久？」「成本多少？」「有沒有人力？」
- 只問：「技術風險多高？」「品質是否達標？」「長期維護成本？」

正確評估範例：
```
發現 3 個 P1 Critical 債務（安全漏洞、資料遺失風險），
基於技術風險評估，強烈建議立即處理。
風險等級：高
預估工時：12 小時
不處理的後果：潛在資料遺失、安全性漏洞
```

### 客觀性原則

所有評估必須基於客觀資料：
```
- 測試通過率：77/77 (100%)
- 靜態分析：0 errors, 0 warnings, 17 infos
- 函式平均長度：23 行
- P1 債務：0 個
- P2 債務：3 個（已標註）
- 品質評分：A-
```

禁止主觀判斷如「程式碼看起來不錯」「應該沒什麼問題」。

### 可行性原則

建議必須具體可執行：
```
債務：BatchEnrichBooksUseCase 實作不完整
位置：lib/use_cases/search/batch_enrich_books_use_case.dart:142-148
建議：整合 EnrichBookInfoUseCase，替換模擬操作為真實呼叫
工時：4-6 小時
```

### 派發建議格式

```
建議派發給 parsley-flutter-developer：
- 理由：識別的 3 個 P2 債務涉及程式碼結構
- 工作量：預估 8-12 小時
- 影響範圍：5 個檔案
最終派發決定由 PM 負責。
```

## 升級機制

Bay Quality Auditor 通常不需要升級。罕見升級情況：

| 情況 | 處理 |
|------|------|
| 無法執行測試（環境問題） | 記錄完整錯誤訊息和已嘗試方案，回報 PM |
| 需要特殊工具但無權限 | 記錄需求，回報 PM |
| 分析超過預期時間 | 停止當前流程，回報 PM |
