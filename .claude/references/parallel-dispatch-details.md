# 並行派發詳細規則

本文件包含並行派發指南的詳細規則、格式和場景。

> 精簡版（常駐載入）：.claude/rules/guides/parallel-dispatch.md

---

## 並行派發 5W1H 格式

使用並行派發時，5W1H 格式：

```text
5W1H-{TOKEN}

Who: rosemary-project-manager (dispatcher)
What: 並行派發 N 個獨立任務
When: Wave X 開始
Where: {目標目錄1}, {目標目錄2}, ...
Why: 任務無依賴，可並行加速
How: [Task Type: Dispatch] 並行啟動 {代理人} x N

並行任務清單：
| Task | Ticket | Agent | Target |
|------|--------|-------|--------|
| 1 | {Ticket-ID-1} | {agent} | {target1} |
| 2 | {Ticket-ID-2} | {agent} | {target2} |

並行安全確認：
- [x] 檔案無重疊
- [x] 測試無衝突
- [x] 依賴無循環
- [x] 資源無競爭
```

---

## 大量並行適用情境

| 適用情境 | 說明 | 範例 |
|---------|------|------|
| 同一層級修改 | 修改都在同一架構層 | 所有 Repository 檔案 |
| 純重命名/格式化 | 無邏輯變更 | 變數重命名跨幾十個檔案 |
| 無邏輯依賴 | 各檔案獨立運作 | 獨立的 Widget 檔案 |
| 批量機械操作 | 規則明確的重複操作 | 導入路徑統一修正 |

## 分析任務並行派發

> 核心洞察：不僅實作任務需要並行化，分析任務也應該考慮多視角並行。
>
> 詳見：.claude/methodologies/multi-perspective-analysis-methodology.md

### 分析任務與實作任務的差異

| 維度 | 實作任務並行 | 分析任務並行 |
|------|-------------|-------------|
| 檔案重疊 | **禁止**（會衝突） | **允許**（只讀分析） |
| 輸出類型 | 程式碼修改 | 分析報告 |
| 輸出衝突 | 會衝突 | 不衝突（各自獨立報告） |
| 彙整需求 | 無 | **需要**彙整者整合報告 |
| 安全檢查 | 嚴格 | 寬鬆 |

### 分析任務並行觸發條件

**符合任一即可考慮啟用**：

| 條件 | 說明 |
|------|------|
| 認知負擔 > 10 | 問題複雜度高 |
| 風險等級 >= 高 | 需要交叉驗證 |
| 原因不明確 | 根因不清楚 |
| 影響範圍 > 3 模組 | 跨模組問題 |
| 安全敏感 | 涉及認證、授權 |

### 分析任務並行安全檢查

```markdown
## 分析任務並行安全檢查

- [ ] 各視角分析範圍清晰（知道各自關注什麼）
- [ ] 各代理人可獨立完成（無需互相等待）
- [ ] 報告格式統一（便於彙整）
- [ ] 彙整者已確定（知道誰負責整合）
```

### 分析任務並行派發格式

```text
5W1H-{TOKEN}

Who: rosemary-project-manager (dispatcher)
What: 多視角並行分析
When: 問題分析階段
Where: {分析對象}
Why: 問題複雜/原因不明/風險高，需要多視角交叉驗證
How: [Task Type: Dispatch] 並行派發 N 個分析視角

分析視角清單：
| 視角 | 代理人 | 分析焦點 |
|------|-------|---------|
| 程式碼 | parsley | 實作細節 |
| 測試 | sage | 測試設計 |
| 架構 | saffron | 設計模式 |

彙整安排：
- 彙整者: saffron-system-analyst
- 產出: 彙整報告
- 決策者: PM
```

### 分析任務並行流程

```
問題/分析需求
    |
    v
[判斷] 需要多視角分析?
    |
    +-- 否 --> 單一代理人分析
    |
    +-- 是
        |
        v
    [選擇視角] 根據問題類型選擇必選和可選視角
        |
        v
    [並行派發] 多個分析代理人（同時）
        |
        +-- 視角 A 代理人 --> 報告 A
        +-- 視角 B 代理人 --> 報告 B
        +-- 視角 C 代理人 --> 報告 C
        |
        v
    [報告彙整] SA 或專門代理人整合
        |
        v
    彙整報告（含交叉驗證、衝突處理）
        |
        v
    [PM 決策] 根據彙整報告派發後續動作
```

### 常見分析場景的視角組合

| 場景 | 必選視角 | 可選視角 | 彙整者 |
|------|---------|---------|-------|
| 測試失敗分析 | 程式碼、測試 | 架構、需求 | SA |
| 架構決策 | 架構、效能 | 安全、整合 | SA |
| 效能問題 | 效能、程式碼 | 架構 | ginger |
| 安全問題 | 安全、程式碼 | 架構 | security |
| 重構決策 | 整合、架構 | 測試、程式碼 | cinnamon |

## 並行派發後驗證詳細規則

> 核心原則：不依賴代理人自我報告，必須用 `git diff --stat` 驗證實際變更。
>
> 來源：歷史教訓 — 26 個檔案修改任務，代理人全部回報完成，實際只有 9 個檔案有變更。

### 驗證步驟

| 步驟 | 指令 | 目的 |
|------|------|------|
| 1 | `git diff --stat` | 列出所有實際變更檔案 |
| 2 | 比對代理人報告 | 確認每個代理人聲稱修改的檔案都有實際 diff |
| 3 | 識別缺失檔案 | 找出代理人報告已完成但實際無變更的檔案 |
| 4 | 補派或記錄 | 缺失檔案重新派發代理人修改，或記錄到 Ticket |

### 禁止行為

| 禁止行為 | 正確做法 |
|---------|---------|
| 僅依賴代理人回報「已完成」 | 必須 `git diff --stat` 驗證 |
| 用 py_compile 替代內容驗證 | py_compile 只驗語法，不驗內容變更 |
| 發現缺失後直接跳過 | 必須補派或記錄到 Ticket |

### 不一致的常見原因

| 原因 | 說明 | 對策 |
|------|------|------|
| IO 競爭 | 多個代理人同時寫入 | 降低並行數量或分批執行 |
| Token 耗盡 | 代理人在分析後 token 用盡，未實際執行 Edit | 確認代理人 max_turns 足夠 |
| Hook/Linter 還原 | 寫入後被自動工具還原 | 檢查 Hook 和 Linter 配置 |
| 代理人誤報 | 聲稱完成但未操作 | git diff 是唯一真相來源 |

## Agent Teams 詳細規則

### 與 Task Subagent 的比較

| 維度 | Task Subagent | Agent Teams |
|------|--------------|-------------|
| 通訊模式 | 無（獨立執行） | 即時（SendMessage） |
| 任務清單 | 無共享 | 共享 Task List |
| 成本 | 1x * N | 3-4x |
| 結果收集 | 各自回報主線程 | Team 內部協調 |
| 適用場景 | 各 Agent 獨立完成 | Agent A 的發現改變 Agent B 的工作 |
| Ticket 整合 | 自動（由主線程管理） | 需要橋接協定 |

### 具體場景表

**Task Subagent 場景**（各 Agent 獨立完成，不互相影響）：

| 具體情境 | 判斷理由 |
|---------|---------|
| 獨立模組各自實作（Widget/Repository/Controller） | 各自獨立，無交互需求 |
| 批量重命名/導入路徑修正/格式化 | 純機械操作 |
| 多視角分析（各視角獨立產出報告） | 各報告不互相依賴，PM 事後彙整 |
| 多來源外部研究（各來源獨立） | 各自查詢不同網站，結論不互相依賴 |
| 測試失敗分析（單一明確原因） | incident-responder 單一代理人足夠 |
| 標準 TDD Phase 開發 | Phase 順序天然序列 |

**Agent Teams 場景**（Agent A 的發現會改變 Agent B 正在進行的工作）：

| 具體情境 | 判斷理由 |
|---------|---------|
| 跨模組測試失敗（大量失敗、原因不明） | 一個代理人的發現會改變另一個的調查方向 |
| 前後端同時開發共用 API | API 契約需雙方即時協商 |
| 元件 + ViewModel 共同設計介面邊界 | 介面契約在工作中定義，非預先確定 |
| 架構審查 + 安全審查（涉及認證/授權） | SA 架構發現可能改變安全評估範圍 |
| 多來源研究（結論相互依賴） | 一個來源的發現影響另一個的查詢策略 |
| 除錯：多假設平行測試（原因不明） | 一個假設被否定後要即時通知其他代理人調整方向 |
| QA 批量測試（多 URL/多頁面/多端點驗證） | 純平行加速，社群驗證有效 |
| 大型重構：多模組同時修改共用介面 | 共用介面變更需即時同步，避免衝突 |

> 「多視角分析」特別說明：預設使用 Task subagent（與「分析任務並行派發」章節一致）。
> 僅在一個視角的發現會即時改變另一視角的分析方向時，才升級為 Agent Teams。

### 並行安全檢查（Agent Teams 補充）

在現有並行安全檢查清單基礎上，Agent Teams 額外確認：

```markdown
- [ ] Ticket 已建立且 metadata 包含 ticketId
- [ ] 入職模板已準備（所有 {var} 已替換）
- [ ] 成本評估已完成（確認 3-4x 成本可接受）
- [ ] 回退策略已確認（失敗時可降級為 Task subagent）
```

### 禁止場景

| 禁止行為 | 原因 | 替代方案 |
|---------|------|---------|
| 簡單獨立任務用 Agent Teams | 成本浪費 | Task subagent |
| DOC 類型用 Agent Teams | 不需要互動 | Task subagent |
| 跨 Wave 任務用 Agent Teams | 違反 Wave 隔離 | 分開處理 |
