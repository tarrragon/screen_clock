# Plan-to-Ticket 映射詳細規範

本文件包含 Plan-to-Ticket 轉換流程的詳細映射規則和 TDD 階段對應表。

> 主要流程：@.claude/pm-rules/plan-to-ticket-flow.md

---

## Step 1：Plan 檔案格式識別

### Plan 檔案標準結構

Plan 檔案通常包含以下結構：

```markdown
# 實作計畫：{功能名稱}

## 概述
{功能描述}

## 實作步驟
1. {步驟描述}
   - 修改檔案：{檔案路徑}
   - 說明：{詳細說明}

2. {步驟描述}
   ...
```

### 解析規則對照表

| 元素 | 識別方式 | 對應 Ticket 欄位 |
|------|---------|----------------|
| 功能名稱 | `#` 標題或 `## 概述` | 根任務 title |
| 實作步驟 | 有序清單項目（`1.` `2.` ...） | 子任務 title |
| 修改檔案 | `修改檔案：` 或檔案路徑模式 | where.files |
| 架構層級 | 檔案路徑推斷（lib/domain/ → Domain） | where.layer |

---

## Step 2：任務類型分類

### 任務粒度檢查標準

每個識別出的任務項目必須通過原子性檢查：

| 檢查項目 | 標準 | 不通過時 |
|---------|------|---------|
| 語義檢查 | 能用「動詞 + 單一目標」表達 | 需要拆分 |
| 修改原因 | 只有一個修改原因 | 需要拆分 |
| 檔案範圍 | 修改檔案數 <= 5 | 需要拆分 |

### 任務類型識別與分類

| Plan 步驟特徵 | 識別關鍵字 | Ticket 類型 |
|-------------|-----------|------------|
| 建立新檔案/功能 | 「建立」「新增」「實作」 | IMP |
| 修改現有功能 | 「修改」「調整」「修正」 | ADJ |
| 分析/研究 | 「分析」「研究」「調查」 | ANA/RES |
| 撰寫文件 | 「撰寫」「更新文件」「記錄」 | DOC |

## Step 4：TDD 階段映射表

### 任務類型與 TDD 階段對應

根據任務類型和複雜度，決定需要哪些 TDD 階段：

| 任務特徵 | SA 審查 | Phase 1 | Phase 2 | Phase 3a | Phase 3b | Phase 4 |
|---------|---------|---------|---------|----------|----------|---------|
| 新功能（複雜） | 需要 | 需要 | 需要 | 需要 | 需要 | 需要 |
| 新功能（簡單） | 不需要 | 需要 | 需要 | 可省略 | 需要 | 需要 |
| 修改/調整 | 不需要 | 不需要 | 可選 | 不需要 | 需要 | 需要 |
| 純文件 | 不需要 | 不需要 | 不需要 | 不需要 | 不需要 | 不需要 |

### SA 前置審查觸發條件

| 條件 | 需要 SA 審查 |
|------|-------------|
| 新功能需求 | 是 |
| 架構變更 | 是 |
| 影響 3+ 模組 | 是 |
| 小型修改/Bug 修復 | 否 |

### Ticket 產生策略

**完整 TDD（新功能）**：

```
根 Ticket：{功能名稱}
├── {ID}.1：SA 前置審查（Phase 0）
├── {ID}.2：功能設計（Phase 1）
├── {ID}.3：測試設計（Phase 2）
├── {ID}.4：策略規劃（Phase 3a）
├── {ID}.5：實作執行（Phase 3b）
└── {ID}.6：重構評估（Phase 4）
```

**簡化 TDD（修改/調整）**：

```
根 Ticket：{修改名稱}
├── {ID}.1：實作執行（Phase 3b）
└── {ID}.2：重構評估（Phase 4）
```

**純文件**：

```
單一 Ticket：{文件名稱}（DOC 類型）
```

## Step 5：依賴識別規則

### 依賴類型與識別方式

| 依賴類型 | 識別方式 | 表示方式 |
|---------|---------|---------|
| 資料依賴 | Task B 使用 Task A 的產出 | B.blockedBy = [A] |
| 架構依賴 | 底層必須先於上層 | Domain → Application → Presentation |
| TDD 依賴 | Phase 順序 | Phase 1 → 2 → 3a → 3b → 4 |
| 共用模組依賴 | 多個 Task 依賴同一模組 | 共用模組先完成 |

### 依賴矩陣產生流程

```
任務清單
    |
    v
[識別] 各任務修改的檔案集合
    |
    v
[比對] 檔案集合交集
    |
    +-- 有交集 → 分析依賴方向
    |               |
    |               +-- A 產出是 B 輸入 → B.blockedBy = [A]
    |               +-- 雙向依賴 → 合併為同一 Ticket
    |
    +-- 無交集 → 可並行
```

## Step 7：Ticket 欄位規範

### 必填欄位清單

每個產生的 Ticket 必須包含以下欄位：

| 必填欄位 | 來源 | 說明 |
|---------|------|------|
| id | 自動產生 | `{版本}-W{wave}-{序號}` 格式 |
| title | Plan 步驟標題 | 「動詞 + 目標」格式 |
| type | Step 2 分類 | IMP/ADJ/ANA/DOC |
| status | 固定 | pending |
| version | 觸發條件 | 目標版本號 |
| wave | Step 6 分配 | Wave 號 |
| priority | 依賴分析 | P0（阻塞路徑）/ P1（正常）/ P2（可延後） |
| where.files | Plan 步驟 | 修改檔案清單 |
| where.layer | 檔案路徑推斷 | 架構層級 |
| blockedBy | Step 5 | 依賴的 Ticket ID |
| acceptance | Plan 步驟 | 驗收條件清單 |
| decision_tree_path | 自動填充 | 決策樹路徑記錄 |

### 驗收條件產生規則（4V 原則）

驗收條件必須符合 4V 原則：

| 原則 | 說明 |
|------|------|
| 可驗證 (Verifiable) | 有明確的確認方法 |
| 可量化 (Quantifiable) | 有具體的量化標準 |
| 可追溯 (Traceable) | 可追溯到來源需求 |
| 可記錄 (Recordable) | 結果可被記錄和保存 |

詳細規範：@.claude/methodologies/acceptance-criteria-methodology.md

## Step 6：並行分組詳細規則

### 並行安全檢查清單

在分組前，每組任務必須通過：

```markdown
- [ ] 檔案無重疊：各任務修改的檔案集合無交集
- [ ] 測試無衝突：各任務的測試可獨立執行
- [ ] 依賴無循環：任務之間無先後依賴關係
- [ ] 資源無競爭：不會同時存取相同外部資源
```

### 分組策略詳解

```
所有待處理 Ticket
    |
    v
[排除] 有依賴的 Ticket（序列執行）
    |
    v
[分組] 無依賴的同階段 Ticket
    |
    +-- 同為 Phase 1 → 可並行
    +-- 同為 Phase 3b → 可並行（檔案無重疊）
    +-- 跨階段 → 不可並行
    |
    v
[排序] 依賴在前，獨立在後
    |
    v
輸出執行順序
```

### Wave 分配策略

| 分配策略 | 說明 |
|---------|------|
| 序列任務 | 依賴鏈中的任務按順序分配到同一 Wave |
| 並行任務 | 無依賴的同類任務分配到同一 Wave（並行執行） |
| 跨 Wave | 前置 Wave 完成後才開始後續 Wave |
