# 驗收暫停點模板 - 詳細檢查清單

> 來源：rosemary-project-manager Agent 定義
> 基於《清單革命》原則的驗收暫停點機制

---

## 暫停點使用規則

- 代理人暫停：執行代理人完成階段後必須主動暫停
- 主線程檢查：PM 在暫停點執行驗收檢查
- 通過後繼續：通過檢查後才能繼續下一階段
- 未通過返回：未通過檢查則返回修正

## 問題發現暫停點：任何階段發現架構問題

**觸發條件**: 代理人識別出架構債務或設計缺陷

**通過標準**:

- [ ] 問題已解決或已記錄到 todolist
- [ ] 不允許繼續（零容忍原則）
- [ ] 問題影響範圍已評估
- [ ] 解決方案已規劃或已執行

**處理原則**: 架構債務零容忍，立即停止功能開發優先修正

## 通用驗收檢查清單（所有 Phase）

- [ ] 工作日誌符合 work-log-template.md 或 ticket-log-template.md 格式

- [ ] 對應 Phase 的驗收條件全部打勾 `[x]`

### 溝通檢查清單完整性

- [ ] Phase 交接溝通確認清單全部打勾

- [ ] 測試通過率 100%（如適用於該 Phase）

- [ ] todolist.yaml 狀態與工作日誌一致

## 主線程強制記錄原則

> 主線程的核心職責不只是派工，更重要的是記錄完整的思考脈絡。

### 記錄時機（強制要求，零例外）

- [ ] 每次與用戶討論後 → 立即記錄討論內容和決策
- [ ] 每次階段性決策後 → 記錄決策依據和思考過程
- [ ] 派工給代理人前 → 記錄派工理由和預期目標
- [ ] 發現問題或變更計畫時 → 記錄問題分析和調整方案
- [ ] 收到代理人回報後 → 記錄驗收結果和下一步規劃

### 記錄內容標準（四個必要元素）

1. **討論記錄** - 用戶需求、疑問、建議和關注點
2. **思考分析** - 問題理解、根本原因、影響範圍
3. **決策依據** - 方案選擇理由、替代方案、關鍵因素
4. **預期效益** - 目標、評估標準、風險和緩解

### 派工前強制檢查

- [ ] 思考過程是否已記錄到 `.0-main.md`
- [ ] 記錄內容是否包含四個必要元素
- [ ] 記錄是否足夠讓他人重新進入狀況
- [ ] 派工理由和預期目標是否明確

### 違規處理

- 禁止「先派工後補記錄」
- 禁止「簡略記錄」或「省略思考過程」
- 禁止「只記錄結論不記錄分析」
- 未完成記錄 = 不得派工
