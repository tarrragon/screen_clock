# Layer 1 代理人自檢模板

> **定位**：Layer 1 自檢是代理人在 complete 前執行的自律審查輪，攔截 60-70% 低階違規（文字品質、禁用字、Schema 結構），降低對 Layer 2 常駐委員的依賴。
>
> **首段聲明**：自檢不替代首次品質要求。首次產出時已符合規範的項目無需重複掃描；本清單目標是抓住「幾乎確定漏掉的」而非「不可能犯的」違規。
>
> **使用時機**：IMP/ANA/DOC ticket 執行完畢、commit 前，依以下三大類執行一輪快速掃描。

---

## 觸發時機

| 時機 | 動作 |
|------|------|
| 完成所有 ticket body 填寫後、commit 前 | 依本清單掃描，發現違規立即修正 |
| 收到 Layer 2 委員報告後 | 對照報告項目確認是否已修正，補充說明 |
| 自檢結果需記錄時 | 寫入 Solution 子章節 `### 自檢結果` |

---

## 類別 A：文字品質

文字品質自檢適用於所有書面產出（規則條款、方法論段落、ticket Solution/Test Results、長段落說明）。

### A1 三明示（Why / Consequence / Action）

**檢查對象**：每段論述（非每句）。

| 掃描訊號 | 判斷 | 處理 |
|---------|------|------|
| 段落只有「禁止 X」，無說明理由或替代做法 | 違規（缺 Why + Action） | 補「為何禁止 X，否則 Y，改做 Z」 |
| 段落描述症狀，無成因說明 | 違規（缺 Why） | 補根因說明 |
| 段落有原則，無操作步驟 | 違規（缺 Action） | 補「下一步應做什麼」 |
| 段落三層齊備 | 通過 | 無需調整 |

**快速測試**：刪掉段落最後一句，讀者還知道「下一步做什麼」嗎？若不知道，Action 層缺失。

### A2 資訊優先序（原則先於示例）

**檢查對象**：技術解釋句、定義句。

| 掃描訊號 | 判斷 | 處理 |
|---------|------|------|
| 句子以「X 是 Y，它不是單純的…，而是…」開頭 | 違規（示例先於原則） | 將核心原則移到句首 |
| 段落開頭是具體範例，末段才說原則 | 違規（順序顛倒） | 原則先行，範例在後，提醒最後 |
| 讀者只讀第一句能知道「這段在說什麼規則」 | 通過 | 無需調整 |

### A3 負向對比有正向錨點

**檢查對象**：禁令列表、反模式說明。

| 掃描訊號 | 判斷 | 處理 |
|---------|------|------|
| 「禁止 X」後無對應的「正確做法是 Y」 | 違規（單向禁令） | 補「改做 Y（因為 Z）」 |
| 反模式清單只有錯誤示例，無正確示例 | 違規 | 每條補正向錨點 |
| 禁令與正確做法成對出現 | 通過 | 無需調整 |

## 類別 B：禁用字與字元集

依 `.claude/rules/core/language-constraints.md` 規則 1/3 執行。

### B1 繁體中文規範

| 禁用詞 | 正確用語 |
|--------|---------|
| 數據 | 資料 |
| 代碼 | 程式碼 |
| 默認 | 預設 |
| 文檔 | 文件 |
| 軟件 | 軟體 |
| 硬件 | 硬體 |
| 信息 | 資訊 |
| 視頻 | 影片 |
| 優化（品質語境） | 最佳化 |

**快速掃描指令**（對產出文字段落使用）：

```
掃描以下詞：數據、代碼、默認、文檔、軟件、硬件、信息、視頻
出現任一詞 → 立即替換為對應繁體用語
```

### B2 Emoji 禁令

**禁止**：在任何產出文字中使用 emoji 符號（Unicode U+1F000 以上裝飾類字元）。

| 掃描訊號 | 處理 |
|---------|------|
| 標題含裝飾性 emoji | 刪除 emoji，保留文字 |
| 列表項含完成/未完成 emoji | 改用 `[x]` / `[ ]` |
| 警示類 emoji | 改用 `[WARNING]` 或「警告：」 |

### B3 簡體字與日文漢字

**禁止**：產出文字中混入簡體字或日文漢字（包括形近但意義不同的字元）。

常見混淆字元：

| 禁用字形 | 說明 | 正確字形 |
|---------|------|---------|
| 这、该、们、为、对（簡體） | 簡體中文 | 這、該、們、為、對 |
| 発、処（日文漢字字形） | 部首筆畫結構與繁體不同 | 發、處（繁體） |

**快速測試**：使用任何輸入法切換至繁體中文後，確認文字不含異體字形。

## 類別 C：Schema 與結構

依 `.claude/references/agent-definition-standard-details.md` 執行責任章節執行。

### C1 Ticket Body Schema 章節必填

| Ticket Type | 必填章節 |
|------------|---------|
| IMP | Problem Analysis / Solution / Test Results |
| ANA | Problem Analysis / Analysis Result（或 Solution） |
| DOC | Solution（文件變更摘要） |

**掃描重點**：

| 掃描訊號 | 判斷 | 處理 |
|---------|------|------|
| 必填章節仍為 `<!-- To be filled by executing agent -->` | 違規（佔位符未替換） | 填入實際內容 |
| 章節存在但僅有「待填寫：...」文字 | 違規（佔位符未替換） | 填入實際內容 |
| 必填章節存在且有實質內容 | 通過 | 無需調整 |

### C2 自定義 H2 禁令（W17-072）

**禁止**：在 ticket body 使用非 Schema 定義的 H2 標題（`## 自訂標題`）。

| 掃描訊號 | 判斷 | 處理 |
|---------|------|------|
| ticket body 出現 `## 實作摘要` / `## 驗證指令與結果` 等非 Schema 章節 | 違規 | 改為 H3 子章節（`### 實作摘要`）寫在對應 Schema 章節下 |
| ticket body 所有 H2 均為 Schema 章節名 | 通過 | 無需調整 |

**Schema 合法 H2 清單**：`Task Summary` / `Problem Analysis` / `重現實驗結果` / `Solution` / `Test Results` / `Context Bundle` / `NeedsContext` / `Exit Status` / `Completion Info`

### C3 Test Results 必含執行摘要

**適用**：IMP ticket 的 Test Results 章節。

| 必含項目 | 說明 |
|---------|------|
| 測試執行指令（或標明不適用） | 如 `npm test`、`npx jest tests/x.test.js` 或「本 ticket 為文件類，無測試指令」 |
| 通過數或結論（或 commit SHA） | 如「全通過」、「35/35 pass」 或「N/A - 純文件修改」 |

## 自檢結果寫入範本

完成自檢後，若有發現需記錄，於 Solution 章節追加：

```markdown
### 自檢結果（Layer 1）

**執行時機**: complete 前
**掃描類別**: A（文字品質）/ B（禁用字）/ C（Schema 結構）

| 類別 | 發現項目 | 處理方式 |
|------|---------|---------|
| B1 | 「數據」→「資料」，共 2 處 | 已修正 |
| C1 | Test Results 章節佔位符未替換 | 已填入測試結果 |
| （無其他發現）| — | — |

**自檢結論**: 通過 / 發現 N 項，已全數修正
```

若自檢無任何發現，可省略 `### 自檢結果` 子章節；無需強制記錄「零違規」。

---

## 與 Layer 2 委員的介面分工

| 層次 | 執行者 | 產出位置 | 說明 |
|------|-------|---------|------|
| Layer 1 自檢 | 執行代理人（自律） | Solution `### 自檢結果` | complete 前執行，修正低階違規 |
| Layer 2 委員審查 | basil-writing-critic（外部） | Test Results（委員報告） | 高價值產出（規則條款、方法論）觸發；發現 Layer 1 遺漏 |

**避免覆寫衝突**：Layer 1 寫入 `## Solution` 的子章節；Layer 2 寫入 `## Test Results`。兩者不重疊，可安全並存。
