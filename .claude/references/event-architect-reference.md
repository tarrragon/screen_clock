## 事件命名規範

### 格式

- **主格式**：`MODULE.ACTION.STATE`
- **替代格式**：`MODULE.CATEGORY.ACTION`
- **範例**：`UI.PROGRESS.UPDATE`、`EXTRACTION.COMPLETED`、`STORAGE.SAVE.COMPLETED`

### 命名規則

| 規則 | 說明 |
|------|------|
| 全大寫 | 事件名稱使用全大寫 |
| 點分隔 | 使用 `.` 分隔模組、動作、狀態 |
| 意圖明確 | 事件名稱應清楚表達意圖和目的 |
| 一致性 | 相同類型的事件使用相同的命名模式 |

## 事件設計最佳實踐

### 單向流設計

事件流應保持單向，防止循環依賴：

```
模組A → 事件總線 → 模組B
         ↓
       模組C
```

### 錯誤處理模式

| 策略 | 適用場景 |
|------|---------|
| 重試 | 暫時性錯誤（網路、資源忙碌） |
| 死信佇列 | 無法處理的事件 |
| 補償事件 | 需要回滾的操作 |

### 效能考量

| 項目 | 建議 |
|------|------|
| 事件大小 | 負載保持精簡，避免傳遞大物件 |
| 處理時間 | 事件處理器應快速返回，耗時操作使用非同步 |
| 記憶體 | 避免在事件中持有大量狀態 |
