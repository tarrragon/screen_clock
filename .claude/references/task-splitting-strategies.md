# 任務拆分策略詳細說明

本文件包含任務拆分的 7 個策略詳細說明和範例。

> 精簡版（常駐載入）：.claude/rules/guides/task-splitting.md

---

## 策略 1：按架構層拆分

**適用情境**：任務跨越多個 Clean Architecture 層級

```
原始任務：「實作新的書籍搜尋功能」
    |
    +-- 子任務 A：Domain 層 - 建立 SearchQuery 值物件
    +-- 子任務 B：Domain 層 - 建立 SearchResult Entity
    +-- 子任務 C：Application 層 - 建立 SearchBooksUseCase
    +-- 子任務 D：Infrastructure 層 - 實作 BookSearchRepository
    +-- 子任務 E：Presentation 層 - 建立搜尋 Widget

執行順序：A → B → C → D → E（由底層向上）
```

**原則**：
- 每個子任務只修改一個架構層
- 由底層（Domain）向上層（Presentation）實作
- 下層完成後才能開始上層

---

## 策略 2：按功能模組拆分

**適用情境**：任務涉及多個獨立模組

```
原始任務：「為所有 Repository 加入快取機制」
    |
    +-- 子任務 A：共用快取基礎設施（先執行）
    +-- 子任務 B：BookRepository 快取（可並行）
    +-- 子任務 C：AuthorRepository 快取（可並行）
    +-- 子任務 D：CategoryRepository 快取（可並行）

執行順序：A 先完成，B/C/D 可並行
```

**原則**：
- 共用模組先完成
- 獨立模組可並行
- 每個模組由同一個代理人完成

---

## 策略 3：按操作類型拆分

**適用情境**：任務包含不同類型的操作

```
原始任務：「重構命名並優化邏輯」
    |
    +-- 子任務 A：重命名操作（機械性，可並行）
    +-- 子任務 B：邏輯優化（需謹慎序列）

執行順序：A 全部可並行，B 序列執行
```

**原則**：
- 機械性操作（重命名、格式化）可大量並行
- 邏輯修改需要謹慎序列
- 不同類型不要混合在同一 Ticket

---

## 策略 4：按 TDD 階段拆分

**適用情境**：完整的功能開發

```
原始任務：「實作書籍評分功能」
    |
    +-- Phase 1 Ticket：功能設計（lavender）
    +-- Phase 2 Ticket：測試設計（sage）
    +-- Phase 3a Ticket：策略規劃（pepper）
    +-- Phase 3b Ticket：實作執行（parsley）
    +-- Phase 4 Ticket：重構優化（cinnamon）

執行順序：嚴格序列（Phase 順序）
```

## 策略 5：批量修正拆分

**適用情境**：Code Review / Audit 產出多個問題需跨多檔案修正

**觸發條件**（任一符合）：
- Review 報告含 3+ 個問題涉及不同檔案
- Lint/Audit 報告需修改 5+ 個檔案
- 遷移/重構任務影響 5+ 個檔案且修正模式相同

**核心原則**：拆分單元是**檔案**，不是問題類型。

**拆分流程**：

```
Code Review 報告（N 個問題，M 個檔案）
    |
    v
[Step 1] 建立「問題-檔案矩陣」
    |
    v
[Step 2] 識別共用檔案，將觸及同一檔案的問題合併
    |
    v
[Step 3] 剩餘獨立檔案按修正複雜度分批
    |
    v
[Step 4] 檔案所有權驗證（策略 6）
```

**問題-檔案矩陣範例**：

```
| 問題 ID | 觸及檔案 |
|---------|---------|
| C-001 | messages.py |
| C-002 | messages.py |
| M-001 | utils.py |
| M-002 | hook_a.py, hook_b.py, ... (22 個) |
| M-004 | messages.py, gate_a.py |
```

**合併規則**：messages.py 被 C-001/C-002/M-004 觸及 → 合併為 1 個 ticket。

**分批基準**：

| 修正複雜度 | 每批檔案數 | 判斷依據 |
|-----------|-----------|---------|
| 機械修正（搜尋/替換級） | 4-6 檔案 | 每檔案改動 < 5 行，無需理解上下文 |
| 結構修正（需理解上下文） | 2-4 檔案 | 每檔案需局部判斷，如函式簽名調整 |
| 邏輯修正（需推理） | 1 檔案 | 每檔案都需要獨立思考 |

**反過度拆分守則**：

| 情境 | 正確做法 | 原因 |
|------|---------|------|
| 每檔案改動 < 3 行 | 合併 4-6 檔案一批 | 管理成本 > 修正成本 |
| 修改 A 後需讀結果才能改 B | 放同一 ticket | 邏輯依賴 |
| 共用元件 + 多使用端 | 共用元件獨立 + 使用端分批 | 共用元件是依賴 |

## 策略 6：檔案所有權隔離（強制前置檢查）

**適用情境**：任何拆分產生 2+ 個 ticket 且預期並行執行

**核心原則**：同一 Wave 中，每個檔案最多只被一個 ticket 寫入。

**檔案所有權矩陣**：

```
| 檔案路徑 | Ticket A | Ticket B | Ticket C | 衝突 |
|---------|----------|----------|----------|------|
| messages.py | W | - | - | 無 |
| utils.py | - | W | - | 無 |
| hook_a.py | - | - | W | 無 |
| gate.py | W | - | W | 衝突! |
```

**衝突解決規則**：

| 衝突類型 | 解決方式 |
|---------|---------|
| Write-Write | 合併到同一個 ticket（強制） |
| Write-Read | 允許（讀不衝突） |

**執行時機**：
1. 拆分完成後、ticket 建立前（強制）
2. 並行派發前的安全檢查（保留，作為二次驗證）

**與 ticket create 的整合**：
- 建立批量 ticket 時，`where.files` 必須填入完整修改檔案清單
- PM 驗證同一 Wave 所有 ticket 的 `where.files` 無交集

## 策略 7：按測試群組拆分（Phase 3b 推薦）

**適用情境**：Phase 3b 實作拆分，Phase 2 定義了多個獨立的 GWT scenario group

**核心原則**：拆分單位是 Phase 2 的 GWT scenario group（測試群組），而非模組或檔案。每個子任務聚焦「通過特定測試群組」。

```
Phase 2 測試設計產出：
  - TC-01~TC-13：連線管理（建立/斷開/重連）
  - TC-14~TC-16,TC-18~TC-34：事件推送+心跳+分頁+並發
    |
    v
拆分為 2 個 Phase 3b 子任務：
  - 子任務 A：通過 TC-01~TC-13
  - 子任務 B：通過 TC-14~TC-16,TC-18~TC-34
```

**與模組拆分的差異**：

| 維度 | 按測試群組（推薦） | 按模組（舊方式） |
|------|-------------------|----------------|
| 子任務定義 | 「通過 TC-01~TC-13」 | 「實作 Module A」 |
| 代理人需讀取 | API 簽名 + 測試案例 | 整個模組設計文件 |
| 完成判斷 | 測試全部通過 | 模組功能完成 |
| context 消耗 | 低（只讀相關區段） | 高（讀完整文件） |

**PM 派發流程**：

1. 從 Phase 2 test-design 識別獨立的 GWT scenario group
2. 每個 group 建立一個子任務，標明 TC 範圍
3. 使用 phase3b-prompt-template.md 提取 API 簽名 + 測試案例 + 常數
4. 確認各子任務修改檔案無交集
5. 並行派發

> 完整 prompt 模板：.claude/references/phase3b-prompt-template.md

## 拆分文件格式

```markdown
## 任務拆分計畫

### 原始任務
- **Ticket ID**: {原始 ID}
- **任務描述**: {描述}
- **複雜度評估**: {高}

### 拆分原因
- 認知負擔指數: {數值} > 10
- 跨架構層級: {數量} > 2

### 拆分策略
{策略名稱}：{策略描述}

### 子任務清單

| ID | 描述 | 代理人 | 依賴 | 可並行 |
|----|------|--------|------|--------|
| {ID}-A | {描述} | {代理人} | 無 | - |
| {ID}-B | {描述} | {代理人} | A | 否 |

### 驗收條件
- 所有子任務完成
- 整合測試通過
- 原始需求滿足
```

## 策略 8：按依賴鏈序列拆分

**適用情境**：步驟之間有嚴格的先後依賴關係，上游產出是下游的輸入。

**核心原則**：

| 原則 | 說明 |
|------|------|
| 依賴方向明確 | 每個 Ticket 用 `blockedBy` 標記上游 Ticket |
| 不可並行 | 依賴鏈中的步驟必須序列執行 |
| 上游先完成 | 上游 Ticket complete + merge 後才能派發下游 |

**典型場景**：

| 場景 | 範例 |
|------|------|
| Hook 拆分 → 修復 → 驗證 | 拆分 Ticket → 修復 bug Ticket → 驗證 |
| 規格 → 實作 → 整合 | 設計 API → 實作模組 A → 實作模組 B（依賴 A） |
| 分析 → 建立 → 執行 | ANA Ticket → 建立後續 IMP → 執行 IMP |

**拆分後驗證**：

- [ ] 每個 Ticket 的 `blockedBy` 是否正確？
- [ ] 依賴鏈中是否有可以並行的步驟？（有的話改用策略 2 或 6）
- [ ] 鏈條最長路徑是否合理？（> 5 步考慮是否能簡化）

**與其他策略的區別**：

| 對比 | 策略 8（依賴鏈） | 策略 4（TDD 階段） |
|------|----------------|-------------------|
| 依賴來源 | 業務/技術依賴 | 流程階段依賴 |
| 靈活性 | 依鏈條結構 | 固定 Phase 1→4 |
| 適用範圍 | 任何序列依賴場景 | 僅新功能 TDD |

> **實戰案例**：典型依賴鏈：先確認 Hook protocol 行為，再拆分 God Hook，最後統一 output helper。
