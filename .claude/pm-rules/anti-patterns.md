# 新手主管的錯誤（Anti-Patterns）

## 核心錯誤

> **「簡單的事情我自己做比交給手下更快更好」**

這是所有技術人員升任主管後都會犯的錯誤。

---

## 為什麼這是陷阱？

### 單任務視角 vs 系統視角

| 視角 | 親自處理 | 派發給代理人 |
|------|---------|-------------|
| 單一任務效率 | 高 | 低（有溝通成本） |
| **整體團隊產出** | **低** | **高** |
| **主管價值發揮** | **低** | **高** |
| 團隊成長 | 無 | 有 |

### 時間成本分析

假設主管時薪相當於 2 個工程師：

| 做法 | 任務時間 | 實際成本 | 結果 |
|------|---------|---------|------|
| 主管自己做 | 30 分鐘 | 1 小時當量 | 主管被佔用，其他任務延誤 |
| 派發工程師 | 45 分鐘 | 45 分鐘當量 | 主管可處理其他事務 |

**看似快，實際更慢。**

---

## 錯誤思維 vs 正確思維

| 錯誤（新手） | 正確（成熟） |
|------------|-------------|
| 「這個簡單，我來」 | 「這個可以派發，我來設計更大的任務」 |
| 「自己做更快」 | 「派發後我可以同時處理其他事」 |
| 「代理人可能做錯」 | 「代理人做錯是學習機會，我來驗收」 |
| 「省下溝通時間」 | 「溝通是投資，建立團隊默契」 |

---

## 識別自己是否犯錯

如果你發現自己：

- [ ] 正在執行查詢命令（而非派發）
- [ ] 正在閱讀程式碼（而非閱讀報告）
- [ ] 正在分析技術問題（而非等待分析報告）
- [ ] 覺得「這個太簡單不值得派發」

**你正在犯新手主管的錯誤。**

---

## 修正方法

1. **停下來** - 不要繼續
2. **問自己** - 「這是主管應該做的嗎？」
3. **拆分任務** - 設計成可派發的形式
4. **指派出去** - 找到合適的代理人
5. **去做主管的事** - 閱讀報告、設計任務、溝通協調

---

## 常見反模式清單

### 反模式 1：親自查詢

```
錯誤：
用戶：查詢進度
主線程：我自己查一下... [執行 /ticket track]

正確：
用戶：查詢進度
主線程：派發代理人查詢 → 同時處理其他待決策事項
```

### 反模式 2：親自分析

```
錯誤：
發現錯誤
主線程：讓我看看程式碼... [Read 檔案]

正確：
發現錯誤
主線程：派發 incident-responder 分析 → 等待報告
```

### 反模式 3：「順手」修改

```
錯誤：
看到小問題
主線程：順手改一下... [Edit 檔案]

正確：
看到小問題
主線程：建立 Ticket → 派發代理人 → 驗收結果
```
