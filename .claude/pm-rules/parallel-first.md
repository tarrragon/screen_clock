# 並行優先策略（Parallel-First Strategy）

## 核心原則

> 決策第一步不是分析任務類型，而是：
> **「這個工作可以讓多少人去做？」**

---

## 並行化評估流程

```
接收任務
    │
    ▼
┌─────────────────────────────────────┐
│ 1. 這個任務可以拆成幾個獨立部分？   │
│    - 識別可分離的子任務             │
│    - 每個部分能獨立完成嗎？         │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 2. 拆分後的部分有依賴關係嗎？       │
│    - 畫出依賴圖                     │
│    - 識別關鍵路徑                   │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 3. 哪些可以同時開始？               │
│    - 無依賴 → 立即並行              │
│    - 有依賴 → 排序後批次並行        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ 4. 我需要親自處理的是什麼？         │
│    - 答案應該是「幾乎沒有」         │
│    - 只有：拆分設計、指派、驗收     │
└─────────────────────────────────────┘
```

## 最大化並行的判斷標準

### 可以並行的情況

| 情況 | 判斷依據 |
|------|---------|
| 檔案無重疊 | 修改的檔案集合無交集 |
| 模組獨立 | 修改的模組沒有直接依賴 |
| 階段相同 | 都在同一個 TDD 階段 |
| 資源無競爭 | 不會同時存取相同外部資源 |

### 必須序列的情況

| 情況 | 處理方式 |
|------|---------|
| 有輸入輸出依賴 | 先完成上游，再開始下游 |
| 修改同一檔案 | 分配給同一代理人 |
| 設計影響實作 | 先完成設計審查 |

## 範例

### 用戶指令

> 「查詢進度，然後更新文件，最後執行測試」

### 錯誤做法（序列、親自處理）

1. 主管查詢進度
2. 主管更新文件
3. 主管執行測試

**問題**：主管被佔用，無法處理其他事務

### 正確做法（並行、派發）

1. 派發代理人查詢進度（不等待）
2. 派發代理人更新文件（不依賴進度結果時可並行）
3. 派發代理人設計測試（不依賴上述結果）
4. 收取報告，驗收，調整

**優點**：最大化並行，主管保持調度狀態

## 研究任務並行化（重要）

> **經驗教訓**：資料蒐集任務是最適合並行化的任務類型，但常被忽略。
> 來源：歷史執行經驗

### 為什麼研究任務更適合並行化？

| 特徵 | 開發任務 | 研究任務 |
|------|---------|---------|
| 依賴關係 | 可能有 | **幾乎沒有** |
| 資源競爭 | 可能有（檔案） | **沒有**（各查各的） |
| 合併難度 | 可能複雜 | **簡單**（整合報告） |
| 並行潛力 | 中等 | **非常高** |

### 範例：外部驗證研究

**錯誤做法**（歷史實際執行）：
```
研究任務（3 個主題）
    ↓
派發 1 個代理人處理全部
    ↓
代理人依序查詢 3 組關鍵字
```

**正確做法**（應該這樣做）：
```
研究任務（3 個主題）
    ↓
並行派發 3 個代理人：
    ├── Agent 1: 授權理論 → Ticket-006.1
    ├── Agent 2: 並行管理 → Ticket-006.2
    └── Agent 3: 管理者思維 → Ticket-006.3
    ↓
收取 3 份報告 → 主線程整合結論
```

### 研究任務並行化檢查清單

收到研究/查詢任務時，先問：

- [ ] 有幾個獨立的搜尋關鍵字/主題？
- [ ] 各主題之間有依賴嗎？（通常沒有）
- [ ] 可以拆成幾個並行的子 Ticket？
- [ ] 最後如何整合結果？

### 典型可並行化的研究場景

| 場景 | 並行方式 |
|------|---------|
| 多個技術方案評估 | 每個方案一個代理人 |
| 多個關鍵字搜尋 | 每組關鍵字一個代理人 |
| 多個文檔查閱 | 每個文檔一個代理人 |
| 競品分析 | 每個競品一個代理人 |

## 數量原則

**無固定上限** - 依任務性質決定

能並行多少就並行多少，主管的工作是：

- 設計並行方案
- 發出派發指令
- 收取和閱讀報告
- 做出決策和調整

---

## 並行派發檢查清單

在派發多個並行任務前，確認：

- [ ] 各任務修改的檔案集合無交集
- [ ] 各任務的測試可獨立執行
- [ ] 任務之間無先後依賴關係
- [ ] 不會同時存取相同外部資源
- [ ] 每個任務有明確的代理人負責
- [ ] 每個任務有獨立的 Ticket
