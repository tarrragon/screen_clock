# ARCH-016: Hook 允許清單過度限制導致工作流阻塞

## 錯誤症狀

主線程（PM）執行低風險小修正時被 Hook 阻擋，被迫走重流程（建 Ticket → 派發代理人 → Review → Merge），修正本身可能只需 1 行。

典型表現：
- 主線程需要補一行 `.gitignore` 忽略 runtime artifact（如 session lock 檔案）
- Hook 基於「保護分支 + 允許路徑清單」雙重攔截
- 允許清單未涵蓋該路徑 → 攔截 → 建議「建 Ticket 派發代理人」
- 實際修改風險極低但流程成本過高（10x 以上）

## 根因分析

### 根因 1：允許清單的預設策略偏向「新增須有充分理由」而非「排除須有充分理由」

Hook 設計時遵循「最小權限」原則，允許清單只列「已知需要的路徑」。當實戰遇到新情境（例如新增一類 runtime lock 檔），清單未更新就成為阻塞源。

預設策略偏差：
- 「新增路徑需證明安全」→ 新增摩擦高
- 「排除路徑需證明必要」→ 更符合實戰

`.gitignore` / `README.md` / `LICENSE` 等 repo 層級配置檔天然適合放行，但容易因未被早期設計覆蓋而遺漏。

### 根因 2：允許清單的維護責任不明確

- Hook 程式碼文件中未明示「誰可以決定新增路徑」
- 文件缺少「遇到攔截時的處理準則」（應該修 hook 還是走繞路？）
- 每次遇到新攔截都像初次遇到

### 根因 3：規則驗證依賴單次實戰觸發

允許清單的「太嚴」只有當主線程實際撞牆才被發現。沒有定期回顧機制檢視：
- 過去 N 個 session 中有多少次主線程被此 Hook 攔截？
- 這些攔截中有多少是「真的該擋」 vs 「清單遺漏」？

## 防護措施

### 規則層：允許清單維護策略

應在 Hook 設計文件明示：

| 情境 | 行動 |
|------|------|
| 攔截 .gitignore / repo 層級小配置 | 放行，這類檔案改動風險低 |
| 攔截主線程高頻小修正（runtime lock / README 補充） | 放行 |
| 攔截產品程式碼（src/ 下檔案） | 維持攔截，走 feature 分支 + 代理人 |
| 攔截有安全風險的檔案（secrets / CI 配置） | 維持攔截，走 Review |

### Hook 設計：放行清單可擴充性

- 允許清單集中管理（本專案：`.claude/hooks/lib/path_permission.py` 的 `ALLOWED_PATTERNS`）
- 錯誤訊息明列「如需新增允許路徑，請建 Ticket 並修改 ALLOWED_PATTERNS」
- 保留「放行 vs 阻擋」的邊界測試，避免誤放行

### 實戰防護：撞牆即修

- 主線程被 Hook 攔截時，第一反應不是「走繞路」，而是「評估是否清單遺漏」
- 如判斷清單遺漏 → 建 IMP Ticket 修 Hook（本錯誤模式觸發的 W10-033 即此模式）
- 如判斷攔截合理 → 尊重 Hook，走 feature 分支 / 代理人

### 定期回顧：Hook 攔截統計

（建議未來實作）
- Hook 日誌統計近 N 個 session 的攔截次數
- 依攔截路徑分群，識別高頻攔截
- 高頻攔截 + 低風險 = 清單遺漏候選

## 典型邊界案例

### 案例 A：.gitignore 補 runtime lock
- 風險：極低（不影響程式邏輯）
- 頻率：中（每次新 runtime 工具）
- 結論：應放行

### 案例 B：package.json 更動依賴版本
- 風險：中高（影響建置 / 可能引入漏洞）
- 頻率：中
- 結論：**應攔截**，走 feature 分支 + 代理人

### 案例 C：README.md 修 typo
- 風險：低
- 頻率：中
- 結論：應放行（已在豁免清單）

### 案例 D：.env / secrets 檔
- 風險：極高
- 頻率：低
- 結論：**永遠攔截**，走 Review

## 驗證方式

修改 Hook 允許清單後：
- 應有 TDD 測試覆蓋「新放行路徑 PASS + 邊界路徑仍 FAIL」
- 錯誤訊息同步更新（訊息與實際允許清單不一致會誤導用戶）
- 至少一次實戰驗證（修 Hook 後立刻測試原本被攔的情境）
