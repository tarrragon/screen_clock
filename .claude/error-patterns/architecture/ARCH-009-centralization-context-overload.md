# ARCH-009: 將決策邏輯集中到單一 skill 造成 context 膨脹

## 基本資訊

- **Pattern ID**: ARCH-009
- **分類**: 架構設計
- **來源版本**: v0.1.0
- **發現日期**: 2026-03-08
- **風險等級**: 中

## 問題描述

### 症狀

- 想解決「邏輯分散 → 流程斷裂」問題，直覺是「把所有決策邏輯集中到 manager skill 統一管理」
- 設計計畫包含：將 17 個 AskUserQuestion 場景、多層 Checkpoint 邏輯全部移入 manager skill
- 執行後發現 skill 膨脹（估計 +3-5K tokens），PM 每次呼叫 `/manager` 都需載入大量規則

### 根本原因（問題誤判）

**將兩個不同問題混為一談**：

| 問題 | 正確解法 |
|------|---------|
| 「規則維護分散，改一處需動多個檔案」 | 建立 Source of Truth 文件，hook/rules 引用之 |
| 「PM 在長 session 迷失，不知道下一步」 | 建立輕量錨點（Re-center Protocol），而非搬移規則 |

注意力渙散的問題不是「缺少規則」，而是「缺少重新定位的儀式和錨點」。把更多規則集中到 skill 不能解決注意力問題，反而增加每次呼叫時的認知負擔。

### 影響

- Skill 膨脹，每次 `/manager` 呼叫消耗大量 context（3-5K tokens）
- 17 個場景的詳細規則並不是 PM 在迷失時需要的資訊——PM 需要的是「現在應該做什麼」這一個答案
- 維護困難：規則更新需要同時修改 askuserquestion-rules.md 和 manager/SKILL.md

## 解決方案

**正確方向**：建立輕量的「Re-center Protocol」（錨點），而非集中化規則文件。

Re-center Protocol 設計原則：
1. 通過查詢（`ticket track list --status in_progress`）讓 CLI 告訴 PM 當前狀態
2. 對照最後完成的動作定位 Checkpoint
3. 輸出「唯一的下一步強制指令」

**關鍵區別**：

| 錯誤方向 | 正確方向 |
|---------|---------|
| 把規則搬進 skill（增加知識量） | 建立查詢儀式（讓資料告訴答案） |
| 17 個場景全部文件化 | 3 步驟快速定位 |
| 被動讀取（靠記憶） | 主動查詢（靠 CLI 狀態） |

## 預防措施

### 設計時的問題

在提出「集中化」方案前，先問：
1. **目標是什麼**：解決「維護分散」還是「注意力渙散」？
2. **使用者在迷失時需要的是什麼**：更多規則，還是一個明確的下一步？
3. **集中化的 context 成本是否合理**：每次呼叫的 token 成本 vs 解決的問題

### 識別信號

| 信號 | 說明 |
|------|------|
| 「把所有 X 集中到一個地方」 | 觸發評估：是否真的需要集中，還是只需要錨點 |
| 設計文件 > 500 tokens | 考慮是否應該拆分或引用 |
| 「方便查詢」但 hook 已有觸發 | hook 負責觸發，skill 只需提供錨點 |

### 五視角評估

當遇到「應該集中到哪裡」的架構決策時，建議啟動五視角評估（`/parallel-evaluation`）：
- 設計一致性視角：是否符合單一職責原則
- 維護性視角：集中化後的實際維護成本
- 觸發可靠性視角：主動 vs 被動哪種更有效
- Context 效率視角：集中化的 token 成本
- 使用者體驗視角：在迷失時真正需要什麼
