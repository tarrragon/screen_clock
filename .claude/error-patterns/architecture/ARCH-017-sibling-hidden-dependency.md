# ARCH-017: 兄弟任務隱藏依賴違反 Atomic Ticket 結構原則

## 核心原則

> **兄弟任務之間預設平行無依賴。隱藏依賴（`relatedTo` / 文件引用 / 多對一 hub）必須透過父子結構（深度）表達，而非扁平在 `blockedBy` 中。單向時序依賴（規格→實作）在嚴格條件下可保留為「串行兄弟」，見下方合法例外章節。**

## 症狀

PM 拆分任務時把有依賴的 Ticket 放在**同一層級**作為兄弟，依賴關係藏在：

- `blockedBy` 欄位（顯式依賴旗標）
- `relatedTo` 欄位（弱關聯標記）
- 文件中的引用（如「引用 .1 的哲學章節」）
- 實務上的執行順序隱含（如 .5 必須在 .1 之後寫）

表面看 5 個兄弟 Ticket，實質是線性/樹狀依賴鏈壓平呈現。

## 實際觸發案例（W10-036 原始拆分）

**原結構**（違反原則）：

```
W10-036
├── .1 (哲學層)
├── .2 (規則層，relatedTo .1 ← 隱藏引用依賴)
├── .3 (references，relatedTo .1 ← 隱藏引用依賴)
├── .4 (Hook 實作，blockedBy .1/.2/.3/.5 ← 顯式但壓平)
└── .5 (實務指南，relatedTo .1 ← 隱藏引用依賴)
```

問題：
- .2/.3/.5 都引用 .1 的哲學章節，.1 是實質 hub
- .4 被其他 4 個阻塞，是實質 tail
- 但 5 者都在同一層級，無法從結構看出依賴序

## 根因

PM 直觀上把 W10-036 視為「一個大任務」，拆子時按「內容面向」劃分（哲學/規則/references/實務/實作），而非按「依賴序」。結果所有面向變兄弟。

**錯誤的抽象**：「子 Ticket = 父任務的某一面向」
**正確的抽象**：「子 Ticket = 父任務樹中一個可獨立完成的節點，兄弟間應無依賴」

## 防護措施

### 設計時檢查（Ticket 拆分前）

**三問**：

1. 這些子任務之間是否**可同時執行**？（彼此無依賴）
2. 若任一子任務需要等待另一子任務完成，依賴能否**透過深度表達**？
3. 父下所有子的完成，是否等價於父責任履行？

**若 Q1 = 否**：依賴者應**下沉一層**成為被依賴者的子，或移到不同 Wave。

### 結構重組範式

**原（反例）**：
```
Parent
├── A (hub, 被 B/C/D 引用)
├── B (relatedTo A)
├── C (relatedTo A)
└── D (blockedBy A)
```

**重組（正例）**：
```
Parent
└── A' (新聚合父，承擔 A 的 hub 責任)
    ├── A.1 (原 A, 基礎層)
    ├── A.2 (原 B, relatedTo A.1)
    ├── A.3 (原 C, relatedTo A.1)
    └── A.4 (原 D, blockedBy A.1/A.2/A.3)
```

A' 作為聚合父，兄弟層級（Parent 下）只有 A' 一個。A' 內部的依賴深度化（A.1 被 A.2-A.4 引用）。

### 拆分決策樹（簡化）

```
拆分父任務時
    |
    v
子任務之間是否有依賴？
    |
    +── 否 → 平行兄弟，適當分派（OK）
    |
    +── 是 → 是否所有依賴指向同一個子？
            |
            +── 是 → 該子應為其他子的父（升格為聚合父）
            |
            +── 否 → 評估是否拆為多個兄弟 + 聚合父組（較複雜）
```

## 合法例外：串行兄弟（單向時序依賴）

**不是所有兄弟 `blockedBy` 都是違規**。當滿足以下 4 條件時，屬於「串行兄弟」合法模式：

1. **單向**：A → B，B 不依賴 A 以外的兄弟
2. **無環**：兄弟依賴形成 DAG
3. **規格→實作時序**：A 為規格/介面/方法論，B 為實作/驗證/Hook
4. **不可深度化**：B 若成為 A 的子會違反內容邊界（如 Hook 不是方法論的子章節）

**合法 vs 違規對照**：

| 維度 | 違規（本 error-pattern 主題） | 合法串行兄弟 |
|------|---------------------------|-------------|
| 依賴結構 | 多對一 hub（B/C/D 都指 A） | 單對一時序（僅 B 指 A） |
| 依賴性質 | hub 面向共用（內容層並列） | 規格→實作（時序層級） |
| 可深度化？ | 可（升格 A 為聚合父） | 不可（違反內容邊界） |
| 隱藏程度 | 扁平化藏 hub 依賴 | 顯式 `blockedBy` 直書 |

**判別重點**：問「若把 B 變成 A 的子任務，內容邊界是否合理？」
- 合理 → 應深度化，原結構違反 ARCH-017
- 不合理 → 串行兄弟合法，保留 blockedBy

## 與 Wave 定義的關係

`atomic-ticket-methodology.md` 已定義 Wave：
- W1: 無依賴，可並行執行
- W2: 依賴 W1 的部分 Ticket

Wave 是**版本層級**的依賴分層，而 ARCH-017 規則是**單一任務鏈內**的兄弟無依賴原則。兩者互補：
- 跨 Wave 依賴：透過 Wave 編號分層
- 同 Wave 內兄弟依賴：透過父子結構分層
