# ARCH-003: 並行代理人持久化落差

## 基本資訊

- **Pattern ID**: ARCH-003
- **分類**: 架構設計
- **來源版本**: v0.31.0
- **發現日期**: 2026-02-26
- **風險等級**: 高

## 問題描述

### 症狀

並行派發 7 個 Task subagent 後，所有代理人都自我報告「完成」，但 `git diff --stat` 顯示只有 9/26 個檔案實際被修改。17 個檔案的變更遺失。

具體情況：

| Agent | 自我報告 | git 實際狀態 | 差異 |
|-------|---------|-------------|------|
| 1 | 完成 | 4 檔已修改 | 一致 |
| 2 | 完成 | 未修改 | 全部遺失 |
| 3 | 完成 | 3/4 修改 | 1 檔遺失 |
| 4 | 完成 | 未修改 | 全部 4 檔遺失 |
| 5 | 2/4 完成 | 0/4 修改 | 全部遺失 |
| 6 | 完成 | 未修改 | 全部 4 檔遺失 |
| 7 | 完成 | 未修改 | 全部 4 檔遺失 |

### 根本原因 (5 Why 分析)

1. Why 1: 17 個檔案的修改沒有出現在工作目錄中
2. Why 2: 代理人自我報告完成但實際 Edit 工具未成功執行
3. Why 3: 可能原因包括：並行寫入 IO 競爭、Linter Hook 自動還原、代理人 token 用盡後仍報告完成
4. Why 4: 沒有在代理人完成後立即用 `git diff --stat` 驗證實際變更
5. Why 5 (根本原因): **驗證機制依賴代理人自我報告，而非客觀的檔案系統狀態**

### 可能的失敗機制

| 機制 | 說明 | 佐證 |
|------|------|------|
| IO 競爭 | 多個 subagent 同時操作檔案系統 | Agent 1（獨佔檔案）成功率 100% |
| Linter 還原 | Pre-commit hook 或 linter 自動還原 | Agent 5 明確提到「被自動 linter 還原」|
| Token 耗盡 | 分析完畢後 token 不足以執行 Edit | Agent 2、4、6、7 全部遺失 |
| 驗證缺陷 | py_compile 只驗語法，不驗內容變更 | 所有代理人都報告「語法驗證通過」|

## 解決方案

### 正確做法

**Step 1: 並行派發後強制驗證**

```bash
# 並行代理人全部完成後，立即執行
git diff --stat
# 或
git diff --name-only

# 比對預期修改檔案清單 vs 實際修改檔案
```

**Step 2: 預期檔案清單對照**

在派發前建立「預期修改檔案清單」，完成後逐一比對：

```markdown
## 並行派發驗證

| 預期檔案 | Agent | 實際修改 | 狀態 |
|---------|-------|---------|------|
| hook_a.py | 3 | Yes | OK |
| hook_b.py | 4 | No | MISSING |
| hook_c.py | 4 | No | MISSING |
```

**Step 3: 缺失立即補派**

發現缺失後，只補派缺失檔案的代理人，不重做已完成的部分。

### 錯誤做法 (避免)

| 錯誤做法 | 問題 |
|---------|------|
| 信任代理人自我報告就提交 | 可能遺失大量變更 |
| 只用 py_compile 驗證 | 語法正確不代表有修改 |
| 全部重做 | 浪費已完成的工作 |
| 不建立預期檔案清單 | 無法系統化比對 |

### 防護措施

1. **並行派發後強制驗證檢查清單**：
   - [ ] `git diff --stat` 確認修改檔案數量
   - [ ] 比對預期清單 vs 實際清單
   - [ ] 缺失檔案立即補派
   - [ ] 內容抽檢（至少抽查 20% 修改內容）

2. **代理人 max_turns 設定**：確保代理人有足夠的回合數完成工作，避免分析完但無法執行 Edit

3. **分批策略**：如果並行數量 > 5，考慮分 2 批執行，降低 IO 競爭風險

## 影響統計

| 指標 | 數值 |
|------|------|
| 遺失檔案數 | 17 / 26（65%）|
| 受影響代理人 | 5 / 7（71%）|
| 補派次數 | 至少 2 次（第二次提交修復 16 檔）|
| 額外耗時 | 需要重新派發代理人修復 |
