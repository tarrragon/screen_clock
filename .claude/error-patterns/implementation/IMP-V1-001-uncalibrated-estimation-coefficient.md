# IMP-V1-001: 估算係數未經實測校準即上線

**類別**: implementation
**嚴重度**: 中
**發現日期**: 2026-06-12
**相關 Ticket**: 1.0.0-W7-004.7（發現）/ 1.0.0-W7-006（校準修復）

---

## 症狀

監測/守門類機制使用「合理猜測」的轉換係數（如 token 估算 `chars // 3`）上線，自估值與權威工具實測值出現倍數級偏差（本案 2.3 倍：hook 自估 16.9k tokens，`/context` 同集合實測約 38.9k tokens）。

**識別特徵**：

- 程式碼中存在轉換係數常數（`CHARS_PER_TOKEN`、壓縮比、平均行寬等），註解寫「保守近似」「可後續校準」但無實測依據與校準日期
- 守門機制輸出的數值與權威來源（平台內建量測、計費報表、profiler）長期無人比對
- 測試只驗證「公式套用正確」（chars // 3 算對了），不驗證「公式本身貼近真值」

## 根因

估算公式的**套用正確性**（單元測試可驗）與**係數真實性**（只能實測校準）是兩個獨立維度；實作與測試流程天然只覆蓋前者。係數的選擇發生在實作初期、權威量測往往要到部署後才可取得，若校準動作沒有被綁定為顯式 trigger（ticket / acceptance），「可後續校準」會永遠停留在註解裡。

**Why 危害放大**：守門類機制的價值在告警靈敏度。低估方向的係數偏差（本案實際已用 86% 預算、hook 顯示 38%）會讓預算守門在真正逼近上限時保持沉默，機制存在但失效——比沒有機制更糟，因為它提供假安心。

## 解決方案

1. 取得權威量測值（本案：新 session `/context` 的 Memory files 逐檔 tokens）
2. 對齊量測集合（排除兩端覆蓋差異，如 user-level 檔案），計算實際係數（本案約 1.3 chars/token，繁中為主混英文）
3. 更新常數，註解記錄校準依據與日期（來源工具、量測日期、集合範圍）
4. 測試斷言改引用常數而非硬編碼係數值，避免下次校準時測試連鎖失敗
5. 加 smoke 驗收：自估值與權威值同集合比對誤差上限（本案 <= 15%）

## 預防措施

| 時機 | 動作 |
|------|------|
| 實作含估算係數的機制時 | 常數註解必須標明「未校準」狀態與校準資料來源；acceptance 預留校準條目或於 complete 前 spawn 校準 ticket（decision-trigger-binding：禁止「之後校準」無 trigger 延後） |
| 驗收守門類機制時 | 驗收條件必含「自估值 vs 權威值同集合比對」，不可只驗公式套用 |
| 中文為主文本的 token 估算 | 不可沿用英文經驗值 chars/3 ~ chars/4；繁中混英實測約 1.2-1.4 chars/token，先以 1.3 為起點再實測校準 |

## 行為模式分析

「保守近似 + 可後續校準」是實作期降低決策成本的合理策略，問題不在猜係數，而在校準動作未綁定 trigger。本案因 Context Bundle 預留校準條款 + 最終驗收強制 `/context` 實測，偏差在上線當日即被攔截；缺少任一環節，偏差會潛伏至預算真正爆表時才暴露。
