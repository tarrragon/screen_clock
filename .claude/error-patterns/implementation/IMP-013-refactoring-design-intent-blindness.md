# IMP-013: 重構設計意圖盲視

## 基本資訊

- **Pattern ID**: IMP-013
- **分類**: 程式碼實作
- **來源版本**: v0.31.1
- **發現日期**: 2026-03-04
- **風險等級**: 中
- **來源 Ticket**: hook-health-monitor v2.0 重構

## 問題描述

### 症狀

重構後的 `check_hook_logs` 函式接收 `logger` 參數但從未使用。初始判斷為「多餘的程式碼」，準備移除。實際上是**設計意圖未被實現**：`_check_single_hook_log` 遇到異常時只回傳結果，沒有寫入日誌檔。

### 根本原因 (5 Why 分析)

1. Why 1: logger 參數存在但未被使用
2. Why 2: v2.0 重寫時從 v1.x 照搬了函式簽名
3. Why 3: v1.x 本身就有相同問題 — logger 被傳入但未在子函式中使用
4. Why 4: 重構時沒有質疑「這個參數為什麼存在？它的設計意圖是什麼？」
5. Why 5 (根本原因): **重構認知模式偏向「簡化移除」而非「質疑理解」**

### 三個錯誤決策

#### 錯誤決策 1：沿用舊設計，未理解新需求

- **行為**：v2.0 重寫時照搬 v1.x 的函式簽名和參數，沒有重新審視每個參數的設計目的
- **根因**：重構時將「保持介面一致」視為安全做法，跳過了「這個參數在新設計中是否有意義？如何使用？」的質疑
- **後果**：繼承了舊版本的設計缺陷

#### 錯誤決策 2：異常處理缺少日誌持久化

- **行為**：`_check_single_hook_log` 的 except OSError 只回傳 severity 和 message，沒有調用 logger 記錄
- **根因**：重構者關注的是「讓函式能正確回傳結果」，忽略了「異常資訊需要持久化到日誌檔」的可觀測性需求
- **後果**：hook 檢查失敗時，只有 stdout/stderr 輸出，失敗原因沒有持久化

#### 錯誤決策 3：將未使用的設計意圖判斷為死程式碼

- **行為**：發現 logger 未被使用後，直覺判斷為「多餘的程式碼，應該移除」
- **根因**：除錯時的認知偏誤 — 看到 unused code 的第一反應是「清理」而非「質疑為什麼存在」
- **後果**：如果真的移除了 logger 參數，會讓設計意圖永遠消失，日誌記錄需求被靜默遺忘

## 防護措施

### 檢查清單（重構時強制）

#### A. 參數沿用審查（每個被保留的參數）

- [ ] 這個參數在舊版中是否被正確使用？
- [ ] 如果舊版也沒用到，它的原始設計意圖是什麼？
- [ ] 在新設計中，這個參數應該在哪裡被使用？

#### B. 未使用程式碼質疑（發現 unused 時強制）

- [ ] **第一步**：不是「移除」，而是「為什麼存在？」
- [ ] 追溯歷史：這個程式碼是何時加入的？加入時的目的是什麼？
- [ ] 判斷類型：是「已完成但不再需要」還是「設計意圖未實現」？
- [ ] 如果是未實現的設計意圖 → 補上實作，而非移除

#### C. 異常處理可觀測性（每個 except 區塊）

- [ ] 異常資訊是否有持久化記錄（日誌檔）？
- [ ] 異常是否對用戶可見（stderr/通知）？
- [ ] 只做 return/pass 而不記錄的 except 是否有正當理由？

## 決策流程改進

### 改進 1：重構前的「參數意圖審計」

```
沿用函式簽名
    |
    v
[新增步驟] 逐一審查每個參數
    |
    +-- 參數在舊版中被使用? → 新版是否保持相同使用方式?
    |
    +-- 參數在舊版中未被使用?
        |
        +-- 原始設計意圖是什麼?（查 git log、註解、需求）
        +-- 在新設計中是否應該被使用? → 是 → 補上實作
        +-- 確認不需要? → 移除並記錄原因
```

### 改進 2：「Unused = 質疑」認知模式

```
發現未使用的程式碼
    |
    v
[強制] 質疑三問
    |
    1. 這段程式碼的原始目的是什麼?
    2. 它是「曾經有用但不再需要」還是「應該有用但沒實作」?
    3. 移除它會不會讓某個設計意圖永遠消失?
    |
    v
只有確認為「曾經有用但不再需要」才可移除
```

### 改進 3：異常處理的「雙通道輸出」

```
except 區塊
    |
    v
[檢查] 是否同時滿足:
    1. 用戶可見（stderr 或通知）
    2. 持久化（日誌檔）
    |
    +-- 兩者都有 → 通過
    +-- 缺少任一 → 補上
```

## 與其他模式的關係

| 模式 | 關係 |
|------|------|
| IMP-003 (作用域迴歸) | 共同根因：重構時影響範圍分析不完整 |
| IMP-005 (import 遷移) | 共同根因：「移動東西」時未追蹤全量引用 |
| IMP-006 (Hook 靜默故障) | 本模式的後果之一：異常未持久化導致問題不可見 |
| **本模式 IMP-013** | **獨特問題**：不是「漏改引用」而是「未質疑設計意圖」 |

## 核心教訓

> **Unused code is a question, not an answer.**
>
> 未使用的程式碼是一個待回答的問題（「為什麼存在？」），
> 而不是一個已知的答案（「應該移除」）。
