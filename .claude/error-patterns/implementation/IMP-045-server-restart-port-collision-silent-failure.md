# IMP-045: 伺服器專案缺乏運作狀況觀測 — 假設能動的除錯陷阱

## 錯誤摘要

| 項目 | 說明 |
|------|------|
| **分類** | Implementation |
| **嚴重性** | High（浪費大量除錯時間在錯誤方向） |
| **發現版本** | v0.2.0 |

## 症狀

- 重啟 Go Backend 後，Flutter app 行為異常（訊息不更新、session 消失）
- 花大量時間除錯程式碼邏輯（FileWatcher、channel、Registry），但根因是伺服器根本沒有成功重啟
- 多次修復、合併、重啟，每次都重複遇到同一問題
- 沒有觀測手段確認伺服器「正在正確運作」，只能從前端異常反推

## 根因分析

### 核心問題：缺乏可觀測性意識

這不只是一個「port 被佔用」的技術問題，而是反映了兩個更根本的缺失：

**1. 開發時沒有考慮可觀測性**

伺服器專案的可觀測性應在開發階段就設計，而非出問題後才補：
- 啟動成功/失敗應有明確的狀態輸出
- 關鍵元件（FileWatcher、EventDispatcher、WebSocket）的運作狀態應持續可見
- 異常狀態（port 衝突、channel 滿、fsnotify 停止）應主動告警，不依賴人工發現

**2. 操作時沒有觀測伺服器運作狀況**

啟動伺服器後直接進行下一步，沒有確認伺服器實際運作狀態。當驗證結果異常時，誤以為是程式碼問題而非環境問題，導致修復方向完全錯誤。

### 具體技術問題

- `go run .` 在 port 被佔用時錯誤訊息被 scrollback 推走
- zellij 送按鍵無回傳值，無法判斷命令是否成功
- 舊 process 的子 process 未被完全清除

## 教訓

### 原則 1：伺服器專案必須設計可觀測性

| 階段 | 應有的可觀測性 |
|------|--------------|
| 啟動 | 明確輸出所有元件啟動狀態和關鍵指標（掃描檔案數、監聽 port） |
| 運行中 | 定期輸出心跳或統計（事件數、連線數、最後事件時間） |
| 異常 | 主動輸出告警（channel 滿、fsnotify 停止、連線中斷） |
| 停止 | 明確輸出 graceful shutdown 狀態 |

### 原則 2：每次環境變更後必須觀測

任何環境變更（重啟、合併、部署）後，第一步是**確認環境狀態**，不是開始使用：
- 讀取 server log 確認關鍵元件啟動
- 確認 port 正確綁定
- 確認 client 能連線
- 確認事件能正常流動

### 原則 3：區分環境問題和程式碼問題

當行為異常時，排查順序：
1. 環境是否正確（process 活著？port 對？版本對？）
2. 資料流是否暢通（事件有進入？有推送？有接收？）
3. 程式碼邏輯是否正確

## 防護措施

### 已實施

1. CLAUDE.md 新增「重啟伺服器注意事項」：kill-verify-start-verify 標準流程
2. Flutter 端新增 `[WS]`/`[Conv]`/`[SessionList]` debug log，可從 terminal 觀測前端收到的訊息

### 建議

1. Go Backend 新增啟動摘要 log：一行輸出所有關鍵指標（掃描 session 數、監聽 port、啟用的功能）
2. 考慮新增 `/health` 端點回傳元件狀態（FileWatcher 最後事件時間、Registry session 數）
3. 考慮定期心跳 log（每 60 秒輸出一次統計）
