# IMP-039: Phase 4b 代理人 context 耗盡 — prompt 過於開放

## 錯誤摘要

| 項目 | 說明 |
|------|------|
| **分類** | Implementation |
| **嚴重性** | 中（任務完全失敗，需重試） |
| **發現版本** | v0.2.0 |

## 症狀

- Phase 4b 派發 cinnamon-refactor-owl 和 fennel-go-developer，兩者均因 context 耗盡未完成任何修改
- 代理人花費大量 context 讀取 Phase 4a 報告、源碼、常數檔案等，在實際修改前即耗盡
- 需要在下一個 session 重新派發

## 根因分析

**行為模式**：Phase 4b prompt 要求代理人「根據 Phase 4a 報告執行重構」，但未提供具體修改位置和程式碼片段。代理人需自行：
1. 讀取 Phase 4a 報告（多份，每份數 KB）
2. 讀取所有相關源碼檔案
3. 理解問題後規劃修改
4. 執行修改

步驟 1-3 的 context 消耗已超過可用額度。

**技術原因**：Phase 4b 的修改通常很小（數行到數十行），但理解「為什麼要修改」需要閱讀大量背景資料。PM 應承擔「理解」的工作，只將「執行」交給代理人。

## 解決方案

Phase 4b 派發時，PM 在 prompt 中直接提供：

1. **具體修改的檔案和行號**（如 `session_registry.go:238-252`）
2. **修復前後的程式碼片段**（讓代理人知道改什麼）
3. **明確的執行步驟順序**（先改 A → 測試 → 再改 B → 測試）
4. **修復清單優先級表**（P0/P1/P2 分類）

## 防護措施

- PM 在派發 Phase 4b 前，必須先讀取源碼和報告，整理出精簡的修復清單
- prompt 應包含足夠資訊讓代理人「只需執行，不需探索」
- 若修復項目超過 5 個，考慮拆分為多個子任務

## 教訓

「理解問題」和「執行修復」是兩個不同的認知負擔。PM 應承擔前者，代理人負責後者。讓代理人同時承擔兩者會導致 context 浪費在探索而非產出上。
