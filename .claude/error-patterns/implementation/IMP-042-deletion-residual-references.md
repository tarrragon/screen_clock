# IMP-042: 簡化/刪除操作後殘留引用未同步清理

## 錯誤摘要

| 項目 | 說明 |
|------|------|
| **分類** | Implementation |
| **嚴重性** | Medium（語義矛盾，不影響功能但誤導讀者） |
| **發現版本** | v0.2.0 |

## 症狀

- 刪除章節或概念後，frontmatter description、其他章節的描述文字仍引用已刪除的名稱或數量
- 具體案例：刪除「核心抽象（遞迴展開器）」章節後，frontmatter 仍寫「遞迴需求展開器」；刪除維度 4-7 後，Layer 2 描述仍寫「沿 7 個維度掃描」
- 表現為文件內部的事實矛盾

## 根因分析

- 刪除操作聚焦於目標區段本身（被刪除的章節），未搜尋同檔案或跨檔案中引用該概念的其他位置
- 人類（和 AI）在執行「刪除」時，心智模型是「移除這個區塊」，而非「移除這個概念的所有痕跡」
- frontmatter 和其他章節的描述文字不在刪除的心智範圍內

## 防護措施

**刪除後全檔搜尋檢查清單**（刪除任何章節/概念後必須執行）：

| 步驟 | 搜尋目標 | 範例 |
|------|---------|------|
| 1 | 被刪除的標題/名稱 | grep「遞迴展開器」「核心抽象」 |
| 2 | 相關數量詞 | grep「7 個維度」「7 維度」 |
| 3 | frontmatter description | 手動檢查 description 是否引用已刪除概念 |
| 4 | 跨檔案引用 | grep 相關檔案（模板、代理人定義等） |

**驗證方式**：三人組多視角審查可有效捕獲此類殘留（本次即由 code-reviewer 發現）。

## 學習要點

- 「刪除」不等於「移除」：刪除是物理操作（移除區塊），移除是語義操作（消除概念的所有引用）
- 簡化任務的審查重點應包含「殘留引用搜尋」
- frontmatter 是最容易被遺忘的殘留位置
