# IMP-053: 一刀切修改忽略程式碼執行路徑差異

## 錯誤症狀

- 批量搜尋替換某個模式（如 `sys.exit(1)` → `sys.exit(0)`），不區分呼叫位置的語義差異
- 修改後部分程式碼行為正確改善，部分程式碼語義被破壞
- Code Review 時不易發現，因為每個改動「看起來一致」

## 根因分析

**直接原因**：將所有 `sys.exit(1)` 視為同一類問題，未區分不同執行路徑的語義。

**行為模式**：
1. 識別出一個正確的問題（Hook 路徑的 `sys.exit(1)` 觸發 CLI hook error）
2. 使用 grep 找出所有出現位置
3. 全部套用同一修復方案
4. 未逐一分析每個位置的上下文和語義

**本案具體表現**：

| 位置類型 | 語義 | 正確做法 | 錯誤做法（一刀切） |
|---------|------|---------|------------------|
| Hook 執行路徑（ImportError） | Hook 系統呼叫 | exit(0) 安全降級 | exit(0) — 正確 |
| `__main__` CLI 工具 | 人工執行，錯誤應報告 | exit(1) 報告錯誤 | exit(0) — 語義錯誤 |
| `run_hook_safely` 異常 | 核心安全包裝 | 記錄日誌，不繞過 CLI bug | exit(0) + stderr — 違反 IMP-048/049 |

## 解決方案

**修復前必做**：對每個修改位置逐一回答：

| 問題 | 目的 |
|------|------|
| 這段程式碼由誰呼叫？（Hook 系統 / CLI 使用者 / 其他模組） | 確定執行路徑 |
| 在這個呼叫者的 context 中，原始行為的語義是什麼？ | 確定語義 |
| 修改後的行為在這個 context 中是否合理？ | 驗證修改正確性 |

**批量修改的安全做法**：
1. grep 找出所有位置
2. 按執行路徑分類（Hook 路徑 / `__main__` / library 方法 / 其他）
3. 每個分類獨立決定修改策略
4. 不同分類可能需要不同（甚至相反）的修改

## 防護措施

| 規則 | 說明 |
|------|------|
| 批量修改前先分類 | 按執行路徑分組，不按檔案或目錄分組 |
| 每組獨立評估 | 同一個 pattern 在不同路徑中可能有不同的正確做法 |
| `__main__` 區塊豁免 | CLI 工具入口的行為不受 Hook 系統限制 |
| 修改核心函式前讀 error-patterns | `run_hook_safely` 等高影響函式有歷史教訓 |

## 關聯

- **相關模式**: IMP-049（hook error 是 CLI bug）、IMP-048（stderr 觸發 hook error）
- **觸發條件**: 任何涉及批量搜尋替換的修改
