# DOC-009: 「靜默處理」用語誤用

## 問題描述

在撰寫 Hook 錯誤處理文件時，將「寫入日誌但不對用戶顯示 stderr」描述為「靜默處理」。這兩者是截然不同的處理方式：

| 用語 | 實際意義 | 是否有記錄 |
|------|---------|-----------|
| 靜默處理 | 完全不記錄（pass/return/空 catch） | 否 |
| 記錄到日誌但不顯示 | 寫入日誌檔，不輸出 stderr | 是 |

## 錯誤範例

```markdown
# 錯誤：
| 已預期但不正常 | `logger.info()` | 否 | 靜默處理 |

# 正確：
| 已預期但不正常 | `logger.info()` | 否 | 記錄到日誌檔，不干擾用戶 |
```

```python
# 以下不是「靜默處理」— 有寫日誌
input_data = read_json_from_stdin(logger)  # 內部呼叫 logger.info()
if input_data is None:
    return 0  # 正常退出（已記錄到日誌）

# 以下才是真正的「靜默處理」
except Exception:
    pass  # 完全不記錄
```

## 根因

撰寫者從「用戶看不到」的角度思考，忽略了「系統有記錄」這個事實。對用戶不可見 ≠ 靜默。

## 防護措施

撰寫錯誤處理文件時，區分三個層級：

1. **對用戶可見**：stderr 輸出、UI 顯示（如 hook error）
2. **記錄到日誌**：寫入日誌檔案，不對用戶顯示 — 可追蹤、可除錯
3. **靜默處理**：完全不記錄 — pass/return/空 catch

只有第 3 種才能稱為「靜默處理」。第 2 種應描述為「記錄到日誌檔，不干擾用戶」。

## 檢查清單

- [ ] 文件中是否有「靜默」一詞？確認是否真的完全不記錄
- [ ] 程式碼註解中的「靜默退出」是否實際上有日誌？改為「正常退出（已記錄到日誌）」
