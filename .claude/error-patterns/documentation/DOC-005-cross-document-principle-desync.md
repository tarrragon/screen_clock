### 症狀

新增原則（如 Context 隔離）到某個文件後，引用或延伸該概念的相關文件出現語義矛盾。例如：

- SKILL.md 新增「一個 session 只做一件事，做完就交接」
- async-mindset.md 仍寫「持續監控多個進行中任務」（暗示跨 session）
- decision-tree.md Checkpoint 0 只涵蓋「子任務」可並行，遺漏「獨立 Ticket」路徑
- 允許表「建立和指派」列為同一職責，但新原則將它們分到不同 session

### 根因

**行為模式**：修改單一文件時，未系統性檢查該文件被哪些其他文件引用或延伸。

具體原因：
1. 新增原則只更新了「定義處」，未更新「消費處」（引用該概念的其他文件）
2. 相關文件之間的引用關係不明顯（非 `@` 引用，而是語義依賴）
3. 修改者假設其他文件不受影響，未執行一致性掃描

**與 ARCH-005 的區別**：ARCH-005 是多份文件各自定義同一規則導致衝突；DOC-005 是新增原則後未同步更新語義相關的文件。

### 影響範圍

| 影響 | 說明 |
|------|------|
| 規則矛盾 | PM 在不同文件看到相反的指引 |
| 行為不一致 | 依據不同文件會做出不同決策 |
| 信任損失 | 規則系統可信度降低 |

### 解決方案

本次修正：
1. async-mindset.md — 「持續狀態」加入 session 範疇限制 + 引用 Context 隔離原則
2. decision-tree.md — Checkpoint 0 擴展為「子任務或獨立 Ticket」
3. SKILL.md — 允許表拆分「建立」和「指派」+ 加 Session 歸屬欄位
4. parallel-dispatch.md — 新增多視角彙整結論回報規則

### 預防措施

**新增原則時的強制檢查清單**：

| 步驟 | 動作 | 驗證方式 |
|------|------|---------|
| 1 | 列出新原則影響的概念關鍵字 | 人工識別 |
| 2 | `grep -r "關鍵字"` 搜尋所有引用該概念的文件 | 全文搜尋 |
| 3 | 逐一檢查語義是否仍一致 | 人工比對 |
| 4 | 更新所有受影響文件 | 同一 commit |

**建議防護機制**：
- 新增原則後，派發 Explore agent 掃描相關文件的一致性（本次即用此方法發現問題）
- 重大原則變更使用 `/parallel-evaluation G`（系統設計審查）
