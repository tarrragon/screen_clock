# DOC-004: CLI 命令通配符表示導致 Agent 錯誤類推

## 基本資訊

- **Pattern ID**: DOC-004
- **分類**: 文件設計
- **來源版本**: v0.1.0
- **發現日期**: 2026-03-04
- **風險等級**: 中

## 問題描述

### 症狀

主線程嘗試呼叫不存在的 `ticket track set-blocked-by` 命令。CLI 回傳 `invalid choice` 錯誤後，主線程改為直接編輯 frontmatter（正確做法，但缺乏引導）。

### 根本原因 (5 Why 分析)

1. Why 1: 主線程呼叫了不存在的 `ticket track set-blocked-by`
2. Why 2: 主線程根據已有的 `set-who`、`set-what` 等命令類推
3. Why 3: SKILL.md 使用 `set-*` 通配符表示 6 個 set 命令
4. Why 4: 文件中缺少「僅此 6 個 set 命令」的明確說明和「不支援的操作」清單
5. Why 5 (根本原因): **CLI 文件使用模式暗示（通配符）而非明確列舉，缺少邊界說明**

### 受影響模式

```
Agent 看到 SKILL.md: "UPDATE 操作（...set-*...）"
                |
                v
觀察到 set-who, set-what 等命令存在
                |
                v
類推 set-blocked-by 也存在
                |
                v
呼叫失敗 → 浪費一次工具呼叫 + 可能選擇錯誤的替代方案
```

### 可偵測性

| 偵測方式 | 可偵測 |
|---------|--------|
| CLI --help | 是（會列出所有有效子命令） |
| 文件檢查 | 否（通配符表示隱藏了邊界） |
| 自動化 | 可以（比對 Skill 文件 vs CLI --help 輸出） |

## 防護措施

### 已完成的修正

1. **track-command.md**：新增「CLI 可修改欄位 vs 手動編輯欄位」對照表和「不存在的操作」清單
2. **SKILL.md**：將 `set-*` 通配符改為明確列出 6 個命令名稱，新增注意事項

### 通用防護原則

| 原則 | 說明 |
|------|------|
| 明確列舉 > 通配符 | 文件中列出所有可用命令，不使用 `set-*` 等隱含所有變體的寫法 |
| 負面清單 | 除了列出「支援的操作」，也要列出「不支援的操作」和正確替代方案 |
| 邊界說明 | 當 CLI 支援某個模式（如 set-{field}）但不支援所有欄位時，必須明確標注範圍 |

### 檢查清單（Skill 文件撰寫時）

- [ ] 是否使用了通配符表示（如 `set-*`、`batch-*`）？→ 改為明確列舉
- [ ] 是否說明了「不支援的操作」？→ 新增負面清單
- [ ] 是否有「CLI 可修改 vs 手動編輯」的對照表？→ 確保欄位操作方式清晰
