# DOC-V1-001: 位置編號引用隨目標文件演進靜默失效

**類別**: documentation
**嚴重度**: 中
**發現日期**: 2026-06-12
**相關 Ticket**: 1.0.0-W8-011（檢視 blog skill 回流時發現）

---

## 症狀

引用方以「位置編號」指向目標文件的內部結構（「第三原則」「見 Stage 3」「如第 4 點」），目標文件改版後該位置不存在或承載不同內容，引用句字面完好、解析不報錯，讀者被靜默導向錯誤或不存在的內容。

**本案實例**：`ask_user_question_reminders.py` 的 checkpoint 提示引用「writing-documents.md 第三原則」，該檔（英文結構）無此錨點——`git show HEAD` grep 計數為 0，引用自寫入起即懸空，每次 ticket complete 都對 PM 顯示一次無效指引，無任何機制告警。

**識別特徵**：

- 引用句含「第 N 原則」「第 N 點」「Stage N」「步驟 N」等位置詞且目標是活文件（會演進的規則 / 方法論 / skill reference）
- broken-link 掃描器不報錯（路徑有效，失效的是文件內定位）
- 引用方與目標文件由不同維護流程更新（hook 文字 vs skill 內容），無同步觸發點

## 根因

位置編號是結構排列的衍生值（derivation），不是穩定事實：目標文件插入、刪除、重排任一單位，後續編號全部位移。比 broken link 更難偵測——連結斷掉會報錯，編號錯位會成功解析到錯的東西（misdirected 優於 broken 的偵測難度）。引用寫入當下編號正確，失效發生在目標文件後續任何一次改版，時間差使作者無從自查。

## 解決方案

1. 引用錨點改用**語意標題**：「見『Indexing by document type』Worklog 段」而非「第三原則」——語意標題在重排後仍可 grep 定位，錯了會明顯找不到（fail loud）
2. 目標文件的結構單位標題承載核心意義（「Stage 3：核心問題」），編號僅作排序前綴；引用一律取語意半邊
3. 修復既有引用時順手驗證目標錨點真實存在（`grep "<語意錨>" <target>`）

## 預防措施

| 時機 | 動作 |
|------|------|
| 寫入跨文件引用（hook 提示文字、規則路由、ticket 模板）時 | 錨點用語意標題；寫完 grep 目標文件確認錨點存在 |
| 重排活文件結構的 commit | 全 repo 掃位置編號式引用：`rg "第[一二三四五六七八九十0-9]+(原則|點|條|章)|Stage [0-9]|步驟 [0-9]" .claude/` 比對受影響引用 |
| 例外 | 發布方凍結的編號（RFC 段號、法條、quality-baseline 規則編號這類「編號即 API、hooks 依賴的穩定錨點」）是 fact 可引用——判準為目標編號是否有凍結承諾 |

## 行為模式分析

寫引用的當下，位置編號比語意標題「更省字」且當下正確，作者無從感知未來失效；目標文件改版者看不到散落各處的引用方。失效是雙盲的——防護只能放在寫入端（語意錨點慣例）與重排端（編號引用掃描），不能依賴讀者發現。本案由 blog 回流的 `reference-by-semantic-title-not-number` 原則在檢視時對照出來，skill 原則與本 error-pattern 互為通用規範與本框架實證案例。
