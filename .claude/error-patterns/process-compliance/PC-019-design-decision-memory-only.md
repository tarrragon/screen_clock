# PC-019：通用架構決策僅記錄到 Memory 未寫入框架文件

## 錯誤模式

| 項目 | 說明 |
|------|------|
| **編號** | PC-019 |
| **類別** | process-compliance |
| **嚴重度** | 中 |
| **發現日期** | 2026-03-23 |
| **來源 Ticket** | v0.1.2 多視角審查 session |

## 症狀

PM 確認了重要的架構設計決策後，僅將決策記錄到 auto memory（`~/.claude/projects/` 下的 memory 檔案），未寫入專案的框架文件（`.claude/rules/`、分析報告等）。

用戶指出：「這是通用的框架，如果只做專案的 memory 記錄是不夠的。」

## 根因分析

1. **記錄層級混淆**：PM 將「跨 session 記憶」和「框架級架構文件」混為一談
2. **Memory 的定位誤用**：Memory 適合記錄用戶偏好、反饋等「個人化」資訊，不適合記錄所有代理人都需要遵循的架構決策
3. **缺乏判斷標準**：何時用 memory、何時寫框架文件，沒有明確的判斷流程

## 正確的記錄層級判斷

| 資訊類型 | 正確位置 | 錯誤位置 | 判斷依據 |
|---------|---------|---------|---------|
| 用戶偏好（溝通方式、效率偏好） | Memory | 框架文件 | 只影響 PM 行為 |
| 反饋修正（「不要做 X」） | Memory | 框架文件 | 只影響 PM 行為 |
| 架構設計決策 | **框架文件**（rules/、分析報告） | Memory | 影響所有代理人 |
| 技術標準決策 | **框架文件**（quality-*.md） | Memory | 影響所有代理人 |
| 版本規劃和目標 | **框架文件**（worklog、todolist） | Memory | 影響任務派發 |
| 外部資源位置 | Memory（reference 類型） | 框架文件 | 外部系統指針 |

**核心判斷準則**：

> 「這個決策是只有 PM 需要知道，還是所有代理人都需要遵循？」
> - 只有 PM → Memory
> - 所有代理人 → 框架文件

## 解決方案

1. 架構決策確認後，**必須**寫入對應的框架文件：
   - 分析報告（如附錄章節）
   - 規則文件（`.claude/rules/`）
   - 或專門的設計文件
2. Memory 可作為**輔助索引**（指向框架文件），但不是決策的唯一儲存位置
3. 確認設計決策時，PM 應主動問：「這個決策需要寫到哪個框架文件？」

## 防護措施

### 行為檢查（PM 自查）

每次記錄設計決策時，問自己：
- [ ] 這個決策是否影響代理人的行為？→ 是 → 寫框架文件
- [ ] 是否只影響 PM 與用戶的互動？→ 是 → 寫 Memory
- [ ] 框架文件已更新？→ 是 → 可額外寫 Memory 作為索引

### 建議的 Hook 改進

考慮建立 Hook：當 PM 寫入 Memory 的內容包含「決策」「架構」「設計」「標準」等關鍵字時，提醒「此決策是否應寫入框架文件？」
