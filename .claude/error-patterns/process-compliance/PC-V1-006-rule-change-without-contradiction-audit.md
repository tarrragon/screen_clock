# PC-V1-006: 規則變更未盤點既有規則矛盾即上線

**類別**: process-compliance
**嚴重度**: 中
**發現日期**: 2026-06-12
**相關 Ticket**: 1.0.0-W7-004（收斂機制設計）/ 1.0.0-W7-007（矛盾盤點修補）

---

## 症狀

新增一條規則或原則時，只驗證「新規則自身正確」，未盤點它與既有規則的指令是否衝突。兩條各自合理的規則對同一動作給出相反指示，執行者依其中一條行動即違反另一條。

**本案實例**：token 收斂後新增「自動載入預算原則」（core/ 規則應為禁令 + 路由的壓縮形態），但既有 `document-writing-style` 仍要求「所有書面文字每段論述必須含 Why/Consequence/Action 三層」、`pm-quality-baseline` 規則 7 升級路徑仍預設「新建 rules/core/*.md」——三條規則互相矛盾，且收斂時未發現。

**識別特徵**：

- 新規則的適用範圍與既有規則重疊（同一檔案層、同一動作），但兩者未互相引用或劃界
- 既有規則的「預設路徑 / 預設形態」與新規則的目標方向相反
- 規則 A 有執法機制（hook / 常駐審查 agent），規則 B 只有事後量測——衝突時 A 必勝

## 根因

規則撰寫流程的驗證焦點是「新內容自身品質」（三明示、格式、二次審查），沒有「與既有規則的一致性盤點」步驟。規則數量增長後，任何新原則都大機率與某條既有規則的適用範圍重疊；不主動 grep 盤點，矛盾只會在執行者實際踩到時暴露。

**Why 危害不對稱**：矛盾的兩條規則不是均等競爭——有執法者（hook 阻擋、常駐審查 agent）的一方會贏，只有事後量測或自律的一方會輸。若膨脹方向恰好是有執法者的一方（本案：三明示有 basil-writing-critic 審查，預算只有 file-size-guardian 事後告警），系統會被執法機制推著違反新原則，新原則形同未生效。

## 解決方案

1. 識別新規則的關鍵概念詞（本案：「三明示」「自動載入」「升級路徑」）
2. `grep -rl "<概念詞>" .claude/rules/ .claude/pm-rules/ .claude/references/` 找出適用範圍重疊的既有規則
3. 逐檔判定：一致（不動）/ 需劃界（補邊界條款，明示兩規則各自的適用範圍）/ 直接矛盾（修改既有規則的預設路徑或例外條款）
4. 矛盾修補與新規則同一批 commit，不留「新規則已上線、舊規則待改」的中間態

## 預防措施

| 時機 | 動作 |
|------|------|
| 新增 / 修改規則或原則時 | 完成自身品質檢查後，加跑一輪概念詞 grep 盤點（quality-baseline 檢查清單「引用一致性」的延伸：不只查引用路徑，也查指令方向） |
| 設計含執法機制的規則時 | 確認執法方向與上位原則一致；執法強度不對稱（hook 阻擋 vs 事後告警）時，優先確保強執法方不與弱執法方矛盾 |
| 機制類變更（預算、流程閘門）落地時 | 盤點「哪些既有規則的預設行為會餵養被防範的問題」——機制防外部輸入，規則矛盾是內生輸入 |

## 行為模式分析

矛盾不是撰寫疏忽，是規則體系成長的必然產物：每條規則寫作當下都只看見自己的場景。本案矛盾由用戶事後檢討「機制是否已入規範」才暴露，距機制上線僅數小時——靠人偶然發現不可複製，靠概念詞 grep 盤點可制度化。
