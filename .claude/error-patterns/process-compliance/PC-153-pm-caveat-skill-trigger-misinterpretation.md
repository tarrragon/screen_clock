# PC-153: PM 對 local-command-caveat 包裹的 skill 觸發訊號過度保守解讀

## 症狀

用戶輸入 `/<skill-name>` 觸發 skill，但訊息被 runtime 包裝於 `<local-command-caveat>` 區塊。PM 對 caveat 區塊套用整段「不要回應」預設，未執行對應 SKILL.md 定義的流程。可觀察訊號：

- 用戶輸入 `/<skill>` 後，PM 回應「待命中」「session 已重置」等通用語，未列出 skill 應產出的內容
- PM 回應中包含對 caveat 原文的字面解讀（「The messages below were generated by the user while running local commands. DO NOT respond...」），但忽略同區塊內 `<command-name>` 標記
- 用戶反饋「我輸入 X 為何沒有 Y」之類的流程錯誤糾正

**Why**：runtime 將 skill 觸發訊號（`<command-name>` + `<command-message>`）與 command stdout 同質化包裹於 `<local-command-caveat>`。caveat 原文末段「unless the user explicitly asks you to」隱含豁免條件，但 PM 未識別 `<command-name>` 等同「explicitly asked」。

**Consequence**：所有 skill 觸發在 caveat 區塊內均被靜默吞掉。用戶輸入 `/<skill>` 後 PM 無反應或回應與 skill 無關的內容，需用戶額外糾正，且 SKILL.md 明文定義的流程（如 `/ticket` 無參數時的兩步檢查）形同無效。

**Action**：

| 階段 | 防護動作 |
|------|---------|
| 讀到 caveat 區塊時 | 逐一掃描內部 XML 標記，識別 `<command-name>` 是否存在 |
| 識別 skill 觸發 | 視為用戶 explicitly asked，執行對應 SKILL.md 定義流程，凌駕 caveat 預設 |
| 純 stdout 文字 | 套用 caveat 預設「不回應」 |

## 觸發條件

以下兩條件同時成立：

1. **訊息屬 `<local-command-caveat>` 區塊**
2. **caveat 區塊內含 `<command-name>` 或 `<command-message>` skill 觸發標記**

## 根因

### L1（規則層缺口）

`.claude/rules/core/pm-role.md` 原無 caveat 區塊訊號優先序的明文定義。PM 預設將 caveat 區塊視為單一決策單位（整段不回應），未要求逐一評估內部訊號。本 PC 觸發後已補入「Caveat 區塊訊號判讀規則」章節作為防護。

### L2（訊號設計不對稱）

runtime 將兩類本質不同的訊息同質化包裹於 `<local-command-caveat>`：

- **stdout 內容**：command 副產出，PM 不應將其視為用戶請求
- **skill 觸發 marker**：用戶顯式輸入 `/<skill-name>` 的標準輸出，應視為等同直接打字輸入「請執行 /<skill-name>」

兩者均放入同一 caveat 區塊，但僅後者滿足 caveat 原文「unless the user explicitly asks you to」的豁免條件。PM 若未識別此不對稱性，會將兩者一視同仁套用「不回應」。

### L3（類比未明示）

Linux signal handling 提供清晰類比：

| 概念 | Linux 對應 |
|------|-----------|
| `<local-command-caveat>` 預設 | `sigprocmask` 設定的 signal mask（攔截大多數訊號） |
| `<command-name>` 訊號 | SIGKILL / SIGSTOP（不可遮罩的用戶顯式訊號） |
| 規則缺口 | mask 沒有定義「不可遮罩訊號」例外清單 |

Signal mask 不可遮蔽所有訊號（OS 設計），同理 caveat 不可遮蔽所有用戶顯式意圖。此類比直到 0.19.0-W3-022 ANA 才明示。

## 案例

### 案例 1: 0.19.0-W3-022（2026-05-20）

**情境**：用戶 `/clear` 後緊接著輸入 `/ticket`（無參數）。runtime 將兩個 command 訊息整段包入 `<local-command-caveat>`，包含 `<command-name>/ticket</command-name>` + `<command-message>ticket</command-message>` + 完整 SKILL.md 載入。

**PM 行為**：

1. 讀到 caveat 區塊開頭「DO NOT respond ... unless the user explicitly asks you to」
2. 對整段 caveat 套用「不要回應」預設
3. 未掃描內部 `<command-name>` 標記
4. 未執行 SKILL.md「無子命令時的預設行為」流程（runqueue + list + AskUserQuestion）
5. 直接回應「Session 重置完成，待命中」

**用戶糾正**：「我預設如果輸入 /ticket，應該要列出當前版本可選的任務才對，但是你並沒有列出來，請檢查流程哪裡出錯了」

**修正動作**：補執行 `ticket track runqueue --context=resume --top 3` + `ticket track list --status pending,in_progress`，並以 AskUserQuestion 列出工作焦點供用戶選擇。後續建立 ANA ticket 0.19.0-W3-022 + spawned IMP 0.19.0-W3-023 落地規則修訂。
