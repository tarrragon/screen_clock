# PC-008: Stub Ticket 認領後未更新驗收條件，導致 complete 失敗

## 基本資訊

- **Pattern ID**: PC-008
- **分類**: 流程合規
- **來源版本**: v0.1.1
- **發現日期**: 2026-03-07
- **風險等級**: 中

## 問題描述

### 症狀

執行 `ticket track complete <id>` 時，CLI 回傳：

```
[Error] 0.1.1-W3-002 有未完成的驗收條件

   未完成項:
   [ ] 任務實作完成
   [ ] 相關測試通過
   [ ] 無程式碼品質警告
```

實際任務是更新 markdown 文件，「相關測試通過」和「無程式碼品質警告」不適用，但這些條件未被勾選也無法跳過。

### 根本原因 (5 Why 分析)

1. Why 1: `complete` 失敗，因為驗收條件未勾選
2. Why 2: 驗收條件是通用模板（`任務實作完成`、`相關測試通過`、`無程式碼品質警告`），不適用於 DOC 類型任務
3. Why 3: Ticket 建立時 5W1H 全部「待定義」（stub 狀態），驗收條件也使用預設值
4. Why 4: 認領 Ticket 時（Checkpoint R），沒有立即驗證並更新驗收條件
5. Why 5（根本原因）：**認領 Stub Ticket 後，未先更新 5W1H 和驗收條件即開始執行**

### 受影響模式

```
Stub Ticket 認領（claim）
        ↓
直接開始執行（沒有更新驗收條件）
        ↓
complete 失敗（通用條件未勾選）
        ↓
緊急手動 Edit Ticket frontmatter（繞過標準流程）
```

## 解決方案

**認領 Stub Ticket 後的必要步驟（Checkpoint R 標準化）**：

1. 閱讀 Ticket 的 5W1H，確認是否有「待定義」欄位
2. 有「待定義」→ 立即更新 5W1H 和驗收條件，確保條件與實際任務類型匹配
3. DOC 類型任務應使用對應驗收條件：
   ```
   - '[x] 文件內容正確反映最新設計/流程'
   - '[x] 無不符合規範的格式問題（如 emoji、簡體詞彙等）'
   ```
4. 更新完後再開始執行

## 預防措施

### 認領 Stub Ticket 時的檢查清單

- [ ] 確認 5W1H 是否有「待定義」欄位，有的話立即填寫
- [ ] 確認驗收條件是否適用於實際任務類型
- [ ] DOC 類型任務：移除「相關測試通過」、「無程式碼品質警告」等不適用條件
- [ ] 確認 `creation_accepted` 狀態（stub Ticket 可能未通過建立後審核）

### 任務類型對應驗收條件模板

| 任務類型 | 建議驗收條件 |
|---------|------------|
| DOC（文件更新） | 內容正確性、格式規範符合、相關文件已同步 |
| IMP（程式實作） | 測試通過、無程式碼品質警告、功能符合規格 |
| ADJ（調整配置） | 配置正確、相關驗證通過、無副作用 |

### 注意：Stub Ticket 的特徵識別

| 特徵 | 識別方式 |
|------|---------|
| 5W1H 含「待定義」 | `ticket track query <id>` 輸出有「待定義」 |
| `creation_accepted: false` | 未通過建立後品質審核 |
| 驗收條件為通用模板 | 「任務實作完成」、「相關測試通過」、「無程式碼品質警告」 |

Stub Ticket 認領前，應先完成設計（補充 5W1H），再執行建立後審核（acceptance-auditor + system-analyst）。
