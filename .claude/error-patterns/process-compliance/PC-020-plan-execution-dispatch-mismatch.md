# PC-020: 計畫-執行派發不一致（敘述與實際 agent 數量不符）

---

## 分類資訊

| 項目 | 值 |
|------|------|
| 編號 | PC-020 |
| 類別 | process-compliance |
| 來源版本 | v0.1.2 |
| 發現日期 | 2026-03-23 |
| 風險等級 | 中 |

### 症狀

1. PM 文字輸出寫「標準三人組」（code-reviewer + code-explorer + linux）
2. 實際 Agent tool 呼叫只派發了 1 個 code-reviewer + 1 個 cinnamon（另一個 Ticket 的 4b）
3. 用戶發現敘述與執行不符後指出，PM 才補派剩餘 2 個 agent

### 根本原因（3 層分析）

| 層級 | 問題 | 說明 |
|------|------|------|
| L1 行為層 | 計畫-執行落差 | 文字描述（思考結果）和工具呼叫（實際行動）各自獨立，無交叉驗證 |
| L2 流程層 | 無派發驗證機制 | 系統有 `git diff --stat` 驗證程式碼變更，但無對應的「派發數量驗證」機制 |
| L3 設計層 | 多 Ticket 並行時認知干擾 | 同時管理 某 Ticket（4a 三人組）+ 某 Ticket（4b 單一 cinnamon）時，注意力分散在「跨 Ticket 並行」，忽略「單一 Ticket 內的並行需求」 |

### 觸發條件

- 多個 Ticket 同時處於不同 Phase，需要不同數量的 agent
- PM 在同一回應中派發不同 Ticket 的不同 Phase

---

## 解決方案

### 正確做法

派發「三人組」時，在同一回應中確認 3 個 Agent tool 呼叫：

```
文字輸出：「派發三人組」
工具呼叫：Agent(code-reviewer) + Agent(code-explorer) + Agent(linux)
自檢：計數確認 3 個 Agent 呼叫
```

### 預防措施

1. **PM 自檢**：派發後在文字輸出中列出實際派發的 agent 名稱和數量，與計畫比對
2. **Hook 偵測**（長期）：偵測文字中「三人組」「多視角」關鍵字，計算同一回應的 Agent 呼叫數，若 < 3 則 WARNING
3. **規則補充**：在 parallel-dispatch.md「多視角審查固定三人組」新增「派發後計數驗證」步驟

### 錯誤做法

| 錯誤 | 說明 |
|------|------|
| 文字寫「三人組」但只派 1 個 | 計畫-執行不一致 |
| 多 Ticket 並行時每個 Ticket 只派 1 個 | 被「一個 Ticket 一個 agent」心智模型影響 |
| 不自檢直接繼續 | 錯誤在用戶發現前不可見 |

| 項目 | 影響 |

| 間接影響 | Phase 4b 重構方向可能偏頗（只有 Bug 視角，缺少架構和簡潔視角） |
| 風險等級 | 中（補派後可修正，但延遲了審查時間） |
