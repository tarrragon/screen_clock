# PC-084: 日文漢字清單誤列繁日共用字 false positive

---

## 分類資訊

| 項目 | 值 |
|------|------|
| 編號 | PC-084 |
| 類別 | process-compliance |
| 風險等級 | 低（偵測錯誤不是安全漏洞，但會打斷正常工作流） |
| 首發時間 | 2026-04-17（W12-002.5 ANA 首版 JAPANESE_ONLY 清單誤列「鑑」） |

## 症狀

設計「日文新字體專屬字」清單時，誤將**繁體中文也使用的漢字**列入，導致：

- 正常的繁體中文文本被掃描工具回報為「日文漢字污染」
- false positive 聚集於以繁體書寫的 error-pattern / rules / methodology 文件
- 若清單用於 Hook 攔截，會打斷正常流程（類比 PC-074）

---

## 根本原因

### 已驗證事實

1. **首次誤判字元**：「鑑」(U+9451) — 繁體中文常用字（「鑑別」、「借鑑」、「鑑於」），同時是日文新字體字形（日文也寫「鑑」，舊字體「鑒」U+9452）
2. **清單設計疏漏**：JAPANESE_ONLY 候選清單直接從「常見日文新字體字」堆砌而成，未逐字驗證「繁體中文是否也用此字元」
3. **W12-002.5 重現次數**：在 3 個獨立繁體文件共 11 處被誤判
   - PC-050-premature-agent-completion-judgment.md：4 處（「鑑別力」）
   - PC-070-pm-hook-signal-agent-failure-inference.md：3 處（「鑑別力」）
   - references/pm-agent-observability.md：4 處

### 真根因

1. **字元集重疊盲區**：
   - 日文新字體漢字大致分三類：
     - (a) 與繁中**共用**字形（如「鑑」「体」「誉」— 但「誉」需驗證）
     - (b) 與繁中**不同**但與簡中共用（如「独」「実」「関」，但這些已在 SIMPLIFIED_CHARS）
     - (c) 日文**專屬**（繁簡皆不用，如「読」「訳」「駅」「両」）
   - 清單應只收類別 (c)，誤將 (a) 納入就會產生 false positive

2. **視覺相似誤判**：
   - 設計者看到「鑑」字形簡潔（相對於舊字體「鑒」），直覺認為是日文簡化結果
   - 實際上「鑑」從唐宋即為繁中正統字形，並非日文創造

3. **清單擴充無驗證流程**：
   - 類比 PC-074 教訓：**常見簡體詞拆字**不可靠（繁簡共用字會混入）
   - **常見日文新字體堆砌**同樣不可靠（繁日共用字會混入）

## 常見陷阱模式

| 陷阱表述 | 為何仍構成誤判 |
|---------|--------------|
| 「日文漢字看起來比繁中簡潔就是新字體專屬」 | 視覺判斷不可靠，許多繁中字本身就簡潔（如「鑑」） |
| 「把新字體對照表的日文欄列入清單就好」 | 對照表日文欄可能與繁中相同（對照後無變化），不構成日文專屬證據 |
| 「日文新字體不會和繁體重疊」 | 錯誤預設，日本「正字法」保留大量漢字原形，與繁中共用字頗多 |

| 層級 | 措施 | 狀態 |

### 首發案例學習要點

- **鑑 U+9451** 是**明確繁日共用**（繁中正統常用字），列入清單必產生 false positive
- **体/誉/豊/拝**是繁中**異體字**。雖繁中主流用正統字，但異體字仍屬繁中合法字形；保守做法「**不列入 JAPANESE_ONLY**」避免打斷異體字使用者
- **日文專屬字**（読/訳/駅/両/発/図/広/実/気/楽/観/検/権/売/関/鉄/転）繁中完全不用，可安全列入

> **注意**：本表為 W12-002.5 首批案例；新發現的繁日共用字或異體字，補充至本表並標註首發來源。

## 建議的清單擴充流程

1. **收集候選字**：從真實污染案例（如 AUQ payload、測試 docstring）蒐集可疑字元
2. **查繁中字典**：對每個字元查：
   - 教育部《異體字字典》（https://dict.variants.moe.edu.tw/）
   - 《漢典》（https://www.zdic.net/）
3. **三類判定**：
   - **繁中不收此字形** → 日文專屬，可加入 JAPANESE_ONLY
   - **繁中收此字形且常用** → 繁日共用，**禁止**加入，補本 PC 清單
   - **繁中收為異體字（少用）** → 保守處理，不加入 JAPANESE_ONLY（避免打斷異體字使用者）
4. **繁中回歸測試**：
   ```python
   # 擴充 JAPANESE_ONLY 後必跑
   trad_chinese_cases = [
       "鑑別能力不足",       # 含 U+9451
       "身体力行",           # 含 U+4F53（若保守列入須驗證）
       "豊饒的田園",         # 含 U+8C4A（若列入須驗證）
   ]
   for text in trad_chinese_cases:
       hits = scan_japanese(text)
       assert hits == [], f"False positive: {text} hits={hits}"
   ```

## 教訓

1. **「共用字」是 CJK 清單設計的通病**：繁簡共用（PC-074）與繁日共用（PC-084）是同一模式的不同維度，所有 CJK 字元集清單都應通過類似驗證
2. **視覺簡潔度不等於簡化字身份**：字形簡潔可能是因為字本身就簡潔，不是誰簡化了誰
3. **擴充清單摩擦力優於下游 false positive**：查字典增加 30 秒摩擦，換取 Hook 啟用後不打斷流程（象限 C，同 PC-074）
4. **清單設計失誤會延遲偵測**：W12-002.5 若直接採用我的首版清單做成 Hook，會在 3 個繁體文件上觸發假警報

---

## 象限歸類

本模式的防護屬 **摩擦力管理 C 象限（增加摩擦）**：擴充清單前多一步查字典驗證增加開發摩擦，換取下游不發生 false positive 打斷。代價（查字典）遠低於收益（避免誤擋正常繁體文本）。與 PC-074 同象限同理由。
