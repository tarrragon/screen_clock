# PC-042: 規則文件過長導致代理人回合耗盡

## 錯誤症狀

分析型代理人（saffron-system-analyst）被派發去分析 PM 規則的缺口，但讀完 2-3 個規則文件後回合就耗盡，無法寫入分析結論。

典型數據：
- decision-tree.md: 548 行
- ticket-lifecycle.md: 164 行
- 2 個 error-pattern 文件: ~100 行
- 4 次 Read = 回合耗盡，0 次 Write

## 根因分析

> **核心洞察**：回合耗盡 = 認知負擔過載的具體訊號。問題不在檔案數量多寡，而在單一檔案體量。20 次 tool call 正常使用下不會不夠——是因為單一檔案過大，Read 成本過高。這是認知負擔理論的具體實踐場景。

**直接原因**：單一檔案體量過大（decision-tree.md 548 行），導致代理人 Read 成本過高，tool call 預算不足以完成分析和寫入。

**深層原因（DDD 視角）**：
1. **domain 邊界違規**：decision-tree.md 承載 6 個 domain（任務路由、TDD 管理、事件回應、完成判斷、完成後路由、代理人管理）——程式碼中不允許一個類別承載 6 個職責，文件也不應該
2. 文件設計時以「PM 自動載入」為前提，但代理人按需讀取，每次都要付出 Read 成本
3. 缺乏「文件體量 + domain 邊界」的品質基線

**行數超標是症狀，domain 混合是病因**。拆分依據不是行數，而是 domain 邊界。行數是量化輔助指標。

## 影響

| 影響 | 說明 |
|------|------|
| 分析任務反覆失敗 | 本 session 中 3 次派發分析代理人，全部在 Read 階段耗盡 |
| PM 被迫親自分析 | 違反「主管不解決問題」原則，但代理人做不到 |
| 規則改善速度降低 | 無法派發代理人分析規則缺口，只能靠 PM 手動 |

## 正確做法

### 規則文件體量評估

| 指標 | 理想值 | 警戒值 | 當前值 |
|------|-------|--------|--------|
| decision-tree.md | < 200 行 | 300 行 | 548 行 |
| 單一 pm-rules 文件 | < 150 行 | 200 行 | 需逐一評估 |
| 代理人需讀取的文件總行數 | < 500 行 | 800 行 | 需逐任務評估 |

### 拆分策略

decision-tree.md 可按「層」拆分：
- decision-tree-core.md: 第一層到第五層（核心決策路徑）
- decision-tree-tdd.md: TDD 相關路徑
- decision-tree-completion.md: 第七層（ANA 檢查 + 完成流程）
- decision-tree-routing.md: 第八層（Checkpoint + 收尾路由）

代理人只需讀取與任務相關的拆分文件，而非整個 decision-tree。

## 預防措施

| 措施 | 類型 | 說明 |
|------|------|------|
| 規則文件體量基線 | 品質規則 | 新增「單一規則文件不超過 200 行」基線 |
| decision-tree 拆分 | 重構 | 按層拆分為 4 個文件 |
| 代理人派發前預估 | 流程 | Read 次數 > 3 時考慮預先提取到 Ticket |

## 發現來源

- 場景：saffron 被派發分析 decision-tree 規則缺口，4 次 Read 後回合耗盡。本 session 3 次分析型派發均失敗在同一模式。
