# PC-020: 修復方向錯誤 — 在消費端補救而非生產端防護

## 錯誤摘要

發現問題後，PM 將修復措施放在「消費端」（resume 時檢查 5W1H 完整性），而非「生產端」（ticket create 時強制 why 必填）。這是治標不治本的修復方向。

## 症狀

- 修復方案聽起來合理（「resume 時加檢查」）但未解決根因
- 問題仍會在每次 resume 時重現（因為建立端沒改）
- 用戶需要介入修正方向

## 根因分析

1. **就近修復偏見**：PM 傾向在「發現問題的地方」修復，而非追溯到「問題產生的地方」
2. **消費端檢查是加法思維**：加一個檢查步驟容易，但忽略���為什麼需要這個檢查
3. **未問「為什麼」**：直接跳到「怎麼做」，沒有先問「為什麼 5W1H 是空的」→ 因為建立時沒要求填

## 觸發條件

- 發現某個流程步驟的輸入資料不完整
- PM 直覺反應是「在這個步驟加驗證」而非「在上游確保資料完整」

## 防護措施

**修復前必問三個問題**：

| # | 問題 | 目的 |
|---|------|------|
| 1 | 這個資料是在哪裡產生的？ | 找到生產端 |
| 2 | 生產端為什麼允許不完整的資料？ | 找到根因 |
| 3 | 我的修復是在生產端還是消費端？ | 確認方向正確 |

**原則**：修復優先在生產端（防止問題產生），而非消費端（問題產生後補救）。

## 類比

| 場景 | 消費端修復（治標） | 生產端修復（治本） |
|------|------------------|------------------|
| 5W1H 不完整 | resume 時加檢查 | create 時強制必填 |
| 資料格式錯誤 | 讀取時加容錯 | 寫入時加驗證 |
| API 回傳 null | 呼叫端加 null check | API 端確保不回傳 null |
