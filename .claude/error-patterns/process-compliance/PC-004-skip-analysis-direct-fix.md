# PC-004: 跳過分析審核直接派發修復導致迴歸

---

## 分類資訊

| 項目 | 值 |
|------|------|
| 編號 | PC-004 |
| 類別 | process-compliance |
| 嚴重度 | 高 |
| 發現版本 | 0.31.1 |
| 發現日期 | 2026-03-05 |

### 症狀

1. UC-02 Export 模組有 62 個編譯 error（跨 i18n、Repository、UseCase、Provider 四個層級）
2. PM 直接派發 parsley 修復全部 62 個 error，未經分析分類和方案審核
3. 修復完成後，新增 25 個測試失敗（原 261 → 修改後 286）
4. 修復後直接跑全量測試（3+ 分鐘），浪費時間且無法快速定位問題

### 根因

**行為模式**：PM 將「62 個編譯 error」視為單一修復任務，跳過 incident-response 流程的分析和審核階段。

**根本原因**：

1. **跳過分析**：62 個 error 跨越 4 個架構層（i18n、Repository、UseCase、Provider），認知負擔遠超 10，但未拆分為獨立子任務
2. **跳過審核**：修復方案未經 parallel-evaluation 審查，代理人在不了解全貌的情況下修改了 13 個檔案
3. **驗證順序錯誤**：修復後直接跑全量測試而非先用單元測試快速驗證，浪費 3+ 分鐘才發現問題

**觸發條件**：
- 編譯 error 看起來是「技術問題」而非「設計問題」，容易讓 PM 降低警覺
- 多個 error 同時出現時，PM 傾向「一次解決」而非「分析再解決」

### 影響範圍

| 影響 | 說明 |
|------|------|
| 新增測試失敗 | 25 個（需回滾修復） |
| 時間浪費 | 代理人執行 ~10 分鐘 + 全量測試 3+ 分鐘 |
| 回滾成本 | 需選擇性回滾 001.6.1 變更，保留 001.6.2 |

### 正確做法

```
62 個編譯 error
    |
    v
[階段 1] 派發 incident-responder 分析
    → 按架構層分類：i18n (36)、Repository (6)、Provider (3)、UseCase (8)、其他 (9)
    → 識別依賴關係：Repository 方法缺失 → ViewModel 無法編譯 → 頁面無法編譯
    → 產出分析報告
    |
    v
[階段 2] 設計方案 + parallel-evaluation 審查
    → 方案 1：在 BookRepository 新增方法（影響範圍大）
    → 方案 2：在 ViewModel 中改用現有方法計算（影響範圍小）
    → 審查：哪個方案不會破壞現有測試？
    |
    v
[階段 3] 按依賴順序逐批修復
    → 先修底層（Repository/UseCase）
    → 再修中層（Provider）
    → 最後修上層（i18n、頁面）
    → 每批修復後用單元測試驗證
```

### 教訓

> **編譯 error 也是「錯誤」**，必須走 incident-response 流程。
> 數量多（>5 個）且跨架構層的 error，認知負擔必定超過閾值，
> 絕對不能合併為單一修復任務直接派發。

> **測試驗證必須從底層開始**。
> 全量測試是最後手段，不是第一選擇。
> 先用單元測試（秒級）快速驗證，比等 3 分鐘全量測試高效得多。
