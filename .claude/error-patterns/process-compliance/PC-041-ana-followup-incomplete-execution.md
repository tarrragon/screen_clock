# PC-041: 分析 Ticket 結論的落地執行不完整

## 錯誤症狀

分析型 Ticket（ANA/DOC）完成後，其結論中的修改項目在落地執行時被部分遺漏。PM 只執行了部分項目就將執行 Ticket 標記為完成。

典型場景：
- 分析結論列出 4 項修改，PM 只建立/執行了 3 項
- 背景代理人的延遲分析補充了額外項目，PM 未納入執行清單
- 執行 Ticket 的驗收條件不夠精確，允許部分完成就通過

## 根因分析

**直接原因**：PM 在 某 Ticket 分析後手動建立某 Ticket 執行 Ticket，但只根據自己的分析寫了 3 項修改。saffron 代理人的分析（延遲完成）補充了第 4 項（phase3b-dispatch-guide.md L27），但 某 Ticket 的驗收條件中沒有包含這一項。

**深層原因**：
1. 分析 Ticket 和執行 Ticket 之間缺乏「結論完整性交叉驗證」——PM 依記憶建立執行 Ticket，而非逐項對照分析結論
2. 背景代理人的分析結果與 PM 的手動分析平行進行，沒有合併機制
3. 執行 Ticket 的驗收條件應直接從分析 Ticket 的 Solution 逐項轉換，而非重新撰寫

## 影響

| 影響 | 說明 |
|------|------|
| 修改不完整 | 規則文件部分修正，遺留的不一致可能在未來引起同一個問題 |
| 分析成本浪費 | 代理人花回合分析的結論未被完整採用 |
| 驗收失效 | 執行 Ticket 通過驗收但實際未完成所有修改 |

## 正確做法

### 分析→執行的轉換流程

1. 分析 Ticket 完成後，**逐項列出** Solution 中的所有修改項目
2. 如有背景代理人的分析，**等待或合併**其結果再建立執行 Ticket
3. 執行 Ticket 的驗收條件 = 分析結論的修改項目 **1:1 對應**
4. 執行完成前，**對照原始分析結論**逐項勾選，而非僅依驗收條件

### 檢查清單

- [ ] 分析 Ticket 的 Solution 有幾項修改？
- [ ] 背景代理人是否有額外發現？（等待通知或查閱 output）
- [ ] 執行 Ticket 的驗收條件是否 1:1 覆蓋所有修改項？
- [ ] 完成前是否逐項對照原始分析結論？

## 預防措施

| 措施 | 類型 | 說明 |
|------|------|------|
| 分析→執行轉換規範 | 規則更新 | 在 ticket-lifecycle 或 decision-tree 中新增轉換檢查步驟 |
| 背景代理人結論合併 | 流程改善 | 等待背景代理人完成後再建立執行 Ticket，或建立後補充 |

## 發現來源

- 場景：PM 手動建立某 Ticket 時只含 3 項修改，saffron 延遲通知補充第 4 項（phase3b-dispatch-guide.md L27），幾乎被跳過
