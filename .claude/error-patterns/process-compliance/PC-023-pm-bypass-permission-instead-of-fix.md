# PC-023: PM 因代理人權限阻擋直接實作而非排除障礙

## 錯誤資訊

| 項目 | 內容 |
|------|------|
| **編號** | PC-023 |
| **分類** | process-compliance |
| **嚴重度** | 中 |
| **發現日期** | 2026-03-24 |
| **相關模式** | PC-022（subagent 權限不足） |

## 症狀

thyme-python-developer 被 Hook 權限限制阻擋編輯 `.claude/skills/ticket/` 檔案。PM 選擇直接實作程式碼，而非排除權限問題後重新派發代理人。

## 根因分析

1. **直接原因**：PM 遇到代理人權限阻擋時，選擇最快路徑（自己做）而非正確路徑（排除障礙後重新派發）
2. **行為模式**：「效率優先」心態導致 PM 跳過職責邊界。雖然 `.claude/skills/*` 在 PM 允許編輯範圍內，但實作程式碼（含邏輯修改和測試撰寫）仍應由開發代理人執行
3. **與 PC-022 的區別**：PC-022 描述權限問題本身；PC-023 描述 PM 遇到權限問題後的錯誤應對行為

## 影響範圍

- PM 承擔了開發代理人的職責，違反 skip-gate 精神
- 缺少代理人的獨立視角和品質檢查
- 設定了「遇阻就自己做」的不良先例

## 解決方案

### 正確流程（遇到代理人權限阻擋時）

```
代理人回報權限阻擋
    |
    v
[Step 1] 診斷原因（Hook 白名單？用戶拒絕？路徑限制？）
    |
    v
[Step 2] 排除障礙
    ├─ Hook 白名單問題 → 更新白名單
    ├─ 用戶拒絕 → 提示用戶授權或調整 permission mode
    └─ 路徑限制 → 確認是否需要擴展代理人權限
    |
    v
[Step 3] 重新派發代理人
    |
    v
[僅在無法排除時] PM 直接執行，且必須記錄原因
```

## 預防措施

1. **派發前檢查**：依 parallel-dispatch.md「派發前路徑權限確認」章節執行
2. **遇阻先排除**：PM 遇到代理人權限問題時，第一反應是「如何讓代理人能做」而非「我自己做」
3. **記錄原因**：如果最終 PM 必須直接執行，在 Ticket Execution Log 記錄為何無法排除

## 檢查清單

- [ ] 代理人回報權限問題時，是否先嘗試排除？
- [ ] 是否確認了權限問題的具體原因（Hook/用戶/路徑）？
- [ ] PM 直接實作是否有記錄原因？
