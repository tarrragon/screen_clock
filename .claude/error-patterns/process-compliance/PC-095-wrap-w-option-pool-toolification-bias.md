# PC-095: WRAP W 階段選項池不完備（Claude 預設「新增工具」結構性偏見）

---

| 項目 | 值 |
|------|------|
| 編號 | PC-095 |
| 類別 | process-compliance |
| 風險等級 | 中（影響 ANA 方向正確性，可能導致投入錯誤類型的 IMP） |
| 首發時間 | 2026-04-18（W10-017 parallel-evaluation Plan 視角揭露） |
| 姊妹模式 | PC-080（WRAP A 階段框架檢查未做）、PC-066（多視角審核發現元架構過度設計） |

## 症狀

執行 WRAP 決策框架時，W 階段（Widen Options）列出多個候選方案，但**所有方案性質皆為「新增工具」**（新增 CLI、新增 Hook、新增 Skill）。結構性排除下列選項類型：

1. 改造既有機制（既有 Hook / Skill / CLI 新增能力）
2. 零工具路線（純文件重構 / 純清單）
3. 載入機制層面改造（SessionStart / Skill Preload / Memory）

導致：

- 方案比較表面多元，實際全指向「新增」而非「改造」
- 機會成本未納入「不做新工具」的選項
- 必須經 parallel-evaluation 的 Plan 視角或用戶即時 reframe 才能揭露

## 實際案例

### 案例 1（W10-017，2026-04-18）

**任務**：分析 PM 常規流程自動化（減少 handoff→/clear 流程的 token 與 tool call 耗損）

**錯誤的 WRAP W 階段**（5 方案實際同一框架）：

| 方案 | 性質 |
|------|------|
| A 全 CLI 化 | 新增 4 個 ticket CLI 子命令 |
| B Hook 攔截 | 新增 3 個 PreToolUse Hook |
| C SKILL Wizard | 新增 1 個 Skill |
| D CLI + Hook 混合 | 新增 CLI + Hook |
| E snapshot 增強 | 改造既有 CLI（部分偏離） |

**共同框架**：「新增工具降低 PM 負擔」

**parallel-evaluation Plan 視角 reframe**：

> W 階段全部假設「新增工具」。整份分析預設「pm-rules 不動 + 新增 CLI/Hook/Skill」，排除「改造既有載入機制」與「零新工具」路線。

**補上的 4 個選項**（揭露後才被看見）：

| 方案 | 性質 | 原本為何被排除 |
|------|------|-------------|
| F SessionStart Hook 自動注入 | 改造既有 Hook 機制 | Claude 視 SessionStart 為「啟動提示」不是「資源注入手段」 |
| G Skill Preload 精簡摘要 | 改造既有 Skill 載入 | Claude 視 Skill 為「觸發式呼叫」不是「預載入 context」 |
| H 純文件重構（46→5 檔） | **零新工具** | Claude 視「改既有文件」比「新工具」更不確定，避開 |
| I Memory 層自動狀態追蹤 | 改造既有 memory 系統 | Claude 視 memory 為「學習型持久化」不是「狀態追蹤」 |

### 案例 2（W10-017.9，2026-04-19，本 Pattern 發現於同一主題的子任務）

**任務**：4 週後基於觀測數據評估 F/G/H/I 是否落地（PM 前台純理論預判）

**發現**：即使 W10-017 已將「F/G/H/I」補上為 9 方案，實際評估後發現 4 選項**強相互關聯**，真實獨立選項只有 1 個（F + H.quickref 整合）：

- G 退化為 F 或等於既有 Skill → G 不獨立
- I 退化為既有 state file → I 不獨立（被 017.8 metrics log 吸收）
- H 原案 46→5 檔不可行（違反 DDD），變種 quickref.md 實為 F 的 payload

**再次揭露的偏見**：Claude 在 WRAP W 階段傾向「列出平行 N 方案」，但**未檢查方案間的強關聯與降級可能**，導致選項池名義完備但實質收斂。本案 4 方案實質只有 1 個獨立路徑，屬「假多樣性」。

## 根本原因

### 真根因

1. **訓練資料偏向「新工具教學」**：Claude 訓練資料中「如何建立 X」的教學遠多於「如何改造 Y」的案例；WRAP W 階段生成候選時偏好熟悉的輸出形式
2. **工具類任務是 Claude 高置信度輸出**：新增 CLI / Hook / Skill 有明確 deliverable 與成功判準；「改既有文件」成果模糊、風險高，Claude 自動避開
3. **方案性質單維檢查缺失**：W 階段只檢查「方案數量 ≥ 3」，未檢查「方案性質是否涵蓋 新增 / 改造 / 零工具」三類
4. **假多樣性（Case 2 新發現）**：即使補上 F/G/H/I 方案數量到 9，若不檢查方案間強關聯，仍可能實質退化為 1 選項

### 為什麼容易發生

- 「新增」比「改造」容易描述（有明確 deliverable 清單）
- Claude 對「改造既有機制」的風險預估偏高（現存機制複雜度、相容性疑慮）
- WRAP W 階段檢查標準只看「列出 N 個」，不看「涵蓋哪些類型」
- 問題描述若用「如何降低 X」框架，預設答案就是「做點什麼」而非「少做什麼」

## 常見陷阱模式

| 陷阱表述 | 為何仍構成選項池不完備 |
|---------|------------------|
| 「我的方案 A-E 是多元的（CLI/Hook/Skill/混合/增強）」 | 5 個方案全是「新增工具」，只是工具類型不同 |
| 「WRAP W 階段已列 5 方案超過建議值」 | 數量達標不等於性質涵蓋完整 |
| 「改造既有機制風險太高所以排除」 | 排除理由應顯式分析，而非結構性略過 |
| 「零工具路線不適用本案」 | 「不適用」結論應在比較後得出，非 W 階段直接略過 |
| 「F/G/H/I 4 方案都已納入」（Case 2） | 即使列出 4 方案，若強相互關聯（G≡F、I≡既有、H 變種併入 F）則實質仍是 1 方案 |

## 防護措施

| 層級 | 措施 | 狀態 |
|------|------|------|
| 流程 | WRAP W 階段**必須**包含三類方案各至少 1 個：新增工具 / 改造既有 / 零工具或文件 | 行為準則（本 PC 後立） |
| 流程 | W 階段方案清單後強制自問：「若本 Ticket 禁止新增任何 CLI/Hook/Skill，還能怎麼做？」 | 行為準則 |
| 流程 | W 階段方案 ≥ 4 個時，必做**關聯性檢查**：每兩方案兩兩問「是否可退化為同一方案或既有機制？」| 行為準則 |
| 工具 | wrap-decision SKILL 補強 W 階段進入條件：要求顯式列出「改造既有」與「零工具」兩類方案 | 待實施（017.10 Phase 2） |
| Memory | 記錄本失誤作為跨 session 提醒 | 已實施（配對本檔） |

## 檢查清單（WRAP W 階段進入 A 階段前）

- [ ] 候選方案是否涵蓋三類：**新增工具** / **改造既有** / **零工具或純文件**？
- [ ] 若禁止「新增 CLI/Hook/Skill」，還能提出哪些方案？列出至少 1 個
- [ ] 任意兩個方案是否可退化為同一方案或既有機制？若是，實質選項數量要重算
- [ ] 「改造既有」方案的成本估算是否因「不熟悉」而虛高？
- [ ] 「零工具」方案是否因「成果模糊」而被心理排除？若是，需重新估算

## 教訓

1. **WRAP W 階段不只是數量規則**：5 個方案全同類型不如 3 個方案涵蓋新增/改造/零工具三類
2. **訓練資料偏誤需主動對抗**：Claude 自然偏向熟悉的輸出形式，必須結構性強制列出反向選項
3. **parallel-evaluation Plan 視角具獨立價值**：本案兩次均由 Plan 視角揭露（或推估），非 PM 自我檢查發現
4. **假多樣性警示**：方案數量達標不等於方案獨立，需做兩兩關聯性檢查（Case 2 新發現）
5. **「改造」比「新增」的心理成本被高估**：Claude 對既有機制的相容性疑慮導致過度保守，實際成本往往低於預估（Case 1 F hook = 第 17 個 SessionStart hook）
