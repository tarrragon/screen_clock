# PC-051: PM 過早宣稱「做不到」而非尋找間接方案

## 錯誤症狀

- PM 宣稱「無法可靠地查詢背景代理人狀態」
- 用戶質疑後，PM 在同一 session 內用 Hook 系統解決了問題
- 原始宣稱是基於「CLI 沒有提供官方 API」的表面判斷

## 根因分析

**思維模式問題**：PM 在遇到工具限制時，直接接受限制而非尋找替代方案。

**行為鏈**：
1. 需求：PM 需要查詢「有多少背景代理人在執行」
2. 調查：Claude Code 沒有官方 API
3. **錯誤結論**：「目前我做不到這一點，因為我無法可靠地查詢正在運行的背景代理人清單」
4. 用戶質疑：「status line 可以看到啊，不能間接查詢嗎？」
5. 正確結論：Hook 系統完全可以在代理人完成時自動推送剩餘數量

**根本問題**：
- PM 只考慮「直接查詢」（呼叫 API），沒有考慮「被動推送」（Hook 自動通知）
- PM 已有 `dispatch_tracker.py` 提供完整 API，但沒有想到用 Hook 串接
- 專案已有 `active-dispatch-tracker-hook.py` 做清理，只差加一行查詢剩餘數量

## 正確的思維模式

| 步驟 | 錯誤做法 | 正確做法 |
|------|---------|---------|
| 1 | 「CLI 沒有 API → 做不到」 | 「CLI 沒有 API → 有其他途徑嗎？」 |
| 2 | 接受限制，放棄 | 列出可能的間接方式（Hook/檔案/流程） |
| 3 | 告訴用戶「無法」 | 提出替代方案讓用戶選擇 |

**關鍵原則**：「沒有官方 API」不等於「做不到」。Hook 系統、檔案系統、流程設計都是替代途徑。在宣稱「做不到」之前，必須先窮盡所有間接方式。

## 防護措施

當 PM 準備說「無法」「做不到」「目前不支援」時，強制自問：

1. 能用 Hook 自動推送資訊嗎？
2. 能用檔案系統間接追蹤嗎？
3. 能改變流程設計來繞過限制嗎？
4. 既有的模組/工具是否已有相關 API 但沒有接線？

**只有四個問題都回答「否」後，才能說「做不到」。**

## 實際案例

**案例（2026-04-10）**：
- PM 說「無法可靠地查詢背景代理人」
- 實際上：`dispatch_tracker.py` 已有 `get_active_dispatches()` API
- 實際上：`active-dispatch-tracker-hook.py` 已在代理人完成時觸發
- 解決方案只需要：(1) 新增 record hook (2) 在清理後加一行 `get_active_dispatches()` 查詢
- 從「做不到」到「完全解決」只差 2 個小修改
