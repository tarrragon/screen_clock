# PC-176: 跨環境設定不一致時歸因「環境差異」而非驗證設定本身

## 摘要

診斷「同一設定在多環境（多機器 / CI / 容器）表現不一致 / 每次同步後又壞」時，第一反應歸因「環境有差異」（裝了不同東西 / 版本 / 路徑不同），而非先驗證「被 git 同步的共用設定本身是否從初次引入就錯」。便利假設把根因指向不可控的環境，掩蓋最可控的根因（一份錯設定被同步到所有環境），並導向「逐環境各自修補」延續跨環境乒乓。修正方向：診斷跨環境不一致前，先確認設定是否被 git 追蹤且單一份、對設定值實機驗證、查 git 歷史，確認設定正確後才談環境差異。

## 症狀

- 用戶回報「同一設定在 A 環境可用、B 環境失敗」「每次同步後又壞」
- 診斷者第一結論為「兩環境裝了不同東西 / 版本不同」，但未實機核對任一環境的實際安裝
- 被用戶質疑「不應該不一樣，一定哪裡搞錯了」後，才查出設定本身一直是錯的
- 設定檔被 git 追蹤且只存一份，卻被假設為「各環境本就不同」

## 根因（便利假設的吸引力）

跨環境不一致有兩種根因，診斷者傾向選便利的那個：

| 根因 | 可控性 | 診斷成本 | 傾向 |
|------|--------|---------|------|
| 環境差異（安裝 / 版本 / 路徑不同） | 低（要逐環境查） | 看似高，常被當「先這樣假設」免責 | 便利假設首選 |
| 共用設定本身錯誤（git 同步一份錯的） | 高（改一處解決所有環境） | 低（讀設定 + 實機驗證） | 常被跳過 |

git 同步的設定**只有一份**，若它錯，會在所有環境一致地錯——表面卻可能因「某環境剛好被另一 session 修對過、尚未被覆蓋」而呈現不一致，強化「環境差異」的錯覺。歸因環境差異等於放棄驗證最可控的根因。

## 案例：codegraph .mcp.json 跨電腦診斷（2026-06-04）

用戶回報 codegraph MCP 在兩台電腦反覆連線失敗、每次 git 同步後又壞。診斷第一版結論為「兩台電腦裝了不同的 codegraph binary（一台 `codegraph`、一台 `codegraph-mcp`）」——基於 git commit message 推斷，未實機核對辦公室電腦。用戶質疑「不應該有同 MCP 但名稱不一樣的問題，一定哪裡搞錯了」後，實機驗證才發現：兩台都是同一個 npm 套件 `@astudioplus/codegraph-mcp`，binary 只有一個名字；`.mcp.json` 的命令從初次引入（commit `c0319bad`）就是憑空猜測的 `codegraph serve --mcp`（PC-172），兩台都報錯。真正根因是「一份錯設定被 git 同步到兩台」+「猜測版反覆覆蓋驗證版」（PC-172 乒乓維度），與環境差異無關。

## 防護（診斷順序）

| 步驟 | 動作 | 目的 |
|------|------|------|
| 1 | 確認該設定是否被 git 追蹤、是否只存一份 | 一份共用設定的不一致，根因多在設定本身 |
| 2 | 對共用設定值實機驗證（PC-172 四步：`file` / 讀 wrapper / 實機啟動 / 對照同類 entry） | 確認設定本身是否正確，再談環境 |
| 3 | 查 git 歷史，看設定初次引入是否驗證過、是否有「猜測 vs 驗證」反覆覆蓋 | 識別乒乓模式 |
| 4 | 前三步確認設定正確後，才驗證環境差異（逐環境核對安裝 / 版本） | 環境差異是排除設定錯誤後的次選假設 |

**Why**：跨環境不一致的最可控根因是「共用設定錯誤」（改一處解決所有環境），但便利假設傾向歸因不可控的環境差異，掩蓋可控根因。

**Consequence**：歸因環境差異會導向「逐環境各自修補」，每環境改成各自以為對的值再 commit，延續跨環境乒乓；真正的單點根因（一份錯設定）永遠沒被修，問題每次同步後重現。

**Action**：診斷「跨環境表現不一致」前，先完成步驟 1-3（驗證共用設定本身），確認設定正確後才考慮環境差異。用戶反問「不應該不一樣」時，視為推翻便利假設的訊號（WRAP reality-test），立即回頭驗證設定。

## 識別訊號表

| 訊號 | 判讀 |
|------|------|
| 「每次同步後又壞」「在 A 可用 B 失敗」 | 優先懷疑共用設定錯誤，非環境差異 |
| 診斷結論含「兩環境裝了不同 X」但未實機核對 | 便利假設，需實機驗證 |
| 用戶反問「不應該不一樣，一定哪裡搞錯了」 | 推翻便利假設訊號，回頭驗證設定本身 |
| 設定檔被 git 追蹤且單一份，卻被假設各環境不同 | 矛盾——單一份設定不可能「本就不同」 |
