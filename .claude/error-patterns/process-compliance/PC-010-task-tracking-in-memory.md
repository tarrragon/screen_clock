# PC-010: 將任務追蹤資訊放入 Memory 而非 Ticket

## 症狀

發現新的開發需求或待辦事項時，將其記錄到 memory 系統（跨對話記憶），而未建立正式 Ticket 追蹤。

## 根因

混淆 memory 和 Ticket 的職責：

| 系統 | 職責 | 適合記錄 |
|------|------|---------|
| Memory | 跨對話偏好、設計決策、行為模式 | 用戶偏好、反覆出現的指導 |
| Ticket | 任務追蹤、進度管理、驗收 | 待辦事項、開發需求、分析任務 |

**行為模式**：遇到新需求時，因為覺得「不是正式 Ticket 範圍」或「下個 session 再做」，就把它放到 memory 裡。但 memory 不會出現在 `ticket track list` 中，容易被遺忘。

## 解決方案

品質基線規則 5：所有發現必須追蹤，發現即建立 Ticket。

| 場景 | 正確做法 |
|------|---------|
| 發現新需求 | `/ticket create` |
| 設計決策理由 | Memory（feedback 類型） |
| 用戶偏好 | Memory（user 類型） |
| 待辦開發任務 | `/ticket create` |

## 預防措施

**判斷規則**：問自己「這個資訊需要有人認領、執行、驗收嗎？」
- 是 → Ticket
- 否（只是偏好或背景知識）→ Memory
