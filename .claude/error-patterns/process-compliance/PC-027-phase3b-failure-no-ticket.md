# PC-027: Phase 3b 失敗時無 Ticket 直接派發修復

## 錯誤模式

Phase 3b 代理人執行失敗（測試失敗、編譯錯誤、context 耗盡）後，PM 未建立修復 Ticket 就直接派發另一個代理人修復，或代理人自行嘗試修復而不回報主線程。

## 症狀

- Phase 3b 代理人回報失敗，PM 立即派發新代理人修復，無對應 Ticket
- 代理人遇到測試失敗後自行嘗試修復，未建立 Ticket 追蹤
- 修復過程無記錄，後續無法追溯問題根因和修復方案
- context 耗盡後 PM 直接重新派發，未分析耗盡原因

## 根因

1. **流程捷徑**：「已經知道問題在哪，直接修就好」的心態，跳過分析和 Ticket 建立
2. **緊迫感**：Phase 3b 是實作階段，失敗會阻塞後續 Phase，PM 急於恢復進度
3. **代理人自治過度**：代理人看到錯誤後自行判斷可修復，未遵循「回報主線程」原則

## 正確做法

```
Phase 3b 代理人回報失敗
    |
    v
[強制] /pre-fix-eval 分析錯誤類型
    |
    v
[強制] /ticket create 建立修復 Ticket
    |
    v
根據錯誤類型派發對應代理人（見 incident-response.md 派發對應表）
    |
    v
修復完成 → 重新執行原 Phase 3b 子任務
```

**context 耗盡的特殊處理**：

```
代理人回報 context 耗盡（prompt too long）
    |
    v
[強制] 建立修復 Ticket
    |
    v
PM 分析耗盡原因：子任務範圍過大？代理人讀取了不必要的檔案？
    |
    v
重新拆分子任務（縮小範圍）→ 派發新代理人
```

## 防護措施

| 措施 | 說明 |
|------|------|
| 失敗 = 立即建 Ticket | 與 PC-026 相同原則，無例外 |
| 禁止無 Ticket 派發修復 | PM 必須先建 Ticket 再派發代理人 |
| 代理人禁止自行修復 | 遇到非預期錯誤必須回報主線程 |
| context 耗盡必須分析根因 | 不可直接重新派發，需先分析並縮小範圍 |
