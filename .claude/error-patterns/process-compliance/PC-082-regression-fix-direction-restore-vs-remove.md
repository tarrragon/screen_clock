# PC-082: 修復 regression 時選還原舊值而非移除（忽略全域規則適用範圍）

---

## 分類資訊

| 項目 | 值 |
|------|------|
| 編號 | PC-082 |
| 類別 | process-compliance |
| 風險等級 | 中（修復方向錯誤導致規則違反持續累積） |
| 首發時間 | 2026-04-17（W13-015 test_track_board emoji regression 修復 session） |
| 姊妹模式 | PC-081（PM 自我檢查比用戶規則更嚴格）— 相反維度：本模式是「PM 修復比用戶規則更寬鬆」 |

---

## 症狀

批量修改造成 regression 後，PM 修復時選擇「還原舊值」作為最低風險路徑，**未檢查原值是否違反其他全域規則**。結果：修復動作本身引入新規則違反。

典型觸發：
1. 批量重構/清理動作（如 emoji → ASCII）
2. 測試或 fixture 依賴被清理的字面值
3. PM 第一反應「還原原值」讓測試恢復
4. 但原值違反全域規則（如 language-constraints 規則 3 禁 emoji）

---

## 實際案例（W13-015）

**W12-002.4.3 批量 emoji 清理**：
- `.claude/skills/ticket/tests/test_track_board.py` 的 `😀` 被改為 `[SMILE]`
- `test_emoji` 預期 `get_char_display_width` 接收單一 emoji 字元返回 2
- `[SMILE]`（7 字元字串）觸發 `unicodedata.east_asian_width` TypeError

**PM 錯誤修復**（W13-015 v1）：
- 將 `[SMILE]` 還原為 `😀`
- 測試恢復 102 passed
- **但規則 3「禁用 emoji」適用於所有地方，含測試數據**

**用戶正確指正**：
> 「修復是把 emoji 去掉 不要用 emoji 驗收，這個做法本身設計有問題，我們也要求不應該再任何地方使用 emoji」

**正確修復**（W13-015 v2）：
- 刪除 `test_emoji`、`test_emoji_characters`、`test_visual_width_with_mixed_emoji` 三個測試
- 功能性驗證由 `test_fullwidth_punctuation`（「，」）+ `test_hiragana`（「あ」）+ `test_hangul`（「가」）覆蓋
- 結果：99 passed（移除 3 個冗餘測試）

## 根本原因

### 真根因

1. **PM 修復選「最低認知負擔路徑」**：
   - 還原舊值 = 1 個動作，低風險
   - 移除測試 = 需評估功能是否被其他測試覆蓋，較複雜
   - PM 預設選簡單路徑，未考慮簡單路徑是否違反規則

2. **批量重構的「還原偏見」**：
   - 批量變更造成 regression 是常見情況
   - 標準建議是「還原造成 regression 的變更」
   - 但此建議預設「被還原的值原本合理」，未檢查該值是否違反全域規則

3. **全域規則適用範圍盲區**：
   - language-constraints 規則 3「禁用 emoji」寫「所有文字輸出」
   - PM 理解為「對用戶可見的輸出」，未包含「測試內部數據」
   - 用戶的解讀是絕對的：任何地方都不該有 emoji

### 為何容易發生

- regression 修復時 PM 處於「快速止血」模式，跳過規則檢查
- 「還原」動作感覺是「回到已知可運作狀態」，安全感強
- 測試數據被視為「技術性使用」，常被豁免全域規則（誤判）

## 常見陷阱模式

| 陷阱表述 | 為何仍構成違反 |
|---------|--------------|
| 「還原是最安全路徑」 | 安全不等於合規；還原違反規則的值是更深的坑 |
| 「測試數據不算正式輸出」 | 測試檔也是專案檔案，全域規則同樣適用 |
| 「功能性 emoji 應保留」 | 功能性使用需證明「等價替代不存在」，本案中文字元可替代 |
| 「用戶會看到再改」 | PC-055/PC-056 教訓：被動等指正是失職，主動檢查是職責 |

---

## 防護措施

| 層級 | 措施 | 狀態 |
|------|------|------|
| 流程 | regression 修復前自問：「還原的舊值是否違反任何全域規則？」 | 行為準則 |
| 流程 | 若原值違反規則，**刪除依賴該值的程式碼**而非還原；缺失功能另補測試 | 行為準則 |
| 流程 | 批量清理腳本應在設計階段考慮「測試數據 / fixture / mock 是否應清理」，不應豁免 | 設計原則 |
| 規則 | language-constraints 規則 3 補充「適用範圍含所有檔案類型（含測試、fixture、mock）」 | 建議實施（W13-016 追蹤） |
| Memory | 記錄本失誤作為跨 session 提醒 | 已實施（配對本檔） |

## 檢查清單（PM 修復 regression 前自我檢查）

- [ ] regression 是否由批量重構造成？是 → 原值可能違反規則
- [ ] 還原舊值前檢查：該值是否違反 language-constraints/document-format 等全域規則？
- [ ] 原值違反規則 → 改「刪除依賴 + 評估功能覆蓋」而非「還原」
- [ ] 功能覆蓋評估：其他測試是否已涵蓋同語意？已涵蓋 → 刪除安全
- [ ] 不確定？建立追蹤 ticket 延後處理，不要先還原埋坑

---

## 教訓

1. **還原不是修復，是逃避**：還原 regression 造成的值若違反規則，本質上是把坑埋深
2. **全域規則適用範圍需放大解讀**：「所有地方」字面意思就是所有地方，不要自行豁免「內部用途」
3. **修復方向是設計決策**：每個 regression 背後都是設計問題的訊號，不要只看最小可行的止血
4. **PC-081 反面教訓**：PC-081 是 PM 自我檢查比用戶規則更嚴（過於保守），PC-082 是 PM 修復比用戶規則更寬（過於寬鬆）；兩者都是「未對齊用戶實際規則」

---

## 象限歸類

本模式的防護屬 **摩擦力管理 C 象限（增加摩擦）**：regression 修復前多一步全域規則檢查增加摩擦，換取下游不引入規則違反。代價（多 1 分鐘查規則）遠低於收益（避免規則違反累積）。
