# PC-093: YAGNI 累積反模式：推測性抽象的延後決策

## 基本資訊

- **Pattern ID**: PC-093
- **分類**: 流程合規（process-compliance）
- **來源版本**: v0.18.0
- **發現日期**: 2026-04-19
- **風險等級**: 中
- **相關 Pattern**: PC-090（推延性 close 反模式）

---

## 問題描述

### 症狀

Phase 1 為「以後可能用到」預留接口或抽象（預留參數、預留例外類別），標註「Phase X 再決定」；Phase X 真的到來時，理由變成「Phase Y 再決定」；如此遞迴延後直到專案結案，預留物變成永久死碼。

### 表現形式

| 表現 | 說明 |
|------|------|
| 「Phase 4 再決定觸發條件」 | Phase 1 預留時的延後話術 |
| 「baseline 顯示需要再做」 | Phase 4 實測未需要時的延後話術 |
| 「保留接口未來擴展彈性」 | 將「不用」偽裝為「保留」 |
| 「移除影響擴展彈性」 | Phase 4 抗拒清理的話術 |

### 累積機制

```
Phase 1: 預留接口（如 use_cache=True 參數）
   ↓ 「Phase 4 再決定」
Phase 3b: 接口仍 noop
   ↓ 「Phase 4 再決定」
Phase 4: 4 視角審查發現 baseline 已達標
   ↓ 「保留以防萬一」
版本 close: 接口仍存在
   ↓ 永久死碼 + 接口噪音
```

## 根因分析

### 直接原因

推測性抽象（speculative abstraction）的代價被低估，益處被高估。

### 深層原因

| 動機類型 | 表面說法 | 深層動機 |
|---------|---------|---------|
| A 損失迴避 | 「移除以後想加會麻煩」 | 對「丟掉已寫程式碼」感到損失 |
| B 預留樂觀 | 「以後一定會用到」 | 對「未來需求」過度樂觀 |
| C 決策疲勞 | 「Phase 4 再決定」 | 把當下決策成本轉嫁給未來 |
| D 邊界模糊 | 「保留彈性無壞處」 | 忽略接口噪音對 caller 的誤導 |

**真根因**：B 和 C 加乘。「未來會用」的樂觀 + 「現在不決定」的疲勞 = 永久死碼累積。

---

## 防護機制

### 規則建議

1. **Phase 4 強制決斷**：所有 Phase 1 預留的接口，Phase 4 必須做出「執行/移除」二選一決策。禁止「Phase 5/6 再決定」此種無限遞迴。

2. **預留接口必須帶 deadline**：Phase 1 引入預留時必須指定觸發條件（如「baseline > 80ms」）和移除條件（如「Phase 4 評估後若 baseline 未觸發則移除」）。

3. **死碼自動偵測**：CI 檢查 `__all__` 中的符號是否真的被測試覆蓋（被引用 ≥ 1 次）；未被引用的預留接口 Phase 4 強制決策。

### Phase 1 設計建議

- 寧可少抽象、實際需要時再加
- 「以後可能會用」應拒絕，等真的需要時加成本不高
- 預留接口必須附明確 use case（不是 vague「擴展彈性」）

---

## 與 PC-090 的關係

PC-090 是 close 時機判斷錯誤（「閘門未達」當「任務完成」），PC-093 是 TD 累積機制（推測性抽象的延後決策）。兩者均涉及「延後判斷」反模式，但聚焦點不同：

| Pattern | 焦點 |
|---------|------|
| PC-090 | 判斷時機錯誤（誤判 close） |
| PC-093 | 累積機制錯誤（永久延後） |
