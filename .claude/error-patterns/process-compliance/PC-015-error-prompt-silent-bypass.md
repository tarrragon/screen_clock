# PC-015: 錯誤提示靜默繞過 — 依照提示修正而非追蹤根因

## 錯誤編號
PC-015

## 分類
process-compliance

## 症狀

CLI 命令回傳非預期錯誤（如「版本偵測錯誤」「找不到 Ticket」），PM 直接依照錯誤提示中的建議操作（如加 `--version` 參數）繞過問題，未追問根因，未建立 Ticket。

**觸發場景**：
- `ticket handoff` 回傳「找不到 Ticket: 0.1.1-W9-002，版本偵測錯誤（當前偵測版本: v0.1.0）」
- PM 直接加 `--version 0.1.1` 重試，成功後繼續工作
- 根因（todolist.yaml 版本配置過期）被忽略

## 根因分析

### 行為模式
PM 將錯誤提示中的「建議操作」視為解決方案，而非問題的症狀。錯誤提示的設計本意是「告訴你如何繞過」，但正確的流程應該是「告訴你可能有 bug」。

### 為什麼會發生
1. **任務導向偏差**：PM 專注於完成當前任務（handoff），對錯誤的反應是「怎麼讓它通過」而非「為什麼它失敗」
2. **錯誤提示設計誤導**：提示中直接給出繞過方案（`--version`），降低了追蹤根因的動機
3. **無強制調查機制**：CLI 錯誤不像測試失敗那樣有 `/pre-fix-eval` 強制流程

## 解決方案

### 即時行為修正
任何 CLI 命令回傳非預期錯誤時：
1. **停下來問**：「為什麼會觸發這個錯誤？」
2. **建立 Ticket**：追蹤根因
3. **才考慮繞過**：且在 Ticket 中記錄繞過方式

### 長期防護
- 在 incident-response 流程中新增 CLI 錯誤調查規則
- 考慮 Hook：當 CLI exit code != 0 時自動提示「是否需要建立 Ticket 追蹤此錯誤？」

## 預防措施

| 措施 | 說明 | 狀態 |
|------|------|------|
| 錯誤即 bug 信號原則 | 任何非預期錯誤都是潛在 bug，不是操作指南 | 已記錄 memory |
| incident-response 擴充 | CLI 錯誤納入強制調查流程 | （追蹤 Ticket） |
| 版本配置一致性檢查 | 偵測 todolist vs 實際 Ticket 版本不一致 | （追蹤 Ticket） |

## 影響範圍

- 嚴重度：中（根因被忽略，持續影響後續命令）
- 發現版本：v0.1.1
- 影響：所有未指定 `--version` 的 ticket 命令偵測到錯誤版本
