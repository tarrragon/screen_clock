# PC-006: 過早統一抽象（DRY 誤用）

## 基本資訊

- **Pattern ID**: PC-006
- **分類**: 流程合規
- **來源版本**: v0.1.0
- **發現日期**: 2026-03-06
- **風險等級**: 中

## 問題描述

### 症狀

Ticket 設計時將「看起來相似的程式碼」定義為「重複」，規劃全面統一包裝，但多視角分析後發現各呼叫的語義、參數、上下文都不同，統一包裝反而增加認知負擔。

### 根因

混淆了「語法重複」和「知識重複」：
- **語法重複**：多處使用 `subprocess.run()`，但各自語義不同（git vs flutter vs uv）
- **知識重複**：多處實作相同邏輯（5 個 hook 各自寫 `get_project_root()`）

DRY 原則的完整定義是「每份**知識**只有一個權威表述」，不是「每個**語法模式**只出現一次」。

### 具體案例

**原始設計**：「統一 subprocess/git 命令包裝到 hook_utils」

多視角分析發現：
- 27 個 subprocess 呼叫中，只有 git 呼叫（9 個）有固定模式值得提取
- flutter/uv/python 呼叫各有不同的 timeout、cwd、shell、錯誤處理語義
- 通用 `run_command()` 包裝等同於重新實作 `subprocess.run` 的別名

**分析後縮小範圍**：只做 `get_project_root()` + `run_git()`，收益明確且風險低。

## 判斷標準

### 值得統一的條件（全部滿足）

| 條件 | 說明 |
|------|------|
| 知識重複 | 多處實作相同的業務邏輯（不只是語法相似） |
| 參數固定 | 呼叫模式的參數組合一致（如 git 呼叫都是 capture_output + timeout=5） |
| 錯誤處理相同 | 失敗時的處理方式相同 |

### 不值得統一的信號

| 信號 | 說明 |
|------|------|
| 包裝函式需要 5+ 個參數 | 接近重新實作原始 API |
| 各呼叫的語義不同 | `subprocess.run(["git",...])` vs `subprocess.run(["flutter",...])` 是不同概念 |
| 統一後讀者需要查原始碼 | 包裝反而增加認知負擔 |
| 只有 1-2 個呼叫 | 不構成「重複」 |

## 防護措施

### Ticket 設計檢查

- [ ] 「統一/包裝/抽象」類 Ticket 是否先做多視角分析？
- [ ] 被標為「重複」的程式碼，是語法重複還是知識重複？
- [ ] 統一後的介面是否比原始 API 更簡單？（如果不是，就不值得）

### 決策原則

> **Good taste 的核心問題**：這個抽象為讀者消除了什麼認知負擔？
> 如果答案是「沒有」，就不要做。
