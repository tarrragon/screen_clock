## TDD Phase 3b 執行流程

### Step 1: 接收 Phase 3a 策略

**從 pepper-test-implementer 接收**：
- [ ] 虛擬碼邏輯完整且無歧義
- [ ] 流程圖涵蓋所有關鍵路徑
- [ ] 架構決策有明確理由
- [ ] 技術債務標記清楚

### Step 2: 確認常數和字串結構

在開始實作前，先確認：
- [ ] 需要的常數已在 `{常數檔案路徑}` 定義（或需新增）
- [ ] 需要的字串已在 `{字串管理目錄}` 定義
- [ ] 需要的錯誤類型已定義

### Step 3: 實作

1. 從常數檔案和字串管理目錄引用，不在程式碼中寫字串或數值
2. 使用 {語言} 標準 log 方式，帶必要的上下文欄位
3. 使用目標版本的最新語法（禁止使用舊版相容寫法）
4. Guard Clause 優先，函式行數控制在範圍內

### Step 4: 測試驗證

```bash
# 執行測試
{測試指令}

# 靜態分析
{靜態分析指令}

# 確認無硬編碼字串（按語言特性搜尋）
{搜尋驗證指令}
```

### Step 5: 品質檢查清單

#### 開始前

- [ ] Ticket 已認領
- [ ] 常數和字串管理結構已確認或已更新
- [ ] 理解了任務完整要求

#### 完成後

- [ ] `{build 指令}` 成功
- [ ] `{lint 指令}` 0 issues
- [ ] `{test 指令}` 100% 通過
- [ ] 無硬編碼字串或數值
- [ ] 函式長度 <= {上限} 行
- [ ] 巢狀深度 <= 3 層
- [ ] 使用目標版本 {目標版本}+ 語法（無降級相容寫法）

2. **禁止使用舊版語法**：預設使用 {目標版本}+ 最新語法，不為舊版相容降級
3. **禁止 `{語言特有錯誤處理禁止行為}`**：必須處理或明確說明忽略原因
4. **禁止修改測試邏輯**：測試本身有問題升級 sage-test-architect
5. **禁止跳過測試**：必須執行 `{test 指令}` 確認 100% 通過
6. **禁止超出 Ticket 範圍**：發現相關問題建立新 Ticket 追蹤
