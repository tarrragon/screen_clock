# 文件撰寫明示性原則（速查 stub）

> **完整規則**：`.claude/references/document-writing-style-details.md`（按需讀取，含二次審查雙清單、正反範例 4 組、反模式句型表、情境速查、邊界表）。本檔僅保留核心原則速查與觸發路由。

所有書面文字（規則/方法論/PC error-pattern/skill description/長段註解）的每段論述都必須對讀者明確交代「為何、後果、下一步」三層。文件如商業邏輯——決策方向和流程都應清晰明示，不依賴讀者自行推論。

## 核心原則：三明示

| 層次 | 問題 | 讀者收穫 |
|------|------|---------|
| Why | 為何有此原則/做法？ | 理解依據（不是憑感覺） |
| Consequence | 不這樣會發生什麼？ | 理解違反成本 |
| Action | 下一步應做什麼？ | 知道如何落實 |

**適用粒度**：每段（非每句）論述。若某段只回答 1-2 項（例如只有「禁止 X」），應主動補足剩餘。

**載入層邊界**：三明示完整論證僅適用**按需讀取層**（references/ / methodologies/ / error-patterns/ / pm-rules/）。**自動載入層**（`rules/core/` 速查 stub、CLAUDE.md、rules/README.md）受 `rules/README.md`「自動載入預算原則」約束，形態為「禁令速查 + 路由」——Why/Consequence 壓縮至一行或路由指向按需層檔案，禁止整段展開（每條規則三明示化約 3 倍 token，預算必回彈）。stub 撰寫規範見 `.claude/references/auto-load-stub-conventions.md`。

## 核心原則二：資訊優先序

同一句話或短段落依序呈現：(1) 核心原則／定義 → (2) 示例／參考 → (3) 提醒／邊界。**禁止**「X 是 Y。它不是單純的…，而是…」這種示例先於原則的句型。寫完一句問「讀者只讀第一句，知道這個概念是什麼嗎？」不知道就把原則移到句首。

## 最高優先原則：二次審查強制執行

完成任何書面文字後、回報完成前，必須執行一輪二次審查掃描補修。**Why**：AI 輸出是機率性的，規則知識只影響輸出分佈，不保證合規。**Action**：依完整版二次審查雙清單掃描，發現違規立即修正並重新掃描。

| 動作 | 是否需要二次審查 |
|------|----------------|
| 新寫一段文字或文件 | 是 |
| 擴寫現有段落 | 是（針對改動部分） |
| 修改既有錯誤（純修正） | 選擇性（確認未引入新問題） |
| 純格式調整（排版） | 否 |

## 何時讀完整規則

| 情境 | 必讀章節（references 詳細版） |
|------|------------------------------|
| 執行二次審查（文章 / 程式碼） | 二次審查雙清單全文 |
| 重寫人性化/隱含表達句型 | 反模式：人性化/隱含表達（6 句型表 + 重寫方向） |
| 撰寫各類書面文字（規則/PC/方法論/skill/註解） | 適用情境速查 + 正反範例對照 4 組 |
| 釐清與 ai-communication / document-format / comment-writing 邊界 | 與其他規則的邊界表 |
