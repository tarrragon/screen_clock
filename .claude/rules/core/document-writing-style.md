# 文件撰寫明示性原則

本文件規範所有書面文字（規則/方法論/PC error-pattern/skill description/長段註解）的論述明示性，要求每段論述都對讀者明確交代「為何、後果、下一步」三層資訊。

> **核心理念**：文件如商業邏輯——所有決策方向和流程都應清晰明示，不依賴讀者自行推論或補完隱含涵義。
> **詳細範例與情境應用**：本檔與 `.claude/rules/core/ai-communication-rules.md`（對話 prompt 品質）、`.claude/rules/core/document-format-rules.md`（Markdown 語法格式）分工互補，不重複。

---

## 最高優先原則：二次審查強制執行

> **核心主張**：理解寫作規範不代表輸出一定遵守規範。每次撰寫或擴寫後，在回報完成之前，必須執行一輪二次審查。不要求第一次完美——用輕量標準快速掃描補修，比強迫首次完美的成本更低，也不影響任務流程。

**Why**：AI 輸出是機率性的，規則知識只影響輸出分佈，不保證每次輸出都合規。若沒有主動審查步驟，違規只有靠外部反饋才能發現，修正成本遠高於自審。

**Consequence**：跳過二次審查會讓錯誤在被人發現前累積，導致後續需要更高成本的修正，並降低讀者對輸出品質的信任。

**Action**：完成任何書面文字後，依下方二次審查清單執行掃描；發現違規立即修正並重新掃描，除非明確被要求延後。

### 二次審查清單（文章 / 文件）

| 審查項目 | 辨識訊號 |
|---------|---------|
| 表格分類有後續說明 | 表格後是否接有解釋文字、辨識訊號或例子？只有表格沒有說明等於只做了分類，沒有傳達理由 |
| 核心原則先行 | 每段開頭是原則，不是示例；若讀者只讀第一句，應能知道這段「是什麼」或「要遵守什麼」 |
| 負向對比有正向錨點 | 每個「禁止 X」後是否緊跟「正確做法是 Y」？只列禁令不教修正方向，讀者不知道往哪走 |
| 無錯字 / 禁用詞 / 簡體字 | 掃描 `language-constraints.md` 禁用詞彙（數據、代碼、默認等）；確認無簡體字混入 |
| 無拼寫 / 語法錯誤 | 繁體中文正確；技術術語英文大小寫符合慣例 |
| 內容中立可重用 | 若涉及本專案特定細節（ticket ID、版本號），已標注或提取為通用原則 |

### 二次審查清單（程式碼）

| 審查項目 | 辨識訊號 |
|---------|---------|
| 邏輯正確性 | 重讀主要邏輯分支，邊界情況是否已覆蓋 |
| 是否需補短註解 | 非顯而易見的決策或限制，是否應補一行說明理由（而非描述做了什麼） |
| 命名自說明 | 新增的變數 / 函式名稱刪掉後，讀者仍能推斷意圖嗎？若不能則需改名或補註解 |
| 無未處理佔位符 | 確認無遺留 `// TODO: implement` 或未替換的模板字串 |

### 觸發時機

| 動作 | 是否需要二次審查 |
|------|----------------|
| 新寫一段文字或文件 | 是 |
| 擴寫現有段落 | 是（針對改動部分） |
| 修改既有錯誤（純修正） | 選擇性（確認修正本身未引入新問題） |
| 純格式調整（排版） | 否 |

## 核心原則：三明示

每段論述必須回答三個問題，讀者讀完後能獨立判斷是否適用、是否需要動作。

| 層次 | 問題 | 讀者收穫 |
|------|------|---------|
| Why | 為何有此原則/做法？ | 理解依據（不是憑感覺或習慣） |
| Consequence | 不這樣會發生什麼？ | 理解違反成本（不只知禁令還知代價） |
| Action | 下一步應做什麼？ | 知道如何落實（不只知原則還知操作） |

**適用粒度**：每段（非每句）論述。原則陳述可一段含三明示；反模式列表可共用一個「為何/後果」，但每項需有「下一步」。

**意圖**：讀者不須靠上下文或其他文件推論隱藏涵義。若某段論述只回答其中 1-2 項（例如只有「禁止 X」），應主動補足剩餘（「因為 Y，否則 Z，遇到 X 時改做 W」）。

## 核心原則二：資訊優先序（句子／短段落層級）

技術論述中，同一句話或短段落的資訊必須依以下順序呈現：

| 順序 | 類型 | 說明 |
|------|------|------|
| 1 | 核心原則／定義 | 這個概念「是什麼」或「必須遵守什麼規則」 |
| 2 | 示例／參考 | 具體實例，跟隨原則出現 |
| 3 | 提醒／邊界 | 注意事項、不可任意轉換等邊界聲明 |

**Why**：顛倒順序（先示例後原則）讓讀者先記一組脫離脈絡的資料，再倒推抽象概念——認知路徑反向，造成主次不分。AI 產出的技術文章特別容易出現此模式（例如「X 是 Y，它不是單純的…，而是…」）。

**Consequence**：讀者必須讀完整句才能理解前文示例「代表什麼意思」，認知負擔上升；不熟悉主題的讀者容易誤以為示例本身就是結論。

**Action**：寫完一句後問「如果讀者只讀第一句，他知道這個概念是什麼嗎？」若答案是「不知道，他只看到一個例子」，把核心原則移到句首。

**反例（示例先，原則後）**：

> Go 欄位使用 `AppName`，JSON 欄位使用 `appName`。這不是任意轉換，而是兩個命名慣例的交界。

**正例（原則先，示例後，提醒最後）**：

> Go exported 欄位必須大寫開頭，JSON payload 常用 camelCase。`AppName` 對應 `appName` 是兩個命名慣例在 API 邊界的交會，不可任意轉換。

## 反模式：人性化/隱含表達

以下表達違反明示性原則，必須重寫為明示結構。

| 反模式句型 | 為何違反 | 重寫方向 |
|-----------|---------|---------|
| 「希望讀者理解…」 | 把責任推給讀者，不交代依據 | 明示：「此原則依據 X，違反會導致 Y」 |
| 「按理應…」「自然而然…」 | 假設共識，實際共識可能不存在 | 明示條件：「當 X 條件成立時，應做 Y」 |
| 「通常來說」「一般情況下」 | 條件模糊，讀者無法判斷是否適用 | 明示邊界：「除了 X/Y 情境外，應做 Z」 |
| 「規則寫得好卻沒人用」 | 描述症狀但不指出成因和修正動作 | 明示因果+動作：「因為 X 機制缺失，導致 Y 發生；修正方向：Z」 |
| 「假設讀者會注意到」 | 把規則的有效性寄託於讀者自律 | 明示強制點：「Hook A 在時機 B 強制檢查；人工 fallback 在 C」 |
| 「理想情況下」 | 未指明理想與現實差距的處理方式 | 明示落差處理：「理想 X；現實 Y；先做 Z 達到 W」 |

**判別準則**：讀完該段後，若讀者仍需「猜」為何、後果、下一步中的任一項，即屬隱含表達。

## 正反範例對照

### 範例 1：規則條文

反例：
> 提交前應確保測試通過。

正例：
> 提交前所有測試必須通過。**Why**: 失敗測試進 main 會汙染後續 CI 和 diff 責任歸屬。**Consequence**: 未通過提交後，下次 CI 失敗時難以辨識是新增 commit 還是既有問題。**Action**: 本地執行 `npm test`，全綠後再 commit；若急需部分提交，用 feature branch 隔離。

---

### 範例 2：PC 錯誤模式論述

反例：
> PM 偶爾會忘記在 context 沉重時觸發 WRAP，導致決策品質下降。

正例：
> Context 使用率 > 60% 時，PM 工作記憶縮小，主動觸發 WRAP 的機率從 80% 降至 20%（PC-066 案例 1-4 統計）。**Why**: 自律機制與最需要它的場景負相關。**Consequence**: 限制性解法（禁止 X / 規避 X）成為預設反應，遮蔽探索性工具（找正確方法做 X）。**Action**: 依 `.claude/hooks/decision-quality-guard-hook.py` 訊號自動偵測 + CLI claim 強制三問；人工 fallback 以 `wrap-decision` SKILL 觸發條件章節為唯一權威對照。

---

### 範例 3：方法論原則說明

反例：
> 命名應該要有意義。

正例：
> 變數名必須是名詞、函式名必須是動詞、布林值以 `is/has/can` 開頭。**Why**: 名稱本身就是文件，讀者不需跳轉查定義才懂用途。**Consequence**: 使用 `data/info/value` 等模糊名會讓讀者每次閱讀都重新推論語意，增加認知負擔。**Action**: Code review 時查命名模糊處；發現時立即重命名（不列為技術債延後）。

---

### 範例 4：Skill description

反例：
> WRAP 決策框架工具。用於各種分析場景。

正例：
> WRAP 決策框架 — 認知偏誤防護、選項擴增與資料充足度閘門。**Use when**: 被困住或連續失敗 2+、宣稱「做不到/無法/禁止」前、偏離核心目標、根因分析。**Why**: 心智自律在壓力下系統性失效；外部框架補救訓練偏誤。**Action**: 輸入條件匹配時主動呼叫 Skill；觸發後依 W→R→A→P 四階段執行並記錄結論。

## 適用情境速查

| 情境 | 明示重點 | 最低要求 |
|------|---------|---------|
| 定義新規則 | Why + Consequence + Action | 原則陳述附至少 1 個違規後果和 1 個修正動作 |
| 撰寫 PC error-pattern | 成因 + 防護 + 觸發條件 | 症狀/根因/案例/防護四段都要可執行下一步 |
| 新增方法論 | 理論依據 + 適用邊界 + 實踐步驟 | 每節開頭交代 Why，結尾交代 Action |
| Skill description | 觸發條件 + 預期產出 + 不適用場景 | 「Use when / Trigger / Do NOT trigger」三段齊備 |
| 長段註解（> 3 行） | 業務情境 + 觸發場景 + 變更影響 | 開頭一句業務意圖，結尾指向變更時需同步的位置 |

## 與其他規則的邊界

| 規則 | 聚焦 | 與本規則差異 |
|------|------|------------|
| `ai-communication-rules.md` 規則 1-5 | 對話、prompt、即時互動品質 | 本規則聚焦「書面文字」（可回讀、可搜尋、長期存在） |
| `document-format-rules.md` | Markdown 語法、檔案命名、狀態標記格式 | 本規則聚焦「論述內容」而非「排版形式」 |
| `comment-writing-methodology.md` | 註解業務情境 + 抽象層級對應 | 本規則為上位原則（三明示）；註解方法論為專項套用 |
| `language-constraints.md` | 繁體中文、禁用詞彙、Emoji 禁令 | 本規則不管字元選擇，只管論述結構 |

**DRY 銜接**：註解專項規則（`comment-writing-methodology.md`）以「引用本規則三明示原則」開頭，不重複定義明示性；本規則在範例段落不深入註解細節，把註解專項留給該方法論。

## 撰寫前檢查清單

提交書面文字前自問：

- [ ] **已執行二次審查？**（表格說明、核心原則先行、負向錨點、禁用詞/簡體字、中立可重用性）
- [ ] 每段論述都能被拆成「為何 / 後果 / 下一步」三層？
- [ ] 技術解釋的資訊優先序正確：核心原則在前，示例居中，提醒在後？
- [ ] 無「X 是 Y。它不是單純的…，而是…」這種示例先於原則的句型？
- [ ] 無「希望」「按理」「自然」「通常」等隱含詞？
- [ ] 條件表達具體到讀者能獨立判別是否適用？
- [ ] 反模式列表有對應的修正方向（不只禁令）？
- [ ] 正反範例至少各 1 個，且可辨識差異點？
- [ ] 與既有規則的邊界章節已寫？

## 與 quality-baseline 的銜接

本規則是 `.claude/rules/core/quality-baseline.md` 規則 5（所有發現必須追蹤）與規則 6（失敗案例學習原則）的**執行前提**。

**Why**: 規則 5 / 6 假設文件能被閱讀並觸發動作；若論述不明示，追蹤條目變成「待詮釋文字」而非「可執行清單」，失敗案例學習無法固化為「下一步該怎麼做」。

**Consequence**: 未明示的規則會堆積為「看似涵蓋但不可執行」的條目，PM 與代理人在壓力情境下跳過該規則，規則 5 / 6 的設計意圖失效。

**Action**: 新增或修訂規則時，先自檢三明示；發現既有規則違反時，依規則 5 建 Ticket 追蹤（範例：將既有 PC error-pattern 套用本規則重寫，作為驗證樣本）。
