# 認知負擔設計原則

本文件定義程式碼和任務設計中的認知負擔管理原則。

> **核心理念**：降低閱讀者的認知負擔是所有設計的終極目標。

---

## 核心概念

認知負擔（Cognitive Load）是閱讀者理解程式碼時需要**同時記住**的資訊量（Miller's Law：工作記憶一次約 7±2 項）。**設計目標**：最小化外在認知負擔，讓認知資源用於理解內在負擔與建立心智模型。

## 量化標準

### 認知負擔指數

```
認知負擔指數 = 變數數 + 分支數 + 巢狀深度 + 依賴數
```

### 閾值表

| 指數 | 評估 | 行動 |
|------|------|------|
| 1-5 | 優良 | 維持 |
| 6-10 | 可接受 | 考慮優化 |
| 11-15 | 需重構 | 建立重構 Ticket |
| > 15 | 必須重構 | 立即處理 |

### 任務拆分閾值

| 條件 | 閾值 | 行動 |
|------|------|------|
| 需追蹤概念數 | > 7 個 | 必須拆分 |
| 需理解檔案數 | > 5 個 | 考慮拆分 |
| 跨架構層級數 | > 2 層 | 必須拆分 |
| 依賴模組數 | > 3 個 | 考慮拆分 |
| 修改檔案數 | > 5 個 | 必須拆分 |
| 目標檔案行數 | > 300 行 | 分析 domain 邊界，考慮拆分檔案 |
| 單一檔案承載 domain 數 | > 2 個 | 必須拆分（DDD 邊界違規） |
| 事件經過處理層數 | > 5 層 | 簡化流程或拆分為獨立處理鏈 |

> **回合耗盡 = 認知負擔過載的具體訊號**：代理人回合耗盡時，不只是「重試」或「PM 代做」，而是系統性訊號——某處的檔案體量或 domain 混合度需要降低。行數超標是症狀，domain 混合是病因。

## 自我檢查清單

| 層級 | 檢查項目 |
|------|---------|
| 函式 | 長度 <= 15 行（理想 5-10）/ 參數 <= 3 / 巢狀深度 <= 3 / 區域變數 < 5 / 名稱是動詞片語 / 只做一件事 / 無隱藏副作用 |
| 類別 | 方法數 <= 10 / 公開方法 <= 5 / 只代表一個概念 / 名稱是名詞 |
| 命名 | 無縮寫或僅用通用縮寫 / 變數是名詞 / 函式是動詞 / 布林 is/has/can 開頭 / 不用 data/info/value 模糊詞 |
| 任務 | 能用「動詞 + 單一目標」描述 / 5 分鐘後還能記住目標 / 不需同時記住超過 7 個概念 |

## 降低認知負擔的原則

| 原則 | 核心要求 | 違反 → 改善方向 |
|------|---------|----------------|
| 1. 單一責任 | 一個函式只做一件事，讀者只需理解一個概念 | 「計算總價並發送通知」（2 責任）→ 拆分 |
| 2. 自說明命名 | 名稱本身就是文件，不需額外資訊 | `calc()` → `calculateMonthlyRevenue()`；`isValid` → `isEmailFormatValid` |
| 3. 避免副作用 | 行為可預測，讀者不需追蹤隱藏狀態 | 移除隱藏狀態變更 |
| 4. 扁平化結構 | 減少巢狀，不需追蹤深層上下文 | 巢狀 4+ → Guard Clause 或提取函式（1-2 優良 / 3 可接受 / 4+ 需重構） |
| 5. 資訊就近 | 相關資訊放在一起，不需跳轉尋找 | 分散資訊集中 |

> 認知負擔是決策樹任務拆分的核心依據：認知負擔指數 <= 10 直接派發，> 10 必須拆分後再派發。
