## 1. 功能概述

### 1.1 問題背景

目前 `hook-completeness-check.py` 只能報告未註冊的 Hook 檔案，但無法自動註冊它們。原因是：

1. **缺少配置資訊**：每個 Hook 需要在 `settings.json` 中定義：
   - `event_type`：Hook 的觸發事件（PreToolUse, PostToolUse, SessionStart 等）
   - `matcher`：觸發的工具名稱（可選，某些事件無需 matcher）
   - `timeout`：執行超時（可選）

2. **配置分散**：Hook 的配置資訊目前存在：
   - Hook 檔案內的註解中（無結構化格式）
   - 口頭協議（非常規方式）
   - settings.json 中（部分已註冊的 Hook）

3. **維護困難**：新增 Hook 時需要手動更新 settings.json，容易遺漏或錯誤配置。

### 1.2 解決方案

實現 **Hook 自動註冊機制**，讓 Hook 自帶 metadata 資訊，系統能自動讀取和註冊：

1. **Metadata 標準化**：定義結構化的 Hook metadata 格式
2. **自動提取**：實作 hook-auto-register.py 能讀取 Hook 的 metadata
3. **自動更新**：根據 metadata 自動生成或更新 settings.json 配置
4. **預覽模式**：提供 --dry-run 模式，不實際修改文件，只顯示預覽

### 1.3 目標成果

完成本 Phase 後，應該產生：

1. **Hook Metadata 規範**（文件）
   - Metadata 格式定義
   - 必填和可選欄位
   - 驗證規則

2. **hook-auto-register.py 架構設計**（文件）
   - 命令行介面設計
   - 核心模組劃分
   - 執行流程

3. **settings.json 更新機制設計**（文件）
   - 配置合併策略
   - 衝突解決規則
   - 備份機制

4. **完整的功能規格文件**（本文件）

## 2. Hook Metadata 規範

### 2.1 設計決策：Docstring vs JSON Sidecar

#### 選項 1：Docstring 型（推薦）

**優點**：
- Hook 檔案完全自足，無需額外文件
- Metadata 與程式碼一起版本控制
- 修改 Hook 時同步修改 metadata

**缺點**：
- 需要解析 Python docstring（略微複雜）
- 多行 metadata 可能影響代碼可讀性

#### 選項 2：JSON Sidecar（替代方案）

**優點**：
- 純 JSON 格式，易於解析
- 不需要修改 Hook 程式碼

**缺點**：
- 需要額外文件（.json）
- Hook 檔案和配置容易不同步

**決定**：採用 **Docstring 型（推薦）** + **JSON Sidecar 型（可選備選）** 雙模式支持
- 優先使用 Docstring（更加簡潔和內聚）
- 允許使用 JSON Sidecar（若 Hook 邏輯複雜）

### 2.2 Metadata 格式 - Docstring 模式

#### 2.2.1 格式定義

Hook 的 docstring 應包含特定格式的 metadata 區塊：

```python
"""
{Hook 功能描述}

HOOK_METADATA (JSON):
{
  "event_type": "PreToolUse",
  "matcher": "Write|Edit",
  "timeout": 10000,
  "description": "hook 簡短描述",
  "dependencies": [],
  "version": "1.0.0"
}
"""
```

#### 2.2.2 規範說明

| 欄位 | 型別 | 必填 | 說明 | 範例 |
|------|------|------|------|------|
| `event_type` | string | 是 | Hook 觸發事件類型 | `PreToolUse`, `PostToolUse`, `SessionStart` |
| `matcher` | string | 否 | 工具匹配規則（空表示無 matcher） | `Write`, `Edit`, `Write\|Edit`, 或空字符串 |
| `timeout` | integer | 否 | 執行超時（毫秒），不設置則使用預設 | `5000`, `10000`, `60000` |
| `description` | string | 否 | Hook 功能簡述 | `"檔案編輯驗證"` |
| `dependencies` | array | 否 | Hook 依賴的其他 Hook | `["ticket-path-guard-hook.py"]` |
| `version` | string | 否 | Hook 版本 | `"1.0.0"` |

#### 2.2.3 具體範例

**例 1：PreToolUse Hook（有 matcher）**

```python
#!/usr/bin/env -S uv run --quiet --script
# /// script
# requires-python = ">=3.11"
# dependencies = []
# ///

"""
Ticket Path Guard Hook - 路徑守衛

檢查編輯/寫入操作是否在允許的檔案路徑範圍內。

HOOK_METADATA (JSON):
{
  "event_type": "PreToolUse",
  "matcher": "Write|Edit",
  "timeout": 5000,
  "description": "檢查檔案編輯權限",
  "version": "1.0.0"
}
"""
```

**例 2：SessionStart Hook（無 matcher）**

```python
#!/usr/bin/env -S uv run --quiet --script
# /// script
# requires-python = ">=3.11"
# dependencies = []
# ///

"""
Ticket Reinstall Hook - Ticket 重新安裝

在 Session 開始時檢查並重建 Ticket 檔案結構。

HOOK_METADATA (JSON):
{
  "event_type": "SessionStart",
  "timeout": 10000,
  "description": "檢查並重建 Ticket 結構",
  "version": "1.0.0"
}
"""
```

**例 3：有依賴的 Hook**

```python
"""
Style Guardian Hook - 風格守衛

檢查檔案編輯是否符合風格要求，依賴於 ticket-path-guard-hook。

HOOK_METADATA (JSON):
{
  "event_type": "PostToolUse",
  "matcher": "Write|Edit",
  "timeout": 5000,
  "description": "檢查檔案風格合規性",
  "dependencies": ["ticket-path-guard-hook.py"],
  "version": "1.0.0"
}
"""
```

### 2.3 Metadata 格式 - JSON Sidecar 模式

作為備選方案，允許 Hook 在 `.claude/hooks/metadata/` 目錄下放置對應的 JSON 文件。

#### 檔案命名規則

```
Hook 檔案：.claude/hooks/my-hook.py
Metadata 檔案：.claude/hooks/metadata/my-hook.json
```

#### Metadata JSON 結構

```json
{
  "hook_name": "my-hook.py",
  "event_type": "PreToolUse",
  "matcher": "Write|Edit",
  "timeout": 5000,
  "description": "Hook 功能簡述",
  "dependencies": [],
  "version": "1.0.0",
  "author": "代理人名稱",
  "created": "2026-02-06"
}
```

### 2.4 Metadata 驗證規則

自動註冊工具在讀取 metadata 時應驗證：

| 驗證項 | 規則 | 失敗處理 |
|--------|------|---------|
| event_type 有效性 | 必須是官方事件類型 | 跳過此 Hook，記錄警告 |
| matcher 格式 | 多個匹配器用 `\|` 分隔 | 自動修正或記錄警告 |
| timeout 有效性 | 必須是正整數 | 使用預設值，記錄警告 |
| 依賴檢查 | 所有 dependencies 中的 Hook 必須存在 | 記錄警告但允許註冊 |

## 3. hook-auto-register.py 架構設計

### 3.1 命令行介面設計

```bash
# 基本用法：掃描並自動註冊所有未登記的 Hook
hook-auto-register.py

# 預覽模式：顯示將要執行的操作，不實際修改
hook-auto-register.py --dry-run

# 詳細模式：顯示詳細的執行過程
hook-auto-register.py --dry-run --verbose

# 強制更新：即使 Hook 已登記也更新其配置
hook-auto-register.py --force

# 指定輸出檔案：將結果寫入特定的 settings.json
hook-auto-register.py --output /path/to/settings.json

# 驗證模式：只檢查 metadata 有效性，不修改配置
hook-auto-register.py --validate-only
```

### 3.2 核心模組劃分

```
hook-auto-register.py
├── MetadataExtractor
│   ├── extract_from_docstring(file_path)
│   ├── extract_from_sidecar(file_path)
│   └── validate_metadata(metadata)
├── SettingsManager
│   ├── load_settings(path)
│   ├── merge_hooks(existing, new)
│   ├── save_settings(path, settings)
│   └── create_backup(path)
├── HookRegistrar
│   ├── scan_hooks_directory()
│   ├── register_missing_hooks()
│   └── generate_report(registered, failed)
└── 主程式
    ├── parse_arguments()
    ├── validate_environment()
    └── execute_registration()
```

### 3.3 執行流程

```
開始
  |
  v
[步驟1] 參數解析與驗證
  ├─ 驗證 settings.json 可訪問
  ├─ 驗證 .claude/hooks/ 目錄存在
  └─ 檢查 --force, --dry-run 等標誌
  |
  v
[步驟2] 掃描 Hook 目錄
  ├─ 列出所有 .py 檔案
  ├─ 排除已知的非 Hook 檔案（從 hook-exclude-list.json）
  └─ 建立 Hook 清單
  |
  v
[步驟3] 提取 Hook Metadata
  ├─ 對每個 Hook 檔案：
  │  ├─ 首先嘗試從 docstring 提取
  │  ├─ 若失敗，嘗試從 JSON Sidecar 提取
  │  ├─ 若均失敗，記錄錯誤並跳過
  │  └─ 驗證提取的 metadata
  └─ 建立 metadata 對映表
  |
  v
[步驟4] 加載現有配置
  ├─ 讀取 settings.json
  ├─ 解析現有 hooks 配置
  └─ 識別已登記和未登記的 Hook
  |
  v
[步驟5] 計算所需變更
  ├─ 對每個未登記的 Hook：
  │  ├─ 根據 event_type 定位配置位置
  │  ├─ 根據 matcher 分類
  │  └─ 建立配置條目
  └─ 建立變更清單
  |
  v
[步驟6] 預覽或應用變更
  ├─ 若 --dry-run：顯示預覽報告
  ├─ 否則：
  │  ├─ 建立備份（settings.json.backup-{timestamp}）
  │  ├─ 合併配置
  │  └─ 寫入 settings.json
  |
  v
[步驟7] 生成報告
  ├─ 已登記 Hook 數量
  ├─ 新增 Hook 數量
  ├─ 失敗/跳過的 Hook 清單
  └─ 執行摘要
  |
  v
結束
```

### 3.4 Exit Code 定義

| Code | 含義 | 場景 |
|------|------|------|
| 0 | 成功 | 所有 Hook 已登記，無錯誤 |
| 1 | 部分成功 | 某些 Hook 無法解析 metadata，但不影響主要功能 |
| 2 | 失敗 | 致命錯誤（settings.json 無法寫入、目錄不存在等） |

## 4. settings.json 更新機制

### 4.1 配置合併策略

#### 策略原則

1. **保留現有配置**：已登記的 Hook 配置不修改
2. **新增缺失配置**：未登記的 Hook 自動新增
3. **版本檢查**：若 Hook 的 metadata 版本更新，可選擇是否更新配置
4. **衝突解決**：若同一 Hook 在多個 matcher 下已登記，不重複新增

#### 配置結構分析

```json
{
  "hooks": {
    "{event_type}": [
      {
        "matcher": "{工具名稱}",  // 或無此欄位（SessionStart）
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/{hook-name}.py",
            "timeout": 5000  // 可選
          }
        ]
      }
    ]
  }
}
```

### 4.2 新增 Hook 的位置規則

| event_type | matcher | 新增位置 |
|------------|---------|---------|
| PreToolUse | 指定值 | 在對應 matcher 的 hooks 陣列末尾 |
| PreToolUse | 空 | 建立無 matcher 的新條目 |
| PostToolUse | 指定值 | 在對應 matcher 的 hooks 陣列末尾 |
| SessionStart | - | 在 hooks 陣列末尾 |
| UserPromptSubmit | - | 在 hooks 陣列末尾 |

### 4.3 衝突解決規則

#### 場景 1：同一 Hook 在相同 event_type + matcher 下已存在

**判定**：
- 比較 command 路徑（去掉參數）
- 若路徑相同，判定為已登記

**處理**：
- --force 未設定：跳過（無需重複新增）
- --force 已設定：更新配置（timeout 等）

#### 場景 2：同一 Hook 在不同 matcher 下登記

**判定**：
- 同一個 Hook 檔案在多個 event_type 或 matcher 下登記

**處理**：
- 允許（某些 Hook 可能需要在多個 matcher 下執行）
- 記錄日誌但不報錯

#### 場景 3：metadata 定義多個 matcher，但只部分登記

**判定**：
- Hook metadata 的 matcher 為 `Write|Edit`
- 但 settings.json 中只在 `Write` matcher 下登記

**處理**：
- 在 --force 模式下，新增缺失的 matcher 條目
- 否則只提醒，不自動修改

### 4.4 備份機制

每次修改 settings.json 時：

1. **自動備份**：
   - 檔名格式：`settings.json.backup-{YYYYMMDD-HHMMSS}`
   - 存放位置：同 settings.json 目錄
   - 保留期限：7 天自動清理

2. **備份驗證**：
   - 確保備份檔案有效的 JSON
   - 備份完成後才修改原檔案

3. **恢復機制**：
   - 若寫入失敗，恢復最新備份
   - 記錄恢復事件到日誌

## 6. 向後相容性設計

### 6.1 現有 Hook 遷移策略

對於已在 settings.json 中登記但不含 metadata 的 Hook：

1. **保留現有配置**：自動註冊不修改
2. **逐步遷移**：
   - Phase 3b 時，可選擇為現有 Hook 新增 metadata
   - 新增 metadata 後可參與自動更新流程

3. **遷移工具**：
   - 提供腳本自動掃描已登記 Hook 的推薦 metadata
   - 手動驗證後加入 Hook 檔案

### 6.2 過渡期運行模式

階段 1（當前）：
- 自動註冊**僅**處理有 metadata 的 Hook
- 無 metadata 的 Hook 保留現狀

階段 2（未來）：
- 支持為無 metadata Hook 建立預設配置
- 基於檔案名稱和功能推測 metadata

## 7. 錯誤處理設計

### 7.1 錯誤分類

| 錯誤類型 | 嚴重程度 | 處理方式 |
|---------|---------|---------|
| metadata 解析失敗 | 警告 | 記錄並跳過此 Hook |
| 無效的 event_type | 警告 | 拒絕註冊，記錄詳情 |
| 無效的 timeout 值 | 警告 | 使用預設值，記錄警告 |
| 依賴 Hook 不存在 | 警告 | 記錄但允許註冊 |
| settings.json 解析失敗 | 致命 | 中止，提示手動修復 |
| settings.json 寫入失敗 | 致命 | 恢復備份，報錯退出 |

### 7.2 錯誤訊息範例

```
[錯誤] my-hook.py: metadata 解析失敗
原因: 無效的 JSON 格式
詳情: 在 "HOOK_METADATA" 區塊找到語法錯誤
建議: 檢查 docstring 中的 JSON 格式

[警告] my-hook.py: event_type 無效
值: "InvalidType"
有效值: PreToolUse, PostToolUse, SessionStart, ...

[警告] my-hook.py: 依賴的 Hook 不存在
依賴: required-hook.py
狀態: 仍進行註冊，但可能運行時報錯
```

## 11. 設計決策記錄

### 決策 1：Metadata 格式選擇

**決定**：採用 Docstring 為主，JSON Sidecar 為備選

**理由**：
- Docstring 更符合 Python 文化
- Hook 檔案和配置內聚性強
- 便於版本控制和修改追蹤

**替代方案考慮**：
- 純 JSON Sidecar：分散配置，容易不同步
- 環境變數：不適合複雜配置

### 決策 2：metadata 欄位設計

**決定**：除 event_type 外，其他欄位可選

**理由**：
- event_type 是必要標識
- matcher, timeout 等有合理預設值
- 簡化現有 Hook 的遷移

### 決策 3：衝突解決策略

**決定**：保留現有配置，--force 模式下允許更新

**理由**：
- 安全第一，避免意外覆蓋
- 用戶可透過 --force 主動控制
- 向後相容
