# SA 前置審查誤報分析（W5-014 專章）

## A. 根因分析

### 觸發位置
`.claude/hooks/process-skip-guard-hook.py:66-76` SKIP_SA_REVIEW 模式

```python
"SKIP_SA_REVIEW": {
    "pairs": [
        ("不需要", "sa"),
        ("跳過", "sa"),
        ("不做", "架構審查"),
        ("不需要", "審查"),
        ("跳過", "系統分析"),
    ],
    ...
}
```

### 為何在 Phase 4 重構誤報

觸發函式 `detect_skip_intent`（line 128-150）只做兩件事：
1. prompt 轉小寫
2. 檢查兩個關鍵字是否都 `in` prompt

**完全沒有讀取**：
- 當前 Ticket 的 type（IMP/DOC/ANA/REF）
- 當前 Ticket 的 current_phase
- Ticket status、wave 等上下文

### 具體誤報場景（本 session）

W5-001 是既有模組的 Phase 4 重構評估，PM 在 prompt 中自然會提及：
- 「重構評估不需要 SA 審查」→ 命中 ("不需要","sa") + ("不需要","審查")
- 「Phase 4 不需要系統分析」→ 命中 ("不需要","sa") 或 ("跳過","系統分析")

**設計意圖**：SA 前置審查應在「新功能或架構變更前」執行（見 `.claude/agents/basil-system-analyst.md` 觸發條件）。
**實際行為**：Hook 不區分任何 Ticket 類型與 phase，一律提醒。

### 落差總結

| 層面 | 應該 | 實際 |
|------|------|------|
| 觸發條件 | 新功能 or 架構變更 | 任何 prompt 含關鍵字對 |
| 排除情境 | DOC/ANA/REF、Phase 4 重構 | 無排除 |
| 上下文感知 | 讀 Ticket frontmatter | 0 |

---

## B. 修正方案

### 方案 1（Ticket 上下文 guard，推薦）

在 `main()` 讀 prompt 前，加入「當前 Ticket 狀態」查詢：

```python
def get_current_ticket_context() -> dict:
    """讀取最近 in_progress Ticket 的 type 與 phase"""
    # 透過 ticket track list --status in_progress --json 取得
    # 回傳 {"type": "IMP", "current_phase": "Phase 4", "ticket_id": "..."}
```

在 `detect_skip_intent` 中對 SKIP_SA_REVIEW 加 guard：
```python
if skip_type == "SKIP_SA_REVIEW":
    ctx = get_current_ticket_context()
    if ctx.get("type") in ("DOC", "ANA", "REF"):
        continue  # 跳過此模式
    if ctx.get("current_phase", "").startswith("Phase 4"):
        continue  # 重構評估不應觸發 SA 審查提醒
```

**優點**：精準過濾已知不適用情境
**缺點**：需新增 CLI 查詢呼叫（每次 prompt 都執行，~50-100ms 開銷）
**誤殺風險**：低（SA 審查原本就不該在 DOC/ANA/REF/Phase 4 觸發）
**實作複雜度**：小（30 行內，含測試）

### 方案 2（關鍵字 pair 精確化）

僅修改關鍵字對，移除高假陽性組合：
- 移除 ("不需要","審查")  <-- 太泛，覆蓋所有「審查」語境
- 保留 ("不需要","sa 審查")、("跳過","sa 前置")、("跳過","系統分析")

**優點**：零 CLI 查詢開銷、實作最快（10 行）
**缺點**：仍可能在 IMP 正常重構時誤報 ("不需要","sa")
**誤殺風險**：中（關鍵字精確化後漏檢真實省略意圖的概率略升）
**實作複雜度**：極小

### 方案 3（兩者合併，最健壯）

方案 2 關鍵字精確化 + 方案 1 Ticket guard。
**推薦**：方案 3。關鍵字層止住最明顯的假陽性，guard 層處理系統性誤報。

## 結論
